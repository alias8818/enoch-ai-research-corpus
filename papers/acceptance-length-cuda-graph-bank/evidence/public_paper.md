## Abstract

Speculative decoding accelerates autoregressive inference by proposing multiple candidate tokens per step, but the number of accepted tokens varies per request. When the post-acceptance decode tail is implemented as a variable-length sequence of small CUDA kernels, per-kernel launch overhead can dominate latency. This paper evaluates a *CUDA graph bank*—a set of pre-captured CUDA graphs, one per possible acceptance length—that selects and replays the graph matching the realized acceptance length, avoiding both the launch overhead of dynamic dispatch and the wasted work of a single max-length guarded graph. On an NVIDIA GB10 (CUDA 13.0) microbenchmark simulating acceptance-length-dependent decode tails with up to 8 acceptance lengths and 4 phases per accepted token, the graph bank reduces mean per-request latency by 1.42×–2.34× over dynamic kernel launches across vector sizes n ∈ {256, 1024, 8192, 65536} and four request distributions. A max-length guarded graph and per-request graph recapture are both strictly inferior alternatives. Checksum verification confirms functional equivalence between the dynamic and graph-bank paths. These results are confined to a synthetic microbenchmark; integration with a production LLM decode engine remains untested.

## 1. Introduction

CUDA graphs reduce kernel launch overhead by capturing a sequence of kernel launches into a single executable graph that can be replayed with near-zero host-side dispatch cost. In autoregressive LLM inference, speculative decoding introduces a variable-length acceptance phase: a proposer generates candidate tokens, and a verifier accepts a prefix of varying length. The subsequent decode tail—the work required to process the accepted tokens—naturally depends on the acceptance length.

A naive implementation launches a variable number of kernels per request, paying launch overhead proportional to the acceptance length. A single max-length guarded graph avoids variable launch counts but executes all kernels up to the maximum acceptance length, wasting work when the actual acceptance length is shorter. Per-request graph recapture incurs capture overhead that typically exceeds the launch savings.

This paper investigates an intermediate design: a *bank* of pre-captured CUDA graphs, one for each acceptance length from 1 to a maximum L. At runtime, the engine indexes into the bank by the realized acceptance length and replays the corresponding graph. This preserves the launch-overhead benefit of graph replay while issuing exactly the work required.

We evaluate this design on an NVIDIA GB10 GPU using a synthetic microbenchmark that models the launch-count sensitivity of acceptance-length-dependent decode tails. We compare four strategies—dynamic launch, graph bank, max-length guarded graph, and per-request recapture—across multiple vector sizes and acceptance-length distributions. We emphasize upfront that this is a CUDA copy-calibration microbenchmark, not an integrated engine validation; the results characterize launch-overhead mechanics only.

## 2. Method

### 2.1 Harness

The benchmark harness is implemented in `src/acceptance_length_graph_bank_bench.cu`, compiled with `nvcc -O3 -std=c++17` (CUDA 13.0, V13.0.88). The harness simulates an acceptance-length-dependent decode tail as a sequence of `accept_len × phases` trivial kernel invocations over a vector of size `n`. Each kernel performs a simple elementwise operation sufficient to exercise launch mechanics without introducing arithmetic-intensity confounds that would dominate execution time in a real transformer.

Four dispatch strategies are implemented:

1. **dynamic**: Launches `accept_len × phases` kernels per request via individual `<<<>>>` calls.
2. **graph_bank**: Pre-captures one CUDA graph for each acceptance length `1..maxL`. At runtime, selects and replays the graph corresponding to the realized acceptance length.
3. **max_guard_graph**: Captures a single graph of length `maxL × phases` kernels. Each kernel receives the acceptance length as a parameter and guards its work with a conditional, so only `accept_len × phases` kernels perform useful work, but all `maxL × phases` kernels are launched.
4. **recapture**: Captures and instantiates a fresh CUDA graph for each request, then destroys it. Included as a negative control to quantify capture overhead.

### 2.2 Parameters

Default sweep parameters:

- `maxL = 8` (maximum acceptance length)
- `phases = 4` (kernels per accepted token)
- Vector sizes: `n ∈ {256, 1024, 8192, 65536}`
- Acceptance-length distributions:
  - **fixed1**: Always accept 1 token (worst case for launch overhead ratio).
  - **short**: Geometric-like distribution with mean acceptance length ≈ 1.748.
  - **uniform**: Uniform over {1, …, 8}, mean ≈ 4.455.
  - **fixedmax**: Always accept 8 tokens (best case for absolute launch savings).

### 2.3 Environment

- **Host**: Linux gx10-efe8 6.17.0-1014-nvidia, aarch64

GPU utilization is sampled at 1 Hz via `nvidia-smi dmon`; we note that this sampling rate is insufficient for characterizing sub-millisecond bursts and serves only to confirm GPU execution occurred.

Two trends are apparent. First, speedup increases with mean acceptance length because the graph bank eliminates a larger number of individual kernel launches. Second, speedup decreases modestly at n = 65536, where per-kernel execution time begins to dominate over launch overhead. This confirms that the benefit is primarily a launch-overhead effect rather than an execution-time effect.

Distribution-level speedup ranges across all vector sizes:

- **fixed1**: 1.42×–1.58×
- **short**: 1.52×–1.81×
- **uniform**: 1.79×–2.18×
- **fixedmax**: 1.89×–2.34×

The speedup is positive in every tested configuration, but the magnitude varies substantially. At n = 65536 with fixed1, the speedup is only 1.42×, indicating that the technique provides diminishing returns as per-kernel work grows.

### 3.2 Negative Controls

The max-length guarded graph (`max_guard_graph`) was slower than the graph bank in every tested configuration because it launches all `maxL × phases` kernels regardless of the actual acceptance length. For example, at n = 1024 with the short distribution, the graph bank achieves 10.109 μs while the max guarded graph requires 29.160 μs—nearly 3× slower—because it launches 32 kernels (8 × 4) even when only approximately 7 (1.748 × 4) are needed on average.

Per-request graph recapture (`recapture`) was consistently slower than the prebuilt graph bank and usually similar to or worse than dynamic launch. At n = 1024 with the short distribution, recapture requires 19.821 μs versus 10.109 μs for the graph bank. The capture and instantiation overhead negates any launch savings, confirming that per-request recapture is not a viable strategy.

### 3.3 Correctness

The `dynamic`, `graph_bank`, and `max_guard_graph` strategies produced identical output checksums (2.177224809 × 10³) for the configuration n = 1024, iterations = 100, maxL = 8, phases = 4, distribution = short. This confirms that the graph-bank path computes the same result as the dynamic path for at least this configuration. Correctness was not checked exhaustively across all configurations.

## 4. Limitations

1. **Synthetic microbenchmark.** The harness models launch-count sensitivity using trivial kernels over flat vectors. It does not execute real transformer attention, GEMM, or KV-cache operations. The absolute latency numbers and speedup magnitudes may change substantially when integrated with arithmetic-intensive kernels whose execution time dominates launch overhead. The reported speedups characterize launch-overhead reduction only; they should not be interpreted as end-to-end inference speedups.

2. **Stable buffers and fixed parameters.** The benchmark uses stable device buffers and fixed kernel parameters across graph captures. A production engine must manage graph-safe pointer lifetimes, ensure allocator and memory-pool stability across graph replays, and handle graph-update compatibility for per-request metadata (e.g., KV-cache pointers, token positions). The cost and feasibility of these mechanisms are not evaluated here and may reduce or eliminate the observed benefit.

3. **Bounded acceptance length.** The bank cardinality equals the maximum acceptance length. This study uses maxL = 8. Production speculative-decoding configurations with larger draft budgets would require proportionally more graphs, increasing memory and capture cost. The trade-off between bank size and benefit is not characterized beyond maxL = 8.

4. **Single GPU architecture.** All results are from an NVIDIA GB10 (aarch64 host, CUDA 13.0). Launch overhead, graph capture cost, and replay cost may differ on other architectures (e.g., data-center GPUs with different driver stacks or CUDA versions). No cross-architecture comparison is available.

5. **Coarse utilization sampling.** The 1 Hz `nvidia-smi dmon` sampling under-represents the actual SM occupancy of sub-millisecond bursty workloads. No Nsight Systems or Nsight Compute profiling was performed, so no detailed stall or occupancy analysis is available.

6. **No end-to-end inference measurement.** Token-level throughput, time-to-first-token, and inter-token latency in a serving context are not measured. The reported speedups apply specifically to the simulated decode-tail portion of a speculative-decoding step.

## 6. Conclusion

A CUDA graph bank keyed by speculative-decoding acceptance length is a viable mechanism for reducing launch overhead in acceptance-length-dependent decode tails. On an NVIDIA GB10 microbenchmark, the graph bank reduces mean per-request latency by 1.42×–2.34× over dynamic kernel launches, with the benefit increasing as the mean acceptance length grows and the number of eliminated launches increases. The approach strictly outperforms both a max-length guarded graph (which wastes work on unneeded kernels) and per-request graph recapture (whose capture overhead exceeds launch savings).

These results are confined to a synthetic microbenchmark and should not be extrapolated to end-to-end inference performance. The critical path to practical adoption requires integration with a production inference engine, where graph-safe memory management, per-request metadata updates, and the interaction with real transformer kernels must be addressed. The approach is most promising when (a) the decode tail consists of many small, launch-overhead-bound kernels, (b) acceptance lengths are bounded enough to keep the graph bank cardinality manageable, and (c) the engine can maintain graph-safe buffer lifetimes across replays. Under these conditions, an acceptance-length graph bank offers a straightforward and empirically validated improvement over both dynamic dispatch and max-length guarded graphs. Whether these conditions hold in a production engine, and whether the observed launch-overhead savings translate to meaningful end-to-end speedups, remain open questions.
