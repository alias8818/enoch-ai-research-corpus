## Abstract

We present a deterministic, black-box integration benchmark for verifying the process-level contract between an agent controller and an opaque agent runner. The harness treats the runner as an external subprocess and validates six integration surfaces: launch and stdout/stderr capture, stdin/environment/working-directory propagation, streaming progress output, large-output capture without pipe deadlock, non-zero exit-status preservation with artifact retention, and timeout detection with process-group termination. In a calibrated run of 60 total case invocations (10 iterations per case) on an aarch64 Linux host with 128 GB RAM and no swap, all 60 checks passed with 0 failures, at an aggregate throughput of 7.39 cases/s and total elapsed time of 8.12 s. The benchmark intentionally uses local deterministic Python stand-ins rather than live LLM-backed agents, isolating integration mechanics from model and network latency. Results are high-confidence for local process-level black-box integration; generalization to full external agent ecosystems remains medium-confidence and would require end-to-end adapter cases exercising real agent runners behind the same contract.

## Introduction

Agent controllers that orchestrate external runner processes depend on a narrow set of operating-system-level surfaces: process launch, standard I/O streams, environment variables, working directory, exit status, and signal handling. Errors in these integration surfaces—pipe deadlocks on large output, lost non-zero exit codes, uncontrolled timeout behavior—can silently corrupt agent outputs or hang controller pipelines. Despite their criticality, these surfaces are often tested only implicitly through end-to-end agent runs that conflate integration failures with model or network failures, making root cause attribution difficult.

This work contributes a deterministic black-box integration benchmark that isolates and verifies the controller–runner process contract. The runner is treated as an opaque command: the harness neither inspects nor depends on runner internals. Six test cases cover the identified failure modes. By using deterministic local Python scripts as runner stand-ins, the benchmark removes model and network variance, enabling reproducible calibration of integration mechanics alone.

The benchmark does not evaluate LLM reasoning quality, agent task correctness, or network resilience. It is narrowly scoped to the question: *can a controller correctly launch, communicate with, and terminate an opaque runner process across the identified integration surfaces?*

## Method

### Design

The benchmark harness (`benchmark/agent_runner_black_box_benchmark.py`, SHA256: `c6b4895a50874e13891fc901cf920f5546ca96a0f0cc30078217d96b7f351060`) implements six test cases, each exercising a distinct integration surface:

| Case | Integration Surface Verified |
|---|---|
| `echo_success` | Successful launch, stdout/stderr capture, zero exit status |
| `stdin_env_cwd` | stdin content, environment variable, and working directory propagation to the runner |
| `streamed_progress` | Incremental stdout lines received before process exit (streaming progress) |
| `large_output_no_deadlock` | Capture of ~640 KB stdout + ~80 KB stderr without pipe-buffer deadlock |
| `failure_exit_preserved` | Non-zero exit status preserved while retaining captured output artifacts |
| `timeout_kill` | Timeout detection and process-group termination within bounded wall time |

Each case is implemented as a short deterministic Python script invoked as a subprocess by the harness. The harness validates observable outputs (stdout bytes, stderr bytes, exit code, elapsed time) against expected values. No model inference, network calls, or external services are involved in the calibrated runs.

### Execution Modes

The harness supports two modes:

- **Smoke mode:** One iteration per case, intended for rapid CI gating.
- **Calibrated mode:** Configurable iterations per case (10 used in the reported run), intended for statistical stability assessment and variance characterization.

### Observations

- The `timeout_kill` case had the highest average elapsed time (0.455 s), reflecting the intentional sleep-then-kill design of the timeout test. The maximum elapsed time of 0.462 s confirms bounded termination.
- The `streamed_progress` case averaged 0.272 s, consistent with a runner that emits incremental output over a short interval before exiting.
- The remaining four cases each completed in under 30 ms on average, confirming low overhead for basic launch-and-capture operations.
- The `large_output_no_deadlock` case successfully captured ~655 KB of stdout and ~82 KB of stderr without any pipe deadlock, validating the harness's concurrent drain strategy for large outputs.
- The `failure_exit_preserved` case confirmed that non-zero exit codes are correctly captured alongside output artifacts, preventing silent failure masking.
- No variance anomalies were observed across the 10 iterations per case, though the sample size is insufficient for formal statistical claims about tail behavior.

## Limitations

1. **Deterministic stand-ins, not real agents.** The calibrated benchmark uses local Python scripts as runner stand-ins. This isolates integration mechanics but does not exercise model inference latency, API rate limits, token streaming protocols, or network failure modes that real agent runners would introduce. The results validate the process contract, not agent task quality or end-to-end correctness.

2. **Single platform, single execution session.** All runs were conducted on one aarch64 Linux host with 128 GB RAM and no swap. Results may differ on memory-constrained systems, different OS kernels, or architectures where process-group signaling behaves differently (e.g., Windows job objects, macOS process groups).

3. **No nested real-agent invocation.** Intentionally, no live Codex or external agent invocations were used in calibrated runs. Adding such cases would confound integration-mechanics measurement with network/API/model variance. This is a deliberate scope boundary, not an oversight, but it means the benchmark does not characterize real-world latency distributions.

4. **Limited failure-mode coverage.** The six cases cover common integration surfaces but do not exhaust all possible failure modes. Notably absent: SIGKILL from external OOM, disk-full artifact writes, concurrent multi-runner scheduling, partial write truncation, and signal races during teardown.

5. **Small iteration count.** Ten iterations per case provides a basic consistency check but is insufficient for formal statistical claims about tail latencies, rare failures, or distribution shape. Larger iteration counts would be needed for production-grade reliability estimates.

6. **Confidence boundary.** Confidence is high for local process-level black-box integration on the tested platform. Confidence is medium for generalization to full external agent ecosystems, where adapter-specific failure modes (authentication, API errors, model hallucination, token-limit truncation) would require separate characterization.

7. **No independent human review.** This draft and the underlying benchmark execution were produced by automated systems. No independent human review of the code, results, or interpretation has been performed. The claim ledger contains no audit-approved claims.

## Conclusion

A deterministic black-box integration benchmark is viable as a local process-level harness for verifying the controller–runner boundary. In a calibrated execution of 60 case invocations across six integration surfaces, all checks passed with zero failures, demonstrating that launch, I/O propagation, streaming, large-output capture, exit-status preservation, and timeout termination can be reliably verified without depending on model or network behavior.

The harness provides a reproducible, CI-compatible gate for integration mechanics. Its scope is deliberately narrow: it proves the process contract, not agent intelligence or end-to-end task correctness. The 0/60 failure rate on deterministic stand-ins is expected and should not be interpreted as evidence of robustness under non-deterministic real-agent conditions.

The recommended next step is to adopt this harness as a CI smoke/calibration gate for the controller, then extend it with optional adapter cases that invoke real agent runner CLIs behind the same black-box contract, enabling end-to-end latency and failure-mode characterization without compromising the deterministic baseline. Such adapter cases would be necessary before drawing any conclusions about production readiness.
