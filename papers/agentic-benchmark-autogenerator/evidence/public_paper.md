## Abstract

We describe a dependency-light method for automatically generating agentic bugfix benchmark tasks from Python repositories with pytest test suites. The method discovers AST mutation candidates in a passing codebase, retains only mutations that cause the test suite to fail, and emits self-contained task bundles comprising a mutated starter repository (with tests removed), evaluator-only hidden tests, a solver prompt, and a restoring oracle patch. In a local prototype evaluation on one small synthetic Python/pytest fixture repository, the generator identified 29 mutation candidates, attempted 10, and accepted 5 as valid task bundles (acceptance rate 0.5 among attempted mutations, ~0.17 among all candidates). Independent validation confirmed that all 5 emitted tasks fail their hidden tests in the buggy state and pass after applying the oracle patch. These results establish feasibility of the core autogeneration loop as a local prototype but do not constitute a corpus-scale scientific claim; external validity remains limited to the single fixture tested. No agent solve-rate data, difficulty calibration, or multi-repository validation was performed.

## Introduction

Evaluating agentic coding systems requires benchmark tasks that are executable, self-validating, and resistant to memorization. Manual benchmark construction is labor-intensive and produces static corpora that may become contaminated as they appear in training data. Automated benchmark generation offers a potential path to scalable, renewable evaluation: if new tasks can be synthesized from existing codebases with minimal human intervention, evaluators can produce fresh task sets on demand.

This work explores a simple generation strategy: mutate a passing Python repository at the AST level, filter for mutations that break the test suite, and package each surviving mutation as a self-contained bugfix task. The resulting task bundles are designed for agentic evaluation: solvers receive a repository and a task prompt but cannot access the hidden tests or oracle patch, while evaluators retain both for automated grading.

The central question is whether this loop—AST mutation, test-based filtering, and bundle emission—can reliably produce tasks that are both executable (the buggy code runs and fails tests) and restorable (a known patch exists that makes all tests pass). We do not address difficulty calibration, corpus-scale validity, or multi-language support in this prototype; those are deferred to subsequent work.

## Method

### Generation Pipeline

The generator (`scripts/autogen_benchmark.py`) operates in four phases:

1. **Baseline verification.** The source repository's test suite is executed. If it does not pass in full, generation aborts.

2. **Candidate discovery.** The source repository's Python files are parsed into ASTs. Mutation candidates are identified at specific source locations. In this prototype, mutation operators are limited to simple transformations: arithmetic operator substitution, comparison operator substitution, boolean operator substitution, and integer constant modification.

3. **Mutation testing.** Each candidate mutation is applied to a copy of the repository, and the test suite is re-executed. Mutations that do not cause at least one test failure are discarded. Mutations that cause test failures are provisionally accepted. A maximum of 10 mutations were attempted in the reported run (controlled by the `--max-tasks` parameter set to 5, with the generator attempting up to twice the target).

4. **Bundle emission.** For each accepted mutation, the generator emits a task directory containing:
   - `repo/`: the mutated repository with test files removed (solver-visible)
   - `hidden_tests/`: the original test files (evaluator-only)
   - `task.md`: a solver prompt describing the bugfix objective
   - `solution.patch`: a unified diff restoring the original code (oracle)
   - `metadata.json`: per-task validation metadata

### Validation Contract

A mutation is accepted only if it satisfies three conditions:

- **Baseline passes:** The unmodified repository passes all tests.
- **Buggy fails:** The mutated repository fails at least one hidden test.
- **Patch restores:** Applying `solution.patch` to the mutated repository causes all tests to pass.

### Independent Validation

After generation, each emitted task bundle is independently validated by copying the starter repo into a temporary directory, injecting the hidden tests, confirming test failure, applying the oracle patch, and confirming all tests pass. This step guards against generator-internal state leakage or filesystem side effects.

## Results

The source fixture repository (`sample_target/`) passed all 6 tests in 0.01 seconds. This fixture is a small synthetic Python/pytest project; it is not a real-world repository.

The naming convention encodes the mutation type and source line (e.g., `binop` for binary operator substitution at line 16). All five accepted tasks involve single-line mutations in a calculator module.

All buggy states exhibited exactly one test failure out of six; all patched states passed all six tests. The independent validation confirmed: `all_buggy_failed = true`, `all_solution_patches_applied = true`, `all_fixed_passed = true`.

### Negative and Mixed Observations

Several aspects of the results temper any strong conclusions:

- **19 of 29 candidates untested.** The generator discovered 29 candidates but only attempted 10 due to the task cap. The true acceptance rate over the full candidate set is unknown.
- **Uniform failure pattern.** All five accepted tasks exhibit the same failure signature (1 failed / 5 passed), suggesting the mutations affect a single test case each. This may indicate limited test coverage or limited mutation diversity rather than a desirable property.
- **No solve-rate measurement.** No coding agent was run against the emitted tasks. Whether these tasks are discriminative for agent evaluation remains unknown.
- **No difficulty variation.** All accepted tasks are single-line operator or constant mutations in the same module. The task set lacks diversity in difficulty, scope, and bug type.

## Limitations

1. **Single-fixture external validity.** All evidence derives from one small synthetic Python/pytest repository. No claim is made about generalization to real-world repositories, other test frameworks, or other languages. The fixture was designed for this smoke test; it is not representative of production codebases.

2. **Primitive difficulty calibration.** Accepted mutations are genuine executable bugfixes, but the mutations are limited to simple operator and constant changes. The resulting tasks may be too easy for capable coding agents. No difficulty scoring or filtering was applied, and no agent solve-rate data exists to characterize difficulty.

3. **Hidden test provenance.** The hidden tests are copied directly from the source repository's visible test suite. A solver that memorizes or reconstructs the original tests could circumvent the evaluation. Production benchmarks should synthesize or carefully split evaluator-only tests.

4. **Narrow mutation operators.** Only arithmetic, comparison, boolean, and integer constant mutations are implemented. The 29 candidates discovered in the fixture reflect this limited operator set. Richer mutation operators (statement deletion, conditional boundary changes, method call replacement) are not yet represented.

5. **Acceptance rate instability.** The observed acceptance rate of 0.5 (5 of 10 attempted) is specific to this fixture and operator set. It should not be treated as a stable parameter; acceptance rates will vary with repository complexity, test coverage, and mutation operator scope.

6. **No agent solve-rate data.** No coding agent was run against the emitted tasks. The tasks are validated as structurally sound (buggy fails, patch restores) but their discriminative power for agent evaluation is unmeasured.

7. **Empty claim ledger.** The structured claim ledger for this run contains no claims and has audit status `blocked_empty_claims`. The observations in this paper are drawn from run logs and decision records rather than from a passed claim/evidence audit. Readers should weight the evidence accordingly.

8. **Missing environment metadata.** Python version, operating system, and hardware details were not recorded in the available artifacts. Future runs should capture these explicitly to support reproducibility.

## Conclusion

A dependency-light pipeline of AST mutation, test-based filtering, and bundle emission can automatically produce executable, self-validating agentic bugfix tasks from a Python/pytest repository. On a single synthetic fixture, the generator identified 29 candidates, accepted 5 valid task bundles, and independent validation confirmed that each bundle correctly fails in the buggy state and passes after applying the oracle patch. This establishes the feasibility of the core autogeneration loop as a local prototype.

The result is not a corpus-scale scientific claim. The critical next steps are: (1) running the generator on multiple real Python/pytest repositories to assess acceptance rates and task quality at scale; (2) measuring solve rates with a baseline coding agent to evaluate discriminative power; (3) introducing hidden-test synthesis or splitting to prevent solver access to evaluator tests; and (4) extending mutation operators and adding difficulty scoring. If these steps produce favorable results, the approach may offer a practical path to renewable, contamination-resistant agentic benchmarks. If they do not—particularly if acceptance rates collapse on real repositories, or if generated tasks prove trivially solvable—the prototype result reported here will remain an interesting but narrow feasibility demonstration.
