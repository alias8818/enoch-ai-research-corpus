## Abstract

We present an asynchronous in-memory key-value backend that grants selective, short-lived leases on cache blocks so that active decode work can protect important blocks from eviction while still permitting capacity-bound admission and measurable fallback behavior. In a local synthetic benchmark at 8 MiB capacity, the prototype processed 32,000 operations at 2,773.9 ops/s with p95 latency of 1.48 ms, evicting 6,576 unpinned records and zero force-evicted records, with zero rejections under steady-state pressure. A lighter smoke test at sub-capacity load achieved 123,045.6 ops/s with p95 latency of 0.016 ms. These results demonstrate that lease-based eviction protection is mechanically coherent in a single-process async setting, but they do not establish model-quality, latency, or throughput effects under real GPU inference workloads. The prototype carries known limitations: byte payloads rather than GPU/UMA tensor pages, no integration with any transformer runtime scheduler, no cross-process coordination, and a heap-rebuild eviction path that is unsuitable for high-churn production use.

## 1. Introduction

Autoregressive language model inference reuses key-value (KV) cache entries across decoding steps and across requests sharing a common prefix. When memory is bounded, the scheduler must decide which blocks to retain and which to evict. Common strategies—least-recently-used, attention-score-weighted, or prefix-partitioned eviction—treat all unoccupied blocks as equally evictable. A decode step that is actively reading a block, however, has a stronger claim on that block than a stale prefix that may never be revisited.

We investigate whether a small async backend can grant selective short-lived *leases* on KV-cache blocks so that:

1. **Live leases prevent eviction.** A block held by an active lease is not a candidate for eviction under normal pressure.
2. **Expired or unleased blocks become evictable.** Once a lease expires or is released, the block rejoins the eviction candidate pool.
3. **All-pinned capacity pressure rejects safely.** When every block is live-leased and capacity is full, new admissions are rejected rather than silently corrupting active holders.
4. **Forced eviction is explicit and counted.** An operator may override lease protection, but only through a deliberate `force=True` path that increments a separate counter.

This paper reports on a single-process, in-memory prototype that implements these semantics and evaluates them under synthetic concurrent workloads. We do not claim results for real inference throughput, model quality, or GPU memory management.

## 2. Method

### 2.1 Backend Design

The prototype (`src/kv_lease_backend.py`) is an async Python in-memory KV store with the following operations:

- **`put(key, value, priority)`**: Insert or update a record. If capacity is exceeded, evict expired or unleased records first (lower priority first, then older access time). If no evictable record exists, reject the insertion.
- **`get(key)`**: Read a record and update its access timestamp.
- **`acquire(key, lease_seconds)`**: Grant a lease on a record, returning a lease ID. Leased records are protected from normal eviction.
- **`renew(lease_id, lease_seconds)`**: Extend an existing lease.
- **`release(lease_id)`**: Release a lease, returning the record to the evictable pool.
- **`snapshot()`**: Return a point-in-time view of backend state.

Eviction ordering uses a min-heap keyed on `(priority, last_access_time)`. When capacity pressure triggers eviction, the heap is rebuilt from the current record set, expired and unleased records are popped in order, and the first successfully evicted record frees space for the incoming insertion. If no evictable record exists and `force=True` is not set, the insertion is rejected.

### 2.2 Test Suite

Four regression tests (`tests/test_kv_lease_backend.py`) verify:

1. **Lease protection**: A leased record survives an eviction sweep that removes an unleased record of equal or lower priority.
2. **All-pinned rejection**: When all records are live-leased and capacity is full, a new insertion is rejected rather than evicting a leased record.
3. **Expiry**: An expired lease is treated as unleased and becomes evictable.
4. **Forced eviction**: With `force=True`, a leased record can be evicted, and the forced-eviction counter increments.

### 2.3 Benchmark Harness

`bench.py` drives concurrent synthetic workloads against the backend, recording operation counts, throughput, latency percentiles, eviction counts, forced-eviction counts, and rejection counts. Two configurations were run:

- **Smoke benchmark**: 400 operations, sub-capacity working set, intended to verify correctness and measure uncontended throughput.
- **Calibrated benchmark**: 32,000 operations, 8 MiB capacity, working set designed to fill capacity and induce sustained eviction pressure.

### 2.4 Environment

The host runs Linux 6.17.0-1014-nvidia on aarch64 with swap disabled. Available memory remained approximately 122.6 GB throughout, so the prototype did not face physical memory pressure. The 8 MiB capacity limit is a software-imposed bound, not a hardware constraint.

## 3. Results

### 3.1 Unit Tests

All 4 of 4 unit tests passed. The lease semantics—protection, all-pinned rejection, expiry, and forced eviction—behave as specified in the prototype's local process model.

The smoke benchmark confirms that the async event loop and lease bookkeeping introduce negligible overhead when the working set fits in capacity.

### 3.3 Calibrated Benchmark

| Metric | Value |
|---|---|
| Operations | 32,000 |
| Throughput | 2,773.9 ops/s |
| p95 latency | 1.4816 ms |
| Evictions | 6,576 |
| Forced evictions | 0 |
| Rejections | 0 |
| Capacity | 8,388,608 bytes (8 MiB) |
| Used bytes (final) | 8,388,608 bytes |

The calibrated benchmark filled capacity exactly (`used_bytes == capacity_bytes`) and sustained admission by evicting 6,576 unpinned records. No forced evictions were necessary: the workload's mix of leased and unleased records always left sufficient evictable candidates. No insertions were rejected, indicating that the lease-protected fraction of the working set remained below total capacity throughout the run.

The p95 latency of 1.48 ms is substantially higher than the smoke benchmark's 0.016 ms. This tail is consistent with the eviction path's heap-rebuild cost under sustained pressure: rebuilding a heap over the full record set on each capacity-exceeding insertion is O(n log n) and likely dominates the observed tail.

### 3.4 Capacity and Eviction Behavior

The backend correctly enforces the capacity bound. When `used_bytes == capacity_bytes`, new insertions trigger eviction of the lowest-priority, oldest-accessed, unleased-or-expired record. The zero-rejection result shows that the workload never reached a state where all records were simultaneously live-leased, which is the only condition under which normal-mode rejection occurs. The all-pinned rejection path was exercised only in unit tests, not under the synthetic benchmark's particular lease distribution. This is a mixed result: the rejection path is verified in isolation but unvalidated under sustained concurrent load.

## 4. Limitations

1. **Byte payloads, not tensor pages.** Records store arbitrary `bytes` values. The prototype has no awareness of GPU/UMA memory handles, CUDA stream semantics, or tensor layout. Results do not transfer to real KV-cache page management without integration work.

2. **No transformer runtime integration.** There is no connection to llama.cpp, vLLM, or any attention-kernel scheduler. The prototype cannot report TTFT, inter-token latency, cache hit rate, or model-quality metrics under real decoding.

3. **Synthetic workload only.** The benchmark uses a synthetic key/value distribution and lease pattern. Real inference workloads exhibit prefix sharing, variable sequence lengths, and attention-weighted reuse patterns not captured here.

4. **Single-process, no crash recovery.** All state is in-memory within one process. There is no cross-process lease coordinator, no persistent lease log, and no recovery protocol for lease consistency after a crash.

5. **Heap-rebuild eviction.** The current eviction implementation rebuilds a heap from all records on each capacity-pressure event. Under high churn with many records, this path is the likely cause of the observed p95 tail and would need replacement by an indexed priority queue or segmented LRU for production use.

6. **No concurrency contention measurement.** The async benchmark runs coroutines within a single event loop. It does not measure contention under true multi-threaded or multi-process access.

7. **All-pinned rejection path untested under load.** The rejection path was verified in unit tests but not triggered during the calibrated benchmark. Its behavior under sustained concurrent pressure where all blocks are leased remains uncharacterized empirically.

## 6. Conclusion

An async selective KV-lease backend is mechanically viable in a single-process prototype. Live leases protect records from eviction, expired leases rejoin the evictable pool, all-pinned pressure rejects safely, and forced eviction is explicit and counted. Under a synthetic 8 MiB capacity-bound workload, the prototype admitted 32,000 operations with 6,576 evictions and zero rejections, confirming that lease-aware eviction can maintain a capacity bound without silent data loss.

However, this prototype does not establish that lease semantics improve inference performance. The gap between byte-payload synthetic benchmarks and real GPU KV-cache management remains large. The heap-rebuild eviction path produces a measurable p95 tail (1.48 ms vs. 0.016 ms uncontended) that would need replacement before production evaluation. The all-pinned rejection path, while correct in unit tests, was not exercised under sustained load. Scientific closure on whether selective leases improve TTFT, inter-token latency, or cache hit rate under real decoding workloads requires integration with a transformer runtime and A/B measurement on prefix-reuse or long-context tasks.
