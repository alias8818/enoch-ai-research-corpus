## Abstract

Multi-agent systems that aggregate results from several autonomous agents face a reliability problem: when agents disagree, a naive aggregation policy such as accepting the most recent claim can accept incorrect results. This paper evaluates a coordination mechanism in which a shared blackboard accepts agent claims only after discharging task-specific, machine-checkable proof obligations. A deterministic local prototype implements this mechanism across three synthetic task types—integer summation, sorting, and graph-path verification—and compares it against a naive last-write baseline. In the main benchmark (3,000 tasks, 3 agents per task, per-agent accuracy 0.70), the proof-obligation blackboard achieves a solved rate of 0.9833 with a false acceptance rate of 0.0000, compared to the baseline's solved rate of 0.7550 and false acceptance rate of 0.2450. An accuracy sweep across five agent accuracy levels (0.40–0.95, 1,500 tasks each) confirms that the proof-obligation policy consistently eliminates false accepts under the synthetic verifier model, though it introduces a rejected-all rate (tasks where no agent produces a valid claim) that is highest when agent accuracy is low (0.1633 at accuracy 0.40). These results are limited to synthetic tasks with known-sound verifiers; external validity for real LLM-agent workflows remains unproven.

## Introduction

When multiple agents contribute solutions to shared tasks, the aggregation policy determines which claims are accepted as results. A simple policy—accepting the most recently submitted claim—provides no guarantee of correctness and can propagate errors. An alternative is to require that each accepted claim satisfy a machine-checkable proof obligation before it is committed to the blackboard.

The core research question is: can a shared blackboard improve multi-agent result reliability by accepting claims only after task-specific proof obligations are discharged?

This work implements a deterministic prototype to test the mechanism in isolation from the variability of live language model agents. The design is intentionally synthetic: agents are simulated with configurable accuracy, and proof obligations are implemented as deterministic Python functions. This isolates the coordination mechanism itself, allowing measurement of its effect on solved rate and false acceptance rate under controlled conditions.

The trade-off is explicit: the proof-obligation blackboard cannot accept a claim that fails verification, which means it may reject all candidates when no agent produces a correct result. This rejected-all rate quantifies the cost of the guarantee.

## Method

### Architecture

The prototype (`research/blackboard_po_experiment.py`) implements two blackboard policies:

1. **Naive last-write (`naive_last_write`):** Accepts the most recently submitted claim without verification. This serves as the baseline.

2. **Proof-obligation blackboard (`BlackboardWithProofObligations`):** Scans candidate claims in submission order and accepts the first claim whose proof obligation passes. If no claim passes, the task is recorded as rejected-all.

Each agent produces a `Claim` object containing a proposed value. Each `Task` carries a `proof_fn`—a deterministic Python function that returns `True` if and only if the claim satisfies the task's correctness condition.

### Task Types and Proof Obligations

Three synthetic task types were implemented, each with a corresponding proof obligation:

| Task Type | Proof Obligation |
|-----------|-----------------|
| Integer sum | Proposed value equals the exact sum of input numbers |
| Sorting | Output list is sorted in non-decreasing order and preserves the input multiset |
| Graph path | Path endpoints match the specified source and target, and every edge in the path exists in the graph |

These proof obligations are sound by construction within the prototype: a correct solution always passes, and an incorrect solution always fails. This assumption is a deliberate simplification; in production, verifier soundness would require independent assurance.

### Agent Simulation

Agents are simulated with a configurable per-task accuracy parameter. With probability equal to the accuracy parameter, an agent produces a correct claim; otherwise, it produces an incorrect one. The random seed is fixed for reproducibility. Each task receives claims from 3 independent agents. This simulation model does not capture correlated failures, task-dependent accuracy variation, or strategic behavior that real LLM agents might exhibit.

### Metrics

- **Solved rate:** Fraction of tasks where the blackboard accepts a correct claim.
- **False acceptance rate:** Fraction of tasks where the blackboard accepts an incorrect claim.
- **Rejected-all rate (proof-obligation only):** Fraction of tasks where no agent's claim passes verification.
- **Mean claim checks per task:** Average number of proof-function invocations before acceptance or exhaustion.
- **p95 check time:** 95th-percentile wall-clock time for a single proof-function invocation.

### Experimental Protocol

Five stages were executed:

1. **Compilation check:** `python3 -m py_compile` on the prototype source.
2. **Unit tests:** 4 tests in `research/test_blackboard_po.py`, all passing.
3. **Smoke test:** 9 tasks (3 per kind), 3 agents, accuracy 0.70, seed 1.
4. **Main benchmark:** 3,000 tasks (1,000 per kind), 3 agents, accuracy 0.70, seed 42. Executed under `/usr/bin/time -v` for resource telemetry.
5. **Accuracy sweep:** 1,500 tasks per accuracy level (0.40, 0.55, 0.70, 0.85, 0.95), 3 agents, seed 42.

All computation is local and deterministic; no external API calls or network dependencies are involved.

## Results

### Smoke Test

The smoke test (9 tasks, accuracy 0.70, seed 1) produced initial confirmation of the mechanism:

| Metric | Naive Last-Write | Proof-Obligation |
|--------|:---:|:---:|
| Solved rate | 0.6667 | 0.8889 |
| False acceptance rate | 0.3333 | 0.0000 |

The small sample size makes these numbers indicative only; the main benchmark provides statistically more meaningful results.

### Main Benchmark

| Metric | Naive Last-Write | Proof-Obligation |
|--------|:---:|:---:|
| Solved rate | 0.7550 | 0.9833 |
| False acceptance rate | 0.2450 | 0.0000 |
| Rejected-all rate | — | 0.0167 |
| Mean claim checks per task | — | 1.3223 |
| p95 check time | — | 0.004864 ms |

The proof-obligation blackboard eliminates false accepts entirely under the synthetic verifier model. The solved rate increases from 0.7550 to 0.9833 because the blackboard searches among multiple candidate claims rather than trusting a single submission. The 0.0167 rejected-all rate represents tasks where no agent produced a correct claim; these tasks are correctly left unsolved rather than incorrectly accepted.

Three patterns emerge from the sweep:

1. **False acceptance is zero across all accuracy levels** under the proof-obligation policy. This is expected given sound verifiers: no incorrect claim can pass verification.

2. **The rejected-all rate decreases as agent accuracy increases**, from 0.1633 at accuracy 0.40 to 0.0000 at accuracy 0.95. This rate reflects the probability that all three agents fail independently, which shrinks as individual accuracy rises. At accuracy 0.40, approximately one in six tasks receives no valid claim from any agent.

3. **Mean claim checks decrease with higher accuracy**, from 1.77 at accuracy 0.40 to 1.05 at accuracy 0.95, because the first claim is more likely to be correct and accepted immediately.

### Resource Usage

The main benchmark completed in 0.09 seconds wall time with a maximum RSS of 21,632 KB. Memory availability before and after the run was approximately 122,663 MB and 122,657 MB respectively on a system with 127,536 MB total RAM and zero swap. The proof-checking overhead is negligible at sub-0.01 ms per check (p95), though this reflects the simplicity of the synthetic verifiers rather than production-scale verification costs.

## Limitations

1. **Synthetic tasks only.** The three task types (integer sum, sorting, graph path) have trivially implementable proof obligations. Real-world tasks—code editing, research summarization, open-ended reasoning—may not admit compact, sound, and cheap verifiers. The benefit demonstrated here does not generalize automatically to domains where proof obligations are incomplete, unsound, or expensive.

2. **Verifier soundness is assumed.** The zero false-acceptance result depends entirely on the correctness of the `proof_fn` implementations. A buggy or incomplete verifier would undermine the guarantee. In practice, verifier correctness is itself a claim that requires independent assurance.

3. **Simulated agents, not live LLMs.** Agents produce correct or incorrect claims with fixed probability, independent of task content. Real LLM agents exhibit correlated failures, task-dependent accuracy, and strategic behavior. The prototype does not model these factors.

4. **No recovery mechanism.** When all claims are rejected, the prototype records the task as unsolved but does not attempt re-prompting, agent reconfiguration, or fallback strategies. The rejected-all rate therefore represents a hard floor on unsolved tasks under the current design.

5. **Small and fixed agent count.** All experiments use 3 agents. The interaction between agent count, accuracy, and rejected-all rate is not explored. Increasing the number of agents would reduce the rejected-all rate (since more independent attempts increase the probability that at least one agent produces a correct claim), but this trade-off is not quantified here.

6. **No adversarial evaluation.** Agents are not incentivized to produce plausible-but-incorrect claims that might exploit verifier gaps. An adversarial setting could reveal weaknesses not captured by random incorrect claims.

7. **Claim ledger audit status.** The structured claim ledger for this artifact was recorded as `blocked_empty_claims` at the time of draft generation, meaning no formal claim-evidence bindings were extracted into the ledger schema. The empirical results reported here are drawn directly from the run notes and project decision record rather than from a formally audited claim ledger.

## Conclusion

A proof-obligation blackboard that accepts agent claims only after discharging machine-checkable verification eliminates false acceptance on synthetic tasks and improves the solved rate relative to a naive last-write baseline. The mechanism's cost is a rejected-all rate that reflects the probability that no agent produces a verifiably correct claim; this rate is highest when individual agent accuracy is low (0.1633 at accuracy 0.40) and approaches zero at high accuracy (0.0000 at accuracy 0.95). Proof-checking overhead is negligible for the simple verifiers tested.

These results support the proof-obligation blackboard as a viable coordination pattern for domains where claims can be paired with sound, cheap verifiers. However, the result is a prototype-level finding from a deterministic simulation: the guarantee depends on verifier soundness, the tasks are synthetic, and the agents are simulated rather than live. The mechanism does not inherently benefit domains where claims cannot be converted into checkable obligations.

The recommended next step is to integrate the proof-obligation blackboard into a real agent loop where each claim must include an executable test, source citation, or formal checker invocation, and to compare against an ungated baseline on tasks that approximate production workloads. Key metrics for such an evaluation would include false accept rate, time-to-valid-answer, verifier cost, and the rate and recoverability of rejected-but-potentially-valid claims.
