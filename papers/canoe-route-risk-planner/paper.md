## Abstract

We evaluate whether a canoe route risk planner can be grounded in publicly available, machine-readable data sources rather than generic safety advice. A dependency-free Python prototype queries the National Weather Service (NWS) point forecast and alert APIs, the USGS Network Linked Data Index (NLDI) for hydrolocation and streamgage discovery, and the USGS instantaneous-values service for recent streamflow observations. For a representative Mississippi River day-paddle route near Saint Paul, Minnesota, the prototype retrieved live forecast, alert, hydrologic-network, and streamgage data in a single 2.5-second run, producing an explainable risk score with source links. Water temperature—a critical cold-water safety parameter—was unavailable from the nearest gage, and several hazard categories remain unaddressed by current public APIs. We conclude that the approach is technically viable for a minimum viable product targeting U.S. waterways with NWS and USGS coverage, but that significant gaps in data availability and threshold calibration must be resolved before any user-facing deployment.

## Introduction

Recreational paddling carries well-documented risks that depend on local weather, streamflow, water temperature, and physical river hazards. Existing safety guidance is predominantly qualitative and generic: consult forecasts, wear life jackets, monitor conditions, and plan alternate takeouts. While this advice is sound, it does not integrate real-time, location-specific data into a structured risk assessment tied to a particular route.

The question motivating this work is whether a canoe route risk planner can be grounded in public, machine-readable data rather than generic advice. Specifically, a minimal planner should be able to:

1. Map a route waypoint to official weather forecast and alert data.
2. Map the same waterway to hydrologic network context and nearby streamgages.
3. Retrieve recent streamflow and gage observations.
4. Produce a transparent risk score with source links and known limitations.

This paper reports on a smoke-test evaluation of these four capabilities using live public APIs from NWS and USGS. The evaluation is deliberately narrow: a single route, a single time point, and a prototype script. Its purpose is to establish whether the data path exists and is traversable, not to validate a production safety tool.

## Method

### Data Sources

The prototype draws on the following public, machine-readable sources:

- **NWS `api.weather.gov`**: The `/points/{lat},{lon}` endpoint maps a latitude/longitude pair to gridpoint metadata, from which `forecastGridData` and forecast links expose JSON timeseries data. Active alerts for the same point are also available.
- **USGS NLDI**: The Network Linked Data Index indexes streamgages, water-quality sites, and watershed outlets to the river network. It supports hydrolocation (snapping a coordinate to a network feature via COMID) and upstream/downstream navigation to discover nearby gages.
- **USGS Water Services instantaneous values**: The IV service retrieves recent and historical streamflow and other time-series parameters as JSON, with efficient filtered queries by parameter code and time window.
- **USGS hydrography products (NHD/NHDPlus/WBD)**: Referenced for potential future use in reach-level attribute enrichment; not queried in the smoke test.

Safety-threshold context was drawn from NWS cold-water safety guidance (cold shock severity at 50–60 °F water temperature), NWS marine criteria (small-craft advisory thresholds varying by geography, typically involving winds of 20–33 kt), and Minnesota DNR paddling safety guidance (boating accidents reported as five times more likely fatal when water temperature is below 60 °F). These sources informed placeholder thresholds but were not formally calibrated.

### Prototype Implementation

A single dependency-free Python script (`scripts/canoe_risk_smoke.py`) was implemented with the following pipeline:

1. **NWS point query**: Submit the route waypoint to `/points/{lat},{lon}` to obtain the forecast office, grid coordinates, and forecast/alert links.
2. **Forecast and alert retrieval**: Pull the next 12 hourly forecast periods and any active NWS alerts for the waypoint.
3. **NLDI hydrolocation**: Snap the waypoint to the nearest hydrologic network feature, obtaining a COMID.
4. **Gage discovery**: Navigate 50 km upstream and downstream along the mainstem from the COMID to discover NWIS streamgages.
5. **Streamflow retrieval**: Pull 48 hours of USGS instantaneous values for discharge (parameter code `00060`), gage height (`00065`), and water temperature (`00010`) when available.
6. **Risk scoring**: Emit a simple `low | moderate | high` risk score with enumerated reasons, based on placeholder thresholds for wind speed, precipitation probability, active alerts, and streamflow trend.

The script uses only the Python standard library. No external packages, build steps, or containerization are required.

### Test Route

The representative route is a Mississippi River day-paddle near downtown Saint Paul, Minnesota:

- Waypoint: `44.9446, -93.0983`
- USGS gage: `05331000` — Mississippi River at St. Paul, MN

This route was chosen as a well-gaged, publicly documented reach with expected NWS forecast coverage, providing a best-case scenario for data availability.

### Execution Environment

The script was executed on a general-purpose Linux worker with system swap disabled. Resource usage was captured via `/usr/bin/time -v` and post-run memory telemetry (`free -h`). No containerization, GPU, or specialized hardware was involved.

### Risk Score

The smoke-test risk score was **`low`**, with zero risk points accumulated. The reasoning was: no active NWS alerts, and maximum forecast wind (10 mph) below the placeholder concern threshold. This result is specific to the test conditions and should not be generalized. A rising discharge trend of ~15% over 48 hours was noted but did not trigger the placeholder threshold; whether such a trend should influence the risk score is a calibration question that the smoke test does not resolve.

## Limitations

This feasibility evaluation has several important limitations:

1. **Not a safety product.** The smoke script is a prototype demonstrating data-path viability. It must display source timestamps, disclaimers, and guidance to check official channels (NOAA Weather Radio, local authorities) before any real-world use. The risk score it produces should not be used as sole go/no-go guidance.

2. **Route-to-gage relevance is imperfect.** Nearby or network-connected gages may not represent conditions at the exact put-in, take-out, or intermediate reaches. Tributary confluences, dams, rapids, and lake conditions can differ substantially from mainstem gage readings. A confidence metric for gage relevance is needed but not yet implemented.

3. **Water temperature is critical but not universally available.** Cold-water immersion risk is a leading factor in paddling fatalities, yet many streamgages do not report water temperature. The test gage—a well-instrumented mainstem site—did not return water temperature for the requested window. Fallback sources (user-entered observations, local monitoring programs, satellite-derived estimates) are required to close this gap.

4. **Several hazard categories are under-covered by public APIs.** Strainers, sweepers, low-head dams, portage closures, ice, outfitter reports, motorboat traffic, campsite availability, wildfire/smoke, and local regulations are either absent from or only partially represented in the queried data sources. Manual curation or additional data partnerships would be necessary.

5. **Thresholds are placeholder heuristics.** The current wind, precipitation, and alert thresholds are intentionally conservative stand-ins. A production planner requires regional calibration and should decompose risk into separate dimensions: river current, lake wind/wave, cold-water exposure, remoteness, daylight, and user skill level.

6. **Single route, single time point.** The smoke test covers one route at one point in time. It does not demonstrate robustness across diverse waterway types, geographic regions, seasonal conditions, or API failure modes. The favorable results (no alerts, low wind, dense gage data) represent a best-case scenario.

7. **No GPX or multi-waypoint support.** The prototype accepts a single coordinate. Real route planning requires sampling along a path defined by multiple waypoints or a GPX trace, with potentially different NWS gridpoints and gage contexts along the route.

8. **No failure-mode testing.** API timeouts, malformed responses, rate limiting, and missing data were not systematically tested. A production system must handle these gracefully.

9. **Claim audit incomplete.** The claim ledger for this paper contains no completed claim-audit entries at the time of generation. The paper review checklist records all 9 items as pending. The findings reported here should be treated as prototype evidence pending independent verification.

## Conclusion

A dependency-free smoke prototype successfully combined NWS point forecasts and alerts, USGS NLDI hydrolocation and network navigation, and USGS instantaneous streamgage values into an explainable canoe-route risk score for a representative Mississippi River route. The core data path—coordinate to forecast, coordinate to hydrologic network, network to gages, gages to observations—is viable and lightweight, completing in under 3 seconds with modest memory use.

However, the evaluation also reveals meaningful gaps. Water temperature, a critical cold-water safety parameter, was unavailable from the nearest gage—a negative result with direct safety implications. Several hazard categories (physical obstructions, dams, local advisories) are not represented in the queried public APIs. Risk thresholds remain uncalibrated placeholder heuristics. The prototype covers a single route at a single time point under favorable conditions and does not address multi-waypoint routes, API failure modes, or regional variation.

These findings support continuing development toward a minimum viable product for U.S. waterways with NWS and USGS coverage, provided that: (a) gage-relevance confidence scoring is added, (b) water-temperature fallback sources are integrated, (c) manually curated hazard overlays for low-head dams, portages, and local advisories are incorporated, and (d) risk output is presented as an explainable checklist with source links and timestamps rather than a single opaque score. The smoke test establishes that the public-data pipeline is traversable; it does not establish that it is sufficient.
