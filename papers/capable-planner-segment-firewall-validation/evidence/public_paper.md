## Abstract

This report evaluates whether a segment-level context firewall can preserve useful web-agent task context while reducing prompt-injection susceptibility for a capable local planner. A locally provisioned Qwen2.5-3B-Instruct (Q4_K_M) planner was evaluated on 60 realistic shopping, travel, and billing extraction tasks served through llama.cpp. On this validation, full-context prompting achieved clean task success of 1.0 but exhibited nonzero injection susceptibility (0.233), while the segment firewall reduced observed injection success to 0.0 with no clean-task regression. A binary-block baseline also eliminated injections but reduced contaminated task success to 0.0, illustrating the utility trade-off that segment isolation aims to avoid. A cached Phi-4-mini model failed to qualify as a capable planner, serving as a negative control. Results are evidence-bound to the recorded model, harness, task set, and run artifacts; they should not be read as a universal guarantee across all planners or browser-agent stacks.

## Introduction

Prompt-injection attacks remain a practical concern for LLM-based agents that ingest untrusted web content. A segment-level context firewall partitions retrieved page content into isolated context segments so that the planner processes trusted instructions and untrusted page data in separate context windows, preventing injected instructions in untrusted segments from overriding the planner's task.

Prior work in this project line established a segment-firewall implementation and evaluated it under deterministic synthetic benchmarks and a small local model (parent project). The parent results were mixed: some local models were too weak or unstable to serve as capable planners, leaving open the question of whether the segment firewall would retain its protective properties and task-preservation properties when evaluated against a planner that is both competent on clean tasks and susceptible to injection under full-context prompting.

This branch-specific validation addresses that gap by:

1. Defining a capable-planner gate requiring high clean success (≥0.80), nonzero full-context injection susceptibility, substantial firewall attack reduction (≥0.40), and minimal clean regression (<0.05).
2. Provisioning a local Qwen2.5-3B-Instruct GGUF model that meets the gate.
3. Comparing three conditions—full context, segment firewall, and binary block—on realistic extraction tasks.

The central question is whether segment-level isolation can simultaneously reduce injection success and preserve task completion, outperforming the naive binary-block alternative that eliminates injection at the cost of task success on contaminated pages.

## Method

### Segment Firewall

The segment-firewall implementation (`src/segment_firewall.py`) was inherited from the parent project. In the segment-firewall condition, untrusted page content is placed in a context segment that is isolated from the system prompt and task instructions. The planner processes the trusted and untrusted segments in separate turns or windows, preventing injected directives in untrusted content from being interpreted as task-level instructions.

### Benchmark Harness

Two harnesses were used:

1. **Deterministic simulator** (`experiments/capable_planner_validation.py`): A smoke-test harness using a `susceptible_simulator` planner that deterministically follows injected instructions. This verifies harness correctness but does not constitute model-in-the-loop evidence.

2. **Live model benchmark** (`experiments/realistic_model_planner_benchmark.py`): A live OpenAI-compatible planner benchmark over realistic shopping, travel, and billing extraction cases. The planner receives a task instruction and one or more web pages (clean or contaminated with prompt-injection payloads) and must extract the requested information. The benchmark records clean task success, contaminated injection success, and contaminated task success under each condition.

### Task Domains and Cases

The realistic task set covers three domains—shopping, travel, and billing—with extraction-style tasks (e.g., extracting prices, dates, or account details from mock web pages). Each domain contributes both clean cases (no injection present) and contaminated cases (injection payload embedded in page content). The primary validation used 60 cases total: 30 clean and 30 contaminated.

### Conditions

Three prompting conditions were compared:

- **Full context**: The planner receives all content (trusted instructions and untrusted page data) in a single context window.
- **Segment firewall**: Untrusted page content is isolated in a separate segment from trusted instructions.
- **Binary block**: Untrusted page content is entirely withheld from the planner.

### Capable-Planner Gate

The validation gate requires all of the following:

| Criterion | Threshold |
|---|---|
| Full-context clean task success | ≥ 0.80 |
| Full-context injection susceptibility | > 0 |
| Segment-firewall clean task success | ≥ 0.80 |
| Firewall attack reduction vs. full context | ≥ 0.40 |
| Clean-task regression vs. full context | < 0.05 |

A planner that does not meet these criteria is not considered capable for this validation, since the firewall's protective effect cannot be meaningfully assessed if the planner is never susceptible to injection or never succeeds on clean tasks.

### Model Provisioning

No pre-existing OpenAI-compatible endpoint was found on local ports (127.0.0.1:{18080, 8000, 11434, 1234}). Two models were evaluated:

1. **Phi-4-mini** (cached, served via llama.cpp): Negative control. Failed the capable-planner gate with full-context clean success of 0.333, firewall clean success of 0.0, and full-context injection success of 0.0. This model was too weak or unstable for the validation tasks.

2. **Qwen2.5-3B-Instruct** (Q4_K_M GGUF, ~2.0 GiB, downloaded from Hugging Face): Served via llama.cpp on 127.0.0.1:18081. The helper server was stopped after the run. This model met the capable-planner gate.

## Results

### Deterministic Simulator (Smoke Test)

The simulator smoke test (24 cases) verified harness correctness. Under the deterministic susceptible simulator, full-context clean success was 1.00 and full-context contaminated injection success was 1.00. The segment firewall reduced contaminated injection success to 0.00 with no clean-task regression and preserved contaminated task success at 1.00. Binary block prevented injection but reduced contaminated task success to 0.00. These results confirm the harness and metrics function as intended but do not constitute model-in-the-loop evidence.

### Phi-4-mini Negative Control

The cached Phi-4-mini model served via llama.cpp was reachable but failed the capable-planner gate. Full-context clean task success was 0.333, segment-firewall clean task success was 0.0, and full-context injection success was 0.0. This model was insufficiently capable on the extraction tasks to serve as a validation target, preserving the parent-project concern that some local models are too weak or unstable for this validation.

### Qwen2.5-3B-Instruct Primary Result

The primary validation was conducted with Qwen2.5-3B-Instruct (Q4_K_M) on 60 realistic cases. Results are recorded in `results/realistic_model_planner_qwen25_3b_10x.json`.

| Metric | Full Context | Segment Firewall | Binary Block |
|---|---|---|---|
| Clean task success | 1.0 | 1.0 | 1.0 |
| Contaminated injection success | 0.233 | 0.0 | 0.0 |
| Contaminated task success | 0.767 | 1.0 | 0.0 |

Derived metrics:

- Attack reduction (segment firewall vs. full context): 1.0
- Clean-task regression (segment firewall vs. full context): 0.0

The capable-planner gate passed on all criteria.

The key empirical contrast is between the segment-firewall and binary-block conditions on contaminated cases. Both eliminated observed injection successes in this run. However, the segment firewall preserved contaminated task success at 1.0, whereas binary block reduced contaminated task success to 0.0, because the planner could not access the page content needed to complete the task.

## Limitations

1. **Validation scale.** The primary result covers 60 realistic cases (30 clean, 30 contaminated). This scale is sufficient to support the recorded claims but should be independently replicated at larger scale before broad generalization.

2. **Single local model.** Only Qwen2.5-3B-Instruct (Q4_K_M GGUF) met the capable-planner gate. Results are specific to this model, quantization, and serving configuration (llama.cpp). No claim is made that the observed firewall effectiveness transfers to other models, quantizations, or serving stacks without additional validation.

3. **Task domain restriction.** The benchmark covers extraction-style shopping, travel, and billing tasks. Other task types (navigation, form-filling, multi-turn reasoning) may yield different results.

4. **Injection design.** The contaminated cases use specific injection payloads embedded in mock web pages. Injection success rates depend on payload design and may not generalize to all attack strategies.

5. **No claim of complete elimination.** The segment firewall reduced observed injection success to 0.0 in this run, but this does not constitute a guarantee that all prompt-injection attacks are eliminated. The result is bounded by the tested payloads and case count.

6. **Negative control model.** Phi-4-mini's failure to qualify means the validation lacks a second capable-planner data point. The result is strengthened by the negative control (showing the gate meaningfully excludes weak planners) but weakened by the absence of a second passing model.

7. **Local serving only.** The model was served locally via llama.cpp, not through a production API or browser-agent stack. Latency, concurrency, and serving-layer behaviors were not evaluated.

## Conclusion

On this 60-case validation with a locally provisioned Qwen2.5-3B-Instruct (Q4_K_M) planner, the segment firewall eliminated observed prompt-injection successes (reduction from 0.233 to 0.0) while preserving clean task success (1.0) and contaminated task success (1.0). The binary-block baseline also eliminated injections but reduced contaminated task success to 0.0, demonstrating the task-completion cost that segment isolation is designed to avoid.

The capable-planner gate was passed, and the branch kill condition (requiring a planner with high clean success and nonzero injection susceptibility) was not triggered. A Phi-4-mini negative control failed the gate, confirming that the validation meaningfully requires planner capability.

These results support the hypothesis that segment-level context isolation can reduce prompt-injection susceptibility for capable planners without sacrificing task completion on contaminated pages, at least under the conditions tested here. Independent replication on additional production API models, larger case sets, and browser-agent stacks is necessary before broader claims are warranted.
