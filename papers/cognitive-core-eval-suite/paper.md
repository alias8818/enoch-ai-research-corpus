## Abstract

We present the Cognitive Core Eval Suite, a compact, dependency-free synthetic evaluation that probes four separable cognitive axes—state tracking, causal intervention reasoning, belief tracking, and program execution—using deterministic items with exact symbolic scoring. The suite generates items via a seeded procedural generator, enabling reproducible item sets and oracle-verified labels. In a prototype validation on a single consumer-grade GPU node (NVIDIA aarch64, Linux 6.17), we establish that a random axis-label baseline yields 0.130 accuracy on 100 generated items while an oracle solver achieves 1.000, confirming that items are internally solvable and far from trivial. A smoke evaluation of Qwen2.5-7B-Instruct (Q4_K_M) on a balanced 24-item subset produces an overall accuracy of 0.708 but reveals pronounced axis-level separation: perfect scores on state tracking (6/6) and causal intervention (6/6), near-perfect belief tracking (5/6), and complete failure on program execution (0/6). This non-uniform profile constitutes the primary finding: the suite can expose qualitatively different capability levels within a single model rather than collapsing to an aggregate score alone. Results are preliminary—limited to 24 items, one model, one seed—and we detail the specific threats to generalization that follow.

## Introduction

Large language models are increasingly evaluated on aggregate benchmarks that report single-number accuracy, making it difficult to identify which underlying cognitive capacities contribute to performance. A model that scores well on factual recall may still fail on multi-step state tracking or false-belief reasoning, and these failures are obscured by aggregate metrics.

We investigate whether a compact, dependency-light synthetic evaluation suite can expose *separable* cognitive core capabilities—distinct axes along which a model's performance varies independently. The four axes we operationalize are:

1. **State tracking**: maintaining and querying mutable state across a sequence of operations.
2. **Causal intervention**: predicting outcomes when a value in a causal chain is altered.
3. **Belief tracking**: reasoning about an agent's false belief about a state of affairs.
4. **Program execution**: computing the output of a small string-manipulation program.

Each axis is instantiated by procedurally generated items with deterministic correct answers, enabling exact symbolic scoring without human annotation. The suite is designed to run locally with no external dependencies beyond Python 3 and a llama.cpp build, making it suitable for rapid iteration on consumer hardware.

The central question we evaluate is: *can a dependency-free synthetic cognitive-core eval be generated, scored, and run locally, and does a real LLM run on that eval show separable axis-level performance rather than uniform capability?*

## Method

### Item Generation

Items are generated by `cognitive_core_eval.py` using a seeded procedural generator. Each item consists of a natural-language prompt encoding a problem in one of the four axes, a set of labeled answer choices, and a symbolically computed correct label. The generator is deterministic given a seed: running with `--seed 34 --per-axis 25` produces exactly 100 items (25 per axis) stored in JSONL format.

A balanced smoke subset of 24 items (6 per axis) is extracted into a separate JSONL file for rapid testing. This smoke subset is the set used for the model evaluation reported here; the full 100-item set was used only for baseline computation.

### Scoring

Scoring is exact: the runner compares the model's extracted answer label against the oracle-computed correct label. No approximate or human-judged scoring is employed. Unit tests (`test_cognitive_core_eval.py`) verify that the generator, scorer, and DSL execution produce consistent results across the full item set. All unit tests passed (see `logs/unit_tests.log`).

### Baselines

Two baselines are computed over the full 100-item set:

- **Random axis-label baseline**: For each item, a label is drawn uniformly at random from the answer choices available for that item's axis. This yields variable expected accuracy per axis depending on choice-set cardinality, not a uniform 0.25.
- **Oracle symbolic baseline**: The oracle solver computes the correct answer symbolically for every item. This baseline confirms that all items are internally solvable and that labels are consistent with the problem statements.

### Model Evaluation

Model evaluation uses llama.cpp's `llama-completion` binary, invoked per item via the `llama` subcommand of `cognitive_core_eval.py`. The model under test is Qwen2.5-7B-Instruct quantized to Q4_K_M (GGUF format), running fully offloaded to GPU (`--ngl 99`) with a context window of 1024 tokens and a generation limit of 12 tokens (`--n-predict 12`). A per-item timeout of 45 seconds is enforced.

Each invocation spawns a separate `llama-completion` process, which introduces per-prompt startup overhead. Latency measurements therefore include process initialization time and should not be interpreted as pure inference latency. This is a known measurement limitation of the hook-prototype evaluation approach.

## Results

### Baseline Verification

| Baseline | Overall | belief_tracking | causal_intervention | program_execution | state_tracking |
|---|---|---|---|---|---|
| Random axis-label | 0.130 (13/100) | 0.04 (1/25) | 0.36 (9/25) | 0.00 (0/25) | 0.12 (3/25) |
| Oracle symbolic | 1.000 (100/100) | 1.00 (25/25) | 1.00 (25/25) | 1.00 (25/25) | 1.00 (25/25) |

The random baseline's overall accuracy of 0.130 is below the naive 0.25 expectation for uniform four-way guessing, reflecting variable choice-set cardinalities across axes. The causal_intervention axis shows an anomalously high random baseline (0.36), suggesting that axis has fewer or more overlapping answer choices, making random guessing more likely to succeed. The program_execution random baseline of 0.00 indicates that axis uses a larger answer space where uniform guessing never hit the correct label in 25 trials.

The oracle baseline of 1.000 confirms that all 100 items are internally consistent and symbolically solvable, ruling out label errors as a confound for the evaluation infrastructure itself.

### Qwen2.5-7B-Instruct Smoke Results (24 items)

| Axis | Accuracy | Correct / N | Mean Latency (s) |
|---|---|---|---|
| belief_tracking | 0.833 | 5 / 6 | 3.73 |
| causal_intervention | 1.000 | 6 / 6 | 3.69 |
| program_execution | 0.000 | 0 / 6 | 5.45 |
| state_tracking | 1.000 | 6 / 6 | 3.83 |
| **Overall** | **0.708** | **17 / 24** | **4.18** |

The model achieves perfect or near-perfect accuracy on three axes and zero accuracy on program execution. The mean latency for program_execution items (5.45 s) is elevated relative to other axes (3.69–3.83 s), with one item requiring 13.32 s—likely reflecting the model's difficulty producing a valid answer within the token budget rather than a hardware bottleneck, given that the GPU throughput calibration showed stable performance.

Total measured inference time for the 24-item smoke was 100.20 s (Python-measured per-item accumulation), with a wall-clock time of 1:40.24 including process overhead. The close agreement between these two figures (100.20 s vs. 100.24 s) suggests that inter-process overhead was minimal relative to inference time for this configuration, though this may not hold for shorter-generation tasks.

### Axis Separation

The key positive result is the non-uniformity of the accuracy profile. An aggregate score of 0.708 obscures the fact that the model is at ceiling on two axes, near-ceiling on a third, and at floor on the fourth. This separation demonstrates that the suite can produce a cognitive profile rather than only a scalar, and that the four axes are not redundant with one another at this model scale and quantization.

However, the extreme values (0.000 and 1.000) on axes with only 6 items each must be interpreted cautiously. With 6 binary trials, a 95% confidence interval for a true accuracy of 0.50 spans approximately [0.19, 0.81]; the observed 0/6 and 6/6 results are consistent with true accuracies well away from 0.50 but do not pin them precisely.

## Limitations

1. **Small sample size.** The model evaluation covers only 24 items (6 per axis) from a single model. No confidence intervals are meaningful at this sample size, and the accuracy estimates—particularly the 0.000 and 1.000 endpoints—may shift substantially with more items. The full 100-item set should be used for model comparisons.

2. **Single model.** Only Qwen2.5-7B-Instruct Q4_K_M was evaluated. Whether the observed axis separation replicates across model families, scales, and quantization levels is unknown.

3. **Per-prompt process overhead.** Each item spawns a separate `llama-completion` process, inflating latency measurements and limiting throughput. A persistent `llama-server` with batched requests would yield cleaner latency data and enable larger-scale runs.

4. **Program execution confound.** The 0/6 score on program_execution may conflate genuine execution failure with answer-format non-adherence. The model may produce a correct computation in natural language but fail to emit the expected label format within 12 tokens. Prompt-ablation experiments (scratchpad-allowed vs. no-scratch variants) are needed to disambiguate this.

5. **Random baseline asymmetry.** The random baseline varies substantially across axes (0.00–0.36), indicating unequal answer-space sizes. This complicates cross-axis accuracy comparisons unless calibrated. A normalized or chance-adjusted metric (e.g., accuracy above chance) would make axes more comparable.

6. **Deterministic generation from a single seed.** Items are procedurally generated from seed 34 only. While this ensures reproducibility, it limits diversity. Multiple seeds and held-out item sets are needed to assess generalization of the observed axis separation.

7. **No human validation of item quality.** Items are machine-generated and oracle-scored but have not been reviewed by human annotators for ambiguity, cultural bias, or pragmatic naturalness. It is possible that some items are solvable by surface-level heuristics rather than the intended cognitive operation.

8. **Claim ledger audit status.** The structured claim ledger for this artifact was flagged as `blocked_empty_claims` at the time of draft generation, meaning no formal claims passed through the claim/evidence audit pipeline. The findings reported here should be understood as prototype observations rather than audit-approved claims.

9. **Hook-prototype evaluation, not production validation.** The model evaluation was conducted via per-item `llama-completion` process invocations (a hook-prototype approach), not via a production-grade inference server with controlled batching, warm caches, and repeated trials. Results are sufficient to demonstrate axis separation but not to characterize latency distributions or throughput.

## Conclusion

The Cognitive Core Eval Suite is a viable prototype for separable cognitive axis profiling. It generates deterministic, oracle-verifiable items across four axes with no external dependencies; its random baseline is low (0.130 overall) and its oracle baseline is perfect (1.000), establishing that items are non-trivial and internally consistent. A smoke evaluation of Qwen2.5-7B-Instruct on 24 balanced items reveals a sharply non-uniform accuracy profile—perfect on state tracking and causal intervention, near-perfect on belief tracking, and zero on program execution—demonstrating that the suite can expose qualitative capability differences that an aggregate score would obscure.

These results are preliminary. The 24-item, single-model, single-seed evaluation is insufficient for drawing general conclusions about model capabilities or benchmark properties. The program_execution axis in particular requires prompt-engineering follow-up to determine whether the observed failure reflects genuine computational limitation or format mismatch. The asymmetric random baselines across axes complicate direct cross-axis comparison without calibration. Future work should evaluate the full 100-item set across multiple models and seeds, migrate to a persistent server for cleaner latency measurement, add prompt-ablation variants to disambiguate execution failure from format non-adherence, and introduce chance-adjusted metrics to normalize across axes with different answer-space cardinalities.
