## Abstract

We investigate whether commit-level critic gating—requiring each candidate commit to pass declared invariants and hidden critic tests before acceptance—reduces regression escape rates compared to free-form batch patch generation by AI coding agents. We report results from a progressive series of pilot experiments: an initial toy smoke test, a 4-candidate storefront pilot, and an 8-candidate harder storefront pilot, each run under both deterministic model-stub candidates and actual Codex-generated patches. Deterministic stub experiments show a consistent regression escape reduction (free-form escapes 1, commit-gated escapes 0) across all pilot sizes. However, actual Codex-exec experiments yield a neutral result on regression escape rate: both free-form and commit-gated lanes achieve zero escaped regressions in both the 4-candidate and 8-candidate pilots. Commit gating did catch one incomplete per-commit patch in the harder 8-candidate Codex run (a gift card credit feature missing required support), forcing a repair, but this did not manifest as a final regression delta because the free-form batch also preserved the relevant invariants. The overhead of commit gating is substantial: wall-time ratios range from 2.53× to 3.34×, and critic-review-time ratios range from 7.92× to 13.38×. We conclude that commit-level critic gating provides a per-commit quality gate that can catch incomplete patches, but in the tested setting with a strong model, it does not reduce the final regression escape rate enough to justify the observed overhead. The harness may be more discriminating when applied to weaker or budget-constrained models, or to production-style tasks with less explicit invariant specification.

## 1. Introduction

AI coding agents increasingly generate multi-file patches in a single batch. A risk of free-form batch generation is that regressions—violations of invariants the developer intended to preserve—can escape detection if the agent's own declared tests are insufficient. Commit-level critic gating proposes an alternative: require each candidate commit to pass both its declared tests and a set of hidden critic invariants before the commit is accepted into the repository. If a commit fails the critic gate, it is blocked and the agent must repair it before proceeding.

The core hypothesis is that commit-level gating reduces the regression escape rate compared to free-form batch generation. A secondary question is whether any such reduction justifies the additional review time and wall-clock overhead.

This paper reports evidence from a progressive series of bounded pilot experiments that test this hypothesis under controlled but increasingly realistic conditions. We deliberately report mixed and neutral findings where they occur, as the discrepancy between deterministic stub results and actual model-generated patch results is itself a finding of interest.

## 2. Method

### 2.1 Harness Design

The experimental harness (`commit_level_critic_messy_pilot.py`) implements two lanes for the same set of candidate code changes:

- **Free-form lane**: The agent receives all maintenance tasks as a single batch, generates patches in one pass, and the result is evaluated against both declared tests and a full set of hidden critic invariants at the end.
- **Commit-gated lane**: The agent receives one candidate commit at a time. Each commit must pass its declared tests plus candidate-scoped critic invariants (base invariants plus features expected to exist by that commit) before acceptance. If the critic gate fails, the commit is blocked and the agent must repair it before proceeding. After all commits are accepted, the final state is evaluated against the full invariant set.

Both lanes operate on the same generated repository fixture and the same set of maintenance requests. The evidence schema records, for each candidate: rationale, invariant claim, declared tests run, critic verdict, blocked/repaired events, diff lines, regression escape count, and review wall time.

### 2.2 Candidate Sources

The harness supports two candidate sources:

- **Deterministic model-stub**: Pre-scripted candidate patches with known regression properties. This source provides a controlled signal for harness correctness and mechanism plausibility, but does not test actual model behavior.
- **Codex-exec**: Actual patches generated by the Codex CLI/API, invoked per-candidate in the commit-gated lane and as a single batch in the free-form lane. This source tests real model-generated patch quality.

### 2.3 Repository Fixtures

Two fixture sizes were used:

- **4-candidate storefront**: Modules for pricing, inventory, authentication, and reports. Changes are relatively isolated per module.
- **8-candidate storefront**: Extended with `storefront/orders.py` and cross-file checkout/cart tasks (bulk line discounts, gift card credits, backorder line support, strict positive quantities). Changes span pricing, inventory, auth, reports, and orders with cross-file coupling.

### 2.4 Invariant Design

Hidden critic invariants are tests not visible to the candidate source. In the commit-gated lane, each gate checks base invariants plus features that should exist by that commit (candidate-scoped). In the free-form lane, the full invariant set is checked only at the end. This asymmetry is intentional: the commit-gated lane receives incremental quality signals, while the free-form lane relies on the agent's own test discipline.

### 2.5 Metrics

- **Escaped regressions**: Count of hidden critic invariant violations in the final repository state.
- **Blocked commits**: Count of commits rejected by the critic gate (commit-gated lane only).
- **Accepted commits**: Count of commits passing the critic gate.
- **Critic-review-time ratio**: Ratio of critic review wall time (commit-gated) to free-form review wall time.
- **Wall-time ratio**: Ratio of total wall-clock seconds (commit-gated) to free-form.

## 3. Results

### 3.1 Initial Smoke Test (Toy Arithmetic Script)

The first experiment used a minimal arithmetic script with deterministic stub candidates.

| Lane | Escaped Regressions | Blocked Commits | Review-Time Ratio |
|------|-------------------|-----------------|-------------------|
| Free-form | 1 | — | — |
| Commit-gated | 0 | 1 | 10.19× |

This established that the harness mechanism can detect and block a regression. However, the candidate commits were scripted rather than model-generated, and the repository was toy-sized, so external validity is weak.

### 3.2 4-Candidate Storefront Pilot

#### 3.2.1 Deterministic Model-Stub

| Lane | Escaped Regressions | Blocked Commits | Review-Time Ratio |
|------|-------------------|-----------------|-------------------|
| Free-form | 1 | — | — |
| Commit-gated | 0 | 3 | 16.90× |

Commit-gated events: coupon alias accepted; SKU hyphen regression blocked/repaired; role substring-authorization regression blocked/repaired; report input-mutation regression blocked/repaired. The deterministic stubs show a clear regression escape reduction.

#### 3.2.2 Codex-Exec (Actual Model Patches)

| Lane | Escaped Regressions | Blocked Commits | Diff Lines | Wall Seconds |
|------|-------------------|-----------------|------------|--------------|
| Free-form | 0 | — | 38 | 68.35 |
| Commit-gated | 0 | 0 (4 accepted) | 99 | 173.03 |

Critic-review-time ratio: 7.92×. Wall-time ratio: ~2.53×.

**Result: Neutral on regression reduction.** Both lanes achieved zero escaped regressions. The free-form Codex batch preserved all hidden invariants without requiring per-commit gating.

### 3.3 8-Candidate Harder Storefront Pilot

#### 3.3.1 Deterministic Model-Stub

| Lane | Escaped Regressions | Blocked Commits | Review-Time Ratio |
|------|-------------------|-----------------|-------------------|
| Free-form | 1 | — | — |
| Commit-gated | 0 | 6 | 34.33× |

The deterministic stubs continue to show a strong regression escape reduction with commit gating.

#### 3.3.2 Codex-Exec (Actual Model Patches)

| Lane | Escaped Regressions | Blocked Commits | Wall Seconds |
|------|-------------------|-----------------|--------------|
| Free-form | 0 | — | 182.22 |
| Commit-gated | 0 | 1 (8 accepted) | 608.61 |

Critic-review-time ratio: 13.38×. Wall-time ratio: ~3.34×.

The single blocked commit was meaningful: candidate `checkout-gift-card-credit` initially left `checkout()` without `gift_card_cents` support despite declared tests passing. The critic blocked it, and Codex repaired it before the next commit. However, this did not produce a regression escape delta because the free-form batch also preserved the full invariant set in its final state.

**Result: Neutral on regression-rate improvement.** Commit gating caught an incomplete per-commit patch, but the free-form lane's final result was equally correct on the regression metric.

### 3.4 Summary of Overhead

| Pilot | Candidate Source | Wall-Time Ratio | Review-Time Ratio |
|-------|-----------------|-----------------|-------------------|
| 4-candidate | Deterministic stub | — | 16.90× |
| 4-candidate | Codex-exec | 2.53× | 7.92× |
| 8-candidate | Deterministic stub | — | 34.33× |
| 8-candidate | Codex-exec | 3.34× | 13.38× |

Commit gating imposes substantial overhead in both wall-clock time and critic review effort. The overhead grows with pilot size and task coupling.

## 4. Limitations

1. **Small sample size.** Each pilot configuration was run once. No statistical confidence intervals can be computed over multiple runs. The results are suggestive but not statistically significant.

2. **Synthetic repository fixtures.** The storefront repositories are synthetic and self-contained. They do not represent the complexity, scale, or dependency structure of real production codebases. External validity to production settings is unknown.

3. **Single model source.** All actual-model results use Codex CLI/API. The behavior of other models—particularly weaker or budget-constrained models—may differ. The project decision artifact explicitly recommends reusing the harness for such models in future work.

4. **Deterministic stub vs. actual model discrepancy.** The deterministic stubs show a strong positive signal for commit gating, while actual Codex runs show a neutral signal. This discrepancy may reflect that the stubs were designed with known regression properties, whereas Codex patches in these bounded tasks happened to preserve invariants. The stubs test the mechanism; the Codex runs test the mechanism's value under current model quality.

5. **Invariant design bias.** The hidden critic invariants were authored by the same process that designed the harness. In a real deployment, invariant authoring is itself a hard problem, and the quality of critic invariants would determine the gate's effectiveness.

6. **No human review baseline.** The experiments compare two automated lanes. No human reviewer was involved, so the results do not address how commit gating compares to or complements human code review.

7. **Review-time metric is synthetic.** The "review time" recorded by the harness measures critic test execution and gate evaluation, not human cognitive effort. The ratios reported (7.92×–34.33×) reflect automated processing overhead, not human time savings or costs.

8. **No GPU or accelerator involvement.** All experiments were CPU-only unit-test pilots using Codex CLI/API for patch generation. No accelerator-heavy model serving was used. Performance characteristics would differ under local model inference with GPU acceleration.

## 6. Conclusion

Commit-level critic gating provides a mechanism for blocking incomplete or incorrect individual commits before they enter the repository. In deterministic stub experiments, this mechanism clearly reduces regression escape rates. However, in actual Codex-exec experiments on bounded storefront tasks, the free-form batch lane also achieves zero escaped regressions, yielding a neutral result on the primary metric of regression escape rate reduction.

The one blocked commit in the 8-candidate Codex run demonstrates that the gate can catch real model-generated incompleteness—a candidate that passed its declared tests but failed hidden critic invariants. This per-commit quality-control function may have value as a control-plane gate even when it does not improve the final regression rate, particularly in settings where intermediate repository states must be valid (e.g., continuous integration, collaborative development).

The overhead of commit gating is non-trivial: wall-time ratios of 2.5–3.3× and critic-review-time ratios of 8–13× in actual model runs. Whether this overhead is justified depends on the cost of regressions in the target environment and the reliability of the model being gated.

The current evidence suggests that commit-level critic gating is most likely to show a regression-rate benefit when applied to models that are more prone to invariant violations than Codex proved to be in these bounded tasks. The project decision recommends reusing the harness for future benchmarks with weaker or budget-constrained models, or with real production-style tasks where invariant violations are more likely to escape a free-form batch.
