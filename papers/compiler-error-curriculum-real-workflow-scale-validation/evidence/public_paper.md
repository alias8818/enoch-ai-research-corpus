## Abstract

We investigate whether compiler and test failure evidence, organized as a curriculum of reusable failure-signature-to-repair-operator mappings, improves automated repair-loop recovery at a realistic multi-file workflow scale compared to a generic no-curriculum repair order. A deterministic validation harness generates 12 families of multi-file Python workflow projects (bytes payload decoding, CSV ingestion, decorated currency parsing, ISO-Z datetime parsing, environment config, nested JSON payloads, mutable default cache leakage, pagination off-by-one, base-directory path resolution, plugin import path, status normalization, and CLI syntax errors), mines failure signatures from training trajectories, and evaluates two strategies under an identical `max_iters=3` budget. On 96 held-out evaluation cases across all 12 families, the baseline (fixed generic repair order) recovers 8/96 cases (0.083), while the error-curriculum strategy recovers 96/96 (1.000), yielding an absolute recovery lift of 0.917 and a relative lift of 11.0×. Curriculum throughput (15.14 samples/sec) also exceeded baseline (8.03 samples/sec). These results support the mechanism at local validation scale, but they derive from synthetically generated workflow-shaped projects and a deterministic policy proxy rather than a learned model or production repository corpus. Generalization to arbitrary production codebases remains unvalidated.

## 1. Introduction

Automated program repair loops commonly iterate over a fixed or randomly ordered set of repair actions when confronted with compiler errors or test failures. This approach ignores the informational content of the failure itself: the error type, message, and location constitute a signature that may map reliably to a specific repair. If such mappings can be mined from prior failures and reused as a curriculum, repair loops could converge faster and recover more cases within a bounded iteration budget.

We test this hypothesis with a local, real-workflow-shaped scale validation. Rather than drawing from a production repository corpus, we generate multi-file Python projects that structurally resemble common workflow patterns (parsing, configuration, import resolution, etc.), inject characteristic failure modes, and evaluate whether a curriculum mined from training trajectories improves recovery on held-out variants of the same families.

The core question is not whether a learned model can generalize across repositories, but whether the mechanism of failure-signature-indexed repair selection yields measurable gains under controlled conditions. A positive result at this scale is a necessary but not sufficient condition for production claims.

## 2. Method

### 2.1 Harness

The validation harness is implemented in `src/real_workflow_scale_validation.py`. It performs the following steps deterministically:

1. **Project generation.** For each of 12 workflow families, generate multi-file Python projects with characteristic failure modes injected. Each family targets a distinct error category (e.g., `ValueError` in currency parsing, `ImportError` in plugin loading, `SyntaxError` in CLI entrypoints).

2. **Training phase.** Execute each training variant through the real Python compiler (`py_compile`) and `unittest` runner. Record the failure signature (error type, message pattern, failing file/line) and the repair operator that resolved it. Store these as a `failure_signature → repair_operator` mapping (the curriculum).

3. **Evaluation phase.** For each held-out evaluation variant, execute under two strategies:
   - **baseline_no_curriculum:** Apply repairs in a fixed generic order, ignoring the observed failure signature.
   - **error_curriculum:** Look up the observed failure signature in the mined curriculum mapping and apply the corresponding repair operator first.

4. **Budget.** Both strategies receive the same `max_iters=3` iteration budget. A case is "recovered" if all compiler errors and test failures are resolved within this budget.

### 2.2 Workflow Families

The 12 families are:

| # | Family | Failure Category |
|---|--------|-----------------|
| 1 | Bytes payload decoding | Type/decode error |
| 2 | CSV ingestion | Parse/format error |
| 3 | Decorated currency parsing | ValueError |
| 4 | ISO-Z datetime parsing | ValueError |
| 5 | Environment config | KeyError / missing env |
| 6 | Nested JSON payloads | KeyError / structure mismatch |
| 7 | Mutable default cache leakage | State mutation bug |
| 8 | Pagination off-by-one | Logic/index error |
| 9 | Base-directory path resolution | FileNotFoundError |
| 10 | Plugin import path | ImportError |
| 11 | Status normalization | Mapping/key error |
| 12 | CLI entrypoint syntax | SyntaxError |

### 2.3 Scale

- **Training:** 3 variants per family × 12 families = 36 training cases.
- **Evaluation:** 8 held-out variants per family × 12 families = 96 evaluation cases.
- Training and evaluation variants differ in concrete data and parameter values but share the same failure-mode structure within each family.

### 2.4 Metrics

- **Recovery rate:** Fraction of evaluation cases fully resolved within `max_iters=3`.
- **Throughput:** Samples processed per second (wall time).
- **Memory:** `MemAvailable` and `SwapFree` from `/proc/meminfo` before and after the full run.

### 2.5 Execution Environment

All runs executed locally with swap disabled. Memory monitoring confirmed no resource exhaustion: `MemAvailable` decreased by approximately 90 MB over the full run (from 122,635,896 kB to 122,545,416 kB), and `SwapFree` remained at 0 kB throughout.

## 4. Limitations

1. **Synthetic projects, not production repositories.** The workflow projects are generated to structurally resemble real patterns (CSV parsing, config loading, plugin imports, etc.), but they are not drawn from an actual codebase or issue tracker. Failure-mode diversity is bounded by the 12 families implemented in the harness.

2. **Deterministic policy proxy, not a learned model.** The curriculum selector performs exact lookup of failure signatures in a mined mapping. It does not involve an LLM fine-tune, embedding similarity, or fuzzy matching. Production deployments would need to handle novel or ambiguous signatures that lack exact matches.

3. **Within-family generalization only.** Evaluation variants share the same failure-mode structure as training variants within each family. The curriculum is not tested on failure signatures absent from the training set. Cross-family or cross-repository generalization is unvalidated.

4. **Fixed iteration budget.** The `max_iters=3` budget is generous for the curriculum (which typically resolves in 1 iteration) but restrictive for the baseline. A higher budget would narrow the gap, though at increased cost.

5. **No LLM-generated repairs.** Repair operators are deterministic code transformations, not model-generated patches. The result validates curriculum-based action selection, not patch generation quality.

6. **Single harness, single language.** All projects are Python. Generalization to other languages, build systems, or error-reporting formats is not addressed.

7. **Ceiling effect.** The 96/96 curriculum recovery rate likely reflects the controlled nature of the benchmark. Real production failures would exhibit more diverse and ambiguous signatures, likely yielding a lower ceiling.

## 6. Conclusion

A deterministic validation harness demonstrates that compiler and test failure evidence, organized as a curriculum of failure-signature-to-repair-operator mappings, substantially improves repair-loop recovery on synthetically generated multi-file Python workflow projects. Under a `max_iters=3` budget, the curriculum strategy recovered 96/96 held-out evaluation cases (recovery rate 1.000) versus 8/96 for a generic no-curriculum baseline (recovery rate 0.083), an absolute lift of 0.917 and relative lift of 11.0×. Curriculum throughput also exceeded baseline by approximately 1.9×.

These results validate the mechanism at local scale: failure-signature-indexed repair selection is effective when the signature space is well-covered by training data. The result is a bounded positive signal, not a production generalization claim. The curriculum selector is a deterministic policy proxy, not a learned model, and the benchmark uses generated workflow-shaped projects rather than production repositories. The necessary next step is to apply the same trajectory schema to a real repository and issue corpus, or to train and evaluate a repair model on mined compiler-error trajectories at broader scale, to determine whether the observed gains persist under the ambiguity and diversity of production failures.
