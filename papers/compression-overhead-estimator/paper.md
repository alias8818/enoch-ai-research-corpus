## Abstract

We present a lightweight, locally runnable estimator that quantifies when byte-level compression overhead is worth paying in pipelines where compressed data bypasses downstream token processing. The estimator measures compression and decompression wall-clock latency, throughput, and compression ratio for synthetic text, code, JSON, and random payloads using standard-library compressors (zlib, bz2, lzma). It computes estimated token savings, break-even downstream processing throughput, and net latency savings under a configurable prefill tokens-per-second assumption. On an aarch64 host with 20 CPUs, compressible 1 MiB synthetic corpora yielded net latency savings of approximately 45–52 seconds at a 5,000 tokens/s downstream assumption, with break-even throughputs ranging from approximately 4.59M to 285M estimated tokens/s—well above typical LLM prefill rates. Random (incompressible) payloads correctly produced no latency win. These results are bounded: synthetic corpora are not representative of production workloads, the estimator does not score semantic compression quality, and the 4 bytes/token heuristic requires replacement with a target-model tokenizer for deployment decisions. The claim ledger for this artifact records no structured claims, and the audit status is blocked-empty-claims; the numerical results reported here should be treated as prototype calibration data rather than validated production benchmarks. The artifact is self-contained, reproducible, and emits machine-readable JSON/CSV metrics.

## 1. Introduction

When data is compressed before entering a downstream processing stage—such as an LLM prefill path—the compression overhead is justified only if the time saved by processing fewer tokens exceeds the time spent compressing. This trade-off is straightforward to state but non-trivial to quantify: it depends on compressor throughput, compression ratio, payload characteristics, and the downstream processing rate.

Prior work on prompt compression for LLMs has focused on semantic methods that reduce token count while preserving task-relevant information. Byte-level compression occupies a different niche: it is lossless, fast, and well-understood, but produces a bitstream that is not directly consumable by token-based pipelines. The question this work addresses is narrower: given a known compression ratio and compressor throughput, is the compression overhead latency-positive or latency-negative for a given downstream processing rate?

We built a self-contained Python CLI tool that answers this question empirically. It does not claim to solve semantic prompt compression; rather, it provides the overhead-economics layer that any such system must account for when deciding whether to compress at all. The scope is deliberately limited to byte-level compression overhead estimation; we do not evaluate whether compressed representations are semantically usable downstream.

## 2. Method

### 2.1 Estimator Design

The estimator (`scripts/compression_overhead_estimator.py`) is a Python 3 CLI using only standard-library compressors: `zlib` (levels 1, 6, 9), `bz2` (levels 1, 9), and `lzma` (levels 0, 6, 9). For each synthetic corpus and compressor configuration, it performs the following steps:

1. **Generate** a synthetic payload of the requested size and type (text, code, json, random).
2. **Compress** the payload, measuring wall-clock compression time.
3. **Decompress** the result, measuring wall-clock decompression time and verifying round-trip correctness.
4. **Compute metrics:**
   - `ratio = compressed_bytes / input_bytes`
   - `estimated_tokens_in = input_bytes / 4` (4 bytes/token heuristic)
   - `estimated_tokens_out = compressed_bytes / 4`
   - `saved_tokens = estimated_tokens_in - estimated_tokens_out`
   - `break_even_prefill_tps = saved_tokens / compression_seconds`
   - `net_saved_ms = (saved_tokens / prefill_tps × 1000) − compression_ms`

The break-even throughput is the minimum downstream processing rate at which compression becomes latency-negative. If the downstream system processes tokens slower than `break_even_prefill_tps`, compression saves net latency. The net-saved metric makes this concrete for a user-specified `prefill_tps`. When `saved_tokens = 0` (incompressible data), `net_saved_ms` is reported as not applicable.

### 2.2 Synthetic Corpus Generation

Corpora are procedurally generated within the estimator. The text, code, and JSON payloads are synthetic and exhibit repetitive structure. This design choice enables reproducibility without external data dependencies, but it means that compression ratios on these corpora are likely higher than on typical real-world data of the same nominal type. The random corpus serves as a negative control: it should be incompressible, producing ratios near or above 1.0.

## 3. Results

### 3.1 Compressible Payloads at 5,000 tokens/s

For compressible synthetic corpora at 1 MiB, compression overhead was small relative to downstream token-processing savings. Selected results at the 5,000 tokens/s prefill assumption:

| Corpus | Compressor | Ratio | Mean compression time (ms) | Net saved latency (ms) |
|--------|-----------|-------|---------------------------|----------------------|
| Code | lzma-0 | 0.005 | 4.753 | 52,160 |
| Text | bz2-1 | 0.068 | 41.941 | 48,845 |
| JSON | bz2-1 | 0.074 | 39.528 | 45,080 |

These figures represent prototype calibration measurements on synthetic data. The very high compression ratios (particularly 0.005 for synthetic code) reflect the repetitive structure of the generated corpora and should not be assumed for production workloads.

### 3.2 Sensitivity at 50,000 tokens/s

At the 50,000 tokens/s prefill assumption, compressible corpora remained latency-positive, though net savings were proportionally smaller (one-tenth the downstream saving per token, with compression time unchanged). This confirms that the latency-positive finding is robust to an order-of-magnitude increase in assumed downstream processing rate for these synthetic payloads.

### 3.3 Break-Even Throughputs

Measured break-even downstream throughputs for compressible payloads ranged from approximately 4.59M to 285M estimated tokens/s. Typical LLM prefill paths operate well below these rates, indicating that byte-compression overhead is unlikely to be the limiting factor for highly repetitive text, code, or JSON on this hardware. However, this conclusion is contingent on the compression ratios observed; less compressible real-world data would yield lower break-even throughputs.

### 3.4 Incompressible Payloads

Random data payloads produced compression ratios near or above 1.0, resulting in `saved_tokens = 0`. The estimator correctly reports no applicable latency win for these cases, with `net_saved_ms` reported as not applicable. This negative-case behavior is an important correctness property: the estimator does not falsely claim savings on data that cannot be compressed.

### 3.5 Decompression and Round-Trip Verification

All runs verified lossless round-trip decompression. Decompression latency was measured but is not the primary gating factor in the prefill scenario modeled here, which assumes the compressed payload is consumed directly downstream without re-expansion. If a pipeline requires decompression before consumption, decompression time must be added to the overhead side of the break-even equation, potentially eliminating net savings.

## 4. Limitations

1. **Synthetic corpora are not production workloads.** The text, code, and JSON payloads are procedurally generated and exhibit higher repetition than typical real-world data. Compression ratios and net savings on actual production corpora may differ substantially. The extreme ratio of 0.005 observed for synthetic code is unlikely to hold for real source code.

2. **No semantic compression quality evaluation.** The estimator models byte-level compression overhead and downstream token-processing savings. It does not assess whether the compressed representation preserves task-relevant information or is compatible with an LLM context window. This is the more consequential unknown: even if overhead is negligible, a compressed bitstream cannot be fed to a standard LLM tokenizer.

3. **4 bytes/token heuristic.** The default token-count estimate uses a conservative 4 bytes/token ratio. Actual token counts depend on the target model's tokenizer and can vary significantly across languages, code vs. natural text, and tokenizer version. Model-specific tokenizer calibration is required for deployment decisions.

4. **No reconstruction/decode cost in the net-savings formula.** The current model assumes the compressed representation is consumed directly downstream. If a decompression or reconstruction step is required before the downstream system can process the data, that cost must be included, potentially eliminating the net savings entirely.

5. **Single-host calibration.** All measurements were taken on one aarch64 host (20 CPUs, ~122.7 GiB RAM). Compressor throughput varies with CPU architecture, clock speed, and library version. Results should not be generalized without re-calibration on target hardware.

6. **No GPU-pathway measurement.** The benchmark is CPU-bound. If compression were offloaded to GPU or integrated into a GPU pipeline, the overhead profile would change. This possibility was not explored.

7. **Claim ledger audit status is blocked.** The structured claim ledger for this artifact contains no extracted claims and has audit status `blocked_empty_claims`. The numerical results reported in this paper are drawn directly from the run notes and calibration logs but have not passed a formal claim/evidence audit.

## 6. Conclusion

We built and validated a self-contained compression overhead estimator that measures compressor latency, throughput, compression ratio, and estimated token savings, then computes break-even downstream processing rates and net latency savings under configurable assumptions. On synthetic compressible corpora, byte-level compression overhead was overwhelmingly latency-positive at both 5,000 and 50,000 tokens/s downstream assumptions, with break-even throughputs orders of magnitude above typical LLM prefill rates. On incompressible data, the estimator correctly reported no benefit.

These findings are bounded and should not be over-generalized. The large net savings reported here depend on high compression ratios that reflect synthetic, highly repetitive data; real-world compression ratios will almost certainly be less favorable. More fundamentally, the real gating question for deploying compression in LLM pipelines is not whether byte-compression overhead is affordable—it almost always is for compressible payloads—but whether the compressed representation can be consumed downstream without semantic loss or expensive reconstruction. The estimator does not address this question.

Scientific closure for a specific production context requires: (1) calibration on representative target corpora, (2) replacement of the 4 bytes/token heuristic with the target model's tokenizer, (3) inclusion of any reconstruction or decode cost in the net-savings formula, and (4) evaluation of whether the compressed representation is semantically adequate for the downstream task. The estimator provides the reusable overhead-economics layer; the domain-specific questions remain open.
