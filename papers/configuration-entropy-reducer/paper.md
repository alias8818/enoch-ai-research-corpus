## Abstract

Configuration file sprawl in software projects is a recognized operational burden, yet few lightweight tools exist to quantify it or identify concrete reduction opportunities without external services, repository metadata, or heavy dependency chains. We present a dependency-free, standard-library-only Python CLI that scans local project trees for configuration files, computes Shannon entropy over configuration families, and detects three classes of reduction opportunity: duplicate basenames, identical content hashes, and repeated configuration keys. Evaluation on a controlled fixture and a corpus of 12 neighboring local project roots shows sub-100 ms scan times per root (median 31 ms, max 81 ms) and 18 total reduction opportunities across the corpus. Duplicate basenames were present in 4 of 12 roots and identical content in 2 of 12 roots. Repeated-key signals appeared in all 12 roots but are known to overcount schema-like and generated JSON files, reducing their precision. Precision and recall against human-reviewed ground truth remain unmeasured. These results support the tool's viability as an advisory measurement assistant but do not yet support use as a production-grade autofix system.

## Introduction

Software projects accumulate configuration files across multiple formats, directories, and abstraction layers over time. This accumulation—configuration sprawl—increases maintenance burden, raises the risk of inconsistent settings across environments, and complicates developer onboarding. The problem is widely recognized in practice but underserved by lightweight measurement tools. Existing approaches tend to require semantic parsing of specific formats, integration with version-control metadata, or heavyweight dependency chains that limit adoptability.

We investigate a minimal hypothesis: a lightweight, dependency-free scanner can make configuration sprawl measurable and identify reduction opportunities using only filesystem-level signals—filename patterns, content hashing, and lexical key extraction—without requiring private repository metadata or external services.

The contributions of this work are:

1. A standard-library-only Python CLI that discovers configuration files by naming convention, computes family-level Shannon entropy, and emits structured, prioritized reduction suggestions in both JSON and Markdown.
2. Empirical evidence from a controlled fixture and a 12-root local corpus demonstrating sub-100 ms scan times and non-trivial reduction opportunity detection.
3. An honest accounting of limitations, particularly the tendency of lexical repeated-key detection to overcount generated and schema-like files, and the absence of measured precision or recall against human-reviewed ground truth.

## Method

### Tool Design

The scanner (`scripts/config_entropy_reducer.py`) operates in a single pass over a given directory tree. It uses only the Python standard library.

**File discovery.** Files are identified as configuration files if they match a known set of basenames (e.g., `.env`, `Makefile`), known suffixes (e.g., `.toml`, `.yaml`, `.cfg`), or contain the substring `config` in their basename. Common generated and heavy directories (`.git`, `node_modules`, `__pycache__`, build output directories, caches) are excluded via a configurable exclusion list.

**Family classification.** Each configuration file is assigned to a "family" based on its format or naming convention (e.g., `env`, `toml`, `yaml`). The distribution of files across families is used to compute Shannon entropy.

**Entropy computation.** Family Shannon entropy is computed as:

$$H = -\sum_{i} p_i \log_2 p_i$$

where $p_i$ is the proportion of files in family $i$. A normalized variant divides by $\log_2 N$, where $N$ is the number of families present.

**Reduction opportunity detection.** Three signal classes are examined:

- **Duplicate basenames** (high priority): Groups of configuration files sharing the same basename in different directories, indicating potential consolidation targets.
- **Identical content** (high priority): Groups of files with matching SHA-256 content hashes, indicating copy-paste duplication.
- **Repeated keys** (medium priority): Configuration keys that appear across multiple files, detected via line-level lexical parsing of `KEY=VALUE` and `KEY: VALUE` patterns. This signal is format-agnostic and intentionally shallow, which makes it broadly applicable but imprecise.

**Output.** The tool emits both JSON and Markdown reports containing file counts, family counts, entropy values, and prioritized reduction suggestions.

### Evaluation Protocol

We evaluate the tool across three conditions:

1. **Controlled fixture.** A synthetic directory (`artifacts/fixture_entropy_demo`) containing three `.env` files (two with identical content) and one `pyproject.toml`, designed to exercise all three detection classes.
2. **Self-scan.** The tool is run on its own project directory with artifact outputs excluded.
3. **Local corpus scan.** Twelve neighboring local project roots are scanned with heavy directories excluded. These are real project trees of varying size from the same development environment, providing ecological validity despite the small and non-random sample.

### Unit Testing

A single unit test (`tests/test_config_entropy_reducer.py`) verifies detection of duplicate basenames, identical content groups, and repeated keys against the controlled fixture.

Scan runs were short CPU and light I/O operations; no memory pressure was observed or expected given the workload. These timing results reflect a prototype execution environment and should not be treated as production-grade benchmarks.

## Limitations

1. **Lexical repeated-key detection overcounts.** The current key extraction is line-level and format-agnostic. Large generated or schema-like JSON files can produce high repeated-key counts that do not represent actionable drift risk. This inflates the repeated-key signal and reduces its precision. The fact that repeated keys were detected in 12/12 roots likely reflects this overcounting rather than a uniformly high true-positive rate.

2. **No parser-aware key paths.** Without format-specific parsing, the tool cannot distinguish a key `name` in a `pyproject.toml` `[project]` section from a key `name` in a `.env` file. Semantic key-path resolution would improve suggestion quality but would introduce dependencies, conflicting with the dependency-free design goal.

3. **No generated-file classification.** The tool does not distinguish handwritten configuration from generated or locked files. A generated-file detector (e.g., based on `@generated` markers, lockfile naming conventions, or `.gitattributes` entries) would reduce false positives, particularly for the repeated-key signal.

4. **Precision and recall are unmeasured.** No human-reviewed ground truth exists for the reduction suggestions. The 18 opportunities identified in the corpus have not been manually validated, so the false-positive rate is unknown. This is the most significant gap in the current evidence.

5. **Small and non-random corpus.** The 12 scanned roots are neighboring local project directories from the same development environment. They are not a random sample of software projects, and generalization to other codebases is not justified by the current evidence.

6. **No autofix capability.** The tool currently operates in an advisory (measurement and suggestion) mode only. No automated modification of configuration files was attempted. Autofix for even the safest case (identical-content canonicalization) would require human review workflows to avoid semantic breakage.

7. **Original specification unavailable.** The linked Notion page returned HTTP 404 when accessed through the available browser surface. The implemented tool therefore validates a plausible operational interpretation of the project idea rather than a specific private specification.

8. **Prototype-level evidence only.** All results come from a single-session prototype implementation and local smoke-test evaluation. No independent reproduction, large-scale corpus evaluation, or production deployment has been performed.

## Conclusion

A dependency-free local scanner for configuration entropy was implemented and evaluated on a controlled fixture and a 12-root local corpus. The tool produces measurable entropy metrics and identifies concrete reduction opportunities with sub-100 ms scan times per root on the sampled environment. Duplicate-basename and identical-content signals provide high-confidence, auditable suggestions that do not require semantic parsing. Repeated-key signals are broadly triggered but suffer from overcounting in the presence of generated and schema-like files, and their precision is unknown.

The evidence supports the tool's viability as an advisory measurement and reduction assistant at the prototype level. It does not yet support use as a production-grade autofix system. The primary gaps are: (a) absence of parser-aware key-path resolution, which limits suggestion quality; (b) lack of generated-file classification, which inflates false positives; and (c) unmeasured precision and recall against human-reviewed ground truth, which leaves the practical utility of the suggestions unvalidated. Future work should address these gaps, establish a manual review rubric for suggestion quality on a broader and more representative repository sample, and prototype safe autofix only for the identical-content canonicalization case, where the risk of semantic breakage is lowest.
