## Abstract

We investigate whether reusable long-language-model contexts can be treated as a capital allocation problem, where each context is routed to one of four storage tiers—full KV cache in fast memory, offloaded cache with transfer penalty, compressed attention index with modeled quality risk, or full recompute—based on observed reuse patterns, lookup intensity, sparsity tolerance, and memory budgets. In a synthetic simulation with 600 contexts and 9,000 requests, the allocator reduces modeled prefill/reuse cost by 30.74% relative to all-recompute with only 0.34% modeled quality loss, satisfying a pre-registered criterion of ≥25% cost reduction with ≤2% quality loss. However, the allocator underperforms an LRU full-cache baseline by 70.05% when GPU memory is ample (14,000 MB), because it over-allocates contexts to compressed indices even when full-cache hits are inexpensive. A five-seed budget sweep confirms that the allocator outperforms LRU only under tight memory constraints (512–2,048 MB), where compressed indices enable reuse that LRU cannot afford. At budgets of 4,096 MB and above, LRU full-cache is substantially cheaper. The offload tier was never selected in any experiment, suggesting a cost-model calibration gap. All results are conditional on a synthetic Zipf/drift workload and modeled—not measured—quality loss. The approach shows promise as a memory-constrained routing layer but should not be interpreted as a universal replacement for full KV caching.

## 1. Introduction

Long-context language models incur substantial prefill costs when the same context is reused across multiple requests. Recent systems work has begun to address this: sparse attention lookup tables can approximate full attention at reduced cost (arXiv:2604.08584), multi-tier KV cache reuse is an active systems target (LMCache technical report), and shared-context benchmarks highlight the importance of the KV-cache lifecycle beyond single-request evaluation (arXiv:2412.10319). Production frameworks document offloaded and quantized KV cache modes while warning that such modes represent memory–latency tradeoffs rather than universally beneficial optimizations (Hugging Face Transformers KV cache documentation).

The central observation motivating this work is that reusable long contexts resemble capital assets: some are "hot" and warrant full retention in fast memory, others are accessed infrequently enough to tolerate compression or offloading, and still others are better discarded and recomputed on demand. Rather than applying a single caching policy uniformly, we ask whether a portfolio-style allocator—assigning each context to the tier that minimizes expected total cost subject to a memory budget—can reduce aggregate inference cost while bounding quality degradation.

We define four allocation tiers:

1. **full_cache**: Retain the full KV/context state in fast (GPU) memory. Zero transfer cost on hit; highest memory consumption.
2. **offload**: Retain the full cache outside fast memory (CPU/SSD) with a modeled transfer penalty on access. Low fast-memory consumption; latency cost on hit.
3. **attention_index**: Retain a compressed or sparse lookup index with a small modeled quality risk. Low fast-memory footprint; potential quality degradation.
4. **recompute**: Discard the cache entirely and pay the full prefill cost again on next access. Zero persistent memory cost; highest compute cost on hit.

We evaluate this allocator in a synthetic simulation and measure cost reduction relative to two baselines: all-recompute (no caching) and LRU full-cache (cache everything in fast memory with least-recently-used eviction). Our pre-registered success criterion, established before running the full experiment, requires at least 25% cost reduction versus all-recompute with no more than 2% modeled quality loss.

## 2. Method

### 2.1 Allocator Design

The allocator maintains per-context statistics including access frequency, recency, estimated lookup intensity, and sparsity tolerance. At each allocation decision point, it computes an expected cost for each tier assignment given the context's observed access pattern and the current memory budget, then selects the tier that minimizes expected total cost. Quality loss is modeled as a function of the compression ratio implied by the attention_index tier; it is not measured by an LLM evaluator.

The cost model assigns:

- **full_cache**: Memory cost proportional to context length; zero access cost.
- **offload**: Near-zero fast-memory cost; access cost proportional to context length and a transfer penalty factor.
- **attention_index**: Reduced memory cost (compression ratio applied); access cost near zero; quality penalty modeled from sparsity.
- **recompute**: Zero memory cost; access cost equal to full prefill cost.

### 2.2 Simulation Environment

Experiments were conducted as CPU-local simulations, not GPU inference runs. The simulator (`src/context_capital_allocator.py`) models context access patterns, tier assignment, and cost accumulation without executing an actual language model. System memory was recorded via `/proc/meminfo` around each run: MemAvailable was approximately 122,557,616 KiB before and 122,557,620 KiB after the default run, with SwapTotal at 0 KiB throughout. These figures confirm the simulation ran in a no-swap environment with ample host RAM; they do not represent GPU memory behavior.

### 2.3 Workload Generation

The synthetic workload uses a Zipf-distributed access pattern with temporal drift, generating 600 distinct contexts and 9,000 requests per run. This models a scenario where some contexts are accessed frequently and others rarely, with popularity shifting over time. We acknowledge that this distribution may not match real agent/assistant reuse patterns.

### 2.4 Experimental Protocol

1. **Smoke test**: 30 contexts, 200 requests. Completed in 0.0022 seconds. Passed before proceeding to full runs.
2. **Default run**: 600 contexts, 9,000 requests, 14,000 MB GPU budget.
3. **Budget sweep**: 5 seeds × 6 GPU budget levels (512, 1,024, 2,048, 4,096, 8,192, 14,000 MB), with 600 contexts and 9,000 requests per seed.

### 2.5 Baselines

- **All-recompute**: Every request pays the full prefill cost. No caching.
- **LRU full-cache**: All contexts cached in fast memory with LRU eviction when budget is exceeded. Peak GPU usage approaches the budget limit.

## 3. Results

### 3.1 Default Run (14,000 MB GPU Budget)

| Metric | Value |
|---|---|
| All-recompute cost | 117,878.36 |
| Allocator cost | 81,642.37 |
| Cost reduction vs. recompute | 30.74% |
| Allocator quality loss | 0.34% |
| LRU full-cache cost | 48,011.38 |
| Cost reduction vs. LRU | −70.05% (worse) |
| Allocator peak GPU | 2,782.23 MB |
| LRU peak GPU | 13,999.98 MB |

The allocator satisfies the pre-registered success criterion (≥25% cost reduction, ≤2% quality loss) relative to all-recompute. However, it is substantially more expensive than LRU full-cache when GPU memory is ample, because LRU can retain hot full caches at low marginal cost while the allocator routes many contexts to attention_index tiers that incur quality risk without sufficient memory savings to justify the tradeoff. The allocator's peak GPU usage (2,782 MB) is far below the budget ceiling, indicating it leaves fast-memory capacity underutilized.

### 3.2 Budget Sweep

The five-seed budget sweep reveals a clear interaction between memory budget and relative performance:

| GPU Budget (MB) | Mean Reduction vs. Recompute | Mean Reduction vs. LRU | Max Quality Loss | Success Rate |
|---:|---:|---:|---:|---:|
| 512 | 29.67% | +22.25% | 0.037% | 100% |
| 1,024 | 30.13% | +16.07% | 0.037% | 100% |
| 2,048 | 30.21% | +4.49% | 0.037% | 100% |
| 4,096 | 30.26% | −14.01% | 0.107% | 100% |
| 8,192 | 33.99% | −34.83% | 0.508% | 100% |
| 14,000 | 34.00% | −66.10% | 0.505% | 100% |

All budget levels satisfy the success criterion against all-recompute. Against LRU, the allocator wins only at tight budgets (512–2,048 MB). At 4,096 MB and above, LRU full-cache is cheaper because it can retain enough hot contexts in fast memory to serve most requests without compression. The crossover occurs between 2,048 and 4,096 MB. The allocator's quality loss remains well below the 2% threshold at all budget levels, peaking at approximately 0.51%.

### 3.3 Offload Tier Selection

Across all experiments, the offload tier was never selected by the allocator. The cost model's transfer penalty made offload uniformly less attractive than either attention_index (lower memory, lower latency) or recompute (zero memory, known cost). This is a negative result: it indicates either that the transfer penalty parameterization is unrealistic, that the synthetic workload does not exercise the access patterns where offloading is beneficial, or both.

### 3.4 Memory Utilization

The allocator consistently uses far less fast memory than the LRU baseline. At the 14,000 MB budget, the allocator peaks at 2,782 MB versus LRU's 13,999.98 MB. This underutilization is a direct consequence of the allocator's preference for attention_index assignments: compressed indices consume little fast memory, but the cost model does not adequately reward filling available fast-memory capacity with full caches when doing so would reduce total cost.

## 4. Limitations

1. **Synthetic workload**: Access patterns are generated from a Zipf distribution with drift, not drawn from real agent/assistant request logs. Real workloads may exhibit different reuse distributions, burst patterns, or context-size distributions that change the relative advantage of each tier.

2. **Modeled, not measured, quality loss**: Quality degradation from the attention_index tier is estimated from sparsity/compression heuristics, not evaluated by an LLM on actual prompts. The 0.34–0.51% quality loss figures should be interpreted as model predictions, not empirical measurements of answer quality.

3. **No real inference stack**: Experiments are CPU simulations. Time-to-first-token (TTFT), throughput, and real memory behavior under GPU inference were not measured. The cost model abstracts away hardware-specific transfer costs, PCIe bandwidth, and GPU memory fragmentation.

4. **Offload tier unused**: The allocator never selected the offload action across any seed or budget level, indicating a cost-model calibration issue or a gap in the workload's coverage of offload-favorable scenarios. Real CPU/SSD bandwidth and transfer latency need measurement.

5. **Over-preference for attention_index at ample budgets**: When fast memory is sufficient for LRU to retain hot full caches, the allocator still routes some contexts to compressed indices, incurring unnecessary quality risk and failing to exploit cheap full-cache hits. A hybrid policy with an LRU/full-cache fallback for hot contexts would likely improve performance at higher budgets.

6. **Single-allocation assumption**: The current model assigns each context to exactly one tier. In practice, a context might benefit from simultaneous full-cache and index retention, or from tier migration as access patterns evolve.

7. **No comparison to other compression baselines**: The simulator compares against all-recompute and LRU full-cache but does not compare against other KV-cache compression methods (e.g., quantized cache, token pruning) that may occupy a middle ground.

## 6. Conclusion

The Context Capital Allocator demonstrates that a portfolio-style routing approach to reusable long-context KV cache can reduce modeled inference cost by approximately 30% relative to all-recompute while keeping modeled quality loss below 0.5%, satisfying the pre-registered success criterion. However, this result is conditional and narrow:

- The allocator outperforms LRU full-cache **only** under tight GPU memory constraints (512–2,048 MB), where compressed indices enable reuse that LRU cannot afford.
- When GPU memory is ample (≥4,096 MB), LRU full-cache is substantially cheaper, and the allocator's over-preference for compressed indices becomes a liability.
- The offload tier was never selected, leaving its viability unvalidated.
- All results depend on a synthetic workload and modeled quality loss, not on real inference measurements or LLM-evaluated answer quality.

The viable niche for this approach is memory-constrained context portfolio routing—scenarios where fast memory is scarce relative to the working set of reusable contexts and some quality-compression tradeoff is acceptable. It should not be presented as a universal replacement for full KV caching.

Next steps include: replaying real agent/assistant request logs to replace the synthetic Zipf/drift workload; integrating the allocator with an actual serving stack (vLLM/LMCache/HuggingFace cache modes) to measure TTFT, throughput, and real memory behavior; adding a hybrid policy that defaults hot contexts to full-cache when memory permits; calibrating the offload tier's transfer penalty against measured CPU/SSD bandwidth; and evaluating answer quality on shared-context tasks using an LLM evaluator.
