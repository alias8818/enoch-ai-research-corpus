## Abstract

We present a coverage-guided, mutation-based red-teaming microbenchmark applied to a real open-source repository. The target is `Pallets/Werkzeug`'s `safe_join` function at commit `cd8be56a8a3cd70ed359cfaeeee6905933b1e07a`, a widely-used path-safety primitive. A path-containment oracle tests whether any non-`None` return from `safe_join` escapes the trusted base directory after normalization. Under a budget of 5,000 generated candidate inputs per target–strategy pair, the current upstream implementation produced zero escaping-path findings with both coverage-guided and random strategies. To assess harness sensitivity, two injected guard-regression mutants were tested under identical conditions: a parent-traversal mutant (`mutant_no_parent_guard`) and an absolute-path mutant (`mutant_no_absolute_guard`). Coverage-guided mutation found 784 and 1,418 findings respectively (2,202 total), versus 126 and 201 for random (327 total), demonstrating that the harness is non-vacuous and that coverage guidance substantially outperforms random generation for detecting these regression classes. These results support the viability of the benchmark method but do not constitute a new vulnerability claim. The study is limited to a single target function on a Linux host with Python-level tracing.

## Introduction

Security-critical patches in widely-used libraries are regression-prone: a fix for one vulnerability class may inadvertently weaken a guard against another. Automated red-teaming of patched code against path-containment oracles offers a complementary assurance layer beyond upstream unit tests, but most published fuzzing and red-teaming work targets compiled binaries with native coverage instrumentation rather than Python library functions evaluated in-process.

This study asks: *can a coverage-guided input generator, operating on Python line-level trace feedback and semantic classifier signals, detect injected guard regressions in a real repository's security function, and does it outperform random generation under equal budgets?*

We select `Werkzeug.security.safe_join` as the target because it is a compact, well-tested, security-relevant function that enforces path containment under a trusted base directory. Rather than claiming a new vulnerability, we evaluate whether the benchmark harness is sensitive enough to detect known regression patterns when they are deliberately introduced, and whether it produces false positives against the current (presumed-correct) implementation.

The contributions are:

1. A concrete real-repo microbenchmark harness that combines Python line-level tracing with semantic classifier feedback for coverage-guided candidate generation.
2. Empirical evidence that the harness produces zero findings against the current `safe_join` implementation while detecting two classes of injected guard regressions with high yield.
3. Quantitative evidence that coverage guidance outperforms random generation by a factor of approximately 6.7× in total mutant findings under equal budgets.

## Method

### Target and Oracle

The target function is `werkzeug.security.safe_join` in the Pallets/Werkzeug repository (commit `cd8be56a8a3cd70ed359cfaeeee6905933b1e07a`, cloned 2026-04-26). The function accepts a trusted base directory and one or more path components and returns a joined path only if it remains under the base after normalization; otherwise it returns `None`.

The oracle is defined as follows: for any candidate input where `safe_join(BASE, candidate)` returns a non-`None` value, normalize the result to an absolute path and check containment under `BASE`. Any joined path that escapes `BASE` after normalization is recorded as a finding.

### Sensitivity Controls (Mutants)

Two local mutants simulate common patch regression patterns:

- **`mutant_no_parent_guard`**: The normalization-based `..` traversal guard is removed, allowing relative path traversal to escape the base.
- **`mutant_no_absolute_guard`**: The guard rejecting absolute path components is removed, allowing an absolute path to discard the trusted base entirely.

These mutants are not present in upstream code; they serve as positive controls to verify that the oracle and generator can detect known-weak variants.

### Input Generation Strategies

Two strategies are compared under equal budgets:

1. **Coverage-guided (CG)**: Candidates are generated by mutating seeds from a corpus. New candidates that increase the set of observed (line, classifier) pairs are retained in the corpus. The classifier assigns semantic categories to inputs based on structural properties (e.g., contains `..`, is absolute, contains URL encoding). Seed: `20260501`.
2. **Random**: Candidates are generated independently with no corpus feedback. Seed: `20260502`.

Both strategies operate within a budget of 5,000 candidates per target–strategy pair.

### Coverage Feedback

Coverage is measured at two levels:

- **Line-level**: Python `trace` events identify which source lines of `safe_join` are executed.
- **Semantic classifier**: Each candidate is classified into one of several categories based on structural properties of the input string. The union of (line, classifier) pairs constitutes the coverage signal.

This differs from native edge coverage or compiled sanitizer instrumentation; it is a coarser but portable signal suitable for in-process Python targets.

**Key observations:**

1. **Zero findings on current implementation.** Both strategies produced zero escaping-path findings against the unmodified `safe_join`, consistent with the upstream test results and the absence of a known vulnerability.

2. **Non-vacuous oracle.** The same oracle and generator detected both injected mutants. Coverage-guided generation found 784 parent-guard findings and 1,418 absolute-guard findings (2,202 total), versus 126 and 201 for random (327 total). The ratio of CG to random total findings is approximately 6.7×.

3. **Faster first finding.** Coverage-guided generation reached the first finding at candidate 3 (parent mutant) and candidate 5 (absolute mutant), compared to candidate 35 and candidate 17 for random.

4. **Corpus expansion.** The coverage-guided strategy retained 29 corpus entries (versus 1 for random, which never adds to the corpus) and exercised 25 unique classifiers (versus 15 for random) on the current target. This indicates that the classifier feedback drives semantic diversity even when no findings are produced.

5. **Resource usage.** Total wall time across all six runs was 15.72s (CPU 99%), with peak RSS of 44,500 KB and zero swaps. The benchmark is lightweight relative to the available memory.

## Limitations

1. **Single-target microbenchmark.** This study examines one function in one repository. Generalization to broader patch red-teaming across multiple repositories, CVE patches, or vulnerability classes is not established.

2. **Coarse coverage signal.** Python line-level tracing plus semantic classifiers is a weaker coverage signal than native edge coverage, basic-block coverage, or sanitizer-guided fuzzing. Small Python functions like `safe_join` offer limited branch diversity, which constrains the benefit of coverage guidance. The modest coverage signal is reflected in the identical unique trace line counts (9) across strategies for the mutants.

3. **Platform-specific behavior.** The benchmark ran on Linux. Werkzeug's `safe_join` includes Windows device-name hardening that is only exercised when `os.name` is `"nt"`. The upstream tests monkeypatch this behavior, but the red-team oracle did not deeply fuzz Windows-specific paths on a non-Windows host. Escaping-path behaviors specific to Windows filesystem semantics are therefore underrepresented.

4. **No new vulnerability.** The zero-finding result for the current implementation is negative evidence under the tested oracle and budget, not a proof of correctness. A different oracle, a larger budget, or a different input grammar could produce different results.

5. **Mutant design.** The two mutants represent specific, manually designed regression patterns. They do not sample the space of all possible regressions, and the high finding counts may reflect the particular simplicity of these regression classes rather than general harness power.

6. **In-process evaluation.** The harness calls `safe_join` directly within the Python process. It does not test integration-level behaviors such as web framework routing, symlink resolution on particular filesystems, or OS-level path normalization differences.

7. **Claim audit status.** The claim ledger for this draft contains no formally audit-approved claims. The limitations noted here, and the results reported, have not undergone independent human review. All findings should be read as prototype-level evidence from a single automated run.

## Conclusion

This microbenchmark demonstrates that a coverage-guided red-teaming harness operating on Python line-level and semantic classifier feedback can detect injected guard regressions in a real open-source security function, with coverage-guided generation outperforming random generation by approximately 6.7× in total findings under equal budgets. The zero-finding result against the current Werkzeug `safe_join` implementation provides bounded negative evidence—consistent with upstream tests—that no path-escape vulnerability exists within the tested input space, but does not constitute a correctness proof.

The primary scientific result is benchmark viability and sensitivity rather than vulnerability discovery. The harness is non-vacuous (it detects mutants), the coverage signal drives corpus diversity (25 vs. 15 classifiers), and the resource footprint is modest. However, the study is confined to a single target on a single platform with manually designed mutants. The 6.7× advantage of coverage guidance should be interpreted cautiously: it reflects performance on two specific, simple regression classes and may not generalize to more subtle vulnerabilities or larger codebases. Extending this approach to a multi-repo benchmark suite with automated mutant generation, native coverage instrumentation, and cross-platform evaluation remains future work.
