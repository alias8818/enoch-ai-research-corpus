## Abstract

We investigate whether a cached in-memory vector index can reduce query-time retrieval latency for a dense/category-aware retriever on the CUAD contract-understanding benchmark without degrading retrieval hit rate or downstream answer retention. In a 240-question benchmark over native long contracts, the cached index reduced p95 query retrieval latency from 126.7 ms to 28.7 ms (4.42×) and p50 latency from 39.5 ms to 15.2 ms (2.60×), while preserving retrieval hit (0.779) and entity-deduplicated answer retention at all tested context budgets exactly. The cached dense/category retriever also maintained its quality advantage over a lexical baseline (hit 0.688, 1024-token retention 0.533). Index construction for 186 unique contracts completed in 7.2 seconds. These results are limited to a single CPU-only environment, a 240-question sample, and one model configuration; external replication is needed before broader claims can be made.

## 1. Introduction

Dense retrieval methods for long legal contracts have demonstrated improved retrieval hit rates and downstream answer retention compared to lexical baselines. A prior project branch established that a dense/category-aware hybrid retriever improved CUAD hit rate from 0.688 to 0.779 and entity-deduplicated answer retention by up to 7.9 percentage points. However, the query-time cost of computing dense vectors for every contract window at retrieval time yielded p50 latency of 39.6 ms and p95 latency of 126.6 ms, which may be prohibitive in latency-sensitive serving settings.

This work tests whether precomputing and caching per-contract window vectors, token sets, and length normalizers in an in-memory index can substantially reduce query-time latency while preserving retrieval quality. We define a strict productionization threshold: the cached path must reduce p95 latency by at least 2× while keeping retrieval hit within 1 percentage point and entity-deduplicated answer retention within 1 percentage point of the uncached dense/category retriever. Failure to meet these simultaneous constraints would terminate the branch.

## 2. Method

### 2.1 Cached Vector Index Design

The implementation adds three components to the existing dense/category retriever:

- **`IndexedChunk`**: A data structure holding a precomputed dense vector, a hashed representation of that vector, a token set for lexical overlap computation, and a length normalizer for each contract window.
- **`CachedVectorIndex`**: An in-memory index mapping contract identifiers to their precomputed `IndexedChunk` sequences. The index is built once per contract before any queries are issued.
- **`retrieve_dense_category_cached`**: A retrieval function that, at query time, looks up precomputed chunks from the index rather than recomputing vectors, token sets, and normalizers. The hybrid dense/category scoring semantics are preserved identically to the uncached path; only the source of the per-chunk features changes.

### 2.2 Scoring Semantics

The cached retriever uses the same hybrid scoring as the uncached dense/category retriever: a combination of dense vector similarity and category-aware lexical overlap, weighted and normalized identically. Because the cached path reads precomputed features that are bit-for-bit equivalent to those the uncached path would compute on the fly, the ranking order for any given query is guaranteed to match. This equivalence is verified by a regression test (see Section 4).

### 2.3 Index Construction

At initialization, the index iterates over all unique contracts in the evaluation set, segments each contract into overlapping windows (140 tokens with 35-token overlap), computes the dense vector, token set, and length normalizer for each window, and stores them in memory. For the primary benchmark, 186 unique contracts were indexed in 7.217 seconds before any query-time retrieval began.

## 3. Experimental Setup

### 3.1 Dataset

We use the CUADv1 corpus, a benchmark for contract understanding consisting of long legal contracts with annotated answerable questions. The primary benchmark samples 240 answerable CUAD questions over native long contracts.

### 3.2 Windowing and Retrieval Parameters

- Window size: 140 tokens
- Overlap: 35 tokens
- Top-k: 16 retrieved windows per query
- Context budgets: 256, 384, 512, 768, 1024 tokens
- Retrievers compared: lexical, dense/category (uncached), dense/category (cached)
- Packing strategies: source-order and entity-deduplicated

### 3.3 Metrics

- **Retrieval hit**: Whether the ground-truth answer span appears in at least one retrieved window.
- **Entity-deduplicated answer retention**: Whether the answer is retained after entity-deduplicated packing at a given context budget.
- **Query retrieval latency**: Wall-clock time per query for the retrieval step only (excluding index construction), reported at p50 and p95.

### 3.4 Hardware and Environment

All benchmarks ran on a single CPU-only machine. GPU utilization remained at 0% throughout (confirmed via nvidia-smi snapshots), consistent with the CPU-local index design. Peak RSS was 756,820 KB. Wall time for the full benchmark (index build + retrieval + packing) was 82.52 seconds, with script-measured retrieval/packing wall time after index build of 41.339 seconds at 174.17 rows/second.

### 3.5 Verification

- All source files compiled without errors (`python3 -m compileall src tests`).
- Eight unit tests passed, including a cached-vs-uncached ranking equivalence regression test.

## 4. Results

### 4.1 Retrieval Quality

The cached dense/category retriever preserved retrieval quality exactly compared to the uncached dense/category retriever across all tested context budgets:

| Retriever | Retrieval Hit | 1024-token Retention |
|---|---|---|
| Lexical | 0.688 | 0.533 |
| Dense/Category (uncached) | 0.779 | 0.613 |
| Dense/Category (cached) | 0.779 | 0.613 |

Entity-deduplicated answer retention was identical at all five tested budgets (256, 384, 512, 768, 1024 tokens) because the ranking order matched exactly between cached and uncached paths.

### 4.2 Retrieval Latency

| Metric | Uncached | Cached | Speedup |
|---|---|---|---|
| p50 query latency | 39.505 ms | 15.194 ms | 2.60× |
| p95 query latency | 126.747 ms | 28.697 ms | 4.42× |

The cached path cleared the pre-specified productionization threshold (≥2× p95 reduction with ≤1 pp quality degradation). The p95 reduction of 4.42× substantially exceeded the 2× minimum.

### 4.3 Smoke Test

A preliminary 24-question smoke test confirmed the same pattern: cached dense/category matched uncached quality exactly (hit 0.708; 512-token retention 0.458) while reducing p95 latency from 99.750 ms to 22.964 ms.

### 4.4 Index Construction Cost

Building the cached index for 186 unique contracts required 7.217 seconds. This is a one-time cost amortized over all subsequent queries against the same contract set.

## 5. Limitations

1. **Sample size**: The primary benchmark uses 240 answerable CUAD questions. Whether these latency and quality results generalize to the full CUAD evaluation set or other contract corpora is unknown.

2. **Single hardware environment**: All measurements were collected on one CPU-only machine. Latency numbers are not portable across hardware; the relative speedup ratios may differ on different architectures, core counts, or memory configurations.

3. **Single model configuration**: Only one dense embedding model and one set of hybrid scoring weights was tested. The caching benefit depends on the relative cost of vector computation versus vector lookup; models with cheaper encoding steps may show smaller speedups.

4. **No concurrent query load**: Benchmarks measured single-query latency in isolation. Under concurrent request patterns, memory bandwidth contention and cache eviction behavior may alter the latency profile.

5. **Index freshness**: The cached index is built once per contract set. If contract content changes between index builds, the index must be reconstructed; incremental update strategies were not evaluated.

6. **Memory footprint**: Peak RSS of 756,820 KB for 186 contracts is modest, but scaling to much larger contract corpora was not tested. Memory growth characteristics with corpus size are not characterized.

7. **No GPU utilization**: The dense encoder in this configuration runs on CPU. Results may differ for GPU-accelerated encoders where the compute-vs-lookup cost ratio shifts.

8. **Exact quality equivalence is an artifact of deterministic precomputation**: The cached path matches uncached quality because it reads bit-for-bit identical features. This guarantee holds only when the precomputed features are computed with the same model, tokenization, and numerical precision as the uncached path would use. Floating-point nondeterminism across different hardware or batch sizes could, in principle, introduce ranking divergences not observed here.

## 7. Conclusion

A cached in-memory vector index reduced p95 query retrieval latency for a dense/category-aware CUAD retriever by 4.42× (126.7 ms → 28.7 ms) and p50 latency by 2.60× (39.5 ms → 15.2 ms) on a 240-question benchmark, with zero degradation in retrieval hit (0.779) or entity-deduplicated answer retention at any tested context budget. The productionization threshold of ≥2× p95 reduction with ≤1 pp quality loss was cleared. The cached retriever maintained its quality advantage over the lexical baseline (hit 0.688, 1024-token retention 0.533). Index construction for 186 contracts required 7.2 seconds.

These findings support the hypothesis that precomputing per-contract window features in an in-memory index is a viable productionization strategy for dense/category retrieval on CUAD. However, the results are bounded to the specific sample, hardware, and model configuration tested. External replication across different hardware, larger corpora, concurrent query loads, and alternative model configurations is necessary before drawing broader conclusions.
