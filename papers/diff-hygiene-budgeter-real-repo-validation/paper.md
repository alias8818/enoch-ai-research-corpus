## Abstract

We present a validation of a dependency-free diff hygiene budgeter designed to serve as a pre-submit guardrail for automated coding agents and human contributors. Unlike naive lines-of-code (LOC) gates, the budgeter scores diffs on multiple hygiene signals—file count, domain spread, generated/lockfile presence, hunk density, and weighted cost—and emits pass/warn/fail decisions. We validate against 40 sampled accepted commits from the public `pallets/click` repository and four synthetic negative-control diffs. The budgeter passed all five predefined acceptance criteria: it accepted 35 of 40 historical commits as pass or warn (12.5% fail rate, within the ≤30% threshold), correctly flagged a scattered 24-file sweep and a lockfile-coupled diff that LOC-only checks passed, and passed a small doc-only control. Mean per-commit runtime was 36.46 ms. These results support viability with caveats: historical accepted commits are proxy positives rather than human-adjudicated hygiene labels, negative controls are synthetic, and thresholds require per-repository calibration with an explicit override path for releases, merge commits, and generated artifact updates.

## Introduction

Automated coding agents and pre-submit hooks need lightweight checks to reject diffs that are noisy, scattered, or structurally risky before they enter a repository's review queue. A common baseline is a lines-of-changed-code (LOC) threshold: if a diff is small enough, it passes. However, LOC alone fails to capture several hygiene-relevant signals. A diff that touches 24 files across multiple domains with only 48 changed lines is arguably riskier than a diff that touches 2 files with 48 changed lines, yet both pass a LOC-only gate at the same threshold. Similarly, a diff that introduces a lockfile alongside a source change carries different risk than a pure source change of the same size.

We implemented a multi-signal diff hygiene budgeter and validated it against a real public repository to address two questions: (1) Does the budgeter add discriminating signal beyond a LOC-only gate? (2) Is the budgeter fast enough for interactive pre-submit use? This paper reports the design, validation method, results, and limitations.

## Method

### Tool Implementation

The budgeter (`tools/diff_hygiene_budgeter.py`) is a dependency-free Python CLI that reads a unified diff from stdin and emits a JSON decision. It computes the following metrics from the diff:

- **File count** and **directory count**
- **Domain classification** (e.g., `source`, `docs`, `config`, `test`) based on path heuristics
- **Generated or lockfile detection** via filename patterns (e.g., `package-lock.json`, `*.min.js`)
- **Hunk count** and **average hunks per file**
- **Changed lines** (added + deleted)
- **Weighted cost**: a composite score that penalizes cross-domain spread, generated-file inclusion, and hunk density beyond simple line counts
- **File entropy** (Shannon entropy over the file distribution)

Thresholds are configurable; defaults used in this validation were calibrated for agent/pre-submit hygiene rather than universal repository policy. The tool emits one of three decisions: `pass`, `warn`, or `fail`, along with a list of specific threshold violations.

### Naive Baseline

For comparison, a naive LOC-only gate was applied to the same commit set. The naive gate uses only `changed_lines` against a single threshold, emitting `pass`, `warn`, or `fail` identically.

### Validation Target

We cloned `https://github.com/pallets/click.git` at commit `fed9049f7a07550d560a91b30c5b0b3e17d54981` and sampled 40 accepted commits from its history. These serve as proxy positive controls: they are commits that were historically accepted into the repository, so a useful hygiene budgeter should pass or warn on most of them. We emphasize that historical acceptance is a proxy for hygiene, not a ground-truth label.

### Negative Controls

Four synthetic negative-control diffs were constructed and inserted into the real checkout:

1. **scattered_python_sweep**: 48 changed lines across 24 files in 3 directories and 2 domains. Designed to test whether the budgeter catches high-file-count, low-LOC sweeps.
2. **small_lockfile_plus_source**: 82 changed lines across 2 files, including one generated/lockfile (`package-lock.json`). Tests lockfile detection at small LOC.
3. **large_lockfile_plus_small_source**: 1,207 changed lines across 2 files, including one lockfile. Tests both LOC and lockfile detection at large scale.
4. **small_doc_only**: 10 changed lines across 5 files, all in the `docs` domain. Tests that the budgeter does not over-flag low-risk multi-file doc edits.

Negative-control files were added to the checkout using `git add -N` (intent-to-add) so that `git diff` would include them. This workaround was discovered after an initial validation run where untracked files were invisible to plain `git diff`.

### Acceptance Criteria

Five binary criteria were defined before the final validation run:

| Criterion | Description |
|---|---|
| `smoke_passes` | The budgeter's own smoke test passes. |
| `accepted_fail_rate_lte_30pct` | At most 30% of sampled accepted commits receive a `fail` decision. |
| `scattered_sweep_flagged` | The `scattered_python_sweep` control receives `fail`. |
| `small_lockfile_flagged_beyond_naive` | The `small_lockfile_plus_source` control receives `fail` from the budgeter but `pass` from the naive gate. |
| `large_lockfile_flagged` | The `large_lockfile_plus_small_source` control receives `fail`. |
| `doc_control_not_failed` | The `small_doc_only` control does not receive `fail`. |

## Results

### Acceptance Criteria

All five criteria passed:

| Criterion | Result |
|---|---|
| `smoke_passes` | PASS |
| `accepted_fail_rate_lte_30pct` | PASS (5/40 = 12.5%) |
| `scattered_sweep_flagged` | PASS |
| `small_lockfile_flagged_beyond_naive` | PASS |
| `large_lockfile_flagged` | PASS |
| `doc_control_not_failed` | PASS |

### Accepted-Commit Decisions

On the 40 sampled accepted commits from `pallets/click`:

| Decision | Budgeter | Naive LOC-only |
|---|---|---|
| pass | 18 | 39 |
| warn | 17 | 1 |
| fail | 5 | 0 |

The budgeter's 12.5% fail rate on accepted commits is within the 30% threshold. The naive gate passed 39 of 40 commits and warned on 1, reflecting its near-total insensitivity to structural hygiene signals. The budgeter's warn rate of 42.5% (17/40) indicates that it frequently identifies borderline cases without outright rejecting them, which may be an appropriate behavior for a guardrail that defers to human judgment.

### Accepted-Commit Failures

The 5 commits that received `fail` from the budgeter were:

| Commit | Description | Changed Lines | Files | Key Violations |
|---|---|---|---|---|
| `2d0490b12cee` | Merge branch 'main' into fix/help-option-shadowing-suggestion | 575 | 22 | files=22 > 18; domains=5 > 4; generated_or_lock=1 > 0 |
| `cf93868770f4` | Merge branch 'main' into flag-value-unprocessed-auto-detect | 488 | 22 | files=22 > 18; domains=5 > 4; generated_or_lock=1 > 0 |
| `5989375dc395` | ParamType typing improvements | 254 | 5 | avg_hunks_per_file=8.4 > 7 |
| `d777956105fd` | Release 8.3.3 (#3350) | 32 | 3 | generated_or_lock=1 > 0 |
| `c06d2d0a6aee` | Release 8.3.3 | 32 | 3 | generated_or_lock=1 > 0 |

These failures are plausible rather than random: two are merge commits with broad cross-domain spread, two are release commits that touch generated artifacts, and one is a large type refactor with high hunk density. A production deployment would need an explicit override path for these categories. It is also possible that some of these commits would be considered hygiene violations by a human reviewer, which the proxy-positive methodology cannot determine.

### Negative-Control Results

| Control | Budgeter | Naive | Changed Lines | Files | Generated/Lock | Key Violations |
|---|---|---|---|---|---|---|
| `scattered_python_sweep` | **fail** | pass | 48 | 24 | 0 | files=24 > 18 |
| `small_lockfile_plus_source` | **fail** | pass | 82 | 2 | 1 | generated_or_lock=1 > 0 |
| `large_lockfile_plus_small_source` | **fail** | fail | 1,207 | 2 | 1 | changed_lines > 1200; weighted_cost=3744 > 1400; generated_or_lock=1 > 0 |
| `small_doc_only` | pass | pass | 10 | 5 | 0 | (none) |

The first two controls demonstrate the budgeter's added discriminating signal: both passed the naive LOC-only gate but failed the budgeter. The `scattered_python_sweep` control changed only 48 lines—well within a typical LOC budget—yet touched 24 files, a pattern consistent with an agent making unfocused edits across the codebase. The `small_lockfile_plus_source` control changed only 82 lines but coupled a lockfile with a source change, a pattern that can introduce dependency surprises. The `large_lockfile_plus_small_source` control failed both gates, as expected given its 1,207-line size. The `small_doc_only` control passed both, confirming the budgeter is not simply hostile to multi-file edits.

### Runtime

Mean per-commit budgeter runtime: **36.46 ms**. Maximum: **47.54 ms**. Total wall time for the full validation run: **2.07 s**. These latencies are compatible with interactive pre-submit use. However, these measurements were taken on a single machine during a validation run against a small-to-medium repository; performance on larger repositories or under concurrent load was not evaluated.

## Limitations

1. **Proxy positive labels.** Historical accepted commits are used as positive controls, but acceptance into a repository does not necessarily imply the commit met a hygiene standard. Some accepted commits may themselves be hygiene violations that slipped through; the budgeter's 12.5% fail rate on accepted commits may partially reflect this rather than constituting pure false positives.

2. **Synthetic negative controls.** The negative-control diffs were constructed by the experimenter and inserted into a real checkout. They test specific hypothesized failure modes (scattered sweeps, lockfile coupling) but do not represent a random sample of real-world hygiene violations. The controls were designed to be caught, which introduces confirmation risk.

3. **Single-repository validation.** All results come from one repository (`pallets/click`). Thresholds and domain-classification heuristics may not transfer to repositories with different structures, languages, or contribution patterns. The generalizability of these results is unknown.

4. **Untracked-file handling.** The initial validation run revealed that plain `git diff` does not include untracked files. The runner was updated to use `git add -N` (intent-to-add) for negative-control files. A production wrapper must either score staged changes or explicitly include untracked files; this is an integration concern, not a tool bug, but it affects correctness if unaddressed.

5. **Threshold calibration.** Default thresholds were set for agent/pre-submit hygiene and are not universally appropriate. The 5 accepted-commit failures (merge commits, release commits, a large refactor) illustrate the need for per-repository configuration and an override mechanism with audit logging. The choice of the ≤30% fail-rate threshold for acceptance is itself a judgment call rather than a principled bound.

6. **No human adjudication.** No human reviewer classified commits as hygienic or unhygienic. The acceptance criteria use proxy measures (fail rate on historical commits, synthetic control outcomes) rather than ground-truth labels. This limits the strength of any claim about the budgeter's accuracy.

7. **Small sample size.** 40 accepted commits and 4 negative controls provide limited statistical power. Confidence intervals on the fail rate are wide, and the results should be interpreted as preliminary evidence of viability rather than a definitive evaluation.

## Conclusion

A dependency-free diff hygiene budgeter was implemented and validated against 40 sampled accepted commits and 4 synthetic negative controls in a real public repository. The budgeter passed all predefined acceptance criteria, demonstrating that multi-signal hygiene scoring (file count, domain spread, lockfile detection, hunk density, weighted cost) adds discriminating signal beyond LOC-only gates. In particular, the budgeter correctly flagged a 24-file scattered sweep and a lockfile-coupled diff that LOC-only checks passed, while allowing a small multi-file doc-only control. Per-commit runtime averaged 36.46 ms, suitable for interactive use.

The 12.5% fail rate on historical accepted commits—concentrated in merge commits, release commits, and a large refactor—indicates that the default thresholds require per-repository calibration and an explicit override path. The use of proxy positive labels and synthetic negative controls, rather than human-adjudicated ground truth, limits the strength of the viability claim. The single-repository, small-sample design further constrains generalizability. The next step is integration into an agent pre-submit hook with repository-specific threshold configuration, staged-diff scoring, and override logging to support threshold tuning from real usage data. Broader validation across multiple repositories and the collection of human-adjudicated hygiene labels would strengthen the evidence base.
