## Abstract

Speculative decoding accelerates autoregressive inference by drafting multiple tokens in parallel and verifying them against a target model, but rejected draft tokens represent wasted computation that must be discarded and rewound. When request domains have materially different draft-target acceptance rates, a single fixed draft length over-speculates on low-acceptance domains and under-speculates on high-acceptance ones. This paper evaluates a waste-minimizing control policy—selecting the smallest draft length *K* per domain whose calibrated throughput is within 97% of that domain's best *K*—using a deterministic synthetic simulator across five heterogeneous domains. In an 11-seed evaluation with 50,000 calibration and 250,000 test cycles per seed, the domain-gated policy reduced wasted draft tokens per output token by 63.5% compared to a fixed *K* = 8 baseline and by 41.0% compared to a global waste-minimizing *K*, while achieving 8.2% higher speedup than the global policy. An oracle-domain policy with perfect domain labels improved only marginally over the 90%-accurate classifier (speedup 2.305× vs. 2.274×), suggesting most value derives from coarse domain stratification rather than perfect labels. Sensitivity analysis shows the margin over global gating degrades gracefully with classifier accuracy but narrows at low accuracy, which is the primary implementation risk. These results are from a deterministic synthetic control-policy simulation and do not constitute a real-model throughput claim.

## Introduction

Speculative decoding reduces latency in autoregressive language model inference by having a small draft model propose *K* candidate tokens, which the target model verifies in parallel. When a draft token is rejected, that token and all subsequent draft tokens are discarded and the key-value cache is rewound. This rejected-token and rewind cost grows with the number of discarded tokens, creating a trade-off: larger *K* increases the opportunity for parallel verification but also increases waste when the draft diverges from the target distribution early.

Production deployments of speculative decoding typically use a single fixed *K* for all requests. However, different request domains—boilerplate text, code, mathematical reasoning, freeform prose, adversarial or noisy inputs—exhibit systematically different draft-target acceptance rates. A fixed *K* calibrated for one domain may over-draft on low-acceptance domains (incurring high waste) or under-draft on high-acceptance domains (leaving speedup unrealized).

This paper asks: can a runtime controller reduce speculative decoding waste by choosing draft length per request domain, rather than using one fixed draft length for all traffic? The proposed policy is a waste-minimizing scheduling decision: for each observed domain, select the smallest *K* whose calibrated throughput is within 97% of that domain's best *K*. It does not alter verifier acceptance logic or output sampling; it only changes how many tokens are drafted before normal target verification.

We evaluate this policy in a deterministic synthetic simulator modeling five domains with heterogeneous acceptance behavior, under a 90%-accurate domain classifier. We compare against no speculation, fixed *K* = 8, a global waste-minimizing *K*, and an oracle-domain policy with perfect labels. We also characterize sensitivity to classifier accuracy across five accuracy levels.

## Method

### Simulator Design

The simulator (`domain_gated_spec_waste_eval.py`) models speculative decoding as a per-cycle process. Each cycle represents one draft-verify step:

1. A domain is drawn from a fixed distribution over five domains: `boilerplate`, `code`, `prose`, `math`, and `noisy`.
2. The drafter proposes up to *K* tokens.
3. The verifier accepts a prefix of the draft. The acceptance length is drawn from a domain-specific distribution calibrated so that `boilerplate` has high acceptance (long useful drafts), `code` has high but less stable acceptance, `prose` has medium acceptance, `math` has lower acceptance, and `noisy` has low acceptance.
4. Output tokens per cycle equal `accepted + 1` (the first rejected token is replaced by the verifier's own token).
5. Speculative waste per cycle equals `K − accepted`, corresponding to the rejected-token and KV-cache-rewind work that a real engine must discard.

The domain classifier has configurable accuracy (default 90%). When the classifier misclassifies a request, the domain-gated policy selects *K* for the wrong domain, potentially resulting in a suboptimal draft length for that request.

### Calibration Phase

For each seed, a calibration phase sweeps candidate draft lengths and records per-domain throughput. The waste-minimizing policy selects, for each domain, the smallest *K* whose throughput is within 97% of the domain's maximum throughput. The global waste-minimizing policy selects a single *K* that maximizes throughput across all domains.

### Policies Compared

| Policy | Description |
|---|---|
| `no_spec` | No speculative drafting (baseline, *K* = 0). |
| `fixed_max` | Always draft *K* = 8. |
| `global_waste_min` | One calibrated *K* for all traffic, chosen to maximize throughput. |
| `domain_gated` | Calibrated *K* per observed domain (90% classifier accuracy). |
| `oracle_domain` | Calibrated *K* per true domain (perfect classifier). |

### Evaluation Protocol

The final evaluation uses 11 independent random seeds. Each seed runs 50,000 calibration cycles followed by 250,000 test cycles. We report mean speedup versus no speculation and mean waste per output token, each with 95% confidence interval half-widths computed across seeds.

Sensitivity to classifier accuracy is evaluated separately at accuracies of 0.45, 0.60, 0.75, 0.90, and 1.00, using 5 seeds, 15,000 calibration cycles, and 80,000 test cycles per seed.

### Computational Environment

All runs were CPU-only on an aarch64 host (Linux, GB10). The GPU remained idle at 0% utilization and approximately 11 W throughout. Maximum RSS was 275,120 kB. Swap was intentionally disabled (SwapTotal: 0 kB). Memory available before and after the final run was approximately 122.7 GB and 122.6 GB respectively, indicating no memory pressure. The simulator is deterministic given a seed and involves no GPU computation or nondeterministic library calls.

## Results

### Main Comparison

Table 1 reports the primary metrics from the 11-seed final evaluation.

**Table 1.** Speedup and waste per output token by policy (11 seeds, 250,000 test cycles each).

| Policy | Speedup vs. no speculation | 95% CI half-width | Waste / output token | 95% CI half-width |
|---|---:|---:|---:|---:|
| fixed_max | 2.093× | ±0.001 | 1.946 | ±0.002 |
| global_waste_min | 2.102× | ±0.001 | 1.205 | ±0.001 |
| domain_gated | 2.274× | ±0.001 | 0.711 | ±0.001 |
| oracle_domain | 2.305× | ±0.001 | 0.694 | ±0.001 |

The domain-gated policy achieves 108.6% of the fixed-max speedup—that is, it is faster than fixed *K* = 8 in this cost model, because fixed *K* = 8 over-drafts low-acceptance domains—while reducing waste by 63.5% relative to fixed *K* = 8. Compared to the global waste-minimizing policy, domain gating reduces waste by 41.0% and improves speedup by 8.2%.

The oracle-domain policy (perfect classifier) achieves only marginally better speedup (2.305× vs. 2.274×) and waste (0.694 vs. 0.711) than the 90%-accurate domain-gated policy. This suggests that most of the value comes from coarse domain stratification rather than perfect domain labels.

### Calibrated Draft-Length Plans

An example calibrated plan from seed 0 illustrates the per-domain adaptation:

- Global one-size-fits-all *K*: 5
- Domain-gated plan: `boilerplate` → 7, `code` → 5, `prose` → 3, `math` → 2, `noisy` → 1
- Oracle plan: identical to domain-gated plan for this seed

The domain-gated and oracle plans coincide in this seed because the 97% threshold selects the same *K* regardless of classifier accuracy when the throughput curve is steep enough near the optimum. In other seeds with noisier calibration, the plans may diverge.

### Sensitivity to Classifier Accuracy

Table 2 reports domain-gated policy performance as classifier accuracy varies.

**Table 2.** Domain-gated speedup and waste reduction vs. global policy, by classifier accuracy (5 seeds, 80,000 test cycles each).

| Classifier accuracy | Domain-gated speedup | Domain-gated waste/output | Waste reduction vs. global |
|---|---:|---:|---:|
| 0.45 | 2.171× | 0.938 | 21.7% |
| 0.60 | 2.179× | 0.863 | 25.3% |
| 0.75 | 2.224× | 0.740 | 38.7% |
| 0.90 | 2.276× | 0.710 | 40.9% |
| 1.00 | 2.312× | 0.690 | 42.4% |

The domain gate remains useful at degraded classifier accuracy, but the margin over global gating shrinks as labels become noisier. At 45% accuracy (barely above chance for five domains), waste reduction versus the global policy drops to 21.7%. This is the primary implementation risk: the value of domain gating depends on having a classifier that is at least moderately accurate.

## Limitations

1. **Synthetic acceptance model.** The simulator uses domain-specific acceptance distributions that are plausible but not derived from any particular target/draft model pair. Real acceptance distributions may have different shapes, heavier tails, or domain correlations not captured here. The absolute speedup and waste numbers should not be interpreted as predictions for any specific production system.

2. **No real-model validation.** This study evaluates a control policy in a deterministic synthetic simulator. It does not measure throughput on actual hardware with a real target/draft model pair. Production speculative decoding involves kernel launch overheads, KV-cache management costs, and memory bandwidth constraints that are abstracted away in this cost model.

3. **Fixed domain distribution.** The simulator draws domains from a fixed distribution. In production, domain mix may shift over time, requiring recalibration or online adaptation.

4. **No overhead accounting for domain classification.** The simulator assumes domain classification is available at negligible cost. In practice, a domain classifier adds latency and compute that must be weighed against the waste reduction gains.

5. **Backend constraints on dynamic draft lengths.** Some speculative decoding implementations (e.g., certain CUDA-graph one-model paths in TensorRT-LLM) may not support dynamic draft lengths per request, requiring fixed lengths or padding. The domain-gated policy assumes the backend can accept a per-request *K*.

6. **Single cost model.** The simulator equates waste with `K − accepted` tokens discarded. Real systems may have asymmetric costs (e.g., draft time may differ from verification time, and KV-cache rewind cost may depend on sequence length).

7. **No multi-turn or context-dependent effects.** Acceptance rates within a single request may change as generation progresses. The simulator treats each cycle independently.

8. **Claim audit status.** The automated claim ledger for this artifact contains no structured claims and is in a blocked state. The metrics and interpretations presented here are drawn directly from the run notes and decision JSON but have not passed a formal claim-evidence audit.

## Conclusion

A domain-gated draft-length controller that selects the smallest per-domain *K* within 97% of each domain's best throughput can substantially reduce speculative decoding waste in synthetic heterogeneous-acceptance settings. In an 11-seed simulation, the domain-gated policy reduced waste by 63.5% versus a fixed *K* = 8 and by 41.0% versus a global waste-minimizing *K*, while improving speedup by 8.2% over the global policy. The gap between 90%-accurate and perfect domain classification was small (speedup 2.274× vs. 2.305×), indicating that coarse domain stratification captures most of the benefit.

However, these results are from a deterministic synthetic simulator and do not constitute a real-model throughput claim. The margin over global gating narrows substantially as classifier accuracy degrades (from 42.4% waste reduction at perfect accuracy to 21.7% at 45% accuracy), which is the primary implementation risk. Scientific closure on production impact requires traces from an actual target/draft pair with per-request domain labels, accepted-prefix lengths, and kernel/runtime overheads.

The recommended next step is a telemetry-only pilot: log true and observed domain, `max_draft_len`, accepted prefix length, rejected/rewound draft tokens, target pass time, draft time, and output tokens from a production system; then replay the logs offline with the domain-gated policy. The controller should be promoted only if replay shows lower rewound draft tokens at equal or better tokens per second, and only if the selected backend supports dynamic draft lengths per request.
