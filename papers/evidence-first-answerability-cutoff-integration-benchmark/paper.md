## Abstract

We investigate an ordering-only integration strategy for retrieval-augmented generation (RAG) prompt construction, termed *evidence-first answerability cutoff*, which reorders retrieved chunks so that evidence-bearing passages precede metadata and context-only passages within a fixed token budget. Compared against a top-k filling baseline on a 144-case production-style benchmark spanning QA, summarization, and extraction tasks, the evidence-first packer achieves approximately 69.8% prompt-token savings and 14.4% mean latency reduction while preserving aggregate task quality (QA F1 0.8741 vs. 0.8722, summary faithfulness 1.000 vs. 1.000, extraction slot F1 1.000 vs. 1.000) under a Qwen2.5-7B-Instruct endpoint. Two of 144 individual QA cases show quality regression exceeding 0.05 F1, indicating localized risk. Deterministic extractive scoring over 1,296 rows confirms that the reordering preserves all evidence required for downstream answers. These results are bounded to a single model, a single machine, and a synthetic operations/RAG corpus; external replication is needed before broader claims.

## 1. Introduction

In retrieval-augmented generation systems, the prompt construction strategy—how retrieved passages are selected, ordered, and truncated to fit a token budget—directly affects both generation quality and inference cost. A common baseline, *top-k filling*, includes the k highest-scoring chunks in retrieval rank order until the budget is exhausted. This can allocate substantial budget to metadata-rich or context-only passages that carry little answer-bearing content, displacing evidence that downstream models need.

The *answerability cutoff* mechanism addresses this by classifying chunks as required, evidence, or context, then applying a budget-aware selection. The evidence-first variant investigated here is an ordering-only integration: it uses the same selected chunks as the standard answerability cutoff but presents required and evidence chunks before metadata and context chunks. The hypothesis is that this reordering preserves task quality while substantially reducing prompt token count, since downstream content that would have been included by top-k filling is deprioritized and often truncated.

This paper reports results from a structured benchmark designed with explicit kill conditions: the branch would be killed if evidence-first saved fewer than 20% prompt tokens versus top-k filling, or if it lost more than 5% relative QA F1, more than 5 percentage points of summary faithfulness, or more than 5% relative extraction slot F1. The evidence-first packer clears all kill thresholds, though two individual QA cases regress, warranting continued monitoring.

## 2. Method

### 2.1 Evidence-First Answerability Cutoff

The evidence-first packer (`answerability_cutoff_evidence_first`) is an ordering-only variant of the standard answerability cutoff. Given a set of retrieved chunks, each annotated with a classification (required, evidence, or context), the packer:

1. Applies the same chunk selection logic as the standard answerability cutoff.
2. Reorders the selected chunks so that required and evidence chunks appear before context and metadata chunks.
3. Fills the token budget in this reordered sequence.

No chunks are added or removed relative to the standard answerability cutoff; only the presentation order changes. The comparison baseline, `topk_fill`, includes chunks in raw retrieval rank order until the budget is exhausted.

### 2.2 Production-Style Benchmark Corpus

The benchmark corpus consists of 48 deterministic operations/RAG records, each exercised under three task families:

- **Risk/mitigation QA**: Question-answer pairs requiring identification of specific operational risks and their mitigations.
- **Grounded product summarization**: Summarization tasks requiring faithful condensation of source material.
- **Structured field extraction**: Slot-filling extraction of structured fields from operational records.

This yields 144 cases (48 records × 3 tasks). Each case mixes gold evidence chunks, metadata/source-note chunks, and cross-record distractors to exercise metadata-first failure modes without using oracle gold sorting.

### 2.3 Prompt Construction

The prompt builder emits chat-style operations assistant prompts with XML-like `<source>` blocks containing stable source IDs, rank, and retrieval score. This design mirrors production RAG prompt shapes.

### 2.4 Evaluation Protocol

Two evaluation phases were conducted:

**Phase 1: Deterministic extractive scoring.** Each case was evaluated at three token budgets (120, 180, 260 tokens) across three packers, yielding 1,296 rows. Scoring is deterministic: it measures whether the evidence required for the ground-truth answer is present in the packed prompt, without invoking an LLM.

**Phase 2: LLM endpoint generation and scoring.** A Qwen2.5-7B-Instruct model (Q4_K_M quantization) was served via llama.cpp server. A 12-case calibration subset was run first, followed by the full 144-case benchmark at budget 180. Generation quality was scored per task family: F1 for QA, faithfulness for summarization, and slot F1 for extraction.

## 4. Limitations

1. **Single model coverage.** All LLM endpoint results use Qwen2.5-7B-Instruct Q4_K_M. Behavior under other model families, sizes, or quantization levels is unknown.

2. **Single-machine, single-session evaluation.** The endpoint benchmark ran on one machine in one session. Variability across hardware, concurrent load, or repeated runs is not characterized.

3. **Synthetic corpus.** The 48-record operations/RAG corpus is synthetic and designed to exercise specific failure modes. Performance on naturally occurring production data or other domains is not established.

4. **Localized QA regressions.** Two of 144 QA cases regressed by more than 0.05 F1. The equal-or-better rate of 0.778 means 22.2% of individual cases showed some quality decrement (though the ≥20%-saving-within-5%-quality rate of 0.986 indicates nearly all cases remain within the quality tolerance). The root cause of these regressions has not been isolated; prompt wording effects are a plausible contributor.

5. **Budget sensitivity.** The endpoint benchmark was conducted at a single budget (180 tokens). Behavior at tighter or looser budgets may differ, though the extractive benchmark covers budgets of 120, 180, and 260.

6. **Deterministic extractive scoring is not LLM scoring.** Phase 1 confirms evidence preservation but does not predict LLM generation behavior. The two regressed QA cases illustrate this gap.

7. **No external replication.** These results have not been independently replicated by other researchers or on other infrastructure.

8. **Confidence assessment.** The project decision assigns confidence "medium" and evidence strength "strong," reflecting that while the results are internally consistent and clear kill thresholds, the coverage limitations above prevent a higher confidence rating.

## 6. Conclusion

The evidence-first answerability cutoff integration—an ordering-only strategy that presents evidence-bearing chunks before metadata and context chunks within a fixed token budget—achieves approximately 69.8% prompt-token savings and 14.4% mean latency reduction relative to top-k filling on a 144-case production-style benchmark, while preserving aggregate task quality across QA, summarization, and extraction under a Qwen2.5-7B-Instruct endpoint. Deterministic extractive scoring over 1,296 rows confirms complete evidence preservation.

Two individual QA cases regress by more than 0.05 F1, indicating that the reordering can occasionally harm specific prompts. These cases should be retained as regression fixtures in any production adoption, and their prompt wording should be inspected for failure-mode analysis.

The project decision recommends promoting the evidence-first ordering-only cutoff into the parent production path, with the two regressed QA cases retained as regression fixtures. This recommendation is bounded by the limitations enumerated above: single model, single machine, synthetic corpus, and no external replication. Broader validation across models, domains, and hardware is needed before generalizing these findings.
