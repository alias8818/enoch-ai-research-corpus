## Abstract

We present the Experiment Autopsy Agent, a dependency-free Python prototype that audits local experiment project directories for closure quality. The agent inventories project structures, parses decision metadata, classifies decision outcomes, verifies confidence and evidence labeling, checks local artifact reference integrity, counts logs and metrics, and computes a closure score. On a corpus of 1,009 sibling experiment projects managed by the Enoch control-plane system, the agent completed a full audit in 3.602 seconds, classifying 796 projects as closed with evidence and flagging 213 for review. Aggregate risk flags included 123 projects with missing confidence labels, 68 with weak or unlabeled evidence, and 51 with broken artifact references. The mean closure score was 97.854 (median 100, minimum 65, maximum 100). These results support the viability of automated closure auditing for review triage, but the approach is bounded in several respects: the agent evaluates structural and metadata completeness, not the scientific validity of the underlying experiments; heuristic artifact-reference detection and string-based decision classification introduce known precision/recall trade-offs that have not been quantified against a labeled ground truth; and 164 of 1,009 projects (16.3%) received an `unknown` decision classification, indicating the limits of the current pattern set. No independent ground-truth validation was performed.

## Introduction

Automated experiment management systems accumulate project directories containing decisions, logs, metrics, and artifacts. As the number of completed experiments grows, determining which projects have reached adequate closure—and which require human review—becomes a triage problem that manual auditing does not scale to address.

The Experiment Autopsy Agent addresses this gap by performing a structured, local audit of each project directory. Rather than evaluating the scientific truth of an experiment's conclusions, the agent checks whether the structural prerequisites for closure are met: a final decision is recorded, confidence and evidence labels are present, referenced artifacts exist on the local filesystem, and logs or metrics are available.

This paper describes the agent's design, implementation as a dependency-free Python prototype, and evaluation on a corpus of 1,009 local experiment projects managed by the Enoch control-plane system. We report both positive findings (fast, actionable triage) and limitations (heuristic fragility, inability to validate domain science, a non-trivial `unknown` classification rate, and the absence of ground-truth validation).

## Method

### Agent Design

The Experiment Autopsy Agent (`src/autopsy_agent.py`) is a single-file, dependency-free Python module. Its audit pipeline for each project directory consists of the following steps:

1. **Directory inventory.** Walk the project tree and catalog files.
2. **Decision parsing.** Read `.omx/project_decision.json` if present, extracting the `project_decision`, `confidence`, `evidence_strength`, `hypothesis_status`, and `artifacts` fields.
3. **Decision classification.** Map the `project_decision` and `hypothesis_status` string values to one of four classes: `positive`, `negative`, `needs_review`, or `unknown`. Classification is string-based, matching against known decision-value patterns.
4. **Confidence and evidence checks.** Flag projects where `confidence` is absent or `evidence_strength` is unlabeled or weak.
5. **Artifact reference verification.** For each entry in the `artifacts` map, check whether the referenced path exists relative to the project root. Flag missing references.
6. **Log and metrics counting.** Count files under `logs/` and `artifacts/autopsy/` (or equivalent metric directories) to assess observational richness.
7. **Closure score computation.** Compute a numeric score from 0 to 100 based on the presence and quality of the above signals. Assign a verdict of `closed_with_evidence` or `review_recommended` based on a threshold.
8. **Report emission.** Output structured JSON and ranked Markdown reports.

### Corpus Evaluation

A corpus runner script (`scripts/run_corpus_autopsy.py`) iterates over all sibling Enoch project directories under a shared parent, invoking the agent on each. Aggregate metrics are collected into `artifacts/autopsy/corpus_metrics.json` and a summary file (`artifacts/autopsy/corpus_summary.txt`).

### Testing

Unit tests in `tests/test_autopsy_agent.py` cover core agent logic. Three test cases were maintained throughout development, covering decision classification, artifact verification, and score computation. During development, two iterations were required to reduce path false positives and classifier false positives, as documented in the run notes. These iterations are recorded in `logs/unit_tests_after_optimization.log` and `logs/unit_tests_after_heuristic.log`, with the final test run logged in `logs/unit_tests_classifier_final.log`.

## Results

Test logs are recorded in `logs/unit_tests_final.log` and `logs/unit_tests_classifier_final.log`. ### Self-Autopsy

Before project finalization, the agent was run on its own project directory. The pre-finalization self-autopsy scored 25/100 with a verdict of `insufficient_closure`. This was expected: at that point, final artifacts and decision metadata had not yet been written. This single data point confirms that the agent can identify an incomplete project, including its own, but it does not constitute rigorous validation of the scoring model.

The high mean and median scores indicate that most projects in this corpus have strong structural closure. However, 213 projects (21.1%) were flagged for review, and 51 projects contained artifact references pointing to files not found on disk, indicating potential data loss or incomplete artifact writing.

The 164 projects classified as `unknown` (16.3% of the corpus) reflect decision values that did not match the agent's string-based classification patterns. This is a non-trivial fraction and suggests either non-standard decision labels or projects in intermediate states. The `unknown` class represents a gap in the classifier's coverage rather than a meaningful scientific category.

The score distribution is heavily right-skewed, with the median at the ceiling of 100. This ceiling effect limits the discriminative power of the score among well-closed projects and suggests that finer-grained scoring dimensions may be needed to differentiate among the 796 `closed_with_evidence` projects.

## Limitations

1. **Heuristic artifact detection.** The agent uses heuristic path detection to verify artifact references. Unconventional reference formats or prose/path mixtures in artifact fields may produce false positives (flagging a valid reference as missing) or false negatives (failing to detect a broken reference). During development, two iterations were needed to reduce path false positives and classifier false positives, as recorded in the run notes. The residual error rate of these heuristics has not been quantified with a labeled ground-truth dataset.

2. **String-based decision classification.** Decision classification relies on matching known string values in `project_decision.json`. If the Enoch system standardizes a canonical decision schema, the classifier should be updated accordingly. Currently, 164 of 1,009 projects (16.3%) fell into the `unknown` class, indicating the limits of the current pattern set. The classifier's precision and recall on each class have not been independently measured.

3. **No domain-science validation.** The agent assesses structural and metadata completeness. It cannot determine whether an experiment's conclusions are scientifically correct, whether its methodology was sound, or whether its evidence supports its stated hypothesis. A project can score 100/100 and still be scientifically wrong. The agent is a closure-quality auditor, not a scientific validator.

4. **Local-only scope.** The agent operates on local filesystem artifacts. Remote or database-stored evidence (e.g., Notion pages) is not audited. In this run, the Notion page associated with the project was not used as authoritative content; local prompt/project metadata were sufficient for the task.

5. **Corpus homogeneity.** The evaluation corpus consists entirely of Enoch-managed projects sharing a common directory structure and metadata schema. Performance on projects with different conventions, different metadata schemas, or different artifact organizations is unknown. Generalization beyond this corpus is not established.

6. **Threshold sensitivity.** The closure score threshold separating `closed_with_evidence` from `review_recommended` was set during development. Sensitivity analysis of this threshold was not performed, and the optimal cutoff may vary by use case. The ceiling effect in the score distribution (median = 100) further complicates threshold selection and limits the score's discriminative utility among well-closed projects.

7. **No independent ground truth.** The corpus audit results have not been validated against a human-labeled gold standard. The verdicts and risk flags are outputs of the agent's heuristics, not independently verified assessments of project quality. Without ground truth, the agent's precision and recall for any verdict or flag category are unknown.

8. **Narrow test coverage.** The unit test suite comprises three test cases. While these cover core logic paths, they do not exhaustively exercise edge cases in artifact detection, decision classification, or score computation. Broader test coverage would be needed before production deployment.

9. **Prototype status.** This implementation is a dependency-free Python prototype evaluated in a single-session autonomous run. It has not undergone integration testing, stress testing, or deployment in a production finalization pipeline. The corpus runtime of approximately 3.6 seconds for 1,009 projects is a prototype benchmark, not a production performance characterization.

## Conclusion

The Experiment Autopsy Agent demonstrates that automated, local closure auditing of experiment project directories is feasible and fast at the prototype level. On a corpus of 1,009 projects, it completed a full audit in under 4 seconds and produced actionable triage: 796 projects classified as closed with evidence, 213 flagged for review. Risk flags identified specific structural deficiencies (missing confidence labels, weak evidence, broken artifact references) that would otherwise require manual inspection to detect.

The result is positive but bounded. The agent is a closure-quality auditor, not a scientific validator. Its heuristic artifact detection and string-based classification introduce known imprecision whose magnitude has not been quantified. The 16.3% `unknown` classification rate indicates incomplete pattern coverage. The ceiling effect in the score distribution (median = 100) limits discriminative power among well-closed projects. No independent ground truth was available to validate the agent's verdicts, and the unit test suite is narrow.

Integration as a pre-close quality gate in automated experiment finalization pipelines is the most natural next step, supplemented by canonical decision-schema support to reduce classification ambiguity. A labeled ground-truth dataset would be required to rigorously evaluate the agent's precision and recall, which this study does not provide.
