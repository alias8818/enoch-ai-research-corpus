## Abstract

We investigate whether a lightweight personalization signal derived from each author's historical per-file line deltas can improve next-commit changed-file prediction relative to a non-personalized repository-wide file frequency and recency baseline. Using chronological train/validation/test splits across four Git repositories (258 pooled held-out test events), author-personalized features improved pooled MRR by +0.067 (95% bootstrap CI [0.030, 0.104]), hit@5 by +0.050 (95% CI [0.012, 0.089]), and recall@5 by +0.057 (95% CI [0.029, 0.087]) over the baseline. However, results are not uniform across repositories: one of four repositories exhibited MRR regression (−0.034) despite concurrent gains in hit@5 and recall@5, indicating that personalization can reorder rankings in ways that harm first-hit position even when top-5 retrieval improves. We conclude that file-delta personalization is a viable blended ranking feature but should not replace global recency/frequency signals without guardrails such as minimum author-history thresholds and validation-monitored fallback.

## Introduction

Coding assistants that suggest or prefetch context for a developer's next edit benefit from predicting which files that developer is likely to change. A natural baseline ranks files by repository-wide change frequency and recency, but this ignores the fact that different authors have distinct editing patterns. If one developer frequently edits configuration files while another focuses on test suites, a personalized signal could improve file selection for context windows or prefetch sets.

This study operationalizes personalization through *file-delta features*: per-author, per-file statistics computed from Git line-level additions and deletions. We ask whether incorporating these signals into a file-ranking function improves prediction of the set of files changed in a developer's next commit, compared to a global frequency/recency baseline alone.

The prediction task is framed as ranking: given a repository and an author at a point in time, rank all previously seen files by estimated likelihood of being changed in the next commit. We evaluate using hit@5, recall@5, and MRR on held-out chronological test events.

## Method

### Data

We used four Git repositories with locally cloned full-depth histories:

| Repository | Total events | Test events |
|---|---:|---:|
| click | 303 | 61 |
| dflash-local | 61 | 13 |
| packaging | 500 | 100 |
| python-dotenv | 418 | 84 |

An initial attempt to use blobless clones was abandoned because `git numstat` performance on blobless/no-checkout working copies was unacceptably slow; full clones were used instead.

### Event definition

Each commit after a warmup period constitutes one event. The target for each event is the set of files changed by that commit. Only files previously seen in the repository's history up to that commit are candidates for ranking.

### Baseline

The non-personalized baseline ranks previously seen files by a combination of repository-wide change count and exponential recency decay, controlled by a time-decay parameter τ.

### Personalized variant

The personalized ranking adds three author-specific features per file:

1. **Author frequency** (`alpha_author_freq`): the number of times the current author has edited the file.
2. **Author churn** (`beta_author_churn`): `log1p(added + deleted)` line counts for the current author on that file.
3. **Signed net delta** (`gamma_signed_delta`): signed sum of (added − deleted) lines by the current author on that file, allowing the model to capture whether an author tends to grow or shrink a file.

These features are combined additively with the global frequency/recency score. All hyperparameters (feature weights and τ) are selected per-repository on the validation set only.

### Train/validation/test split

Events are split chronologically within each repository: 60% train, 20% validation, 20% test. Hyperparameters are tuned on the validation split; all reported metrics are on the held-out test split.

### Metrics

- **hit@5**: whether any actually changed file appears in the top-5 ranked files.
- **recall@5**: fraction of actually changed files appearing in the top-5.
- **MRR**: mean reciprocal rank of the first actually changed file in the ranked list.

### Uncertainty estimation

We report paired bootstrap 95% confidence intervals over held-out test events, resampling events with replacement (10,000 iterations). The reported deltas are personalized metric minus baseline metric on the same resampled events.

### Computational cost

The analysis is CPU-only and lightweight. The main benchmark completed in 3.50 s wall clock with 21,276 KB maximum RSS; the bootstrap analysis took 2.84 s with 21,260 KB maximum RSS. Zero swaps occurred. A smoke test on the local dflash repository (39 events, 8 held-out test events) completed in 0.07 s with 16,084 KB maximum RSS and ~121.65 GB available memory remaining. No GPU or long-run stability testing was required.

## Results

### Pooled results

Across 258 pooled held-out test events, personalization improved all three primary metrics:

| Metric | Δ (personalized − baseline) | 95% bootstrap CI |
|---|---:|---:|
| MRR | +0.067 | [0.030, 0.104] |
| hit@5 | +0.050 | [0.012, 0.089] |
| recall@5 | +0.057 | [0.029, 0.087] |

All three confidence intervals exclude zero, supporting the conclusion that personalization provides a positive pooled effect.

### Per-repository results

| Repository | ΔMRR | Δhit@5 | Δrecall@5 | Selected hyperparameters |
|---|---:|---:|---:|---|
| click | −0.034 | +0.016 | +0.028 | α=2.0, β=0.1, γ=−0.02, τ=5 |
| dflash-local | +0.008 | +0.077 | +0.077 | α=0.0, β=0.02, γ=0.02, τ=5 |
| packaging | +0.141 | +0.080 | +0.095 | α=1.0, β=0.0, γ=−0.02, τ=5 |
| python-dotenv | +0.061 | +0.036 | +0.030 | α=2.0, β=0.02, γ=−0.02, τ=5 |

Three of four repositories showed positive MRR deltas; all four showed positive hit@5 and recall@5 deltas. The `click` repository is a notable counterexample: while hit@5 and recall@5 improved, MRR decreased by −0.034, meaning that personalization moved the first relevant result to a lower rank on average even as it pulled more relevant files into the top five.

### Hyperparameter patterns

The selected `gamma_signed_delta` values were consistently small (±0.02), suggesting that signed net deltas provide a subtle signal. The `alpha_author_freq` weight varied from 0.0 (dflash-local, where author frequency was not helpful) to 2.0 (click, python-dotenv). The `beta_author_churn` weight was small across all repositories (0.0–0.1), indicating that churn magnitude is a weaker signal than raw author frequency. The recency parameter τ was fixed at 5 across all repositories.

## Limitations

1. **Proxy task, not end-to-end evaluation.** The benchmark measures next-changed-file prediction accuracy, not the quality of LLM-generated code or coding-agent task success. A production deployment would require an A/B test over actual coding tasks to confirm that improved file prediction translates to improved downstream outcomes.

2. **Small and heterogeneous sample.** Four repositories with 258 total test events provide limited statistical power. Repository characteristics (size, author count, commit frequency) vary substantially, and the results may not generalize to repositories with different development patterns.

3. **MRR regression risk.** The `click` repository demonstrates that personalization can harm first-hit ranking even when top-5 retrieval improves. Using personalization as the sole ranking criterion carries risk; it is safer as a blended feature.

4. **Overfitting on sparse histories.** Early smoke testing on a very small repository showed that validation-selected signed-delta personalization can overfit and reduce held-out performance when author histories are sparse. This motivates minimum author-history thresholds before applying personalization.

5. **No private user data.** Only public or locally available Git histories were used. The method does not require private personalization data, but this also means it was not evaluated on proprietary repositories with potentially different collaboration patterns.

6. **Static hyperparameter selection.** Hyperparameters were selected once per repository on the validation split. In a production setting, ongoing adaptation or online learning might yield different results.

7. **Claim audit pending.** The claim ledger for this draft contains no formally audit-approved claims. The limitations noted in the ledger ("Model-authored draft; human claim audit required") flag that the findings have not undergone independent verification.

8. **Missing readiness audit signal.** The paper review pipeline flagged a missing readiness audit signal, indicating that the automated review process did not fully complete all verification stages.

## Conclusion

Author-personalized per-file delta history provides a measurable improvement in next-changed-file prediction over a global frequency/recency baseline, with pooled MRR improving by +0.067 (95% CI [0.030, 0.104]) across 258 held-out test events. However, the effect is not uniform: one repository exhibited MRR regression despite top-5 retrieval gains, and sparse author histories carry overfitting risk.

The appropriate role for file-delta personalization is as one feature within a blended context ranker, not as a standalone replacement for global recency and frequency signals. Guardrails are warranted: minimum author-history thresholds before enabling personalization, validation-monitored fallback to the baseline when personalization yields non-positive validation deltas, and capping personalization feature weights to limit ranking distortion. End-to-end evaluation in a live coding-agent context is necessary before making production claims about downstream task improvement.
