## Abstract

We present a bounded, locally executable reproduction harness for the core mechanism of *Expert Upcycling: Shifting the Compute-Efficient Frontier of Mixture-of-Experts* (arXiv:2604.19835): expanding a trained Mixture-of-Experts (MoE) model from $E$ experts to $mE$ experts by duplicating existing experts and extending the router, while keeping the Top-$K$ active-expert count fixed, then continuing training. The harness implements a tiny Top-1 MoE token classifier on a deterministic synthetic task, compares three expansion strategies (uniform duplication, gradient-utility-guided duplication, and random-new initialization), and records throughput and memory telemetry on an NVIDIA GB10 GPU. Both smoke and calibration runs confirm that warm-started duplicated experts preserve the parent function and converge substantially faster than randomly initialized new experts. Gradient-utility duplication slightly outperformed uniform duplication in both runs, but the gap is small and under-seeded, so we do not generalize it. This work validates the harness and the warm-start mechanism at toy scale; it does **not** reproduce the paper's 7B–13B validation-loss results or its claimed ~32% GPU-hour savings.

## Introduction

Mixture-of-Experts (MoE) architectures improve model capacity by conditionally activating subsets of expert parameters per token. A recent line of work investigates whether a trained MoE with $E$ experts can be efficiently expanded to $mE$ experts—thereby increasing model capacity—without training a larger model from scratch. The *Expert Upcycling* proposal (arXiv:2604.19835) introduces a mechanism: duplicate existing experts to fill the expanded expert pool, extend the router's gating matrix with copies of the original routing weights, keep Top-$K$ fixed, and continue pretraining. The paper reports that this warm-started expansion saves approximately 32% GPU hours compared to training an $mE$-expert model from scratch at the 7B–13B parameter scale, and further proposes that per-expert gradient norms can guide non-uniform duplication.

Reproducing these claims at full scale requires source model checkpoints, large training corpora, and substantial compute budgets. Our goal is narrower: build a concrete, locally runnable harness that tests the *core mechanism*—that expert duplication plus router extension preserves warm initialization and outperforms random initialization—at a scale that fits on a single NVIDIA GB10 GPU. We treat this as a bounded mechanism reproduction, not a full paper reproduction.

Specifically, we ask:

1. Does warm expert duplication preserve the parent model's function closely enough that continued training starts near the parent's loss?
2. Does warm duplication outperform randomly initializing new experts?
3. Does gradient-utility-guided duplication offer measurable advantages over uniform duplication at this scale?
4. What are the throughput and memory characteristics of this workload on GB10?

## Method

### Harness Design

The harness (`expert_upcycling_harness.py`) implements a tiny Top-1 MoE token classifier with the following pipeline:

1. **Base training.** Train a base MoE with $E = 4$ experts on a deterministic synthetic token-classification task until convergence.
2. **Utility computation.** Compute per-expert utility as the squared gradient norm over that expert's parameters.
3. **Expansion to $2E = 8$ experts** using three candidate strategies:
   - **Uniform:** Warm-copy every original expert once (each old expert produces one duplicate).
   - **Utility:** Warm-copy high-gradient-norm experts first, using the computed utility ranking to decide which experts get duplicated.
   - **Random-new:** Copy router rows for the new expert slots but leave the new expert weights randomly initialized (cold start).
4. **Continued pretraining (CPT).** Continue training the expanded model with the same Top-$K$ ($K = 1$) and a small loss-free routing-offset balance heuristic.
5. **Measurement.** Record validation loss, accuracy, expert counts, throughput (tokens/s), CUDA peak memory allocation, and system `MemAvailable` from `/proc/meminfo`.

### Environment

All runs were executed inside a Docker container (`nvcr.io/nvidia/vllm:26.03-py3`) with PyTorch 2.11.0 and CUDA available, running on an NVIDIA GB10 GPU. The host system is an aarch64 Linux machine with ~122 GB available system memory and swap intentionally disabled. GPU utilization was sampled externally at 1 Hz via `nvidia-smi`.

### Run Modes

Two run modes were executed:

- **Smoke:** A short validation that the harness runs correctly on GB10 and produces sensible metrics.
- **Calibrate:** A longer run with more training steps, intended to produce more stable loss values and throughput measurements.

The smoke run confirms that the harness executes on GB10 without errors. Warm-started (uniform and utility) expanded models converge to near-zero loss, while the random-new model remains near 0.54—roughly where an untrained model would be on this task. The maximum absolute loss increase immediately after expansion (relative to the base model) is approximately $4.9 \times 10^{-4}$, indicating that duplication preserves the parent function with only minor perturbation.

The calibration run, with more training steps, yields a more converged base model (loss $\approx 2.1 \times 10^{-6}$) and confirms the same pattern. The initial perturbation from expansion is extremely small ($3.87 \times 10^{-8}$). Both warm-start strategies converge well below the base loss after CPT, while the random-new strategy barely improves beyond the base loss—its final loss ($2.64 \times 10^{-6}$) is slightly above the base loss, suggesting it has not yet begun to benefit from the expanded capacity within the allotted CPT budget.

Utility duplication achieves the lowest final loss in both runs, but the advantage over uniform duplication is modest (a factor of ~1.7× in the calibration run). Given that this is a single deterministic task with a single seed, we do not consider this difference statistically meaningful.

The workload is far from memory-bound on this hardware. CUDA peak allocation increases from ~120 MB (base) to ~134 MB (expanded), a modest ~12% increase consistent with the doubling of expert parameters. System memory availability remains stable throughout, and swap is irrelevant (disabled). The low power draw (~28 W) and moderate utilization reflect the tiny model size relative to GB10's capacity.

Expanded-model throughput (~1.63M tokens/s) is higher than base throughput (~1.22M tokens/s). The throughput similarity across all three expansion strategies confirms that the expansion method does not materially affect inference speed.

## Limitations

This work has several important limitations that constrain the generality of its conclusions:

1. **Toy task and model scale.** The harness uses a synthetic token-classification MoE with 4 experts and small hidden dimensions. This is orders of magnitude smaller than the 7B–13B models studied in the source paper. Mechanisms that hold at toy scale may not transfer directly to large-scale pretraining.

2. **No full-scale reproduction.** We did not reproduce the paper's validation-loss curves, benchmark accuracies, or the claimed ~32% GPU-hour savings. Those claims require source model checkpoints, real training data, and substantially larger compute budgets.

3. **Single seed and deterministic task.** All results come from a single run on a deterministic synthetic task. The utility-over-uniform advantage, while consistent across smoke and calibration runs, is based on one seed and one task. No confidence intervals or statistical significance tests are reported.

4. **No from-scratch $2E$ baseline.** The harness compares warm-started expansion against random-new initialization of the added experts, but it does not compare against training a full $2E$-expert model from scratch. The source paper's GPU-hour savings claim requires this comparison, which we did not implement.

5. **Utility metric simplicity.** Per-expert utility is implemented as the squared gradient norm over expert parameters. The source paper may use a more sophisticated utility measure; our implementation is a straightforward approximation tested only on this one task.

6. **GPU sampling resolution.** External GPU utilization was sampled at 1 Hz via `nvidia-smi`. Short sub-second computation phases can be under-sampled, and the reported utilization and power figures should be treated as coarse estimates.

7. **Routing-offset heuristic.** The harness uses a small loss-free routing-offset balance heuristic during CPT. This is a simplification whose interaction with the expansion mechanism at larger scale is unknown.

## Conclusion

We built and executed a bounded reproduction harness for the core expert-upcycling mechanism on an NVIDIA GB10 GPU. At toy scale, the results are consistent with the source paper's central mechanism: duplicating trained experts and extending the router preserves the parent model's function with negligible immediate loss increase, and continued training from this warm start converges substantially faster than from random initialization of new experts. Gradient-utility-guided duplication showed a small but consistent advantage over uniform duplication across both run modes, though the evidence is insufficient to support a general scientific claim about its superiority.

These findings validate the harness as a functional testbed but do not constitute reproduction of the source paper's scale-dependent claims about validation-loss improvement or GPU-hour savings. Closing those claims would require real MoE language-model checkpoints, training data, multi-seed comparisons, a from-scratch $2E$ baseline, and sufficient training budget to measure gap closure. The recommended next step is to scale this harness to a tiny causal language modeling task with multiple seeds and a from-scratch $2E$ baseline before attempting larger-scale claims.
