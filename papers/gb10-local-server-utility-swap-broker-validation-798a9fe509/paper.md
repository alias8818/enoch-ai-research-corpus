### Key run-note evidence

- ## Pressure calibration and multiseed live replay (2026-04-19)
- - Started `llama-server` from `/tmp` (not the project directory) on `127.0.0.1:8011` using cached Phi-4-mini Q4_K_M weights and verified `/v1/models` before replay.
- - Calibration pass 1 (`artifacts/pressure_calibration/s24_c6_cap350_scale0.12`) used `-np 6 -c 4096`, 24 sessions, concurrency 6, capacity 350, token scale 0.12. It produced some HTTP 400 context-limit errors but showed pressure separation: request p95 -15.0% for utility vs FIFO, session p95 -3.7%, churn -8.9%, throughput -7.3% (utility lower throughput).
- - Calibration pass 2 (`artifacts/pressure_calibration/s36_c6_cap220_scale0.14`) increased pressure to 36 sessions, capacity 220, token scale 0.14. It also hit context-limit errors under `n_ctx_seq=768`; utility improved session p95 by 10.3% and request p95 by 8.0%, with nearly flat throughput. These calibration runs were retained as tuning evidence but not used as the clean decision run because of request errors.
- - Restarted `llama-server` with `-np 4 -c 8192` (`n_ctx_seq≈2048`) to remove context-limit errors while preserving pressure, then ran three seeds at 36 sessions, concurrency 4, capacity 220, token scale 0.14. Artifacts are under `artifacts/pressure_multiseed/s36_c4_cap220_scale0.14_multiseed/`.
- - Clean multiseed aggregate: utility broker reduced session p95 latency by mean 16.8% (median 18.9%, min 11.9%, max 19.5%) with 0 request errors for both policies. Request p95 reduction was mixed but positive on average: mean 17.5%, median 21.3%, min -3.2%, max 34.4%. Scheduler churn improved by mean 5.3%; throughput was essentially flat at mean +0.7% with per-seed range -1.6% to +4.4%. Deadline misses stayed 0 for both policies.
- - Hardware telemetry during the clean decision run recorded mean GPU utilization in the ~25-42% range for FIFO and ~28-34% for utility depending on seed, CPU utilization around 2-3%, min MemAvailable around 112-116 GiB, and no swap. This is materially stronger than the first low-utilization smoke, but still a local Phi-4-mini validation rather than a large production model run.
- - Captured final server `/metrics` and `/slots` into the multiseed artifact directory, then stopped the helper `llama-server`; no project-local helper server remains intentionally running.
- ## Interpretation after pressure run
- - Branch kill condition is not supported for the latency target: under a real OpenAI-compatible llama.cpp server and pressure-calibrated workload, utility scheduling cleared the 10% session-p95 threshold on all three clean seeds.
- - Evidence is mixed for non-latency resource metrics: cache-churn reduction and throughput gains did not reach 10%, so the live evidence supports a latency-oriented broker benefit rather than a broad throughput/cache-churn win.
- - This branch can be finalized positive with medium confidence: the mechanism survived live-server validation and produced durable p50/p95/tok/s/telemetry artifacts, while remaining caveated by small model choice and modest GPU utilization.
