## Abstract

We present a dependency-free converter that transforms native Codex session JSONL into a generalized, host-neutral tool-policy event stream supporting staged rollout modes (`observe`, `shadow`, `enforce`). The converter was evaluated offline over 50 recent completed Codex sessions containing 1,358 tool calls across 14 tool families, producing 2,667 hash-chained policy/result events with a pairable call/result pairing rate of 1.000 and stable hash validation. The policy classifier issued non-trivial decisions (1,223 allow, 132 review, 3 deny) at a mean processing cost of 0.244 ms per policy decision and peak resident memory under 28 MiB. A key negative finding drove a design correction: naively treating all native call records as pairable produced a pairing rate of 0.964, because `web_search_call` records in native Codex JSONL emit no separate output row. Explicitly modeling outputless native call records resolved this. Results are limited to offline replay over local sessions; no live tool call was blocked, and the policy classifier is a compact demonstrator rather than a complete security policy. Production deployment closure requires integration with live Codex/OMX pre/post tool hooks.

## Introduction

Automated coding agents increasingly invoke external tools—shell execution, file patching, web search, state mutation—within sessions that may span many turns. Governing these tool calls at scale requires a policy layer that can observe, shadow-check, or enforce decisions on every invocation while remaining neutral to the specific host runtime.

A practical rollout of such a layer faces several engineering constraints:

1. **Heterogeneous tool signatures.** Different tools produce different result structures; some (e.g., `web_search_call`) produce no separate result record at all in native logs.
2. **Staged deployment.** A policy layer must support incremental rollout from observation through shadow to enforcement without schema changes.
3. **Audit integrity.** Every policy decision and its paired result (when one exists) must be hash-chained for tamper-evident replay.
4. **Bounded cost.** The layer must impose negligible overhead relative to tool execution itself.
5. **Payload hygiene.** Arguments and results must be sanitized and redacted before durable emission.

This work tests whether a single, dependency-free converter can address all five constraints by processing native Codex JSONL session logs into a generalized event stream. We do not attempt to demonstrate live enforcement; rather, we ask whether the event schema, pairing logic, policy classifier, and hash chain are feasible at the offline-replay level, and what design corrections emerge from real session data.

## Method

### Converter design

The converter (`src/tool_policy_event_rollout.py`) is a single-file, dependency-free Python module. It reads completed tool-using Codex session logs from `~/.codex/sessions/2026/*/*/*.jsonl`, parses native `response_item` tool calls and results, and emits two generalized event kinds:

- **`policy_decision`**: Contains tool name, call ID, rollout mode, policy decision (`allow`, `review`, or `deny`), effective decision, reason string, sanitized arguments, context flags, and hash-chain fields linking to the prior event.
- **`tool_result`**: Contains paired call ID, result status/exit code (when available), sanitized result payload, and hash-chain fields.

Both event kinds carry bounded and redacted payloads. Hash chaining uses the previous event's hash to form a tamper-evident sequence.

### Outputless native call records

A critical design element is the explicit modeling of outputless native call records. In native Codex JSONL, `web_search_call` records have no corresponding result row. The converter marks these as outputless policy decisions and excludes them from the pairable-result denominator. This distinction was not present in the initial implementation and was added after a negative finding (see Results).

### Policy classifier

The classifier is intentionally small, designed to demonstrate the event rollout surface rather than serve as a complete security policy. Its rules are:

- **`allow`**: Default for observation and shadow pass-through.
- **`review`**: Applied to potentially sensitive write or system-path-like shell operations and patch operations.
- **`deny`**: Applied to destructive shell patterns or explicit escalated sandbox requests under `approval_policy=never`.

In `shadow` mode, `deny` decisions are recorded but not enforced (the effective decision remains `allow`). In `enforce` mode, `deny` decisions are effective. No live tool call was intercepted or blocked in either mode during this project; enforcement is simulated by setting effective-decision fields in emitted events.

### Evaluation protocol

The converter was run in three configurations:

1. **Smoke test** (`--limit 3 --mode shadow`): 3 sessions to verify basic functionality.
2. **Shadow rollout** (`--limit 50 --mode shadow`): 50 sessions with shadow-mode policy decisions.
3. **Enforce simulation** (`--limit 50 --mode enforce`): Same 50 sessions with enforce-mode policy decisions.

All runs emitted JSONL event files, session summaries, CSV summaries, and JSON metrics files. Unit tests verified: (a) destructive `exec_command` becomes `deny` in enforce mode and remains call/result paired; (b) secret-bearing outputless `web_search_call` is redacted and excluded from the pairable-result denominator.

### Negative finding: outputless call records

The initial implementation treated all native call records as pairable, producing a pairing rate of 0.964 in the first main run. Investigation revealed that `web_search_call` records in native Codex JSONL have no separate output row, creating 49 unpaired calls. The converter was updated to model outputless native call records explicitly, which resolved the pairing failure to 1.000. This is a substantive rollout requirement: any host-neutral policy layer must account for tools whose native representation lacks a separate result record, rather than treating the pairing gap as noise.

## Limitations

1. **Offline replay only.** This is a converter over existing Codex logs, not a host-integrated pre-dispatch enforcement hook. No live tool call was blocked or altered by this project. The enforce-mode run sets effective-decision fields in emitted events but does not intercept or block any tool call.

2. **Compact policy classifier.** The policy rules are intentionally minimal, designed to demonstrate the event rollout surface. A complete security policy would require substantially more rules, tool-specific heuristics, and likely human review workflows.

3. **Non-random corpus.** The 50 sessions are recent local sessions from a single machine, not a randomized production sample. Decision distributions and tool-family frequencies may not generalize.

4. **Simulated enforcement.** Production enforcement closure requires integration with the live Codex/OMX pre/post tool boundary, which was not tested here.

5. **Single-host evidence.** All runs were on one machine with ample memory (approximately 117 GiB available). Memory and throughput figures may differ under constrained conditions.

6. **No adversarial evaluation.** The policy classifier was tested against naturally occurring tool calls, not against deliberately crafted evasion attempts.

7. **Unreviewed claim ledger.** The project's claim ledger contains no formally registered claims at the time of draft generation. The findings reported here are grounded in the run notes, metrics artifacts, and decision JSON but have not undergone independent claim audit.

## Conclusion

A dependency-free converter successfully transformed native Codex JSONL from 50 completed sessions into a generalized, host-neutral tool-policy event stream with 1.000 pairable call/result coverage, stable hash chains, and non-trivial policy decisions (1,223 allow, 132 review, 3 deny). Processing cost was 0.244 ms per policy decision with peak memory under 28 MiB.

The principal design finding was negative: naively assuming all native call records have paired results produced a pairing rate of 0.964, failing on the 49 `web_search_call` records that lack separate output rows in native Codex JSONL. Explicitly modeling outputless native call records resolved this failure and is a necessary design element for any host-neutral policy layer.

These results constitute moderate-to-strong local feasibility evidence for the generalized event schema, but they remain offline replay evidence. The recommended successor step is to wire this schema into the live Codex/OMX pre/post tool boundary, preserving outputless-call semantics and redaction, and then run a prospective shadow rollout before enabling hard enforcement.
