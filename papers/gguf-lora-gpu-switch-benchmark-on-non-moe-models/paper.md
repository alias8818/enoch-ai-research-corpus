## Abstract

We benchmark GGUF-format LoRA adapter switching on a non-Mixture-of-Experts (non-MoE) language model served via llama.cpp's `llama-server`. Using Qwen2.5-0.5B-Instruct quantized to Q4_K_M on an NVIDIA GB10 (aarch64, CUDA 13.0), we measure three quantities: (1) the server-side control-plane latency of switching adapter scales via the `POST /lora-adapters` HTTP endpoint, (2) the per-request generation throughput penalty of applying a LoRA adapter relative to base inference, and (3) whether alternating between two already-loaded adapters imposes additional overhead beyond the LoRA compute path itself. The control-plane switch completes in a median of 0.173 ms (n = 100, max 0.426 ms). Request-level LoRA inference reduces mean generation throughput by approximately 30.5% relative to base (260.5–262.0 tok/s vs. 377.2 tok/s at n_predict = 64). Alternating between two pre-loaded adapters shows no clear additional throughput penalty beyond the LoRA compute cost. These results are specific to a single small model on a single host and do not address cold adapter loading from disk, larger models, or long-duration stability.

## Introduction

LoRA (Low-Rank Adaptation) adapters enable parameter-efficient fine-tuning by injecting small trainable rank-decomposition matrices into a frozen base model. In serving scenarios, the ability to switch between multiple LoRA adapters at low latency — whether at the session level or the individual request level — is attractive for multi-tenant inference where different users or tasks require different adapter behaviors.

The GGUF model format and the llama.cpp inference stack support loading multiple LoRA adapters at server startup and adjusting their application scales at runtime via HTTP API endpoints. Prior discussion of LoRA switching performance in the community has largely focused on Mixture-of-Experts (MoE) architectures, where adapter switching interacts with expert routing. The behavior of GGUF LoRA switching on dense (non-MoE) models has received less systematic measurement.

This work provides a focused, reproducible local benchmark addressing three questions:

1. What is the server-side latency of switching LoRA adapter scales via the HTTP control plane, independent of any generation work?
2. What throughput penalty does request-level LoRA application impose on a small dense model relative to base inference?
3. Does alternating between two different already-loaded adapters at the request level impose additional overhead beyond the cost of applying a single adapter repeatedly?

We conduct all experiments on a single NVIDIA GB10 system using Qwen2.5-0.5B-Instruct Q4_K_M as the base model and two converted GGUF LoRA adapters (8.5 MiB each). We report results with full distributional detail and explicitly note the scope limitations of this single-model, single-host study.

## Method

### Hardware and Software Environment

All experiments ran on an NVIDIA GB10 system (aarch64, Linux kernel 6.17.0-1014-nvidia) with CUDA driver version 580.142 and CUDA runtime 13.0. System swap was intentionally disabled and confirmed at 0 kB throughout all runs. Available system memory before the main benchmark was approximately 121.6 GB; after completion, approximately 120.1 GB.

The inference server was `llama-server` built from the llama.cpp repository at git commit `05e141a`. The base model was `Qwen2.5-0.5B-Instruct-Q4_K_M.gguf`, a 4-bit quantized GGUF checkpoint of the Qwen2.5-0.5B-Instruct dense transformer model.

### LoRA Adapters

Two PEFT-format LoRA adapters were converted to GGUF format using `convert_lora_to_gguf.py` (same commit `05e141a`) with output type f16:

- **Adapter 0:** `qwen_lora_smoke_limit2` → `qwen_lora_smoke_limit2.gguf` (8.5 MiB)
- **Adapter 1:** `qwen_lora_expanded_e8` → `qwen_lora_expanded_e8.gguf` (8.5 MiB)

Both adapters were loaded at server startup. The server was launched with `--lora-init-without-apply`; however, the server's `GET /lora-adapters` endpoint reported both adapters at scale 1.0 regardless. For this reason, all experiments relied on explicit request-level `lora` overrides and explicit `POST /lora-adapters` scale payloads rather than trusting the initial scale state.

### Benchmark Design

Three benchmark phases were executed:

**Smoke benchmark.** A short validation pass with 30 switch iterations and 12 completion iterations at `n_predict = 24`. This confirmed the harness and server were functioning correctly. Results are logged but not reported as primary data.

**Main benchmark.** 100 switch iterations (control-plane latency measurement) and 30 completion iterations per label at `n_predict = 64`. Completion labels were: base (no request-level LoRA), adapter 0 (request-level LoRA override), and adapter 1 (request-level LoRA override). Ten completions per label were generated using the same prompt.

**Repeated-vs-alternating isolation pass.** A focused experiment with `n_predict = 64` and 12 completions per phase across three phases: (a) adapter 0 applied repeatedly, (b) adapter 1 applied repeatedly, and (c) adapter 0 and adapter 1 alternated on successive requests. This isolates whether inter-adapter switching adds overhead beyond the LoRA compute path.

### Measurement Methodology

Control-plane switch latency was measured by alternating `POST /lora-adapters` payloads to change adapter scales between 0.0 and 1.0, with no generation triggered. The harness recorded wall-clock time for each HTTP round-trip.

Completion throughput was measured as `n_predict / wall_time_seconds` for each request, where `n_predict = 64` and wall time covers the full HTTP request-response cycle. The same prompt text was used across all completion requests.

GPU telemetry was sampled via `nvidia-smi` during the main run. Server logs were monitored for cache-clearing events triggered by request-level LoRA changes.

## Results

### Control-Plane Switch Latency

The `POST /lora-adapters` endpoint for switching adapter scales between active and inactive states exhibited the following latency distribution (n = 100):

| Statistic | Latency (ms) |
|-----------|-------------:|
| Min       | 0.084        |
| Median    | 0.173        |
| P90       | 0.256        |
| Max       | 0.426        |
| Mean      | 0.176        |

The switch operation is consistently sub-millisecond on this hardware and server configuration. The tight clustering of the median, p90, and mean suggests a stable, low-overhead control-plane path.

### Request-Level LoRA Throughput Penalty

Generation throughput at `n_predict = 64` across the three completion conditions:

| Condition | Mean wall time (ms) | Mean throughput (tok/s) |
|-----------|--------------------:|------------------------:|
| Base (no LoRA) | 175.85 | 377.24 |
| Adapter 0 (request LoRA) | 256.69 | 260.54 |
| Adapter 1 (request LoRA) | 255.27 | 262.03 |

Applying either LoRA adapter at the request level reduced mean generation throughput by approximately 30.5% relative to base inference. The two adapters produced effectively equivalent throughput (difference < 1.5 tok/s, well within expected request-level variance). Server logs confirmed that request-level LoRA changes triggered cache clearing (`clearing cache for lora change. 2 loras -> 2 loras`), which is a contributing factor to the observed penalty alongside the additional LoRA matrix computations.

### Repeated vs. Alternating Adapter Application

| Phase | Mean wall time (ms) | Median wall time (ms) | Mean throughput (tok/s) |
|-------|--------------------:|----------------------:|------------------------:|
| Adapter 0 repeated | 260.65 | 254.36 | 262.80 |
| Adapter 1 repeated | 253.03 | 253.21 | 263.85 |
| Alternating 0/1 | 253.82 | 253.25 | 263.20 |

The alternating condition did not show a measurable additional throughput penalty relative to repeated application of either adapter. The slightly elevated mean wall time for the adapter 0 repeated phase (260.65 ms vs. ~253 ms for the other two phases) is attributable to a single outlier at 332 ms; the medians across all three phases are essentially identical (~253–254 ms). We conclude that for already-loaded adapters on this configuration, inter-adapter switching at the request level adds no clear overhead beyond the LoRA compute path and associated cache clearing.

### GPU and Memory Posture

During the main benchmark's completion phase, `nvidia-smi` sampling recorded peak GPU utilization at approximately 92%, with a temperature of 44 °C and power draw of 30.26 W. At server shutdown, CUDA memory allocation was: total 124,546 MiB, free 117,289 MiB, model 373 MiB, context 3 MiB, compute 298 MiB. No swap activity occurred.

## Limitations

This benchmark has several important scope restrictions:

1. **Single model and quantization.** Only Qwen2.5-0.5B-Instruct at Q4_K_M was tested. The 30.5% throughput penalty and sub-millisecond switch latency may not generalize to larger models, different quantization levels, or higher-rank LoRA adapters.

2. **Single host.** All measurements were taken on one NVIDIA GB10 system. Different GPU architectures, memory bandwidths, and driver versions may yield different results.

3. **Pre-loaded adapters only.** Both LoRA adapters were loaded into GPU memory at server startup. This benchmark does not measure the latency or throughput impact of cold-loading a new adapter from disk after the server has started, which would involve disk I/O and GPU memory allocation.

4. **Short duration.** The main benchmark consisted of 100 switch iterations and 30 completion iterations per label. This is sufficient to characterize the performance shape of the system but does not constitute a long-duration stability or memory-leak test.

5. **Server scale initialization quirk.** The `--lora-init-without-apply` flag did not produce the expected initial scale of 0.0; the server reported scales of 1.0 at startup. All reported measurements use explicit request-level overrides and POST payloads, which are reliable, but this server behavior is a caveat for deployments that rely on the initial scale state.

6. **Cache clearing confound.** Request-level LoRA changes trigger KV cache clearing in llama-server. The observed 30.5% throughput penalty conflates the cost of LoRA matrix computation with the cost of cache invalidation. Disentangling these two contributions would require a modified server build that suppresses cache clearing, which was not performed.

7. **Small adapter size.** Both adapters were 8.5 MiB. Larger adapters (higher rank, more target modules) may exhibit different switching and compute overhead profiles.

8. **Evidence classification.** These results constitute a local, short-duration llama.cpp hook-prototype benchmark on a single host. They are not the product of a production-validated deployment, a multi-model sweep, or a CUDA copy calibration pass. The evidence quality is best characterized as local reproducible short benchmark.

## Conclusion

On an NVIDIA GB10 system serving Qwen2.5-0.5B-Instruct Q4_K_M via llama.cpp, GGUF LoRA adapter switching is practically viable. The control-plane cost of switching adapter scales via HTTP is sub-millisecond (median 0.173 ms). Request-level application of a LoRA adapter reduces generation throughput by approximately 30.5% relative to base inference, a penalty that includes both LoRA matrix computation and KV cache clearing. Alternating between two already-loaded adapters at the request level imposes no clear additional overhead beyond the cost of applying a single adapter repeatedly.

These findings establish a local performance baseline for GGUF LoRA switching on a small dense model. Whether the 30.5% throughput penalty scales proportionally or differently with larger models, higher LoRA ranks, or different quantization levels remains an open question. The absence of cold-load-from-disk measurements is a notable gap for deployment planning, as real-world multi-adapter serving scenarios may require dynamic adapter loading not covered by this benchmark.
