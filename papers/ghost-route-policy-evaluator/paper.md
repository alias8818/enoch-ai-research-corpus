## Abstract

Broken access control remains a persistent web-application risk. One contributing mechanism is *authorization drift*: a mismatch between the routes an application serves and the authorization policies that guard them. We define four classes of ghost-route defect—uncovered routes, underprotected routes, stale policies, and ambiguous policy overlaps—and implement a dependency-free static cross-checker that compares a route table against a policy table to flag these defects before runtime. In a synthetic benchmark of 5,000 routes and 5,200 policies with injected drift, the evaluator achieved 1.0000 recall and 0.9851 precision (F1 = 0.9925), completing in 6.99 seconds with 43 MB peak memory. A five-seed replication at 3,000 routes confirmed stability (mean precision 0.9931, mean recall 1.0000, minimum precision 0.9888). The 144 false positives in the main run arose from conservative wildcard-overlap flagging, an acceptable bias for a CI guard at the tested threshold. These results support the viability of static route/policy drift detection as a pre-deploy check, though validation against real framework routers and production policy bundles remains necessary.

## Introduction

Web applications typically maintain two independent tables: a *route table* that maps HTTP paths and methods to handler code, and a *policy table* that maps the same paths and methods to authorization rules. When these tables diverge—because a route is added without a corresponding policy, a policy is broadened beyond a route's intended minimum role, or a retired policy is left in place—authorization defects can persist undetected until exploited at runtime.

We term such mismatches *ghost-route defects* and define four specific classes:

1. **Uncovered route** — a live route has no matching policy rule.
2. **Underprotected route** — a matching policy grants roles broader than the route's intended minimum role.
3. **Stale policy** — a policy rule covers no live route.
4. **Ambiguous policy overlap** — multiple matching policies disagree on allowed roles, making behavior order- or engine-dependent.

The security relevance of these defects is grounded in established risk frameworks. Broken access control has consistently ranked as a top web-application risk (OWASP Top 10, A01:2025). API-specific guidance identifies object- and function-level authorization failures as a major concern (OWASP API Security Top 10, 2019). The growing adoption of externalized policy engines such as Open Policy Agent, which exposes policy-as-code and REST integration patterns, makes the route/policy interface a natural target for static verification.

This paper presents the Ghost-Route Policy Evaluator, a static cross-checker that accepts route and policy tables as JSON inputs and reports findings across all four defect classes. We evaluate it on synthetic corpora with known injected drift and assess whether it meets a predeclared viability threshold (precision ≥ 0.98, recall ≥ 0.98).

## Method

### Implementation

The evaluator (`src/ghost_route_policy_evaluator.py`) is a dependency-free Python 3.12 CLI tool with two modes:

- **Evaluate mode** (`evaluate --routes ... --policies ...`): accepts two JSON files, cross-checks them, and reports findings. Exits nonzero when real findings are detected, enabling use as a CI guard.
- **Benchmark mode** (`benchmark --routes N --inject-rate R --seed S --outdir DIR`): generates a synthetic route/policy corpus with known ground-truth drift, runs the evaluator, and scores predicted findings against injected semantic truth.

Path-template normalization supports three common conventions: `/x/{id}`, `/x/:id`, and wildcard `*` segments. HTTP method normalization handles `ANY` and `*` as catch-alls. The matching algorithm is conservative: when multiple wildcard policies could match a route, all are considered, and any disagreement in granted roles is flagged as an ambiguous overlap.

### Defect detection logic

For each route, the evaluator collects all policies whose path pattern and HTTP method intersect with the route. It then classifies:

- If no policies match: **uncovered route**.
- If matching policies grant roles strictly broader than the route's declared minimum role: **underprotected route**.
- For each policy that matches no route: **stale policy**.
- If multiple matching policies disagree on allowed roles: **ambiguous policy overlap**.

### Synthetic benchmark design

1. Generate a clean corpus of N routes and N × 1.04 policies with exact expected authorization.
2. Inject known drift at rate R: removed policies (creating uncovered routes), weakened roles (creating underprotected routes), retired policies (creating stale policies), and broad wildcard overlaps (creating ambiguous overlaps).
3. Run the evaluator and compare predicted finding keys against the injected semantic ground truth.
4. Compute precision, recall, and F1.

### Viability rule

The evaluator is declared viable if precision ≥ 0.98 and recall ≥ 0.98 on the main benchmark. This threshold was predeclared in the project decision record prior to running the main benchmark.

## Limitations

1. **Synthetic corpora only.** The evaluator has not been validated against route tables or policy bundles harvested from a real application. Production route tables may include framework-specific routing features (middleware chains, conditional routing, mounted sub-applications) that the current template-matching logic does not model.

2. **Conservative path-pattern matching.** The evaluator uses template-based path normalization (`{id}`, `:id`, `*`), not a full symbolic route-language solver. Path-pattern intersection is deliberately conservative: it may flag overlaps that a more precise solver would resolve as non-overlapping. This is the direct cause of the 144 false positives in the main benchmark.

3. **No framework router integration.** The evaluator compares abstract route and policy tables. It does not hook into Express, FastAPI, Rails, or any other framework's actual routing engine. Whether the abstract tables faithfully represent what the framework enforces at runtime is untested.

4. **No real policy engine integration.** Validation against OPA/Rego policy bundles or other externalized authorization systems has not been performed.

5. **Quadratic scaling with wildcard density.** The conservative matching strategy checks pairwise intersections among wildcard policies. As the number of wildcard policies grows, evaluation time increases quadratically. The throughput drop from 1,000 to 5,000 routes (3,757 to 715 routes/s) reflects this cost. Services with very large wildcard-heavy policy tables may experience slower performance.

6. **Single defect model.** The four defect classes modeled here do not cover all forms of authorization drift. Context-dependent policies (e.g., attribute-based access control) and dynamic policy evaluation results are outside the current static model.

7. **No dynamic validation.** The evaluator is purely static. It does not send requests to a running service to confirm that flagged defects are exploitable. Some false positives may correspond to routes that are additionally protected by middleware not represented in the policy table.

8. **Confidence assessment.** The project decision record assigns medium confidence to the viability conclusion, reflecting the gap between synthetic benchmark success and the absence of real-framework validation.

## Conclusion

The Ghost-Route Policy Evaluator demonstrates that static cross-checking of route tables against policy tables can detect authorization drift with high recall and acceptable precision in synthetic settings. On a 5,000-route corpus with injected defects, it achieved 100% recall and 98.5% precision, completing in under 7 seconds with negligible memory. Five-seed replication confirmed stability above the predeclared viability thresholds.

The conservative matching strategy produces a predictable false-positive pattern—wildcard-policy overlap flags—that is acceptable for a CI guard at the tested threshold but would require tuning or suppression mechanisms for production adoption. The evaluator's viability is established only for synthetic corpora; its effectiveness on real framework route tables and real policy bundles remains an open question.

The result supports continuing this line of work toward a CI/pre-deploy guard. The recommended next step is integration with one real target stack: extract routes from a small FastAPI or Express application, map route metadata to expected minimum roles, load an exported authorization table, and compare evaluator findings against manual review and at least one dynamic authorization test per critical route.
