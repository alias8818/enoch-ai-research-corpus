## Abstract

We present a prototype harness for replaying labeled historical request logs through candidate decision policies in shadow mode, producing counterfactual audit records without production side effects. The harness enforces immutable replay events, append-only shadow audit trails, and side-effect leak accounting. In calibrated synthetic experiments with 1,000 labeled events, two sample shadow policies—a segment-aware policy and a conservative policy—yielded accuracy deltas of +0.204 and +0.053 over the production baseline (0.744), with changed-action precisions of 0.981 and 0.949 respectively. Zero side-effect leaks were observed across all runs. JSONL round-trip replay reproduced all metrics deterministically. These results demonstrate that the harness can answer its intended research question—estimating whether a candidate policy corrects production errors, introduces regressions, and how often it diverges—under synthetic conditions. However, all labels are deterministic synthetic data; no real production logs were available. Scientific closure on real-world utility requires validation against private labeled production data.

## Introduction

Shadow mode deployment is a standard technique for evaluating candidate decision policies against live production traffic without exposing users to untested decisions. A persistent challenge is obtaining reliable ground-truth labels for historical decisions, since production outcomes are often delayed or incomplete. Without labeled data, it is difficult to quantify whether a candidate policy would correct production errors, introduce regressions, or merely change decisions without improving outcomes.

This work addresses a specific sub-problem: given a set of labeled replay events—each recording a production action and a delayed label action—can a harness replay those events through candidate shadow policies, emit durable audit evidence, and summarize counterfactual quality metrics without any production side effects?

The contribution is a self-contained Python prototype that:

1. Defines an immutable `ReplayEvent` JSONL schema separating `production_action` from delayed `label_action`.
2. Replays events through shadow policies that produce counterfactual decisions only.
3. Writes append-only `ShadowRecord` audit rows rather than calling any production actuator.
4. Reports production accuracy, shadow accuracy, accuracy delta, action-match rate, would-change rate, changed-action precision, corrected errors, regressions, net corrections, mean confidence, audit record count, elapsed time, and side-effect leak count.

We evaluate the harness on deterministic synthetic data and report results honestly, including cases where a shadow policy offered no net improvement.

## Method

### Replay Event Schema

Each replay event is an immutable record with the following fields:

- `event_id`: unique identifier
- `features`: input features for policy evaluation
- `production_action`: the action taken by the production system
- `label_action`: the delayed ground-truth correct action

The separation of `production_action` and `label_action` is deliberate: it models the realistic setting where the correct action is not known at decision time and only becomes available later.

### Shadow Policy Interface

A shadow policy receives the same features as the production system and returns a counterfactual action along with a confidence score. It never calls any production actuator. Two sample policies were implemented for calibration:

- **Segment-aware shadow**: A policy that uses segment-level feature information to select actions, designed to correct known production errors in certain feature segments.
- **Conservative shadow**: A policy that defaults to the production action unless confidence in an alternative exceeds a threshold, designed to minimize regressions.

These policies are synthetic demonstrators included to exercise the harness, not production candidates.

### Audit and Metrics

For each replay event and shadow policy, the harness emits a `ShadowRecord` containing the event ID, production action, shadow action, label action, and confidence. From these records, the following metrics are computed:

| Metric | Definition |
|---|---|
| Production accuracy | Fraction of events where `production_action == label_action` |
| Shadow accuracy | Fraction of events where `shadow_action == label_action` |
| Accuracy delta | Shadow accuracy minus production accuracy |
| Action-match rate | Fraction of events where `shadow_action == production_action` |
| Would-change rate | Fraction of events where `shadow_action != production_action` |
| Changed-action precision | Among events where shadow differs from production, fraction where shadow matches label |
| Corrected errors | Events where production was wrong, shadow was correct, and they differed |
| Regressions | Events where production was correct, shadow was wrong, and they differed |
| Net corrections | Corrected errors minus regressions |
| Side-effect leak count | Number of times a shadow decision was routed to a production actuator (target: zero) |

### Deterministic Synthetic Data

Because no private labeled production logs were available in the project directory, the harness includes a deterministic synthetic data generator seeded by a user-provided random seed. This generator produces `ReplayEvent` records with known label distributions, enabling reproducible smoke and calibration runs. The synthetic data is not intended to model any specific production distribution.

### JSONL Round-Trip

The harness supports both in-memory generation (`--generate-synthetic`) and replay from a saved JSONL file (`--input-jsonl`). The JSONL round-trip test verifies that saving generated events to disk and replaying them reproduces the same metrics, confirming that the audit pipeline is deterministic and the schema is self-contained.

| Metric | Production | Segment-aware shadow | Conservative shadow |

| Accuracy delta | — | +0.150 | 0.000 |
| Would-change rate | — | 0.250 | 0.100 |
| Changed-action precision | — | 0.800 | 0.500 |
| Corrected errors | — | 4 | 1 |
| Regressions | — | 1 | 1 |
| Net corrections | — | 3 | 0 |
| Side-effect leaks | 0 | 0 | 0 |

The conservative shadow matched production accuracy exactly, with a net correction of zero. Its changed-action precision of 0.500 indicates that when it diverged from production, it was correct only half the time—a neutral result. The segment-aware shadow showed a positive accuracy delta but also introduced one regression.

### Calibrated Run: 1,000 Synthetic Events

| Metric | Production | Segment-aware shadow | Conservative shadow |
|---|---|---|---|
| Accuracy | 0.744 | 0.948 | 0.797 |
| Accuracy delta | — | +0.204 | +0.053 |
| Would-change rate | — | 0.212 | 0.059 |
| Changed-action precision | — | 0.981 | 0.949 |
| Corrected errors | — | 208 | 56 |
| Regressions | — | 4 | 3 |
| Net corrections | — | 204 | 53 |
| Mean confidence | — | 0.704 | 0.707 |
| Side-effect leaks | 0 | 0 | 0 |
| Elapsed time | — | 9.11 ms | (included) |
| Audit records | — | 2,000 total | (1,000 events × 2 policies) |

The segment-aware shadow corrected 208 production errors while introducing 4 regressions, yielding a net correction of 204. The conservative shadow corrected 56 errors with 3 regressions, for a net correction of 53. Both policies maintained zero side-effect leaks.

The segment-aware shadow's high changed-action precision (0.981) indicates that nearly all of its divergences from production were correct. However, its would-change rate of 0.212 means it would alter roughly one in five production decisions—a substantial operational divergence that may carry integration risk not captured by these metrics alone.

The conservative shadow's lower would-change rate (0.059) and lower net corrections (53 vs. 204) reflect the expected tradeoff: fewer interventions yield fewer corrections and fewer regressions.

### JSONL Round-Trip Determinism

Replaying the saved JSONL file from the calibrated run reproduced all production and shadow policy metrics exactly, with the exception of elapsed time and file path metadata. This confirms that the replay pipeline is deterministic and the JSONL schema captures all information needed for reproducible evaluation.

### System Environment

All runs were executed on an aarch64 host (Linux 6.17.0-1014-nvidia) with 121 GiB total memory and 116 GiB available. No swap was configured. The prototype is pure Python with no third-party dependencies.

## Limitations

1. **Synthetic labels only.** All results derive from deterministic synthetic data with known label distributions. No real production logs or delayed labels were available. The positive metrics (accuracy deltas, changed-action precisions) reflect properties of the synthetic generator and sample policies, not real-world policy performance. Scientific closure requires validation against private labeled production data.

2. **Sample policies are not production candidates.** The segment-aware and conservative shadow policies are simple demonstrators included to exercise the harness. Their metric profiles should not be interpreted as evidence that any specific policy architecture is superior in production.

3. **No distributional realism.** The synthetic generator does not model any specific production feature distribution, temporal drift, label delay distribution, or class imbalance. Metrics on synthetic data may not generalize.

4. **No latency or throughput claims.** The 9.11 ms elapsed time for 1,000 events reflects a pure-Python in-memory loop on a high-memory machine with no I/O contention. It is not a benchmark of production replay throughput.

5. **Single-seed calibration.** The calibrated run used seed 13; the smoke run used seed 7. Different seeds may yield different absolute metric values, though the harness's structural properties (zero side-effect leaks, deterministic JSONL round-trip) are seed-independent.

6. **No integration risk modeling.** The would-change rate measures decision divergence but does not capture downstream system effects, operational complexity, or rollback cost when a shadow policy is promoted to production.

7. **Small regression counts.** The regression counts (4 and 3 out of 1,000 events) are small in absolute terms, making confidence intervals on regression rates wide. These counts should not be treated as precise estimates.

8. **No human annotation.** The original project plan called for independent labeling of 486 review-relevant replay rows. This was not performed; all labels are synthetic.

9. **Claim ledger audit status: blocked.** The automated claim extraction pipeline produced no structured claims for this artifact, and the claim ledger audit status is `blocked_empty_claims`. The metrics and observations reported here should be read as prototype-level evidence pending independent verification, not as audit-approved claims.

## Conclusion

We implemented and verified a self-contained Python harness for labeled replay in shadow mode. The prototype enforces immutable replay events, append-only shadow audit records, and side-effect leak accounting. On deterministic synthetic data at two scales (20 and 1,000 events), the harness produced complete counterfactual quality metrics with zero side-effect leaks. JSONL round-trip replay confirmed determinism.

The two sample shadow policies exhibited distinct tradeoff profiles: the segment-aware policy achieved higher accuracy gains at the cost of higher decision divergence, while the conservative policy offered modest gains with minimal divergence. Both introduced small numbers of regressions.

These results establish that the harness can answer its intended research question under synthetic conditions. They do not establish that any specific shadow policy is production-ready, nor that synthetic metric magnitudes will hold on real data. The project decision records confidence as "medium" and evidence strength as "prototype_moderate," with the hypothesis status "supported_by_prototype." The critical next step is connecting the JSONL schema to real production request logs with delayed labels and running replay on private labeled data before any live shadow deployment.
