## Hypothesis

A replay drift ledger that records normalized request hashes, open-weight model fingerprint hashes, semantic response hashes, tool/state hashes, and a chained entry hash should detect and localize drift at a real API boundary better than output-only and raw-transcript baselines, while avoiding false positives from benign response formatting changes.

This follow-up deepens the prior local deterministic API harness by replacing the hand-coded completion service with a cached open-weight GPT-2-family model served through a local HTTP endpoint.

## Experiment Design

Implemented `scripts/open_weight_replay_drift_ledger.py`.

Directly tested:
- A real HTTP API boundary: local `POST /v1/completions` calls through `urllib`, not in-process function calls.
- Cached open-weight GPT-2-family models loaded from Hugging Face artifacts with NumPy + safetensors:
  - `sshleifer/tiny-gpt2` for smoke and fast full harness confirmation.
  - `distilgpt2` for the bounded Tier 3 run.
- Fixed seeds: `11, 17, 23, 31, 43`.
- 100 prompt traces per full model run, each with two model-backed API steps and a deterministic tool/state observation.
- 700 replay cases per validator: 100 each for `none`, `format_drift`, `request_drift`, `decoder_drift`, `model_fingerprint_drift`, `api_response_drift`, and `tool_state_drift`.

Validators and controls:
- `ledger_full`: request + model fingerprint + semantic response + tool + state hashes.
- `output_only`: final token only.
- `raw_transcript`: raw JSON responses only.
- `ledger_raw_response`: full ledger except raw response hashes replace semantic response hashes.
- Ablations: `ledger_no_request`, `ledger_no_model_fingerprint`, `ledger_no_response`, `ledger_no_tool_state`.

Only proxied or controlled:
- Drift was injected at the local serving boundary; this did not wait for natural provider-side model updates.
- The traces used next-token GPT-2 completions and a deterministic tool, not broad chat/instruction agents with real external side effects.
- No streaming, retries, rate limits, API billing, or provider failover behavior was tested.

Direct/full evidence required to make this publication-ready:
- Repeat the same ledger and metrics against a live provider or an open-weight serving stack with real chat/tool traces, streaming/retry paths, and longitudinal natural drift windows.
- Use larger instruction-tuned models and externally meaningful tasks rather than next-token GPT-2 continuations.

## Interpretation

The bounded direct open-weight validation supports the mechanism. The full ledger cleanly separated material replay drift from benign formatting drift and localized the first changed component. Baselines exposed the expected weaknesses: output-only missed most non-final-output drift, raw transcript comparison detected all injected material changes but also falsely flagged every benign formatting drift case, and ablations lost exactly the component-specific localization signal they removed.

This is not paper-positive evidence. The strongest claim is scoped to controlled open-weight GPT-2-family HTTP replay with injected drift. It does not yet establish robustness under live provider changes, instruction-tuned chat behavior, streaming deltas, retry semantics, rate limiting, tool side effects, or naturally occurring longitudinal drift.

## Final Decision

Decision: `finalize_negative` with `research_outcome: useful_signal`.

Stop rationale: bounded open-weight evidence supports the replay-ledger mechanism with direct metrics, baselines, fixed seeds, controls, ablations, and a 100-trace DistilGPT-2 run, but the drift is still injected and the tasks are too narrow for publication readiness.
