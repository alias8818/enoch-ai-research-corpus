## Abstract

We investigate whether supplying historical failure-log memory to an LLM-backed patch-generation agent improves its ability to repair code from redacted CI failure logs. A local benchmark harness (`benchmark_l2p_memory.py`) was constructed to create small Python repair tasks and invoke `codex exec` (model: `gpt-5.4-mini`) under three memory policies: no memory (redacted current log only), raw memory (full historical incident transcript), and memory card (concise log-derived actionable memory card). Across three synthetic tasks, the no-memory policy solved 2/3 tasks (66.7%), while both raw-memory and memory-card policies solved 3/3 (100.0%). The discriminating task required inferring a legacy SKU-prefix mapping (`rs-` → `pro`) absent from the redacted current log but present in historical memory. Mean patch-generation time was lower under memory-supplied policies (raw: 16.75 s, card: 16.24 s) than under no memory (25.78 s), though this difference is confounded by task-level variance and the small sample. These results are preliminary: only three synthetic tasks, one model, and a single stochastic seed were tested. The benchmark is useful as a reproducible local acceptance harness, but no broad scientific claim is warranted from this evidence alone.

## Introduction

Automated program repair from CI failure logs is an emerging application of large language models. In practice, the failure logs available to a patching agent are often redacted or incomplete—sensitive identifiers, internal hostnames, and proprietary values may be stripped before the log reaches the repair system. A natural question arises: can historical memory of prior incidents compensate for information lost to redaction?

This paper reports on a small, controlled experiment testing whether supplying historical log-derived memory to an LLM patch-generation agent measurably improves repair success on tasks where the current failure log has been redacted. Three policies for memory supply are compared—none, raw transcript, and concise memory card—across three synthetic Python repair tasks designed so that at least one task requires information only available through historical memory.

The contribution is a reproducible benchmark harness and an initial empirical signal that log-derived memory can lift repair success on redacted-log tasks. The scope is narrow: three tasks, one model, one seed. The result is best interpreted as a viability signal and a methodological template, not as a generalizable scientific conclusion.

## Method

### Benchmark Design

A local harness (`benchmark_l2p_memory.py`) was implemented to construct small Python repair tasks and invoke `codex exec` as the LLM-backed patch generator. Each task consists of:

- A Python implementation file containing a bug.
- A `current_failure.log` file containing a redacted CI failure log.
- A `memory_context.md` file whose content varies by policy.
- Hidden pytest tests injected only after Codex exits, ensuring the patcher cannot read target tests.

The patcher must infer the smallest plausible implementation patch from the log and optional memory context, then edit the code. The harness validates the patch by running the hidden tests post-hoc.

### Memory Policies

Three policies were tested:

1. **no_memory**: Only the redacted current failure log is available. The `memory_context.md` file indicates no historical memory is supplied.
2. **raw_memory**: The full, noisy historical incident transcript is supplied verbatim in `memory_context.md`.
3. **memory_card**: A concise, log-derived actionable memory card is supplied in `memory_context.md`, summarizing the key inference from historical incidents.

### Tasks

Three synthetic Python repair tasks were constructed, each requiring a distinct domain inference:

- **sku_tier_aliases**: Infer that the legacy `rs-` SKU prefix maps to `pro`. The redacted current log omits this mapping; it is only available through historical memory.
- **window_units_si**: Infer that quota-window MB/GB units are decimal SI (1000-based), not binary (1024-based). This inference is available from the current redacted log alone.
- **ticket_slug_normalization**: Infer ASCII/separator slug canonicalization rules. This inference is also available from the current redacted log alone.

The first task was designed to be memory-dependent; the latter two were designed to be solvable from the current log alone, providing a baseline to detect whether memory supply degrades performance on tasks that do not require it.

### Execution

A smoke calibration was run on a subset of tasks and policies before the full benchmark. The full benchmark executed all 3 tasks × 3 policies = 9 runs sequentially.

**Full benchmark command:**


### Failure Analysis

The sole failure occurred on `sku_tier_aliases` under the no-memory policy. Without historical memory, Codex produced a plausible but incorrect generic normalization patch that failed hidden reseller-prefix tests. With either raw historical memory or the concise memory card, Codex added the required `rs- → pro` mapping and passed.

On the two tasks solvable from the current log alone (`window_units_si`, `ticket_slug_normalization`), all three policies passed, indicating that memory supply did not degrade performance on non-memory-dependent tasks in this small sample.

### Patch Time Observations

Mean patch time was lower under both memory-supplied policies than under no memory. However, this difference is driven primarily by the `sku_tier_aliases` task, where the no-memory run took 38.0 s—substantially longer than any other run in the experiment, possibly reflecting a longer search or more edits before the agent produced its (incorrect) patch. For the two tasks where all policies passed, patch times were broadly comparable across policies (range: 14.1–22.0 s). No statistical test is applied given the sample size of one run per cell, and the patch-time difference should not be interpreted as a causal effect of memory policy on generation speed.

## Limitations

1. **Small task set.** Only three synthetic repair tasks were tested. This is insufficient to estimate the general effect size of log-derived memory on LLM patch success.

2. **Single model and seed.** Only one model (`gpt-5.4-mini`) and one stochastic seed per task-policy combination were run. Variance across models, temperatures, and seeds is uncharacterized.

3. **Memory cards are frozen fixtures.** The memory cards used in this benchmark were pre-authored benchmark fixtures, not produced by an LLM memory-extraction pipeline. A separate experiment is needed to measure end-to-end quality when memory cards are themselves LLM-generated from raw logs.

4. **Ceiling effect on two tasks.** Two of three tasks were solvable from the redacted current log alone, limiting the measured lift from memory to at most one task. A task set with more memory-dependent items would provide a stronger test.

5. **Synthetic tasks only.** All tasks are small, self-contained Python functions. Real CI repair tasks involve larger codebases, more complex failure modes, and noisier logs.

6. **No repeated measures.** Each task-policy pair was run once. The observed lift of +1/3 tasks could be due to stochastic variation in the LLM's output.

7. **Patch time confound.** The observed difference in mean patch time between policies is not controlled for task difficulty or edit complexity and should not be interpreted as a causal effect of memory policy on speed.

8. **Prototype-grade evidence.** This benchmark is a local harness invoking a single LLM endpoint. It is not a production-validated system, and the results reflect prototype behavior under controlled synthetic conditions.

## Conclusion

In a small synthetic benchmark of three Python repair tasks, supplying historical log-derived memory to an LLM patch-generation agent improved hidden-test pass rate from 2/3 to 3/3. The improvement was concentrated in a single task (`sku_tier_aliases`) that required a domain-specific mapping absent from the redacted current log. Concise memory cards matched raw historical transcripts on pass rate while being more compact, and showed slightly lower mean patch time in this tiny sample—though the patch-time difference is not statistically meaningful at this sample size.

These findings are consistent with the hypothesis that log-derived memory can compensate for information lost to log redaction, but they do not establish it. The sample is too small, the tasks too synthetic, and the experimental design too narrow (one model, one seed, no repeated measures) to support a general claim. The primary value of this work is the benchmark harness itself—a reproducible local acceptance test for log-to-patch memory interventions—and the initial viability signal it produces.

Future work should: (a) incorporate real historical CI logs and patches from existing repositories; (b) run repeated seeds and multiple models to characterize variance; (c) add an explicit LLM memory-condensation stage and score extraction errors separately from patch-generation errors; and (d) scale the task set to include more memory-dependent items that better stress the intervention.
