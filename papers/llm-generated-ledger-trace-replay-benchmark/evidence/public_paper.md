## Abstract

We evaluate whether semantically structured trace scenarios improve seeded-defect detection in a deterministic synthetic double-entry ledger, compared to naive random, state-aware random, and no-reversal ablation baselines. Using a scripted scenario-guided trace generator as a proxy for LLM-produced accounting workflows, we replay 20,000 traces of length 200 against 10 seeded ledger mutants (800,000 mutant–trace pairs) and supplement with a length-robustness sweep across lengths 50, 100, 200, and 400 (5,000 traces each). Scenario-guided traces achieve a 98.4% mutant kill rate, versus 67.1% for state-aware random, 68.0% for the no-reversal ablation, and 92.4% for naive random. The no-reversal ablation kills zero mutants in three reversal-specific defect classes, directly confirming that reversal scenarios drive detection of those faults. However, no actual LLM-generated traces were evaluated; the scenario generator is a deterministic scripted proxy. Only one toy ledger oracle and 10 seeded mutants were tested. One mutant (`withdraw_off_by_one`) was better detected by state-aware random (0.9816) than by scenario-guided traces (0.84325). The result supports the mechanism that semantic trace structure exposes edge-case defect classes that validity-preserving random replay avoids, but it does not constitute evidence that LLM-generated traces would exhibit the same advantage.

## Introduction

Trace-replay testing—executing recorded sequences of operations against a system under test and checking invariants—is a well-established technique for validating stateful systems. In financial software, double-entry ledger implementations are particularly attractive targets because they admit strong invariants (balanced accounts, non-negative balances where required) that can be checked after each operation.

A practical question for trace-replay testing is: what traces should be replayed? Randomly generated traces are easy to produce but may systematically avoid semantically interesting edge cases such as overdraft attempts, self-transfers, or reversal of already-reversed operations. Conversely, traces that deliberately target semantic edge cases may detect defect classes that random traces miss, but at the cost of higher invalid-operation rates and more complex generation logic.

This study asks a narrowly scoped question: in a deterministic synthetic double-entry ledger mutation benchmark, do semantically structured trace scenarios detect more seeded ledger defects than naive random, state-aware random, and no-reversal ablation baselines? We use a scripted scenario-guided generator as a local proxy for the kind of accounting workflow traces an LLM might produce when prompted for realistic ledger scenarios. We do not evaluate actual LLM outputs in this study.

## Method

### Ledger Oracle

We implemented a deterministic double-entry ledger oracle in Python (`src/ledger_trace_benchmark.py`) supporting five operation types:

- **Deposit**: credit a single account.
- **Withdrawal**: debit a single account, subject to an overdraft check.
- **Transfer**: debit one account and credit another, subject to overdraft check on the sender.
- **Fee**: debit an account and credit a revenue account.
- **Reversal**: reverse a prior operation by reference, subject to reference validity.

The oracle enforces double-entry balance invariants after every operation and rejects invalid operations (e.g., overdrafts, bad reversal references) while recording the rejection.

### Seeded Mutants

We seeded 10 mutants into the oracle, each introducing a single defect class:

| Mutant | Defect class |
|---|---|
| `no_overdraft_check` | Withdrawal/transfer ignores insufficient funds |
| `transfer_allows_self` | Transfer permits sender == receiver |
| `fee_wrong_account` | Fee debits wrong account |
| `rounding_error` | Systematic rounding drift |
| `withdraw_off_by_one` | Withdrawal debits one unit less than requested |
| `bad_ref_accepted` | Reversal accepts invalid reference |
| `reverse_deposit_wrong_sign` | Reversal of deposit posts with wrong sign |
| `reverse_fee_no_revenue_debit` | Reversal of fee omits revenue-side debit |
| `reverse_wrong_op_type` | Reversal misidentifies operation type |
| `double_reverse_allowed` | Reversal of a reversal is permitted |

These mutants cover overdraft checks, fee posting, transfer semantics, rounding, bad reversal references, and reversal posting errors.

### Trace Generators

We compared four trace generators:

1. **Naive random control** (`naive_random_control`): Unconstrained random operation selection from the five types with random parameters. No state awareness.

2. **State-aware random baseline** (`state_random_baseline`): Random operation selection conditioned on current ledger state to produce mostly valid operations (e.g., only withdrawing from accounts with sufficient balance, only reversing existing references).

3. **Scenario-guided** (`scenario_guided`): Deterministic semantic scenario generator producing structured sequences targeting overdrafts, exact-balance withdrawals, self-transfers, bad reversal references, fee reversals, and deposit reversals. This generator is a scripted proxy for LLM-produced accounting workflow traces.

4. **No-reversal ablation** (`scenario_no_reversal_ablation`): Identical to scenario-guided but with all reversal scenarios removed. This ablation directly tests whether reversal scenarios are responsible for detecting reversal-specific mutants.

### Experimental Protocol

**Main run:** 20,000 traces per generator, length 200, seed start 424242. This produces 800,000 mutant–trace replay rows (20,000 traces × 10 mutants × 4 generators).

**Length-robustness sweep:** 5,000 traces per generator at lengths 50, 100, and 400, seed start 777000.

**Kill criterion:** A mutant is "killed" by a trace if replaying that trace against the mutant produces a different final state or invariant violation compared to replaying the same trace against the correct oracle.

**Invalid operation rate:** The fraction of operations within a generator's traces that the correct oracle rejects (e.g., overdraft attempts, bad references). This measures how much of a generator's budget is spent on semantically invalid requests.

**Statistical analysis:** Wilson 95% confidence intervals for kill rates and two-proportion z-scores for pairwise contrasts, computed in `src/analyze_results.py`.

**Environment:** Python 3.13.11, 117 GiB available memory, swap disabled. Main run completed in 174.4 seconds.

## Results

### Main Run (Length 200, 20,000 Traces)

| Generator | Kill rate | 95% CI | Invalid op rate |
|---|---:|---:|---:|
| Scenario-guided | 0.98407 | [0.98351, 0.98461] | 0.20157 |
| Naive random | 0.92351 | [0.92233, 0.92466] | 0.49050 |
| No-reversal ablation | 0.68009 | [0.67804, 0.68213] | 0.10356 |
| State-aware random | 0.67081 | [0.66875, 0.67287] | 0.01463 |

Scenario-guided traces achieve the highest kill rate (98.4%), a +31.3 percentage point lift over state-aware random (z = 262.16), a +30.4 percentage point lift over the no-reversal ablation (z = 257.17), and a +6.1 percentage point lift over naive random (z = 91.23). All contrasts are statistically significant at any conventional threshold.

Naive random achieves a surprisingly high kill rate (92.4%) but at the cost of a 49.1% invalid operation rate—nearly half of all operations are rejected by the oracle. State-aware random has the lowest invalid operation rate (1.5%) but also the lowest kill rate (67.1%), because it systematically avoids the edge-case operations that expose mutants.

### Per-Mutant Analysis

Scenario-guided traces killed 9 of 10 mutants at a rate ≥ 0.99755. The weakest target was `withdraw_off_by_one` at 0.84325, which was above naive random's rate for that mutant (0.44315) but below state-aware random's rate (0.9816). This one mutant represents a case where the scenario-guided generator's specific edge-case targeting was less effective than the state-aware generator's broader valid-operation coverage for that particular defect class.

The no-reversal ablation killed 0.0 of three mutants: `bad_ref_accepted`, `reverse_deposit_wrong_sign`, and `reverse_fee_no_revenue_debit`. This directly confirms that reversal scenarios are the mechanism behind detection of these reversal-specific faults.

The state-aware random baseline killed 0.0 of three mutants: `no_overdraft_check`, `transfer_allows_self`, and `bad_ref_accepted`. Because state-aware random conditions on valid state, it systematically avoids the invalid operations (overdrafts, self-transfers, bad references) that would expose these defects.

### Length-Robustness Sweep

| Length | Scenario-guided | Naive random | State-random | No-reversal ablation |
|---:|---:|---:|---:|---:|
| 50 | 0.93940 | 0.68832 | 0.58054 | 0.66984 |
| 100 | 0.97786 | 0.84426 | 0.63204 | 0.67746 |
| 200 | 0.98407 | 0.92351 | 0.67081 | 0.68009 |
| 400 | 0.98244 | 0.95508 | 0.69368 | 0.67854 |

The scenario-guided advantage persists across all tested lengths. At length 50, the advantage over state-aware random is +35.9 percentage points; at length 400, it is +28.9 percentage points. The scenario-guided kill rate appears to plateau near 98% at length 200, while naive random continues to improve with length (reaching 95.5% at length 400), suggesting that naive random eventually achieves broad coverage through volume but at much higher invalid-operation cost.

The no-reversal ablation and state-aware random baselines show limited improvement with increasing length, plateauing near 68% and 69% respectively, consistent with their systematic avoidance of edge-case operations.

Across the main run and all sweeps, the benchmark produced approximately 1.4 million mutant–trace replay rows and roughly 270 million mutant operation applications.

## Limitations

This study has several important limitations that constrain the generality of its conclusions:

1. **No actual LLM-generated traces.** The scenario-guided generator is a deterministic scripted proxy. It was designed to approximate the kind of semantically structured traces an LLM might produce when prompted for accounting workflows, but no LLM was called during this study. Whether actual LLM-generated traces exhibit the same kill-rate advantage remains untested.

2. **Single toy oracle.** Only one double-entry ledger implementation was evaluated. The oracle is intentionally simple (five operation types, no multi-currency, no time-dating, no partial settlements). Real ledger systems are substantially more complex, and the defect classes encountered in production may differ from the 10 seeded mutants.

3. **Small mutant set.** Only 10 seeded mutants were tested. Mutation testing is a proxy for real defect detection, and the specific mutants chosen determine which generator strategies appear effective. A larger and more diverse mutant set could change the relative rankings.

4. **One defect class where scenario-guided underperforms.** The `withdraw_off_by_one` mutant was killed at a lower rate by scenario-guided traces (0.84325) than by state-aware random (0.9816). This illustrates that semantic targeting is not uniformly superior; for some defect classes, broad valid-operation coverage may be more effective.

5. **No cross-implementation portability evidence.** The results apply only to the specific oracle and mutant set tested. Whether the same relative advantage holds for other ledger implementations is unknown.

6. **Invalid operation rate trade-off.** Scenario-guided traces have a 20.2% invalid operation rate, meaning roughly one in five operations is rejected by the oracle. While lower than naive random's 49.1%, this still represents a substantial fraction of wasted execution budget. The optimal trade-off between invalid operation rate and kill rate depends on the cost model of the target deployment.

7. **Claim ledger audit status.** The automated claim ledger for this artifact reports `blocked_empty_claims` with no structured claims extracted. The results reported here are drawn directly from the run notes and result files, not from a passed claim/evidence audit.

## Conclusion

In a deterministic synthetic double-entry ledger mutation benchmark with 10 seeded mutants, semantically structured trace scenarios detect substantially more defects than state-aware random (+31.3 pp), no-reversal ablation (+30.4 pp), and naive random (+6.1 pp) baselines at trace length 200. The advantage persists across lengths 50, 100, 200, and 400. The no-reversal ablation directly confirms that reversal scenarios are responsible for detecting reversal-specific faults, killing zero mutants in three reversal-dependent defect classes.

However, these results are bounded by significant scope limitations. The scenario-guided generator is a scripted proxy, not an actual LLM. Only one toy oracle and 10 mutants were tested. One mutant (`withdraw_off_by_one`) was better detected by state-aware random than by scenario-guided traces, illustrating that semantic targeting is not uniformly superior. Whether the observed advantage transfers to LLM-generated traces, real ledger implementations, or production defect distributions remains an open empirical question.

The supported mechanism—that semantic trace structure exposes edge-case defect classes that validity-preserving random replay avoids—provides useful engineering evidence for ledger trace-replay benchmark design. A direct validation using actual LLM-generated traces against richer ledger implementations is needed before drawing conclusions about LLM-generated trace effectiveness.
