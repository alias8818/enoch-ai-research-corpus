## Abstract

Speculative decoding accelerates autoregressive inference by drafting candidate tokens with a smaller model and verifying them against the target. In long-context serving, however, the key-value (KV) caches of both target and draft models compete for the same finite GPU memory budget. A fixed speculative depth $k$ can push the system into out-of-memory (OOM) risk under high context lengths, or waste compute on low-acceptance speculation when pressure is low. We propose a KV-pressure speculation governor that dynamically adjusts speculative depth based on projected KV memory pressure and an exponential moving average (EMA) of draft acceptance rate. In a synthetic, GB10-calibrated analytic simulator spanning 80 scenarios (70B-class target, 3B draft), the governor reduces OOM-risk scenarios from 32 (fixed $k{=}8$) to 24, while winning 36 of 48 comparable scenarios with an average relative throughput gain of 44.65% and zero losses. For n-gram-style speculation without a draft model, the governor does not improve OOM-risk reduction; its value is limited to throttling low-acceptance waste, which remains unquantified. All results are synthetic. Scientific closure requires backend-integrated measurements on a serving engine under production workloads.

## Introduction

Speculative decoding reduces per-token latency in autoregressive language model serving. A small draft model proposes $k$ candidate tokens; the target model verifies them in a single forward pass. When acceptance rates are high, the effective tokens-per-forward-pass ratio increases substantially; when acceptance rates are low, the overhead of drafting and rejecting dominates, potentially reducing throughput below the non-speculative baseline.

Existing deployments typically configure speculative depth as a static parameter. This is adequate for short-context workloads where the KV cache occupies a small fraction of GPU memory. In long-context scenarios—multi-turn conversations, document-grounded retrieval, large-context reasoning—the target model's KV cache alone can consume most of the available memory. Adding a draft model's KV cache at a fixed depth can push the system past its memory budget, causing OOM errors or forcing admission control to reject requests.

Conversely, when context is short and memory is abundant, a conservative fixed $k$ leaves throughput unrealized. The optimal speculative depth depends on both the current memory pressure and the recent acceptance behavior of the draft model.

This paper introduces a **KV-pressure speculation governor**: a runtime policy that adjusts speculative depth as a function of (1) projected KV memory usage relative to the available budget and (2) an EMA of the draft acceptance rate. The governor throttles depth downward when memory pressure is high or acceptance is low, and expands depth when both conditions are favorable.

We evaluate the governor in a dependency-free analytic simulator calibrated against CUDA memory allocation benchmarks on an NVIDIA GB10 platform. We compare against a fixed $k{=}8$ baseline across 80 scenarios varying context length, batch size, and memory budget. We also examine the n-gram (no-draft-model) regime.

Our contributions are:

1. A concrete policy specification for KV-pressure-driven speculative depth adjustment, with defined integration points for serving engines.
2. Synthetic evidence that the governor reduces OOM-risk scenarios by 25% (from 32 to 24) relative to fixed $k{=}8$, while improving throughput in 36 of 48 comparable scenarios with zero losses.
3. A negative result: for n-gram speculation, the governor does not improve OOM-risk reduction, though it may provide value in throttling low-acceptance waste—a benefit that remains unquantified.
4. An explicit delineation of the gap between synthetic simulation and backend-integrated validation.

## Method

### Platform Calibration

Before constructing the simulator, we validated the memory allocation characteristics of the target platform. The host is an NVIDIA GB10 system running Linux 6.17 with 127.5 GB RAM, zero swap (`SwapTotal: 0`), and CUDA 13 driver. We wrote a CUDA smoke benchmark (`cuda_mem_smoke.cu`) that allocates and touches (writes to) a specified amount of device memory, measuring effective touch bandwidth.

| Allocation | Result | Effective Bandwidth |
|-----------|--------|-------------------|
| 256 MiB   | Pass   | ~101 GB/s         |
| 4,096 MiB | Pass   | ~97 GB/s          |
| 8,192 MiB | Pass   | ~97 GB/s          |
| 16,384 MiB| Pass   | ~109 GB/s         |
| 32,768 MiB| Pass   | ~109 GB/s         |

All allocations succeeded up to 32 GiB. The system has zero swap, confirming that the memory budget is strictly bounded by physical device memory. The `nvidia-smi` tool reports the GB10 device but returns `Not Supported` for memory usage queries, so we relied on direct CUDA allocation for calibration. These results confirm that the platform can allocate and touch large contiguous device memory regions at consistent bandwidth, but they do not validate the simulator's KV cache model or the governor's runtime behavior.

### Analytic Simulator

We implemented `kv_pressure_governor.py`, a dependency-free analytic simulator that models:

- **Target model**: 70B-class, with per-layer KV cache size parameterized by hidden dimension, number of layers, and data type.
- **Draft model**: 3B-class, with proportionally smaller KV cache.
- **Context length**: Varied across scenarios.
- **Batch size**: Varied across scenarios.
- **Memory budget**: Varied across scenarios, derived from the GB10 calibration under the no-swap assumption.
- **Acceptance rate**: Modeled as a function of speculative depth and context characteristics.

The simulator computes whether a given (context length, batch size, memory budget, speculative depth) configuration fits within the memory budget, and estimates throughput as a function of acceptance rate and depth. The throughput model assumes that accepted tokens contribute linearly to throughput and that rejected draft tokens impose a fixed overhead proportional to depth. This is a simplification; real serving engines exhibit non-linear cost structures due to batching, paging, and scheduling interactions.

### Governor Policy

The governor policy is specified in `.omx/specs/kv-pressure-governor/spec.md`. At a high level:

1. **Pressure estimation**: At each scheduling decision point, compute projected KV memory usage for the target model, the draft model (if present), and any paged cache overhead. Compare against the available memory budget to obtain a pressure ratio $p = \text{projected\_kv} / \text{budget}$.

2. **Acceptance EMA**: Maintain an exponential moving average $\alpha$ of the draft acceptance rate over recent verification steps.

3. **Depth selection**: Choose speculative depth $k$ as a monotonically increasing function of $(1 - p)$ and $\alpha$, clamped to $[k_{\min}, k_{\max}]$. When $p$ exceeds a threshold, $k$ is reduced to $k_{\min}$ or speculation is disabled entirely. When $\alpha$ is low (indicating the draft model is not well-aligned with the target), $k$ is also reduced to avoid wasted compute.

4. **Routing fallback**: If the backend does not support runtime depth changes, the governor degrades to an admission/routing policy that decides whether to enable speculation at all for a given request.

The policy requires two runtime telemetry signals: projected KV memory usage (or the components from which it can be computed) and per-step draft acceptance statistics. Both signals are exposed in current serving engines: vLLM exposes `kv_cache_memory_bytes` and `gpu_memory_utilization` in its cache configuration, and llama.cpp exposes `--spec-draft-n-max` and acceptance statistics in its speculative decoding mode.

### Scenario Grid

We generated 80 scenarios in `artifacts/metrics/kv_governor/scenario_grid.csv`, varying context length, batch size, and memory budget around realistic operating points for a 70B target + 3B draft configuration on a GB10-class device.

For each scenario, we evaluated:
- **Fixed $k{=}8$**: Whether the configuration fits in memory, and estimated throughput.
- **Governor**: Whether the configuration fits in memory (with dynamically chosen $k$), and estimated throughput.

We also evaluated an n-gram speculation regime (`artifacts/metrics/kv_governor_ngram/summary.json`) where no draft model KV cache is present.

## Results

### OOM-Risk Reduction

Across the 80 scenarios:

| Policy | OOM-Risk Scenarios |
|--------|-------------------:|
| Fixed $k{=}8$ | 32 |
| Governor | 24 |

The governor reduces OOM-risk scenarios by 8 (a 25% relative reduction). This reduction occurs because the governor throttles speculative depth when memory pressure is high, shrinking the draft model's KV cache footprint and keeping the total within budget. The 24 remaining OOM-risk scenarios represent configurations where even the minimum speculative depth (or no speculation) exceeds the memory budget—these are fundamentally infeasible regardless of policy.

### Throughput Comparison (Both-Fit Scenarios)

Of the 80 scenarios, 48 are scenarios where both the fixed-$k$ policy and the governor fit within the memory budget. In these comparable scenarios:

| Metric | Value |
|--------|------:|
| Governor wins | 36 |
| Governor losses | 0 |
| Ties | 12 |
| Average relative throughput gain | +44.65% |

The governor wins in 75% of comparable scenarios and loses in none. The 12 ties correspond to scenarios where the governor selects $k{=}8$ (the same as the fixed policy), which occurs when memory pressure is low and acceptance is high—precisely the conditions under which fixed $k{=}8$ is already optimal.

The 44.65% average throughput gain arises from the governor's ability to increase $k$ beyond 8 when memory is abundant and acceptance is high, and to reduce $k$ below 8 when acceptance is low (avoiding wasted draft compute), while the fixed policy cannot adapt. This figure should be interpreted cautiously: it is a property of the synthetic throughput model, not of a real serving engine. The magnitude of the gain depends on the simulator's assumptions about the cost of rejected draft tokens and the benefit of additional accepted tokens, neither of which has been validated against real measurements.

### N-Gram Speculation

For the n-gram regime (no draft model, token-level n-gram speculation):

- **OOM-risk reduction**: The governor does not improve OOM-risk reduction relative to fixed n-gram speculation. This is expected: n-gram speculation does not allocate a draft model KV cache, so the memory savings from reducing depth are negligible.
- **Low-acceptance throttling**: The governor still provides value by reducing speculative depth when the n-gram acceptance rate is low, avoiding wasted verification compute. However, quantifying this benefit requires real backend measurements, as the synthetic model does not capture the actual cost structure of n-gram verification in a serving engine.

This is a genuinely negative result for the OOM-reduction dimension: the governor's primary advantage—reducing draft KV footprint—does not apply when there is no draft model.

## Limitations

1. **Synthetic simulation only.** All results derive from an analytic simulator, not from a real serving engine. The simulator makes simplifying assumptions about KV cache layout, memory fragmentation, paging overhead, and the cost of depth changes. Real-world performance may differ substantially, and the 44.65% throughput figure in particular should not be treated as a prediction of real-world speedup.

2. **No backend-integrated measurements.** The governor has not been implemented in vLLM, llama.cpp, or any other serving engine. The policy spec identifies integration points, but the runtime cost of depth adjustment, the latency of acceptance-rate feedback, and the interaction with continuous batching have not been measured.

3. **N-gram regime is inconclusive.** The governor's value in n-gram speculation is limited to low-acceptance throttling, and this benefit could not be quantified in the synthetic model. Backend measurements are required.

4. **Single hardware profile.** All calibration was performed on a single GB10 host with zero swap. Results may not generalize to other GPU architectures, multi-GPU configurations, or systems with swap or CPU offloading.

5. **Static acceptance model.** The simulator models acceptance rate as a function of depth and context, but does not capture the dynamic, workload-dependent nature of acceptance in real deployments. The EMA smoothing parameter was not tuned on real data.

6. **No multi-tenant evaluation.** The scenario grid varies batch size but does not model the complex scheduling interactions of a multi-tenant serving cluster with mixed context lengths and arrival rates.

7. **Pressure-only gating is insufficient.** A governor that considers only memory pressure (without acceptance telemetry) would fail to distinguish high-value from low-value speculation. Both signals are necessary. This is a design constraint, not merely an implementation detail.

8. **Runtime depth change may not be supported.** If the serving engine does not support changing speculative depth at runtime, the governor degrades to an admission/routing policy. The throughput implications of this degradation have not been evaluated.

9. **Claim audit status.** The structured claim ledger for this artifact is currently empty (`audit_status: blocked_empty_claims`). No formal claim-evidence audit has been completed. The results reported here should be treated as preliminary and unverified by independent audit.

## Conclusion

A KV-pressure speculation governor that dynamically adjusts speculative depth based on memory pressure and draft acceptance rate is a viable approach for long-context LLM serving, subject to the constraint that both telemetry signals are available and the backend supports runtime depth changes. In synthetic evaluation across 80 scenarios calibrated to an NVIDIA GB10 platform, the governor reduces OOM-risk scenarios by 25% relative to a fixed $k{=}8$ baseline and achieves an average throughput gain of 44.65% in scenarios where both policies fit, with zero throughput losses.

However, these results are bounded by significant caveats. The evaluation is entirely synthetic; the throughput gains are properties of the simulator's cost model, not measurements. The n-gram regime shows no OOM-risk improvement. The governor requires both memory-pressure and acceptance-rate telemetry, and it requires a serving backend that supports runtime speculative depth changes. Without these, the policy degrades to an admission/routing decision whose performance has not been evaluated. The structured claim audit has not been completed.

The path to scientific closure requires: (1) implementation of the governor in a real serving engine (vLLM or llama.cpp), (2) measurement of the runtime cost of depth adjustment under continuous batching, (3) evaluation on production-representative workloads with varying context lengths and arrival patterns, and (4) comparison against alternative adaptive policies (e.g., acceptance-only throttling, cost-model-based scheduling). Until such measurements are available, the governor should be regarded as a promising but unvalidated prototype.
