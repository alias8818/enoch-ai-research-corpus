## Abstract

Speculative decoding accelerates autoregressive language model inference by drafting candidate token sequences and verifying them against the target model in a single forward pass. We evaluate whether a code-aware trace and suffix cache—exploiting the high degree of repeated identifiers, imports, braces, and boilerplate in code workloads—can serve as a viable drafting source for speculative and lookahead decoding. Using an oracle upper-bound simulation over held-out Python standard library source files (152,514 regex tokens from 20 files), we measure the reduction in required forward passes under static, online, and combined cache modes. The combined mode with a maximum draft length of 16 tokens achieves an oracle forward-pass reduction of 1.33×, with a proposal hit rate of 0.963 and an accepted token rate of 0.457. Static and online modes alone yield 1.12× and 1.26× reductions, respectively. These results are oracle upper bounds obtained under idealized candidate selection and regex-level tokenization; they do not represent target-model tokenizer IDs, implementable candidate ranking, or measured wall-clock speedup. The gap between these oracle figures and achievable production speedup remains unquantified. We discuss the limitations in detail and outline the steps required to reach a validated prototype.

## Introduction

Autoregressive language model inference is memory-bandwidth-bound: each generated token requires a full forward pass through the model, even when the output is highly predictable. Speculative decoding addresses this bottleneck by generating draft token sequences from a cheaper source and verifying them in a single target-model forward pass, accepting all tokens up to the first mismatch.

Code generation and editing workloads exhibit strong local repetition: the same import blocks, function signatures, closing braces, and identifier patterns recur within and across files in a repository. This raises the question of whether a cache of previously observed token suffixes—built from code traces—can serve as an effective drafting source, potentially without requiring a separate draft model.

We investigate this question through an oracle upper-bound evaluation. Rather than building a full speculative decoding system, we measure the best-case reduction in forward passes achievable if the drafter always selects the optimal cached suffix for each position. This establishes a ceiling on the utility of any suffix-cache-based drafting strategy and informs whether further implementation investment is warranted.

Our evaluation uses Python standard library source code as a proxy for code workloads with repetitive structure. We compare three cache modes: *static* (cache built before inference from training files), *online* (cache updated during inference from observed tokens), and *combined* (both). We measure proposal hit rates, accepted token rates, and the resulting oracle forward-pass reduction. A separate llama.cpp smoke test provides a reference point for target-model inference throughput on the evaluation hardware but was not integrated with the cache drafter.

## Method

### Benchmark Design

We constructed a local benchmark (`scripts/code_trace_suffix_bench.py`) that performs the following steps:

1. Scans Python source files from `/usr/lib/python3.12`, limited to 20 files.
2. Splits files into a training set (used to build the static cache) and a held-out evaluation set.
3. For each position in the held-out files, queries the suffix cache for candidate continuations of varying length (up to `max_draft` tokens).
4. Under oracle selection, chooses the candidate that maximizes the number of accepted tokens (i.e., the longest prefix match against the ground-truth continuation).
5. Records the number of forward passes that would have been required with and without speculative drafting.

This is a toy simulation: no model inference occurs, no target-model verification is performed, and the "forward passes" counted are a theoretical construct derived from the oracle acceptance pattern.

### Tokenization

Tokens were produced by regex-based splitting rather than a target-model tokenizer (e.g., BPE). This is a significant limitation. Regex tokens do not correspond to the token vocabulary of any real target model, and hit rates measured at the regex-token level may not transfer to model-tokenizer-level hit rates. The direction and magnitude of the bias introduced by this choice are unknown. We treat this as a known source of unquantified error and discuss implications in the Limitations section.

### Cache Modes

- **Static**: The cache is populated from training files before evaluation begins and is not updated during inference. This mode tests whether cross-file repetition in the training set provides useful drafting candidates.
- **Online**: The cache starts empty and is populated only from tokens observed during the evaluation run itself. This mode tests whether intra-file local repetition provides useful drafting candidates.
- **Combined**: The cache is pre-populated from training files and is also updated online during evaluation. This mode captures both cross-file and intra-file repetition.

### Oracle Candidate Selection

At each position, if multiple cached suffixes match the context prefix, the oracle selects the candidate whose continuation has the longest prefix match with the ground truth. This is not implementable in practice—it requires knowledge of the ground truth—but it establishes the upper bound on drafting utility for any ranking heuristic.

### LLM Baseline

A separate smoke test (`scripts/llama_smoke.py`) measured single-stream inference throughput for `phi4mini` via `llama.cpp` on the evaluation machine, yielding 32.33 tokens/second. This provides a reference point for target-model inference speed on the evaluation hardware. It was not integrated with the cache drafter and does not represent a speculative decoding measurement.

### Environment

The evaluation was conducted on a machine with at least 116 GB available memory (minimum observed `MemAvailable`: 121,723,856 KB), zero swap, and no GPU constraints reported during the benchmark. The benchmark is CPU-only and does not involve model inference. The llama.cpp baseline was measured on the same machine.

## Results

### Oracle Forward-Pass Reduction

| Mode | Oracle Forward-Pass Reduction (×) |
|------|-----------------------------------|
| Static | 1.12 |
| Online | 1.26 |
| Combined | 1.33 |

The combined mode with `max_draft=16` achieves the best oracle upper bound of 1.33× reduction in forward passes over the held-out evaluation set of 152,514 regex tokens. This means that, under oracle selection, the number of forward passes required to generate the held-out corpus would be reduced by approximately 25% (from N to N/1.33).

### Hit and Acceptance Rates

Under the best configuration (combined, `max_draft=16`):

- **Proposal hit rate**: 0.963 — the fraction of positions at which the cache contained at least one candidate suffix matching the context prefix.
- **Accepted token rate**: 0.457 — the fraction of drafted tokens (among all drafted tokens in proposals that hit) that matched the ground truth.

The high proposal hit rate indicates that the cache frequently finds a context match in this domain. The moderate accepted token rate indicates that while a match is found, the cached continuation often diverges from the actual continuation within a few tokens, limiting the effective draft length.

### Mode Comparison

The online mode (1.26×) substantially outperforms the static mode (1.12×), suggesting that intra-file local repetition within the evaluation files is a stronger drafting signal than cross-file repetition from the training set. The combined mode (1.33×) captures both effects, with the improvement over online-only being modest (0.07× absolute), indicating diminishing returns from adding the static cache on top of the online cache.

### LLM Baseline

The `llama.cpp` smoke test with `phi4mini` achieved 32.33 tokens/second on the evaluation hardware. This measurement was not integrated with the cache drafter and serves only as a reference for target-model inference speed. No speculative decoding wall-clock measurement was performed.

## Limitations

This section enumerates the known limitations of the current evaluation. Several of these represent unquantified sources of bias whose direction and magnitude are unknown.

### Oracle Selection Is Not Implementable

The reported forward-pass reductions assume oracle candidate selection—always choosing the cached suffix that maximizes accepted tokens. Any implementable ranking heuristic (frequency, recency, longest context match) will perform at most as well and likely strictly worse. The gap between oracle and heuristic performance is unknown and must be measured empirically. This limitation introduces optimistic bias of unknown magnitude.

### Regex Tokenization vs. Model Tokenizer

Tokens were produced by regex splitting, not by the target model's BPE tokenizer. This affects results in two ways:

1. **Token boundaries differ.** A regex token like `def` may correspond to a single BPE token or be split across multiple BPE tokens, changing the effective draft length and acceptance dynamics.
2. **Vocabulary mismatch.** The cache stores regex-token suffixes that cannot be directly verified by the target model without re-tokenization. The hit and acceptance rates at the BPE level may be higher or lower than those reported here; the direction of the bias is unknown.

Replacing regex tokens with target-model tokenizer IDs is a necessary next step. Until this is done, the reported metrics cannot be interpreted as predictions of real-system behavior.

### No Wall-Clock Speedup Measurement

The reported metric is a reduction in the *number* of forward passes, not wall-clock speedup. Actual speedup depends on:

- The latency of cache lookup relative to a forward pass.
- The overhead of verifying draft tokens in the target model (which may differ from single-token verification latency).
- The distribution of accepted draft lengths, which determines how many tokens are produced per forward pass on average.
- Any overhead from cache maintenance and data structure operations.

A 1.33× reduction in forward passes does not directly translate to 1.33× wall-clock speedup. If cache lookup and draft verification introduce non-trivial overhead, the wall-clock speedup could be substantially lower. This has not been measured.

### Dataset and Domain Specificity

The evaluation uses Python standard library source code. This domain has strong structural regularity (PEP 8 conventions, repeated import patterns, consistent indentation). Other code domains (e.g., JavaScript, C++, or less structured codebases) may exhibit different repetition profiles. Non-code text (natural language, prose) is out of scope. Generalization to other domains is unsupported by the current evidence.

### Small File Sample

The benchmark was limited to 20 files from the Python standard library. While the held-out token count (152,514) is non-trivial, the file sample may not be representative of broader code workloads, particularly those involving repeated agent trajectories or multi-file editing sessions. The variance of the reported metrics across different file samples is unknown.

### No Real Agent/Code Trace Evaluation

The cache is motivated by workloads where the same agent or developer revisits and re-edits code, producing repeated traces. The current evaluation uses static source files, not dynamic editing traces. The degree to which real agent trajectories (e.g., SWE-Bench sessions) exhibit higher suffix reuse is an open question. It is plausible that real editing traces would show higher acceptance rates due to exact repetition of previously generated output, but this has not been demonstrated.

### No Integration with Speculative Decoding Framework

The current evaluation is a standalone simulation. It has not been integrated with any speculative decoding framework (e.g., vLLM, llama.cpp with draft-model support). The interaction between the cache drafter and the target-model verification step—including the cost of re-tokenization, the handling of partial acceptance, and the scheduling of cache lookups—has not been evaluated.

## Conclusion

A code trace suffix cache shows promise as a speculative drafting source for code workloads, with oracle upper-bound forward-pass reductions of up to 1.33× on held-out Python standard library code. The high proposal hit rate (0.963) confirms that code-like text frequently produces context matches in the cache. However, the moderate accepted token rate (0.457) indicates that cached continuations often diverge quickly from the actual output, limiting the draft length that can be verified in a single forward pass.

These results are oracle upper bounds under idealized conditions: regex-level tokenization, oracle candidate selection, and no wall-clock overhead accounting. The achievable speedup in a real system will be lower, and the magnitude of the gap remains unknown. The online cache mode outperforming the static mode suggests that intra-file repetition is the dominant signal, which has implications for cache design in production systems.

The evidence supports continued prototype development—specifically, integration with a target-model tokenizer, replacement of oracle selection with implementable ranking heuristics, and measurement of wall-clock speedup in an actual speculative decoding framework. However, the current results do not justify a production speedup claim. The decision to continue (`promising_continue`, medium confidence) reflects this assessment: the oracle upper bound is non-trivial, but the path from oracle to achievable speedup involves multiple unquantified gaps.
