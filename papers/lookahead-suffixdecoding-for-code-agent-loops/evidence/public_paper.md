## Abstract

SuffixDecoding—a model-free, CPU-side speculative decoding method that proposes token sequences from suffix-tree caches—has been proposed for accelerating high-repetition workloads such as code editing and agentic loops. This study evaluates SuffixDecoding's verifier-step reduction potential on code-agent-style traces using a teacher-forced replay simulator backed by the production `arctic_inference.suffix_decoding.SuffixDecodingCache` implementation (arctic-inference 0.1.2). On synthetic agent/code traces with repetitive plan/inspect/patch/test/result structure, the simulator achieves up to 8.46× step reduction at a maximum speculation width of 32 tokens, with per-lookup CPU latency of approximately 22.6 μs. On local project artifact traces exhibiting less repetition, step reduction reaches 2.53× at the same width. However, these are replay step-reduction ratios—not end-to-end LLM throughput measurements—and the zero-accept step rate remains substantial (38–55%), indicating that realized speedup will depend heavily on target-model verification cost, batch scheduling, and KV-cache behavior. A planned investigation into direct fusion of LookaheadDecoding with SuffixDecoding was deferred after finding no off-the-shelf runtime composing the two systems; LookaheadDecoding operates via a HuggingFace/LLaMA generation patch, while vLLM SuffixDecoding is ArcticInference-backed, making their integration a nontrivial runtime engineering effort. We conclude that SuffixDecoding is a promising candidate for code-agent acceleration but that full throughput validation and Lookahead fusion both require further engineering before performance claims can be closed.

## 1. Introduction

Speculative decoding accelerates autoregressive language model inference by proposing multiple candidate tokens from a fast draft source and verifying them in a single target-model forward pass. When the draft source accurately predicts the target distribution, the verifier accepts long token sequences, reducing the number of target-model forward passes required per generated token.

Two recent approaches are relevant to code and agentic workloads:

**LookaheadDecoding** uses the target model's own Jacobi iteration matrix to draft multiple tokens in parallel, requiring no separate draft model. The LookaheadDecoding paper reports speedups of up to 1.8× on MT-Bench and up to 4× on multi-GPU code completion (arXiv:2402.02057). The reference implementation (`lade` package) currently supports LLaMA-family models and requires HuggingFace generation patching rather than integration with vLLM's speculative decoding pipeline.

**SuffixDecoding**, implemented in ArcticInference and integrated into vLLM 0.17.1 via `speculative_config={"method":"suffix"}`, is a model-free method that builds global and per-request suffix trees from observed token sequences and proposes candidates via CPU-side tree lookups. It is explicitly targeted at high-repetition workloads including code editing and agentic loops, as documented in the vLLM and ArcticInference project documentation.

Code-agent loops—where an LLM repeatedly plans, inspects, patches, and tests code artifacts—exhibit strong repetition patterns across turns: similar function signatures, repeated test harness boilerplate, and recurring plan/inspect/patch/test/result structural tokens. This makes them a natural fit for suffix-tree-based speculation.

The present study asks two questions:

1. What verifier-step reduction does SuffixDecoding offer on code-agent-style traces?
2. Can LookaheadDecoding and SuffixDecoding be composed for further acceleration?

We address question (1) via a replay simulator and report question (2) as a deferral analysis after finding the two systems incompatible at the runtime level without significant engineering.

## 2. Method

### 2.1 Replay Simulator Design

We employ a teacher-forced replay simulator rather than a live LLM inference pipeline. The simulator treats held-out text as the exact target-model output. At each step:

1. The SuffixDecoding cache proposes up to $K$ candidate tokens drawn from prompt context, current response history, and global suffix trees.
2. The simulator accepts the longest prefix of the proposed sequence that matches the held-out output.
3. If no tokens match, the simulator advances by one token (simulating a rejected speculation step where the target model must generate one token at full cost).
4. The process repeats until all held-out tokens are consumed.

Baseline greedy decoding requires one target-model step per output token. The step reduction ratio is defined as:

$$\text{step\_reduction\_x} = \frac{\text{total\_output\_tokens}}{\text{total\_verifier\_steps}}$$

This ratio is an **upper bound** on actual speedup. It does not account for target-model verification overhead, KV-cache management, batch scheduling effects, or draft-proposal latency in a real serving system. A step reduction of $N\times$ means the number of target-model forward passes is reduced by a factor of $N$ in the idealized replay model; real throughput speedup will be lower to the degree that verification cost, scheduling overhead, and zero-accept steps erode the gain.

### 2.2 Implementation

The simulator uses the production `arctic_inference.suffix_decoding.SuffixDecodingCache` class from `arctic-inference==0.1.2`, not a hand-written mock. A correctness smoke test (`scripts/test_suffix_replay.py`) verified that accepted tokens match held-out text exactly before running benchmarks. The smoke test passed (1 passed in 0.03s).

### 2.3 Datasets

Two datasets were constructed:

- **`local_code_agent_artifacts`**: 18 requests, 2,493 tokens. Token traces drawn from this and nearby project markdown, JSON, and code artifacts, capped for a smoke benchmark. These exhibit moderate repetition typical of real project files with varied content.
- **`synthetic_agent_code`**: 17 requests, 3,740 tokens. Generated code-agent traces with repeated plan/inspect/patch/test/result sections and varied filenames and function names. These exhibit high structural repetition designed to approximate the repetitive patterns found in automated code-editing workflows.

Both datasets are small and serve as smoke benchmarks rather than comprehensive evaluations.

### 2.4 Speculation Widths

We sweep maximum speculative tokens $K \in \{4, 8, 16, 32\}$ to characterize how step reduction scales with proposal width.

### 2.5 Environment

- Host: GB10, NVIDIA GPU available (CUDA 13.0, driver 580.159.03)
- Memory: ~122 GB available, swap disabled (SwapTotal: 0 kB)
- Python 3.12.3, virtual environment at `.venv`
- `arctic-inference==0.1.2`
- Benchmark max RSS: ~82 MB; wall-clock completion under 1 second
- No GPU was utilized during the replay benchmark (CPU-only suffix-tree lookups)

## 3. Results

### 3.1 Step Reduction

| Max speculative tokens ($K$) | Overall step reduction | Local artifact step reduction | Synthetic agent/code step reduction | Lookup CPU time (μs) |
|---:|---:|---:|---:|---:|
| 4 | 2.628× | 2.050× | 3.235× | 19.9 |
| 8 | 3.572× | 2.418× | 5.238× | 21.3 |
| 16 | 4.200× | 2.516× | 7.586× | 22.1 |
| 32 | 4.362× | 2.526× | 8.462× | 22.6 |

Step reduction increases with $K$ but exhibits diminishing returns beyond $K = 16$, particularly for the local artifact traces where step reduction plateaus near 2.53×. Synthetic agent/code traces continue to benefit up to $K = 32$, reaching 8.46× step reduction. The overall step reduction (combining both datasets) reaches 4.36× at $K = 32$.

The plateau on local artifact traces suggests that the repetition structure in real project files has a finite effective depth that saturates the suffix-tree proposal mechanism at moderate speculation widths. The synthetic traces, by design, contain deeper repetition patterns that continue to yield returns at higher $K$.

### 3.2 Per-Step Acceptance Statistics

| Metric | Local artifacts | Synthetic agent/code |
|---|---|---|
| Requests | 18 | 17 |
| Total tokens | 2,493 | 3,740 |
| Total verifier steps | 987 | 442 |
| Step reduction | 2.53× | 8.46× |
| Mean accepted per step | 2.54 | 8.46 |
| Mean proposed per step | 9.52 | 22.05 |
| Zero-accept step rate | 54.7% | 38.5% |

The zero-accept step rate—the fraction of verifier steps where no proposed token matched the held-out output—is notably high. Over half of all steps on local artifact traces produce zero accepted tokens, meaning the target model would need to advance one token at full cost with no speculation benefit. On synthetic traces, the rate is lower but still substantial at 38.5%.

This metric is critical for projecting real speedup: each zero-accept step incurs the full cost of a target-model forward pass plus the wasted draft-proposal cost, directly reducing the net acceleration.

### 3.3 CPU Lookup Cost

Per-lookup CPU time ranges from 19.9 μs ($K = 4$) to 22.6 μs ($K = 32$), showing minimal sensitivity to speculation width. This is consistent with suffix-tree lookup being a tree-walk operation whose cost depends on tree depth and branching factor rather than the maximum allowed proposal width. The sub-25 μs latency suggests that CPU-side proposal cost will not be a bottleneck relative to GPU-side target-model verification in a live system.

### 3.4 Resource Usage

The benchmark process consumed a maximum RSS of approximately 82 MB and completed in under 1 second of wall-clock time. No swap was used. System memory available after the smoke test was approximately 122 GB, indicating negligible memory pressure from the suffix-tree cache at this dataset scale.

### 3.5 Lookahead + SuffixDecoding Fusion Analysis

No off-the-shelf runtime was found that composes LookaheadDecoding with ArcticInference/vLLM SuffixDecoding. The two systems occupy separate runtime surfaces:

- **LookaheadDecoding**: Implemented as a HuggingFace/LLaMA generation patch (the `lade` package), requiring custom generation hooks and currently supporting LLaMA-family models. It operates outside the vLLM serving framework.
- **vLLM SuffixDecoding**: Implemented within the vLLM serving framework via ArcticInference, configured through `speculative_config={"method":"suffix"}`. It operates inside vLLM's speculative decoding pipeline.

Merging these would require engineering a unified verification structure that handles both Jacobi-iteration-based drafts from LookaheadDecoding and suffix-tree-based drafts from SuffixDecoding within vLLM's speculative decoding pipeline. This integration was not attempted in this project and is deferred as future work.

It is worth noting that SuffixDecoding already provides adaptive speculation of draft length within vLLM/ArcticInference, which partially overlaps with the "lookahead" concept of proposing multiple tokens in advance. Whether a combined approach would yield additive or multiplicative benefits—and whether the two methods would interfere at the verification level—remains an open question.

## 4. Limitations

1. **Replay simulation, not live inference.** The step-reduction ratios reported here are upper-bound proxies derived from a teacher-forced replay simulator. Actual throughput speedup depends on target-model forward-pass cost, verification kernel overhead, KV-cache behavior, vLLM scheduler effects, and batch composition. The step-reduction ratio may overstate or understate real speedup depending on these factors. No live LLM inference was conducted.

2. **High zero-accept step rate.** A zero-accept rate of 38–55% means a substantial fraction of speculation steps yield no benefit. In a real system, each zero-accept step incurs the full cost of a target-model forward pass plus the wasted draft-proposal cost. The net speedup will be lower than the step-reduction ratio suggests, potentially substantially so if verification cost dominates.

3. **Small, partially synthetic datasets.** The benchmark uses 35 requests totaling 6,233 tokens. The synthetic traces are designed to exhibit high repetition and may overestimate performance on real agent workloads with more diverse output. The local artifact traces, while drawn from real project files, are a convenience sample of limited size and may not be representative of broader code-agent workloads.

4. **No target-model verification.** The simulator assumes the draft proposal is either accepted or rejected based on exact match with held-out text. In a real speculative decoding system, the target model's verification pass may reject tokens that the replay simulator accepts (if the target distribution diverges from the held-out text), or accept tokens via probabilistic resampling that the simulator would reject. The replay model does not capture this distributional dynamics.

5. **Lookahead fusion untested.** The deferral of LookaheadDecoding integration means we cannot report on whether composing the two methods yields additive or multiplicative benefits, or whether they interfere at the verification level. This remains an open question.

6. **Single hardware configuration.** All results are from a single host (GB10) with no GPU utilization during the replay benchmark. CPU lookup times may differ under concurrent load. No multi-GPU or multi-request scheduling effects were tested.

7. **No batch-level analysis.** The replay simulator processes requests sequentially. In a real serving system, batch composition and scheduler decisions affect both speculation effectiveness and throughput. These effects are not captured.

8. **Claim audit status.** The claim ledger for this artifact reports `blocked_empty_claims` with no structured claims extracted. This draft should not be treated as having passed a strict claim/evidence audit.

## 6. Conclusion

SuffixDecoding demonstrates meaningful verifier-step reduction on code-agent-style traces in a replay simulator: 2.53× on local project artifacts and 8.46× on synthetic repetitive agent/code traces at $K = 32$, with CPU lookup latency of approximately 22.6 μs per call. These results are consistent with SuffixDecoding's design intent for high-repetition workloads and with prior reports of agentic/SWE-Bench gains from the ArcticInference and vLLM communities.

However, three important caveats temper these findings:

1. **The step-reduction ratio is an upper-bound proxy, not a measured throughput speedup.** The high zero-accept step rate (38–55%) will reduce real-world gains, potentially substantially, depending on the ratio of verification cost to baseline generation cost.

2. **The datasets are small and partially synthetic.** The synthetic traces are designed to exhibit high repetition and may overestimate performance on more diverse real agent workloads. The local artifact traces, while drawn from real files, constitute a convenience sample of only 18 requests and 2,493 tokens.

3. **The planned investigation into composing LookaheadDecoding with SuffixDecoding was deferred** after finding that the two systems operate on incompatible runtime surfaces (HuggingFace/LLaMA patch vs. vLLM/ArcticInference), and no public implementation composes them. Whether a combined approach would yield additive, multiplicative, or interfering benefits remains unknown.

The viable path forward is a suffix-first evaluation: install the full vLLM 0.17.1 stack with a small code-oriented model on GB10, run an A/B comparison with and without `speculative_config={"method":"suffix","num_speculative_tokens":16}`, and measure actual throughput, accepted-token rates, latency, and GPU utilization. Only after closing the gap between replay step reduction and measured throughput should Lookahead fusion be revisited as a runtime engineering effort.
