## Abstract

Tool-using language model agents frequently face prompts that superficially resemble tool-call requests but in fact require direct answers. We term these *near-miss tool calls* and investigate whether they expose a tool-overcall failure mode that is invisible under standard aggregate accuracy evaluation. We construct a synthetic and weak-label dataset of 2,500 examples spanning four source types and five blind-spot categories (quoted tool requests, explicit no-tool constraints, provided observations, hypothetical search plans, and stale/current disambiguation), and evaluate five deterministic routing policies. We find that a similarity-centric policy achieves 0.902 accuracy on ordinary non-near-miss examples but only 0.182 on near-miss examples—a 72-point collapse. A targeted abstention rule recovers near-miss accuracy to 0.998 while preserving tool-needed recall at 0.854, yielding an 81.6-point uplift over the similarity-centric baseline (stability range: 79.8–82.5 points across 8 seeded runs). The near-miss slice reverses system rankings in every stability seed. These results constitute moderate positive evidence that near-miss examples expose a distinct and actionable failure signal. However, all data are synthetic or weak-model-generated rather than drawn from production logs, and the findings have not been validated on real agent traces. Confidence in the hypothesis is assessed as medium.

## 1. Introduction

Language model agents equipped with tool-calling capabilities must decide, for each user prompt, whether to invoke an external tool or respond directly. Standard evaluation of such agents typically reports aggregate accuracy across a mixed corpus, implicitly assuming that errors are distributed uniformly across prompt types. This assumption may mask systematic failures on specific prompt categories—particularly prompts that contain surface cues associated with tool use (keywords like "search," "look up," or "find") but where the correct action is to answer directly.

We define a *near-miss tool call* as a prompt that a tool-cue-driven policy would route to a tool call, but where the correct label is a direct answer. Examples include: prompts that quote a tool name without requesting its invocation, prompts that explicitly state no tool is needed, prompts that already contain sufficient observations, hypothetical search plans that should not be executed, and questions about stale versus current information where the answer is already available.

The central hypothesis of this work is that near-miss examples expose a tool-overcall failure mode that is hidden by ordinary aggregate accuracy, and that a simple targeted abstention rule can substantially recover performance on this slice without catastrophically degrading tool-needed recall.

We report results from a two-phase experiment: an initial pilot with 1,500 synthetic examples and an augmented run with 2,500 examples incorporating weak-model traces and structured blind-spot categories. The project decision is `finalize_positive` with hypothesis status `supported`, evidence strength `moderate`, and confidence `medium`.

## 2. Method

### 2.1 Dataset Construction

The dataset was generated by `src/near_miss_experiment.py`, a reproducible MVP generator that produces examples across three classes:

- **`call_tool`**: Prompts that genuinely require tool invocation.
- **`answer_direct`**: Ordinary direct-answer prompts with no tool-cue resemblance.
- **`near_miss`**: Prompts that superficially resemble tool-call requests but whose correct label is a direct answer.

In the augmented run (2,500 examples), each example carries three additional fields:

- **`source`**: One of `synthetic_template` (n=1,419), `weak_model_trace` (n=454), `blind_spot_template` (n=375), or `realistic_trace` (n=252).
- **`blind_spot`**: One of five categories:
  - `explicit_no_tool_constraint` (n=321): Prompts explicitly stating no tool should be used.
  - `provided_observation_sufficient` (n=310): Prompts where the necessary information is already provided.
  - `quoted_tool_request` (n=75): Prompts that mention a tool name without requesting invocation.
  - `stale_current_disambiguation` (n=96): Prompts where stale information suffices and current retrieval is unnecessary.
  - `hypothetical_search_plan` (n=73): Prompts describing hypothetical searches that should not be executed.
- **`trace`**: Whether the example includes a tool-trace context.

The class balance in the augmented dataset is: `answer_direct` = 250, `call_tool` = 625, `near_miss` = 1,625. The near-miss class is intentionally overrepresented to stress-test routing policies.

### 2.2 Routing Policies

Five deterministic routing policies were evaluated:

1. **Similarity-centric**: Routes based on surface-level tool-cue similarity. This policy treats the presence of tool-associated keywords and patterns as evidence for tool invocation.
2. **Trace-naive**: Routes based on trace structure without attending to near-miss cues.
3. **Conservative**: Applies a high threshold for tool invocation, preferring direct answers when uncertain.
4. **Targeted near-miss fix**: Incorporates an explicit abstention rule that detects near-miss patterns (quoted tool names, explicit no-tool constraints, provided observations, hypothetical plans, stale/current cues) and overrides tool-cue signals.
5. A fifth policy included in the evaluation harness (predictions recorded in `results/predictions.jsonl`).

### 2.3 Evaluation Protocol

Each policy was evaluated on accuracy stratified by class (`near_miss`, `call_tool`, `answer_direct`), source, and blind-spot category. System rankings were computed on the full dataset and on the near-miss slice separately. Ranking stability was assessed by re-running the evaluation with 8 different random seeds, recording whether the near-miss slice changed the ranking relative to the non-near-miss slice.

## 3. Results

### 3.1 Pilot Run (1,500 Examples)

The initial pilot established the basic phenomenon:

| Metric | Value |
|--------|-------|
| Near-miss direct-answer lures | 675 |
| True tool-needed prompts | 525 |
| Ordinary direct-answer prompts | 300 |
| Similarity-centric accuracy (ordinary) | 0.902 |
| Similarity-centric accuracy (near-miss) | 0.253 |
| Targeted fix uplift over similarity-centric | +74.7 points |

The near-miss slice changed the system ranking compared with the ordinary non-near slice. Dominant failure clusters were false tool calls on web-search, JSON-extraction, and code-search lures.

### 3.2 Augmented Run (2,500 Examples)

The augmented run confirmed and strengthened the pilot signal:

| Metric | Value |
|--------|-------|
| Similarity-centric near-miss accuracy | 0.182 |
| Trace-naive near-miss accuracy | 0.092 |
| Conservative near-miss accuracy | 0.969 |
| Conservative tool-needed recall | 0.854 |
| Targeted near-miss fix accuracy | 0.998 |
| Targeted fix uplift vs. similarity-centric | 81.6 points |
| Uplift stability range (8 seeds) | 79.8–82.5 points |
| Ranking changed on near-miss slice | True |
| Ranking changed rate (8 seeds) | 1.000 |

Key observations:

1. **Collapse on near-miss**: The similarity-centric policy's accuracy drops from 0.902 on ordinary examples to 0.182 on near-miss examples—a 72-point decline. The trace-naive policy performs even worse at 0.092.

2. **Targeted fix effectiveness**: The targeted near-miss abstention rule achieves 0.998 accuracy on the near-miss slice, an 81.6-point improvement over the similarity-centric baseline, with this uplift stable across all 8 seeded runs (range: 79.8–82.5).

3. **Recall trade-off**: The conservative policy achieves high near-miss accuracy (0.969) but at the cost of reduced tool-needed recall (0.854). The targeted fix achieves higher near-miss accuracy (0.998) while the tool-needed recall figure is not separately reported for this policy in the available metrics, so we cannot confirm whether it avoids the conservative policy's recall penalty.

4. **Ranking instability**: The near-miss slice reversed system rankings in every stability seed (rate = 1.000), indicating that the ranking change is robust to resampling variation within this synthetic dataset.

5. **Blind-spot variation**: The five blind-spot categories showed differential difficulty. The `explicit_no_tool_constraint` (n=321) and `provided_observation_sufficient` (n=310) categories were the most populated, while `hypothetical_search_plan` (n=73) and `quoted_tool_request` (n=75) were less represented, limiting per-category statistical power.

## 4. Limitations

1. **Synthetic and weak-label data**: All 2,500 examples are either rule-generated from templates or produced by weak model traces. None are drawn from production agent logs or human-annotated real interactions. The near-miss phenomenon may manifest differently—or not at all—in real deployment conditions.

2. **No human validation of blind spots**: The five blind-spot categories were designed a priori. Whether these categories correspond to actual failure modes in deployed agents has not been established through human annotation or log analysis.

3. **Deterministic policies only**: The evaluated policies are hand-coded deterministic rules, not learned models. Whether a trained classifier or fine-tuned language model would exhibit the same near-miss vulnerability—and respond to the same targeted fix—remains untested.

4. **Class imbalance by design**: The dataset intentionally overrepresents near-miss examples (1,625 of 2,500). This amplifies the near-miss signal for analysis but does not reflect the natural prevalence of near-miss prompts in any real workload.

5. **Recall trade-off characterization is incomplete**: The tool-needed recall for the targeted near-miss fix policy is not separately reported in the available metrics. The conservative policy's recall penalty (0.854) demonstrates that abstention comes at a cost, but the targeted fix's position on this trade-off curve is not fully characterized.

6. **No external replication**: The experiment was conducted once within a single automated pipeline. No independent replication on different data, models, or hardware has been performed.

7. **Stability testing is internal**: The 8-seed stability test varies the random seed within the same generator and evaluation harness. It does not test stability across different generators, annotation protocols, or model families.

## 6. Conclusion

This work presents moderate positive evidence that near-miss tool-call examples—prompts that superficially resemble tool-invocation requests but require direct answers—expose a tool-overcall failure mode that is invisible under standard aggregate accuracy evaluation. In a synthetic dataset of 2,500 examples, a similarity-centric routing policy collapsed from 0.902 accuracy on ordinary prompts to 0.182 on near-miss prompts, while a targeted abstention rule recovered near-miss accuracy to 0.998 with an 81.6-point uplift that was stable across 8 seeded runs. The near-miss slice reversed system rankings in every stability seed.

These findings are bounded by significant limitations: all data are synthetic or weak-model-generated, blind-spot categories lack human validation, policies are deterministic rather than learned, and no external replication has been performed. The project decision assesses confidence as medium and evidence strength as moderate.

The recommended next step, as recorded in the project decision, is to collect a small human- or log-derived trace sample to validate the weak-generated blind spots against real agent behavior. Until such validation is performed, the results should be interpreted as a proof-of-concept demonstration on synthetic data rather than as evidence of a confirmed phenomenon in deployed systems.
