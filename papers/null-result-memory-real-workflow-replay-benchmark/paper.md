## Abstract

Autonomous coding agents frequently issue probe commands (e.g., `grep`, `find`, `cat`) that return empty or error results—null observations—while exploring unfamiliar codebases. A natural optimization is to memoize these null results and skip re-execution of equivalent probes. This paper evaluates a conservative exact normalized null-result memory on real recorded Enoch/Codex agent workflow logs. Across 475 workflows comprising 3,765 recorded command observations, the memory identified 105 null observations but found zero repeated null keys under normalized command equivalence. The resulting command reduction was 0.00% and elapsed-time savings were 0 ms, with all non-null discoveries preserved. The negative result holds under both per-workflow and global memory scopes. We conclude that exact normalized null-result memoization is not a viable optimization on the sampled real command workflows. The likely cause is that agent probe commands are highly contextual—differing by path, port, search expression, or project—so that exact string-level normalization fails to capture meaningful equivalence. More aggressive semantic generalization may yield savings but introduces false-skip risk and requires separate validation with labeled equivalence classes.

## Introduction

Autonomous software engineering agents explore codebases by issuing shell commands: searching for files, grepping for patterns, inspecting process state, and reading configuration. A significant fraction of these probes return empty results or errors—null observations—indicating that the sought resource does not exist, a service is not running, or a pattern is absent. If the agent could remember that a particular probe previously yielded nothing, it might skip re-execution and reduce latency and resource consumption.

The concept of null-result caching appears in several domains: query result caching in databases, negative caching in DNS, and memoization of pure function calls. However, autonomous agent commands are not pure functions: they depend on filesystem state, running processes, and network conditions. A safe memoization strategy must restrict itself to read-only probe commands and must ensure that skipping a command never suppresses a non-null discovery.

This paper tests the simplest viable design: an exact normalized null-result memory that skips a probe command only when (a) the command is a read-only-ish probe, (b) the same normalized command has previously produced a null result, and (c) the non-null discovery stream is preserved. We evaluate this design on real recorded Enoch/Codex agent workflow logs. The result is negative: across all tested configurations, zero commands were skipped. We report this null result because it constrains the design space for agent memory systems and prevents deployment of an optimization that would add complexity without benefit on real workloads.

## Method

### Data Source

The benchmark replays historical real Codex/Enoch execution logs collected from sibling project paths matching `../*/.omx/logs/enoch/*.stderr.log`. Each log entry contains a recorded shell command observation, command status, elapsed milliseconds, and output size. The harness parses these entries without re-executing the historical commands, making the benchmark side-effect-free.

### Null-Result Memory Design

The memory implements a conservative skip rule:

1. **Eligible commands.** Only read-only-ish probe commands (`rg`, `grep`, `find`, `fd`, `ls`, `cat`, `sed`, `awk`, inline Python inspection) are eligible for memoization.
2. **Null classification.** An observation is classified as null if it yielded no useful payload or a null/error marker.
3. **Skip condition.** A command is skipped only if its normalized key matches a previously recorded null-result key.
4. **Safety invariant.** Non-null observations are never skipped. The benchmark asserts that the ordered non-null discovery stream is identical with and without memory.

Command normalization produces a key from the command string. Under exact normalization, two commands share a key only if their normalized forms are identical.

### Memory Scopes

Two memory scopes were tested:

- **Per-workflow memory.** A repeated null result can only skip later matching null probes within the same workflow.
- **Global memory.** A repeated null result can skip later matching null probes across all workflows.

### Benchmark Harness

The harness (`scripts/null_result_memory_replay.py`) replays the recorded observation stream and applies the memory policy. For each run, it records:

- Total observations and workflows
- Null observations and unique null keys
- Skipped null commands and command reduction percentage
- Recorded elapsed time saved
- Whether the non-null discovery stream is preserved

The harness was validated with a unit test suite (`tests/test_null_result_memory_replay.py`) and a smoke run before calibrated measurements.

In every configuration, the number of unique null keys equals the number of null observations, meaning no null-result key was ever encountered more than once. The memory had no opportunity to skip work.

### Discovery Preservation

The non-null discovery stream was preserved in all runs. In the broadest run (calibrated 1000 / global), both baseline and memory configurations produced 3,660 non-null discoveries, identical and in the same order. This confirms that the safety invariant holds trivially (since no commands were skipped), but also verifies that the harness correctly tracks the discovery stream.

### Memory Scope Comparison

Per-workflow and global memory scopes produced identical results on the 250-log subset (71 null observations, 71 unique null keys, 0 skips in both cases). Expanding to global scope did not create cross-workflow key collisions because the null keys were already unique within workflows. This indicates that the lack of repetition is not an artifact of scope limitation.

## Limitations

1. **Recorded-log replay, not live execution.** The benchmark measures command-saving opportunity in real workflows without re-executing historical commands. It does not measure whether a live agent with null-result memory would adapt its behavior—for example, by issuing fewer redundant probes if it knew results were cached. A live A/B test could yield different results if agents exhibit behavioral adaptation.

2. **Conservative null classification.** The skip rule is intentionally conservative: only read-only-ish probe commands with null/error markers are eligible. This design choice prioritizes safety (no false skips) over coverage. The negative result is specifically about exact/safe null-result memoization, not about broader semantic memory strategies. A less conservative classification might identify more null observations, but at the cost of potential false skips.

3. **Exact normalization.** Command keys are normalized by exact string matching. Two commands that are semantically equivalent but textually different (e.g., `grep -r "pattern" src/` vs. `grep -r "pattern" lib/`) produce different keys. The data shows that real agent probes are highly contextual: they differ by path, port, grep expression, or project, so exact normalization rarely identifies duplicates. Whether approximate or semantic normalization could yield savings without unacceptable false-skip rates remains an open question.

4. **Corpus specificity.** The logs come from Enoch/Codex workflows on a single machine. Other agent systems, project types, or workflow patterns may exhibit different null-result repetition rates. The negative result should not be generalized beyond the sampled corpus without additional evidence.

5. **Temporal dynamics.** The replay treats each log as a static snapshot. In live execution, the underlying system state changes over time (services start, files are created), which could either increase or decrease the utility of null-result memory depending on whether state changes invalidate cached nulls.

6. **No semantic equivalence labels.** The benchmark lacks ground-truth labels for when two different commands probe the same underlying fact (e.g., "is service X running?"). Without such labels, we cannot evaluate semantic generalization strategies that might yield savings at the cost of false-skip risk.

7. **Confidence boundary.** Confidence is high for the specific claim about exact normalized null-result memoization on this corpus, but only medium for the broader null-result memory family. A semantic memory variant could still be useful but requires labeled equivalence classes or live A/B safety validation that this benchmark does not provide.

## Conclusion

Exact normalized null-result memoization does not reduce command count or elapsed time on the sampled real Enoch/Codex agent workflows. The core finding is that null observations, while present (105 in the broadest run of 3,765 observations), are never repeated under exact normalized command keys. Each null probe is contextually unique—differing by path, expression, port, or project—so the memory has no opportunity to skip redundant work.

This negative result constrains the design space: any practical agent memory optimization must go beyond exact command matching. Candidate directions include semantic generalization (e.g., service-state memory, project-structure facts, dependency-presence caches), but these introduce false-skip risk and require explicit safety validation with labeled equivalence classes. We recommend against deploying exact normalized null-result memoization as an optimization. If pursuing the broader idea, future work should build separate benchmarks with false-skip labels for the semantic equivalence classes of interest—such as service availability, project directory structure, and dependency presence—rather than relying on exact command string matching.
