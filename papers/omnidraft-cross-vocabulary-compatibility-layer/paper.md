## Abstract

Speculative decoding accelerates autoregressive inference by having a small draft model propose candidate tokens that a larger target model verifies in parallel, but the mechanism assumes shared token vocabularies between draft and target models. When vocabularies differ, draft token IDs have no direct correspondence to target token IDs, and standard verification cannot proceed. The OmniDraft approach (arXiv:2507.02659) proposes an online n-gram cache that translates draft-token sequences into target-token equivalents, enabling cross-vocabulary speculative decoding. This paper reports on a minimal prototype implementation of that compatibility layer. Using deterministic toy tokenizers to isolate the algorithmic mechanism, we verify that exact-span draft-token n-gram mappings can be learned online from shared text, that greedy longest-match translation correctly merges sub-token sequences (e.g., `f`, `la`, `ke` → `flake`), and that constituent draft-token probabilities can be aggregated via multiplication. Calibration over 2.8 million draft tokens yields a translation throughput of approximately 3.33 million draft tokens per second with zero swap usage and stable memory. These results constitute positive evidence for the feasibility of the compatibility-layer mechanism in isolation. They do not constitute evidence for end-to-end speculative-decoding wall-clock speedup, which remains unproven in this project environment because no real tokenizers, language models, or speculative-decoding loops were employed.

## Introduction

Speculative decoding reduces the latency of autoregressive language model inference by having a small draft model propose candidate tokens that a larger target model then verifies in parallel. A fundamental assumption of standard speculative decoding is that the draft model and target model share the same token vocabulary. When this assumption is violated—for example, when a Llama-68M drafter is paired with a Vicuna-7B, Qwen2-7B, or Llama3-8B target—the draft model's token IDs have no direct correspondence to the target model's token IDs, and the verification step cannot proceed.

The OmniDraft paper (arXiv:2507.02659, NeurIPS 2025) addresses this by introducing an online n-gram cache that maps sequences of draft tokens to target tokens. The key insight is that a target tokenizer may merge into a single token what the draft tokenizer emits as multiple sub-tokens: for instance, the target token `flake` corresponds to the draft-token sequence `f`, `la`, `ke`. By learning such mappings from shared text and applying them during drafting, the compatibility layer enables cross-vocabulary speculative decoding. The original paper reports end-to-end speedups of 1.5–2× for a Llama-68M drafter paired with various 7B–8B targets.

The present work isolates and validates the compatibility-layer mechanism itself, independent of any particular language model or speculative-decoding loop. We implement a minimal n-gram translation layer, verify its correctness on deterministic toy tokenizers, and measure its throughput and memory footprint. This approach trades ecological validity (real models, real tokenizers) for experimental control (known ground-truth mappings, reproducible toy inputs). We are explicit about what this prototype proves and what it does not.

## Method

### Design

The compatibility layer implements three operations:

1. **Online n-gram learning.** Given a shared text span, the layer tokenizes it with both the draft tokenizer and the target tokenizer, then aligns the resulting token sequences to identify exact-span mappings where a contiguous sequence of draft tokens corresponds to a single target token. These mappings are stored in an n-gram cache keyed by the draft-token tuple.

2. **Greedy longest-match translation.** Given a sequence of draft tokens proposed by a draft model, the layer scans left-to-right, attempting to match the longest cached n-gram at each position. If a match is found, the corresponding target token replaces the draft-token span; otherwise, the individual draft token is emitted as-is (or flagged as unmappable, depending on the calling context).

3. **Probability aggregation.** When a draft-token n-gram is merged into a single target token, the layer computes the target-token probability as the product of the constituent draft-token probabilities. This preserves a point probability estimate under the assumption of conditional independence of draft-token probabilities given the prefix.

### Implementation

The prototype is implemented in `src/omnidraft_compat.py`. It uses deterministic toy tokenizers rather than real Hugging Face tokenizers. The toy draft tokenizer splits words into fixed sub-token fragments (e.g., `"flake"` → `["f", "la", "ke"]`), while the toy target tokenizer retains whole words as single tokens (e.g., `"flake"` → `["flake"]`). This design directly models the vocabulary-mismatch scenario described in the OmniDraft paper.

No GPU is required. No Hugging Face model downloads occur. The prototype depends only on Python and `psutil` for memory telemetry.

### Evaluation Protocol

Two evaluation phases were conducted:

- **Smoke test** (100 iterations): Translates a small volume of draft tokens, records cache-hit statistics and memory usage. Outputs structured metrics to `metrics/smoke.json`.

- **Calibration** (50,000 iterations): Translates a large volume of draft tokens to measure sustained throughput and memory stability. Outputs structured metrics to `metrics/calibration.json`.

Both phases use `scripts_run_eval.py` and report memory telemetry via `psutil` (available memory, swap usage).

### Unit Tests

Three unit tests in `tests/test_omnidraft_compat.py` verify:

1. N-gram learning correctness (the cache correctly records learned mappings).
2. Longest-match merge behavior (greedy translation prefers longer matches over shorter ones).
3. Input validation (the layer rejects invalid inputs gracefully).

## Results

### Unit Tests

All three unit tests pass (3/3). The test output is recorded in `logs/unittest.log`.

### Smoke Test

Over 100 iterations, the smoke test translated 5,600 draft tokens. The n-gram cache learned the mapping `("f", "la", "ke") → "flake"`. Cache hits totaled 800. Available memory remained at approximately 118,242 MB with zero swap usage. Full metrics are recorded in `metrics/smoke.json`.

### Calibration

Over 50,000 iterations, the calibration run translated 2,800,000 draft tokens in 0.840069 seconds, yielding a throughput of approximately 3,330,607 draft tokens per second. Cache hits totaled 400,000. Available memory remained stable at approximately 118,238 MB with zero swap usage. Full metrics are recorded in `metrics/calibration.json`.

### Interpretation

The compatibility layer correctly learns and applies cross-vocabulary n-gram mappings in the toy setting. Throughput is high and memory overhead is negligible for the cache sizes tested. The probability-aggregation mechanism (constituent probability multiplication) produces valid point estimates under the independence assumption, though this assumption is not independently validated here.

The calibration throughput of ~3.33 million draft tokens per second substantially exceeds the rate at which a typical small draft model generates tokens, suggesting that the translation layer itself would not be a bottleneck in an end-to-end speculative-decoding pipeline—provided that real-tokenizer alignment exhibits similar cache-hit patterns. This latter condition is untested.

## Limitations

The following limitations are substantive and should temper any extrapolation of these results:

1. **Toy tokenizers only.** No real Hugging Face tokenizers or language models were downloaded or benchmarked. The deterministic toy tokenizers model a simplified version of vocabulary mismatch. Real tokenizers exhibit more complex alignment patterns (partial overlaps, multi-token targets, ambiguous mappings) that this prototype does not exercise.

2. **No end-to-end speculative decoding.** The prototype validates the translation mechanism in isolation. It does not implement or measure a speculative-decoding loop with draft-model proposal, target-model verification, acceptance-rate tracking, or residual distribution correction. The 1.5–2× speedup reported in the original OmniDraft paper is therefore not reproduced or contradicted here.

3. **No online distillation or adaptive draft length.** The OmniDraft paper describes additional mechanisms (online distillation of the drafter, adaptive draft-token count) that interact with the compatibility layer. These are not implemented.

4. **Independence assumption for probability aggregation.** Multiplying constituent draft-token probabilities assumes conditional independence, which does not hold in general for autoregressive models. The impact of this approximation on acceptance rates is unknown.

5. **Cache scaling untested.** The calibration run exercises 400,000 cache hits, but the number of distinct n-gram entries is small. Performance under large, diverse caches (e.g., full vocabulary of a real tokenizer pair) is not characterized.

6. **No target-model verification.** The translated target-token proposals are never verified against a target model's distribution, so the quality of the translated proposals—in terms of acceptance probability—is not measured.

7. **Claim audit status.** The claim ledger for this artifact records an audit status of `blocked_empty_claims`, indicating that no structured claims were extracted for formal evidence-grounded audit. This paper should not be treated as having passed strict claim/evidence audit.

## Conclusion

A minimal OmniDraft-style cross-vocabulary compatibility layer is viable as a local prototype. The implemented layer learns exact-span draft-token n-gram mappings from shared text, greedily translates future draft-token proposals into target-token proposals via longest-match, and aggregates constituent probabilities by multiplication. Calibration confirms a translation throughput of approximately 3.33 million draft tokens per second with stable, negligible memory overhead in the toy setting.

These results validate the narrow compatibility-layer mechanism—the core algorithmic response to the vocabulary-mismatch problem described in the OmniDraft paper—but they do not validate end-to-end speculative-decoding wall-clock speedup. The gap between this prototype and a production system includes real-tokenizer alignment complexity, target-model verification, residual distribution correction, online distillation, adaptive draft length, and acceptance-rate measurement. If the project advances beyond this research prototype, the recommended next step is to build a real-tokenizer harness using one small draft tokenizer and one target tokenizer, integrate into a speculative-decoding loop, and measure acceptance rate and wall-clock speedup on a representative workload before any long-running evaluation.
