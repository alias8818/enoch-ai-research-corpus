## Abstract

We present a guarded, trust-weighted memory store designed as an additive extension to the OMX (oh-my-codex) memory Model Context Protocol (MCP) surface. The existing OMX memory server provides unweighted project memory and notepad tools with no trust semantics, provenance tracking, or quarantine mechanisms. We implement a dependency-free Node.js prototype that introduces four MCP-style tools—`guarded_memory_put`, `guarded_memory_query`, `guarded_memory_observe`, and `guarded_memory_stats`—storing records in a separate JSON file to avoid breaking schema compatibility with existing tools. The prototype enforces working-directory path confinement, atomic writes via temp-file-plus-rename, input validation, trust and evidence range checks, tag normalization, trust-weighted query sorting, and automatic quarantine when effective trust falls below 0.2. Unit tests (5/5 passed), a smoke test confirming trust update behavior (effective trust 0.880797 after one confirmation), and a 1000-record benchmark (967 inserts/sec, 6.39 ms query returning 20 records) provide prototype-level evidence of functional viability. Three substantive limitations constrain interpretability: full-file JSON rewrites per insert (unsuitable for large stores), caller-supplied source trust (unauthenticated in the prototype), and the trust function's status as deterministic ranking logic rather than calibrated probability. The result is classified as a viable prototype not yet upstream-integrated.

## Introduction

Agent memory systems face a fundamental tension between accumulation and reliability. As autonomous agents store observations, plans, and inferred facts, the absence of trust differentiation means all memories are treated equally—regardless of whether they originate from verified tool output, uncertain inference, or contradicted claims. This creates a risk of persistent contamination: a single erroneous observation, once stored, carries the same weight as a confirmed result.

The OMX agent framework (oh-my-codex v0.15.0) provides a memory MCP server with project-memory and notepad tools backed by `.omx/project-memory.json` and `.omx/notepad.md`. A scan of the installed `dist/mcp/memory-server.js` confirms that the current surface contains no trust-weighted fields, no quarantine mechanism, and no guarded memory tools. The counts of relevant features in the existing codebase are:

| Feature | Count in Existing Codebase |
|---|---|
| `sourceTrust` references | 0 |
| `effectiveTrust` references | 0 |
| `quarantine` references | 0 |
| `guarded_memory` tool definitions | 0 |

This absence establishes that any trust-weighted memory capability would be genuinely additive rather than duplicative.

This paper investigates whether OMX's current memory MCP surface can be extended with a guarded, trust-weighted memory store that is locally runnable, backward-compatible, and empirically testable. We do not attempt to show that the trust model is calibrated or that the prototype is production-ready; rather, we assess whether the integration is feasible at the prototype level and identify the gaps that must be closed before production use.

## Method

### Design Constraints

The prototype was designed under three constraints:

1. **Additive compatibility.** New tools must not modify or break existing `project_memory_*` and `notepad_*` tools. Storage is placed in `.omx/guarded-trust-memory.json`, deliberately separate from `.omx/project-memory.json`, avoiding a breaking schema migration.
2. **Zero new dependencies.** The implementation uses only Node.js built-in modules to match OMX's dependency posture.
3. **Local testability.** All behavior must be verifiable with standard Node.js tooling without external services.

### Tool Contracts

Four MCP-style functions are exposed:

| Tool | Purpose |
|---|---|
| `guarded_memory_put` | Insert a memory record with content, source trust, evidence level, and tags |
| `guarded_memory_query` | Retrieve records filtered by tag and minimum trust, sorted by effective trust descending |
| `guarded_memory_observe` | Record a confirmation or refutation observation against an existing record, updating effective trust |
| `guarded_memory_stats` | Return aggregate store statistics |

### Trust Model

Each memory record carries two scalar fields:

- **Source trust** (`sourceTrust`): a value in [0, 1] representing the assessed reliability of the source at insertion time.
- **Evidence level** (`evidence`): a value in [0, 1] representing the strength of supporting evidence.

The **effective trust** is computed as a deterministic function of source trust and evidence. Observations modify effective trust: confirmations increase it; refutations decrease it. When effective trust falls below 0.2, the record is automatically quarantined and excluded from normal query results.

The trust function is deterministic and produces ordinal distinctions useful for ranking and thresholding. It is not a statistically calibrated probability, and no calibration dataset or reliability analysis has been performed. We return to this limitation below.

### Safety Guards

The implementation enforces:

- **Working-directory confinement.** File paths are resolved relative to the working directory; traversal outside it is rejected.
- **Atomic writes.** Records are written to a temporary file and renamed, reducing corruption risk on interrupted writes.
- **Input validation and size bounds.** Content length, tag format, and numeric ranges are validated before storage.
- **Tag normalization.** Tags are lowercased and stripped of surrounding whitespace.

## Limitations

We identify five limitations that constrain the interpretability and production readiness of these results.

**1. Full-file JSON rewrites.** Each insert rewrites the entire JSON store. This is adequate for small-to-medium project memory stores (hundreds to low thousands of records) but will degrade for large, long-lived stores. The current benchmark characterizes performance at only one record count (1000); insert throughput may decrease nonlinearly as store size grows. Before high-volume production use, the storage backend should migrate to append-only NDJSON or SQLite.

**2. Caller-supplied source trust.** In the prototype, `sourceTrust` is provided by the caller at insertion time. This is appropriate for testing but insufficient for production, where source trust should be bound to authenticated tool identity, signed provenance, or MCP caller context. A malicious or confused caller could assign artificially high source trust to unreliable content, undermining the quarantine mechanism.

**3. Trust function is not calibrated probability.** The effective trust score is a deterministic ranking and quarantine logic. It produces ordinal distinctions useful for sorting and thresholding, but it should not be interpreted as a statistically calibrated probability that a memory record is correct. No calibration dataset or reliability analysis has been performed. Presenting it as such would be misleading.

**4. Single-host, single-configuration evidence.** All results were collected on one Linux arm64 host with zero swap and ample available memory (122 GB). Performance characteristics may differ substantially on other platforms, under memory pressure, or with different Node.js versions. The benchmark is a single-point measurement, not a characterization of the performance distribution.

**5. No concurrent access testing.** The prototype was tested under single-process access. Concurrent writes from multiple OMX sessions were not evaluated and may cause data loss under the current full-file rewrite approach, since there is no file-locking mechanism.

## Conclusion

A guarded, trust-weighted memory store can be integrated into OMX's memory MCP surface as an additive extension with low compatibility risk. A dependency-free Node.js prototype demonstrates working-directory-confined storage, atomic writes, input validation, trust-weighted querying, refutation-based quarantine, and backward compatibility with existing OMX memory tools. Unit tests (5/5 passed), a smoke test (effective trust 0.880797 after confirmation), and a 1000-record benchmark (967 inserts/sec, 6.39 ms query) provide prototype-level evidence of functional correctness and adequate performance for project-scope use.

The result is classified as `viable_prototype_not_upstream_integrated`. Five substantive limitations—full-file rewrites, caller-supplied trust, uncalibrated trust scores, single-host evidence, and untested concurrent access—must be addressed before production deployment. The recommended next step is an upstream patch to OMX's `memory-server` adding the four `guarded_memory_*` tools while preserving existing tools unchanged, with storage in `.omx/guarded-trust-memory.json`, accompanied by a migration to an append-only storage backend and authenticated source trust binding.
