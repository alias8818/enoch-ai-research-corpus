## Abstract

We present a prototype registry system that assigns stable, content-derived identifiers to declarative skill artifacts (OMX/Codex `SKILL.md` files), records their metadata and declared inter-skill relationships, and validates bond integrity across a registry. The registry scans skill directories, computes bond IDs from content hashes, persists snapshots as JSON, and detects drift by comparing current hashes against saved snapshots. In local evaluation, the prototype scanned 3 fixture skills and 36 installed skills with zero validation issues across both sets; all 5 unit tests passed. The system is dependency-free and operates entirely on local filesystem artifacts. Limitations include a conservative YAML frontmatter parser that handles only a subset of the YAML specification, absence of cryptographic signing, no remote registry or synchronization layer, no access-control model, and no formal JSON Schema governing the output format. Results are preliminary and reflect single-session prototype validation only; no multi-user, concurrent, or long-running stability data were collected.

## Introduction

Declarative skill definitions—structured metadata files describing capabilities, triggers, and inter-skill relationships—are used in agent orchestration frameworks to configure and compose agent behaviors. As the number of skill artifacts grows, several problems emerge: skill identities are unstable across edits (a renamed or modified skill may silently break downstream references), declared relationships between skills cannot be verified without re-scanning every file, and content drift goes undetected when skill definitions change without updating dependent references.

A *skill bond registry* addresses these problems by providing a durable mapping from skill content to stable identifiers, recording source locations and metadata, validating declared relationships, and enabling drift detection against persisted snapshots. This paper describes the design and prototype evaluation of such a registry for the OMX/Codex skill ecosystem.

The core contributions are:

1. A content-derived bond ID scheme (`skillbond:<hash>`) that provides stable identity tied to skill content.
2. A validation model that checks for duplicate names, duplicate triggers, broken declared bonds, and content drift.
3. A dependency-free Python implementation with CLI commands for scanning, validating, and querying skill registries.
4. Local evaluation against fixture skills and a live installation of 36 skills.

These contributions are at prototype stage. No claim is made regarding production readiness, performance at scale, or suitability for adversarial environments.

## Method

### Design

The registry operates on `SKILL.md` files, each containing a YAML frontmatter block with skill metadata (name, triggers, declared bonds to other skills) followed by a Markdown body. The registry performs the following steps:

1. **Discovery.** Recursively scan a root directory for files named `SKILL.md`.
2. **Parsing.** Extract the YAML frontmatter using a conservative subset parser that avoids a full YAML library dependency. Retain the Markdown body for hash computation.
3. **Bond ID assignment.** Compute a SHA-256 hash over the full file content and assign the identifier `skillbond:<hex-prefix>`.
4. **Snapshot persistence.** Write the registry state as a JSON artifact containing each skill's bond ID, source path, name, triggers, declared bonds, and content hash.
5. **Validation.** Check for:
   - Duplicate skill names within the registry.
   - Duplicate trigger phrases across skills.
   - Declared bonds referencing skill IDs not present in the registry.
   - Content drift by comparing current content hashes against a previously saved snapshot.
6. **Query.** Support lookup of skills by name, bond ID, or trigger phrase without re-scanning the filesystem.

A distinction is made between *declared bonds* (explicit frontmatter entries asserting a relationship to another skill) and *weak mentions* (skill names appearing in the Markdown body). Weak mentions are retained as evidence but do not fail validation when unresolved. This design choice reflects the ambiguity inherent in body-text references: they may refer to examples, placeholders, or external plugins rather than genuine missing dependencies. The threshold between "external reference" and "missing dependency" is not formally defined, which is an acknowledged limitation.

### Implementation

The prototype is implemented in Python with no external dependencies. The module structure is:

- `src/skill_bond_registry/registry.py` — core `BondRegistry` class with `scan()`, `validate()`, and `query()` methods.
- `src/skill_bond_registry/cli.py` — command-line interface exposing `scan`, `validate`, and `query` subcommands.
- `tests/test_registry.py` — unit test suite.

The conservative frontmatter parser handles the YAML subset commonly found in OMX skill files but does not support the full YAML specification. Skills using advanced YAML features (anchors, complex nested structures, non-string scalars) may not parse correctly.

### Evaluation Procedure

The prototype was evaluated in two configurations:

1. **Fixture test.** A set of 3 synthetic `SKILL.md` files was placed in a temporary directory. The registry scanned, persisted, and validated these fixtures.
2. **Installed skill scan.** The registry scanned `~/.codex/skills`, containing 36 installed OMX/Codex skill files. Validation was run against the resulting registry.

Unit tests were executed via `python -m pytest -q`. No performance benchmarks were collected; scan and validation completed interactively for both configurations with no perceptible delay. No CUDA, GPU, or hardware-specific calibration was involved; this is a pure filesystem and CPU operation.

## Limitations

1. **Conservative YAML parser.** The frontmatter parser handles a subset of YAML. Skills using advanced YAML features (anchors, complex nested structures, non-string scalars) may not parse correctly. A full YAML library would resolve this at the cost of adding a dependency.

2. **Weak mention ambiguity.** The decision to not fail validation on unresolved weak mentions is pragmatic but may mask genuine missing-skill problems in some configurations. The threshold between "external reference" and "missing dependency" is not formally defined, and no sensitivity analysis was performed on this design choice.

3. **No cryptographic signing.** Bond IDs are content-derived hashes but are not signed. A compromised or tampered registry snapshot would not be detectable without an external trust mechanism.

4. **No remote registry or synchronization.** The prototype operates entirely on local filesystem artifacts. There is no mechanism for publishing, discovering, or synchronizing registries across machines or teams.

5. **No access control.** Any process with filesystem access can read or overwrite the registry JSON. No authorization model is defined.

6. **No formal JSON Schema.** The registry output format is implemented in code but not governed by a machine-checked schema, limiting interoperability and automated validation by downstream consumers.

7. **Notion content unavailable.** The project metadata referenced a Notion URL, but no local export or content from that page was available during the run. Closure is based entirely on local prototype and installed skill evidence.

8. **Prototype scope.** These results reflect a single-session prototype evaluation on a single machine. No multi-user, concurrent, or long-running stability tests were performed. The zero-issue validation result across 36 installed skills should not be generalized to larger or more heterogeneous skill ecosystems without further testing.

9. **No performance characterization.** Scan and validation times were not instrumented. The interactive completion time provides only a qualitative indication of adequacy for the tested scale.

## Conclusion

The OMX Skill Bond Registry prototype demonstrates that a dependency-free local registry can scan declarative skill files, assign content-derived bond IDs, persist registry snapshots as JSON, validate declared inter-skill relationships, and detect content drift. In local evaluation against 3 fixture skills and 36 installed skills, the registry produced zero validation issues, and all 5 unit tests passed.

These results are preliminary and should not be interpreted as evidence of production readiness. The prototype does not address cryptographic signing, remote synchronization, access control, or formal schema governance, all of which would be necessary for production deployment. The conservative YAML parser and the weak-mention validation policy introduce known edge cases whose frequency in real-world skill ecosystems is uncharacterized. The evaluation is limited to a single session on a single machine with no concurrency, adversarial, or long-running stability data.

If the registry concept is pursued toward production, the recommended next steps are: (1) add signature support over registry snapshots and a JSON Schema file, (2) define an explicit frontmatter bond contract in OMX skill authoring documentation, and (3) integrate registry scanning and validation into CI pipelines that fail on declared relation errors or unexpected drift.
