## Abstract

Autonomous controller systems that manage long-running compute projects often include terminal recovery or reset actions for runs that end in error states. A purely state-based reset policy—one that resets any run in an error or stale state—risks destroying or re-queuing work that has already produced durable, terminal output. This paper evaluates an output-aware reset gate that inspects project output evidence (project decisions, run notes, artifacts, logs) before permitting a terminal reset. In a historical audit of 1,207 wake-gate run records, a state-only baseline identified 18 reset candidates (all in `error` state), yet all 18 possessed durable terminal output or useful intermediate artifacts. The output-aware gate suppressed all 18 resets, reducing false-positive and review-required resets from 18 to 0. Five deterministic synthetic fixture cases also passed. These results support adding an output-aware guard before destructive terminal reset actions, though the observed corpus does not exercise the no-output reset path under real conditions, limiting recall estimation for genuinely reset-worthy runs.

## Introduction

Autonomous control planes for compute workloads must handle terminal recovery: deciding what to do when a run reaches an error state, becomes stale, or otherwise appears to have failed. A common approach is to treat the process state as authoritative—if the run is in an `error` state, reset and re-queue it.

This state-only policy has a known failure mode: a run may reach an error state *after* producing durable, useful output. The process may have crashed during cleanup, lost its heartbeat, or encountered a non-fatal error after writing a terminal project decision. Resetting such a run discards completed work and may cause redundant re-execution.

This paper investigates whether an output-aware reset gate—inspecting project output evidence before permitting a terminal reset—can eliminate this class of false-positive resets. The gate policy is:

1. Before resetting a terminal or stale run, inspect output evidence: project decision files, run notes, result and artifact directories, and logs.
2. If a terminal project decision exists, suppress reset and ingest the result.
3. If useful output exists without terminal closure, suppress reset and route to `needs_review`.
4. Allow reset only when no useful output exists.

We evaluate this policy through a deterministic fixture harness and a historical audit of 1,207 local wake-gate run records.

## Method

### Policy Definition

The output-aware reset gate operates as a decision function mapping (run state, output evidence) to a recovery action:

| Run State | Output Evidence | Gate Action |
|-----------|----------------|-------------|
| Error / stale | Terminal project decision | Suppress reset; ingest |
| Error / stale | Useful output, no terminal decision | Suppress reset; route to `needs_review` |
| Error / stale | No useful output | Allow reset |

A run is classified as a *state-only reset candidate* if its wake-gate record shows `error` state, stale `wake_ready` state, or stale `callback_ready` without delivered callback evidence. Output evidence is classified as:

- **Terminal decision**: `.omx/project_decision.json` exists with a `project_decision` field.
- **Nonterminal useful output**: Presence of `run_notes.md`, `results/`, `artifacts/`, or `logs/` without a terminal decision.
- **No useful output**: None of the above evidence paths exist.

### Implementation

The evaluation harness (`scripts/output_aware_terminal_recovery_eval.py`) is a dependency-free Python script that performs three functions:

1. **Synthetic fixture validation**: Exercises five deterministic test cases covering each branch of the gate policy (terminal decision present, useful output without decision, no useful output, stale callback-ready, and edge-case combinations). Fixtures are constructed in-memory and evaluated against the gate logic.

2. **Historical audit**: Scans wake-gate run records under `~/.local/state/omx-wake-gate/runs`, resolves each record to its project directory, inspects output evidence, and classifies each run under both the state-only baseline and the output-aware gate.

3. **Comparison**: Counts baseline reset candidates, output-aware suppressed resets, output-aware allowed resets, and false-positive or review-required resets under each policy.

### Environment

The evaluation ran on an aarch64 host (`gx10-efe8`, Linux 6.17.0-1014-nvidia) with Python 3.12.3, approximately 116 GiB available memory, no swap (intentionally disabled for the GB10 posture), and an NVIDIA GB10 GPU at idle (0% utilization, approximately 10.82 W). The GPU was not used by the evaluation harness.

All five deterministic fixture cases passed, confirming that the gate logic correctly routes each output-evidence category to the expected action.

All 18 state-only reset candidates were in `error` state. Every one of these 18 runs possessed durable output evidence (terminal decisions or useful artifacts). Under the output-aware gate, all 18 resets were suppressed—either ingested as completed or routed to `needs_review`—and zero false-positive or review-required resets remained.

The 7 runs classified as having no useful output were not in a state that triggered reset candidacy under the baseline policy, so the gate's allow-reset branch was not exercised by the historical corpus.

## Limitations

1. **No real no-output reset candidates in the historical corpus.** All 18 baseline reset candidates had useful output. The gate's allow-reset branch (the case where reset is genuinely appropriate) was validated only through synthetic fixtures, not by real runs that truly lacked output. This means the evaluation estimates precision but not recall for the reset-worthy population.

2. **Historical audit relies on durable output only.** Live terminal scrollback was not available for old runs. It is possible that some runs classified as having no useful output had transient terminal output that was not captured to durable storage. The gate's behavior for such cases depends on what evidence the control plane retains.

3. **Harness validates the decision rule, not a deployed production mutation.** The evaluation is a post-hoc audit of historical records against a policy specification. It does not demonstrate that integrating the gate into a live control-plane terminal recovery path is free of integration hazards, race conditions, or performance regressions.

4. **Single corpus, single host.** All 1,207 run records come from one local wake-gate instance on one host. Generalization to other workloads, controller configurations, or failure modes is not established.

5. **The `running` state record.** One run was in `running` state at scan time. This run was not a reset candidate under either policy, but its presence confirms the corpus includes non-terminal states that the gate does not act on.

6. **Potential for spurious artifacts.** The gate treats the presence of files such as `run_notes.md` or `results/` as evidence of useful output. A run that created these files without making substantive progress could cause the gate to incorrectly suppress a needed reset. The historical corpus did not exhibit this case, but it remains a risk in production.

## Conclusion

An output-aware reset gate that inspects project output evidence before permitting terminal recovery resets eliminated all 18 false-positive resets identified by a state-only baseline in a 1,207-run historical audit. Five synthetic fixture cases confirmed correct routing for each policy branch. The result is a positive local signal that the gate addresses its targeted failure mode—destructive reset of runs that have already produced durable output.

However, the historical corpus did not contain any reset candidates lacking useful output, so the gate's allow-reset branch remains validated only by synthetic fixtures. The 100% false-positive reduction figure should be interpreted with this asymmetry in mind: it reflects the composition of the observed corpus rather than a guarantee of gate correctness across all failure modes. Production deployment should monitor for cases where the gate incorrectly suppresses a reset that is genuinely needed (i.e., runs with spurious artifact files but no real progress). The recommended next step is to implement the gate as a production control-plane guard before terminal reset actions, with observability for both suppressed and allowed resets.
