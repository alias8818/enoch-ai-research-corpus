## Abstract

Speculative decoding accelerates autoregressive inference by having a small drafter model propose tokens that a larger verifier model checks in parallel. When the serving stack applies prompt compression to the verifier for long-context efficiency, a context mismatch arises if the drafter still conditions on the uncompressed prompt. This paper investigates whether the drafter should instead condition on the same compressed prompt as the verifier. We conduct proxy experiments using GPT-2 and DistilGPT-2 on synthetic long-context prompts, measuring exact one-token speculative acceptance across four compression-awareness conditions. When both target and draft models see the same answer-critical compressed prompt (~61 tokens vs. 1024 tokens), one-token acceptance rises from 0.026 to 0.737 relative to the compression-unaware baseline—a 28.1× relative improvement—while draft forward-pass time decreases by 3.6×. A same-model control (GPT-2 as both target and draft) isolates the distribution-alignment effect from model-capability mismatch, showing that prompt mismatch alone reduces acceptance from 1.000 to 0.727. Over-compressing the drafter to question-only prompts (~18 tokens) degrades acceptance to 0.334, indicating that compression must preserve verifier-relevant evidence rather than merely minimize length. These results support the mechanism claim that compression-aware drafting improves draft–verifier alignment and reduces draft prefill cost, but they do not establish end-to-end serving speedup or answer-quality preservation under lossy compression on real workloads.

## 1. Introduction

Speculative decoding is a lossless inference acceleration technique in which a small drafter model proposes candidate future tokens and a larger target (verifier) model checks them in a single forward pass. The technique's effectiveness depends on the alignment between the draft distribution $q$ and the verifier distribution $p$: when the drafter's proposed tokens are likely under the verifier's distribution, more tokens are accepted per verification pass, and fewer verification steps are needed.

Long-context inference intensifies two known bottlenecks for speculative decoding. First, the drafter's key-value cache grows with context length, consuming memory and increasing prefill latency. Second, a drafter conditioned on a long prompt may develop a distribution that diverges from a verifier operating on a shortened or compressed version of the same prompt. Prior work on long-context speculative decoding has identified excessive draft-model KV memory, short-versus-long context mismatch, and tree-attention inefficiency as key obstacles (LongSpec, arXiv:2502.17421). Approaches such as retrieval-based context shortening for the drafter have been proposed to address the memory bottleneck (RAPID, arXiv:2502.20330), but the distribution-alignment consequences of asymmetric prompt compression between drafter and verifier remain understudied.

This paper asks: if the serving stack compresses the verifier's prompt, should the drafter also condition on that same compressed prompt? We term a drafter that sees the compressed prompt *compression-aware* and one that still sees the full prompt *compression-unaware*.

We report proxy experiments that measure exact one-token speculative acceptance—the probability that a token sampled from the draft distribution would be accepted by the verifier—across four conditions varying compression awareness. The experiments use GPT-2 as the target model and DistilGPT-2 as the draft model on synthetically generated long-context prompts, with a same-model control (GPT-2 as both target and draft) to isolate distribution-alignment effects from model-capability mismatch.

## 2. Method

### 2.1 Experimental Design

For each synthetic long-context case, the evaluation script constructs three prompt variants:

1. **Full prompt** (`full`): A noisy long prompt truncated left to 1024 GPT-2 tokens, preserving the answer suffix.
2. **Compressed prompt** (`compressed`): An answer-critical compressed prompt averaging approximately 61 tokens.
3. **Question-only prompt** (`question_only`): An over-compressed prompt averaging approximately 18 tokens, retaining only the question without supporting evidence.

These prompt variants define four experimental conditions:

| Condition | Target prompt | Draft prompt |
|---|---|---|
| Baseline (full/full) | Full | Full |
| Compression-unaware | Compressed | Full |
| Compression-aware | Compressed | Compressed |
| Over-compressed | Compressed | Question-only |

The baseline condition establishes acceptance when both models see the full prompt. The compression-unaware condition represents the naive deployment where the verifier receives a compressed prompt for efficiency but the drafter is left unchanged. The compression-aware condition tests the hypothesis that matching the drafter's context to the verifier's compressed context improves alignment. The over-compressed condition tests whether further shortening the drafter's prompt beyond the verifier's compressed context is beneficial.

### 2.2 Acceptance Metric

The primary metric is exact one-token speculative acceptance, defined as:

$$\alpha = \sum_x \min\bigl(p(x),\, q(x)\bigr) = \mathbb{E}_{x \sim q}\left[\min\left(1,\, \frac{p(x)}{q(x)}\right)\right]$$

where $p(x)$ is the verifier's next-token distribution and $q(x)$ is the drafter's next-token distribution. This metric captures the probability that a single token sampled from the draft distribution would be accepted under the standard speculative decoding acceptance rule. It is a necessary but not sufficient condition for high multi-token speculative acceptance: high one-token acceptance is compatible with poor multi-token acceptance if the distributions diverge after the first token, and conversely, moderate one-token acceptance may still yield useful speedup in a tree-structured draft scheme.

### 2.3 Models and Hardware

- **Main experiment:** Target: GPT-2 (124M parameters), Draft: DistilGPT-2 (82M parameters), 40 synthetic cases.
- **Same-model control:** Target: GPT-2, Draft: GPT-2, 20 synthetic cases.
- **Hardware:** GB10 server with swap disabled. System MemAvailable remained above 115 GiB throughout all runs. GPU utilization peaked at 57–68% during inference passes. The GB10 driver did not expose GPU memory totals through `nvidia-smi`; system MemAvailable was used as the memory indicator.

### 2.4 Software and Execution

All experiments were run using PyTorch and HuggingFace Transformers within a Python virtual environment. A smoke test (3 cases) preceded the main runs to validate pipeline correctness. The evaluation script, results, and logs are listed in the Referenced Artifacts section.

## 3. Results

### 3.1 Main Experiment: GPT-2 Target, DistilGPT-2 Draft

| Condition | Mean acceptance (1-tok) | Target tokens | Draft tokens | Target fwd (ms) | Draft fwd (ms) |
|---|---:|---:|---:|---:|---:|
| Full target / full draft | 0.0263 | 1024.0 | 1024.0 | 16.927 | 5.990 |
| Compressed target / full draft (unaware) | 0.0262 | 60.9 | 1024.0 | 2.783 | 5.990 |
| Compressed target / compressed draft (aware) | 0.7368 | 60.9 | 60.9 | 2.783 | 1.645 |
| Compressed target / question-only draft | 0.3339 | 60.9 | 18.4 | 2.783 | 1.744 |

The compression-aware condition improved one-token acceptance from 0.0262 to 0.7368 relative to the compression-unaware condition, a 28.1× relative improvement. Draft forward-pass time decreased from 5.990 ms to 1.645 ms (3.64× speedup), driven by the reduction in draft prompt length from 1024 to approximately 61 tokens.

The over-compressed (question-only) condition achieved only 0.3339 acceptance despite having the shortest draft prompt, confirming that compression quality matters: the drafter must retain verifier-relevant evidence, not merely minimize token count.

The baseline (full/full) condition showed similarly low acceptance (0.0263) as the compression-unaware condition (0.0262). Both conditions present the drafter with the full 1024-token prompt; the difference is that the compression-unaware condition presents the verifier with a compressed prompt while the baseline presents the verifier with the full prompt. The low absolute acceptance in both cases reflects the substantial distributional mismatch between GPT-2 and DistilGPT-2 on these prompts—a mismatch that compression awareness partially corrects but does not eliminate.

### 3.2 Same-Model Control: GPT-2 Target, GPT-2 Draft

| Condition | Mean acceptance (1-tok) | Target tokens | Draft tokens | Target fwd (ms) | Draft fwd (ms) |
|---|---:|---:|---:|---:|---:|
| Full target / full draft | 1.0000 | 1024.0 | 1024.0 | 24.248 | 9.558 |
| Compressed target / full draft (unaware) | 0.7272 | 60.9 | 1024.0 | 2.742 | 9.558 |
| Compressed target / compressed draft (aware) | 1.0000 | 60.9 | 60.9 | 2.742 | 2.633 |
| Compressed target / question-only draft | 0.5171 | 60.9 | 18.4 | 2.742 | 2.844 |

With an identical model as both target and draft, the full/full condition achieves perfect acceptance (1.0000) as expected—when $p = q$, every token is accepted. Prompt mismatch alone (compression-unaware condition) drops acceptance to 0.7272, demonstrating that distributional divergence from asymmetric context is a real and measurable effect even without model-capability mismatch. Restoring compression awareness recovers acceptance to 1.0000 while reducing draft context from 1024 to approximately 61 tokens and draft forward time from 9.558 ms to 2.633 ms.

The question-only condition again underperforms (0.5171), reinforcing that over-compression harms alignment. Notably, the question-only draft forward time (2.844 ms) is slightly higher than the compressed draft forward time (2.633 ms) despite shorter input, which may reflect measurement noise at this scale.

### 3.3 Mixed and Negative Observations

Several aspects of the results warrant cautious interpretation:

- **Low absolute acceptance in the main experiment.** Even the best condition (compression-aware) achieves only 0.7368 one-token acceptance with the GPT-2/DistilGPT-2 pair. This reflects genuine distributional divergence between models of different capacity and training. Whether this acceptance level translates to meaningful end-to-end speedup in a real speculative decoding loop remains unestablished.

- **The baseline and compression-unaware conditions are nearly identical.** The 0.0263 vs. 0.0262 difference is within measurement noise. This is expected—both conditions present the drafter with the same full prompt—but it also means the experiment provides no evidence about whether compressing the verifier alone (without adjusting the drafter) helps or hurts speculative decoding efficiency. The comparison of interest is between compression-unaware and compression-aware drafting, not between baseline and compression-unaware.

- **No multi-token acceptance data.** Real speculative decoding benefits depend on the acceptance rate of multi-token draft sequences, which can differ substantially from one-token acceptance. A drafter with high one-token acceptance but rapidly diverging subsequent tokens would yield less speedup than the one-token metric suggests.

## 4. Limitations

1. **Synthetic prompts and small models.** The experiments use synthetically generated long-context prompts and GPT-2-family models (124M and 82M parameters). These serve as a mechanism test, not a production workload benchmark. Results may not generalize to larger models, real retrieval-augmented generation contexts, or naturalistic long documents.

2. **One-token acceptance proxy.** The metric measures exact one-token acceptance, which is a necessary but insufficient indicator of multi-token speculative decoding performance. Real speculative decoding uses tree-structured draft sequences and multi-token acceptance; acceptance rates for longer draft sequences may differ substantially from the one-token rates reported here.

3. **No end-to-end speedup measurement.** The experiments measure acceptance probability and per-pass latency but do not implement a full speculative decoding loop. End-to-end tokens-per-second, time-to-first-token, and overall speedup were not measured.

4. **Compression quality not evaluated.** The experiments assume that the compressed prompt preserves answer quality. They do not establish that lossy compression maintains output correctness on real tasks. A compression method that discards verifier-critical information would undermine both answer quality and the acceptance gains reported here.

5. **GPU memory reporting.** The GB10 driver did not expose GPU memory totals through the standard `nvidia-smi` path. System MemAvailable was used as a proxy, which does not capture VRAM allocation directly.

6. **Single compression strategy.** Only one compression method (answer-critical extraction) and one over-compression method (question-only) were tested. The space of compression strategies and their interaction with speculative acceptance remains largely unexplored.

7. **Random seeds not fixed.** Synthetic prompt generation and sampling stochasticity were not controlled via fixed seeds, which may affect exact numerical reproduction.

8. **Software version pins not recorded.** Exact versions of PyTorch, Transformers, and other dependencies were not captured in the run notes, complicating strict reproducibility.

9. **Claim ledger audit status.** The project's claim ledger contains no structured claims and carries an audit status of `blocked_empty_claims`. The claims in this paper are inferred from the run notes and decision JSON rather than from a formally audited claim ledger.

## 6. Conclusion

This study provides local proxy evidence that speculative drafters benefit from conditioning on the same compressed prompt as the verifier, rather than retaining the full uncompressed prompt. In the main experiment (GPT-2 target, DistilGPT-2 draft), compression-aware drafting improved one-token acceptance from 0.026 to 0.737 (28.1× relative gain) and reduced draft forward-pass time by 3.6×. The same-model control isolated the distribution-alignment effect, showing that prompt mismatch alone reduces acceptance from 1.000 to 0.727 even when drafter and verifier have identical capabilities. Over-compression to question-only prompts degraded acceptance, confirming that compression must preserve verifier-relevant evidence.

These findings support the mechanism claim that compression-aware drafting improves draft–verifier distribution alignment and reduces draft prefill cost. However, the evidence is scoped to synthetic prompts, small models, and a one-token acceptance proxy. The results do not establish end-to-end serving speedup, answer-quality preservation under lossy compression, or generalization to production-scale models and workloads. The recommended next step is to prototype compression-aware drafting in a real speculative decoding engine (e.g., llama.cpp, vLLM, SGLang, or TensorRT-LLM) with a real compressor/retriever, and measure accepted tokens per verifier pass, end-to-end latency, output quality, and resource utilization under realistic conditions.
