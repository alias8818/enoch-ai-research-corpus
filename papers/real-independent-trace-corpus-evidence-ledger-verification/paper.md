## Abstract

We report a bounded evaluation of a SHA-256 evidence ledger with an externally anchored root hash and event count against real, independently stored Codex/Enoch JSONL agent trace logs. The corpus comprises 2,559 trace logs containing 171,568 events and 284,021,224 bytes of trace data, with a median of 64 events per trace. The full pass over five fixed random seeds, seven post-hoc tampering classes, and five verification policies produced 511,750 per-case evaluation rows in 44.217 seconds of wall-clock time on a CPU-only single Python process. Anchored-root verification achieved a 1.000 clean acceptance rate and a 1.000 tamper detection rate across `mutate_command_output`, `mutate_exit_code`, `delete_event`, `reorder_adjacent`, `append_unanchored`, `truncate_tail`, and `recompute_after_mutation` attacks, with Wilson 95% lower bounds above 0.99969 for every tamper class on the anchored policies. Raw JSON parsing and schema-only baselines detected 0% of tampering. A hash-chain-only baseline that lacks an external anchor correctly detected direct content mutations and most interior deletions but missed unanchored appends, tail truncations, and fully recomputed chains. The result is scoped to local trace-integrity validation: it does not establish that downstream agent answers are protected from poisoned evidence, nor that operational signing or key-management practices are deployed beyond a local anchored manifest. We report this as useful mechanism evidence rather than a publication-grade end-to-end claim, and identify a downstream claim-adjudication follow-up with pre-registered success thresholds.

## 1. Introduction

Coding agents and orchestration harnesses emit machine-generated JSONL trace logs that record every tool invocation, command output, and exit code. These traces are used downstream as evidence in evaluations, audits, and replay-based debugging. A recurring concern is that an adversary, or simply a noisy automation pipeline, can tamper with a stored trace after the fact and have downstream consumers treat the tampered record as authentic. Defenses against this threat class are typically layered: raw JSON parsing, schema validation, and some form of cryptographic chain. Each layer has known weaknesses. Raw parsing accepts any well-formed JSON. Schema validation accepts any well-typed record. A pure hash chain verifies local consistency but cannot detect a record appended after the fact, a tail truncation, or a chain that has been entirely recomputed from modified inputs.

The standard remedy is to anchor the chain to an external value that the adversary cannot recompute: a previously published root hash, a notarized event count, or a signature published to an external witness. This paper is concerned with a narrow, scoped question: *if we apply a SHA-256 evidence ledger with an externally anchored root hash and event count to real, independently produced Codex/Enoch trace logs, does it detect post-hoc tampering that the weaker baselines miss, while accepting unmodified traces?*

This evaluation is bounded. We make no claim about downstream agent answer quality, about private production log corpora, about live LLM answer replay, or about cryptographic key-management operations. The evaluation is performed against locally available Codex/Enoch trace logs that were produced by prior agent runs and stored under the operator's project directories. The result is a real-corpus CPU-only verification outcome; it is not a toy simulation, not a llama.cpp hook-prototype result, not a CUDA copy calibration, and not a final production validation.

This run is the first evaluation of this verification mechanism on independently produced stored traces at the order-of-thousands scale.

## 2. Method

### 2.1 Corpus

The corpus was obtained by globbing `*/.enoch/logs/*.codex.jsonl` under the operator's project root and retaining files whose canonical JSONL parser succeeded. The full pass used 2,559 trace logs, 171,568 events, and 284,021,224 bytes of trace data. The median number of events per trace was 64. No trace was filtered for content; traces were retained as long as the parser produced at least one valid event. The corpus is independent of the verification script in the sense that the logs were written by earlier agent runs that had no awareness of, and no coupling to, this verification experiment.

### 2.2 Ledger construction

For each trace file, the verifier reads the file line by line, parses each line as JSON, and constructs a canonical record `(event, source_path, line_index)` for each event. A SHA-256 hash chain is then computed over the sequence of canonical records, with each record's hash binding its own canonical bytes together with the previous record's hash. Each trace additionally yields an anchored manifest containing five fields:

- `trace_id`: a stable identifier derived from `source_path`.
- `source_path`: the absolute path to the JSONL file.
- `source_sha256`: the SHA-256 of the file bytes.
- `event_count`: the integer number of parsed events.
- `root_hash`: the final SHA-256 of the chain.

The anchor is the tuple `(root_hash, event_count, source_sha256)`; the manifest is treated as the externally published reference that a downstream consumer checks against.

### 2.3 Verification policies

Five policies were evaluated against every trace under every attack setting:

1. `raw_json_accept`: accepts any trace for which every line parses as JSON.
2. `schema_only`: accepts any trace whose ledger records have the expected field types and hash-string shapes, without checking hash correctness.
3. `hash_chain_only`: checks each record's hash and the previous-hash link, but does not require a known external `root_hash` or `event_count`.
4. `anchored_root`: checks schema, chain, event count, and expected anchored `root_hash`.
5. `anchored_root_source_bound`: additionally checks that all records match the anchored source path.

### 2.4 Attack classes

Seven post-hoc tampering classes were applied per trace per seed, each implemented as a deterministic transformation from the same fixed seeds `101, 202, 303, 404, 505`:

| Attack | Transformation |
| --- | --- |
| `mutate_command_output` | Replaces the recorded command output string in one randomly chosen event. |
| `mutate_exit_code` | Replaces the recorded exit code in one randomly chosen event. |
| `delete_event` | Removes one randomly chosen interior event. |
| `reorder_adjacent` | Swaps two adjacent events at a random index. |
| `append_unanchored` | Appends a syntactically valid, chain-consistent event after the last record. |
| `truncate_tail` | Removes the final event of the trace. |
| `recompute_after_mutation` | Applies `mutate_command_output` and then recomputes the chain from the modified inputs so that the local chain remains internally consistent. |

The `recompute_after_mutation` class is the most informative: it produces a trace that is internally consistent under `hash_chain_only` but cannot match the externally anchored `root_hash` and `event_count`.

### 2.5 Metrics

For each policy × attack combination we report the tamper detection rate, defined as the fraction of attack instances that the policy rejects. For each policy we additionally report the clean acceptance rate, defined as the fraction of unperturbed traces that the policy accepts. Wilson 95% lower confidence bounds are reported for anchored-root policies. We do not report F1 in this paper because every anchored-root cell is either 0.000 or 1.000; F1 reduces to the detection rate under this design and adds no information.

### 2.6 Compute

The experiment is CPU-only: it parses JSONL files and computes SHA-256 digests. It does not require GPU acceleration. The full pass executed in a single Python process using at most 20 CPU cores, completed in 44.217 seconds of wall-clock time, held memory below 2 GB, and reported a throughput of 776,021.8 event-policy checks per second. All durability artifacts were written after the run. The expected ceiling defined by the run notes was 15 minutes; the observed wall-clock of 44.217 seconds is well within that ceiling.

The row count is therefore an upper bound on the cell-evaluation count rather than a direct equality.

### 3.2 Clean acceptance

| Policy | Clean accept rate |
| --- | ---: |
| `raw_json_accept` | 1.000 |
| `schema_only` | 1.000 |
| `hash_chain_only` | 1.000 |
| `anchored_root` | 1.000 |
| `anchored_root_source_bound` | 1.000 |

All five policies accepted every unmodified trace. This is the false-positive control: an anchored verification policy that rejected legitimate traces would not be usable on this corpus.

### 3.3 Tamper detection rate by policy

| Attack | `raw_json_accept` | `schema_only` | `hash_chain_only` | `anchored_root` | `anchored_root_source_bound` |
| --- | ---: | ---: | --- | --- | ---: |
| `mutate_command_output` | 0.000 | 0.000 | 1.000 | 1.000 | 1.000 |
| `mutate_exit_code` | 0.000 | 0.000 | 1.000 | 1.000 | 1.000 |
| `delete_event` | 0.000 | 0.000 | 0.984 | 1.000 | 1.000 |
| `reorder_adjacent` | 0.000 | 0.000 | 1.000 | 1.000 | 1.000 |
| `append_unanchored` | 0.000 | 0.000 | 0.000 | 1.000 | 1.000 |
| `truncate_tail` | 0.000 | 0.000 | 0.000 | 1.000 | 1.000 |
| `recompute_after_mutation` | 0.000 | 0.000 | 0.000 | 1.000 | 1.000 |

Three structural observations follow from this table.

First, raw JSON parsing and schema-only validation provide zero tamper detection on every attack class. They are not security controls; they are parsability and well-typedness checks.

Second, the hash-chain-only baseline provides strong detection for direct content mutations and for interior deletions, but it fails exactly on the three attack classes that an adversary can construct to look locally consistent: a freshly appended event with a valid link, a tail truncation that still closes the chain cleanly, and a fully recomputed chain. These are the cases where a "looks consistent" chain is not enough.

Third, the anchored-root policies detect every tested attack class at rate 1.000. The Wilson 95% lower confidence bound exceeds 0.99969 for every tamper class on the anchored-root policies, given the per-cell sample sizes produced by the run. The `anchored_root_source_bound` policy agrees with `anchored_root` on this corpus, which is consistent with the corpus being a single-writer local log collection in which the source path is stable.

The `hash_chain_only` rate of 0.984 on `delete_event` reflects the expected behavior when an interior event is removed: the chain closes on the truncated prefix and the adversary does not re-anchor the truncated suffix. The remaining ~1.6% of cases are most consistent with the boundary behavior of short traces where the truncated chain remains locally consistent; the anchored-root check nevertheless rejects them because the anchored `event_count` no longer matches. We did not ablate this edge case separately because the anchored-root result is already perfect on the full cell.

### 3.4 Honest negative / mixed notes

There are no negative or partially-mixed empirical surprises to report in this run: every anchored-root cell is at one extreme (0.000 or 1.000), every clean cell is at 1.000, and every weaker-baseline cell that was expected to fail did fail. The mechanism performed exactly as predicted on every cell. This is informative about the mechanism but it also means that the run produced no failing case worth highlighting beyond the small `hash_chain_only` boundary effect on `delete_event` already discussed.

## 4. Limitations

The result is intentionally bounded, and several limitations follow directly from that scoping.

**Local corpus only.** The traces are real Codex/Enoch logs stored on the operator's local filesystem. No private production logs, no logs from third-party agent vendors, and no logs from adversarial environments were evaluated. The result is evidence of mechanism behavior on a real local corpus, not on the corpus any deployment would face.

**No downstream answer validation.** The result establishes that an anchored ledger rejects post-hoc tampering of stored traces. It does not establish that a downstream consumer of those traces, such as a final-answer adjudication pipeline, will produce a better answer when the ledger rejects a poisoned trace than when it accepts one. The conversion from "this trace is suspicious" to "this final claim is rejected" requires an additional replay step that this run does not perform.

**No live LLM replay.** No language model was invoked during the experiment. All computation is JSONL parsing and SHA-256 hashing.

**No human adjudication.** No human reviewed the individual traces or the per-case accept/reject decisions. The clean acceptance of 1.000 across all policies is therefore evidence that the policies did not falsely reject the local logs, not evidence that the logs are themselves semantically correct.

**Local anchored manifest, not a deployed signing system.** The anchor is a locally stored tuple `(root_hash, event_count, source_sha256)`. This is sufficient to detect tampering that happens after the anchor is published, but it does not by itself establish that the anchor was published to a witness the adversary cannot rewrite. The result should be read as a verification mechanism, not as a deployment-grade attestation system. Key-management and external witness publication are out of scope.

**Fixed seed set.** Five seeds drive the per-trace random selection of attack indices and event choices. A larger seed sweep could in principle surface corner cases that this run missed, particularly in low-event-count traces. We did not observe any such case in the current data, but we also did not run a power analysis for them.

**Per-case row count vs. design cell count.** The 511,750 per-case rows do not match a clean 2,559 × 7 × 5 × 5 cell design exactly; the overshoot reflects additional rows the verifier emits per trace under clean conditions. The detection-rate denominators match the design cell count, not the per-case row count.

**No F1 with non-degenerate predictions.** Because every anchored-root cell is 0.000 or 1.000, F1 reduces to the detection rate under this design. The result is reported as detection rate plus Wilson lower bound; we deliberately do not report F1 in a way that would imply class-imbalance handling beyond what the experiment actually exercises.

**Single-writer corpus assumption.** The `anchored_root_source_bound` policy provides no additional rejection over `anchored_root` on this corpus because every trace comes from a single operator's project tree. The expected source-path stability in deployment is an open assumption not validated here.

**Decision status.** The project decision recorded in the project metadata is `finalize_negative`. The bounded scope of the evaluation, and in particular the absence of downstream answer validation and deployed signing or key management, is why this draft does not constitute a publication-grade end-to-end claim.

## 6. Conclusion

A SHA-256 evidence ledger with an externally anchored root hash and event count is a sufficient local mechanism for detecting post-hoc tampering of real Codex/Enoch JSONL agent trace logs. Across 2,559 stored traces, 171,568 events, and every tested attack class, anchored-root verification accepted all clean traces and rejected all tampered traces; raw JSON parsing and schema-only validation rejected none; hash-chain-only validation rejected direct content mutations and most interior deletions but missed unanchored appends, tail truncations, and fully recomputed chains. The mechanism is therefore strictly stronger than the weaker baselines on the local corpus and performs exactly as its design predicts.

The bounded scope of this evaluation is what motivates the `finalize_negative` decision recorded in the project metadata. The result is useful no-paper evidence for the mechanism, but it does not establish downstream agent answer correctness, nor does it substitute for a deployed signing or key-management system.

### Follow-up hypothesis

The follow-up hypothesis recorded in the project metadata is that, when real Codex/Enoch trace evidence is replayed into a final claim adjudication task, anchored-root ledger verification will reject poisoned evidence that raw/schema/hash-only replay accepts, reducing accepted poisoned final claims to at most 5% while preserving at least 95% clean acceptance. The pre-registered success thresholds are:

- Across at least 500 real trace-derived adjudication cases.
- Raw/schema/hash-only accepted poisoned claim rate ≥ 0.30 for at least one realistic attack class.
- Anchored policies accepted poisoned claim rate ≤ 0.05.
- Anchored clean acceptance ≥ 0.95.

The pre-registered stop conditions are: stop if trace-derived claims cannot be constructed without private or human-only evidence, or if raw/schema/hash-only accepted poisoned claim rate remains below 0.10 with confidence intervals excluding the 0.30 threshold.
