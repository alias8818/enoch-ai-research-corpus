## Abstract

We investigate whether docstring-backed property testing can be integrated into a real third-party Python repository as a locally runnable quality gate, moving beyond prior toy-level demonstrations. We implement a two-layer gate—doctest example execution and docstring-anchored Hypothesis property checking—and integrate it into `more-itertools/more-itertools` as a pytest hook. The gate executes 710 docstring examples (zero failures) and 9 docstring-anchored Hypothesis properties (1,800 generated examples, all passing). A mutation probe with three synthetic behavioral mutants shows the property layer kills all three, while the full upstream test suite (718 tests, 19,896 subtests) continues to pass with the gate integrated. Gate runtime is approximately 1.24 seconds. These results support the bounded hypothesis that real-repo integration is viable for docstring examples plus manually curated docstring-anchored properties. They do not demonstrate fully automatic property extraction from arbitrary natural-language docstrings, nor do they establish generalizability beyond this single repository.

## Introduction

Property-based testing and doctest are complementary verification strategies. Doctest anchors executable examples in documentation strings, providing low-effort regression coverage for literal input–output pairs. Property-based testing, as implemented by Hypothesis, generates many inputs to check invariants spanning the semantic space of an API. A natural question arises: can docstrings serve as the anchoring layer for property tests, so that each property is traceable to a behavioral claim in the documentation?

Prior work on this concept has remained at the toy-simulation level: small, self-contained modules where docstring text and property definitions coexist by construction. The open question is whether this idea survives contact with a real, mature, third-party repository—where docstrings are written for human readers, not for property extraction, and where the existing test suite must remain undisturbed.

This study addresses that question in a bounded form. We select `more-itertools/more-itertools`, a mature pure-Python library with substantial behavioral documentation, and attempt to integrate a docstring-backed property gate as a local pytest hook. We report on execution correctness, mutation-killing capability, runtime overhead, and integration friction. We do not claim generality beyond this single repository.

## Method

### Target Repository

We cloned `more-itertools/more-itertools` at commit `bd8475cc1e9b2efdbc055ab4af4d4693a9c5cb7a`. Static inspection of the repository revealed:

| Module | Public APIs | With docstrings | With doctest examples |
|---|---|---|---|
| `more_itertools/more.py` | 145 | 120 | 113 |
| `more_itertools/recipes.py` | 56 | 53 | 50 |

The high ratio of doctest-bearing docstrings made this repository a favorable candidate for a docstring-backed gate. This selection bias is a known limitation (see Section 5).

### Gate Architecture

We implemented `scripts/docstring_property_gate.py` with two layers:

**Layer 1: Docstring example gate.** This layer runs `doctest.testmod` on `more_itertools.recipes` and `more_itertools.more` with `ELLIPSIS` and `NORMALIZE_WHITESPACE` option flags. It reports per-module failure counts and exits non-zero on any failure.

**Layer 2: Docstring-anchored property gate.** This layer runs Hypothesis properties for nine selected public APIs. Each property is required to remain anchored to behavioral phrases or examples in the corresponding function's docstring. The nine covered APIs and their property summaries are:

| API | Property description |
|---|---|
| `flatten` | One-level flatten equivalence |
| `chunked` | Concatenating chunks round-trips to original list; chunk sizes follow documented `n` behavior |
| `sliced` | Concatenating slices round-trips to original tuple; slice sizes follow documented `n` behavior |
| `take` | Output equals input prefix |
| `ilen` | Count equals consumed iterable length |
| `first` | Result equals first element or default |
| `last` | Result equals last element or default |
| `unique_everseen` | First-seen unique order is preserved |
| `windowed` | Windows match contiguous slices when iterable length is at least width `n` |

Each property was configured with 200 Hypothesis examples, yielding 1,800 generated property examples in total.

### Integration Mechanism

The gate script supports an `--emit-pytest` flag that writes a pytest test file, `tests/test_docstring_property_gate_integration.py`, into the target repository. This file imports the gate logic and fails if any doctest or property check fails. The integration was local only; no upstream pull request was submitted.

### Mutation Probe

The gate script supports a `--mutation-probe` flag that introduces three synthetic behavioral mutants and checks whether the property layer detects each one:

1. **`chunked drops remainder`**: Mutates `chunked` to discard the final partial chunk.
2. **`take returns suffix`**: Mutates `take` to return the suffix rather than the prefix.
3. **`unique sorts output`**: Mutates `unique_everseen` to return sorted output rather than first-seen order.

These mutants were chosen to represent plausible regressions that literal example-based tests might fail to catch, since they preserve correct behavior for some inputs while violating documented semantic invariants.

### Execution Environment

All commands were executed in a Python virtual environment on a machine with approximately 120 GB available memory, no swap, and an idle NVIDIA GB10 GPU. The task was entirely CPU-bound; the GPU was not utilized. Dependencies included `pytest`, `hypothesis`, and the `more-itertools` package installed in editable mode with its testing requirements.

## Limitations

1. **Manual property curation.** The nine Hypothesis properties were manually selected and anchored to docstring phrases. This study does not demonstrate fully automatic property extraction from arbitrary natural-language docstrings. The viability of semi-automatic or fully automatic property suggestion remains an open question.

2. **Single-repo evaluation.** Integration was performed on one repository (`more-itertools`) chosen partly for its favorable docstring-to-API ratio. Results may not transfer to repositories with sparser or less structured docstrings.

3. **Local integration only.** The gate was integrated locally and not submitted as an upstream pull request. Upstream acceptance involves social and maintenance considerations not addressed here.

4. **Narrow property coverage.** Although the doctest layer covers 710 examples across two modules, the property layer covers only 9 of approximately 201 public APIs. Broader semantic coverage would require property generation or scoring infrastructure beyond this study's scope.

5. **Small mutation probe.** Three synthetic mutants provide only a lower bound on the property layer's fault-detection capability. A systematic mutation analysis with a larger and more representative mutant set was not performed.

6. **No comparison baseline.** We did not compare the docstring-anchored property gate against alternative quality-gate strategies (e.g., coverage-guided fuzzing, schema-based testing) on the same repository. The marginal benefit of the property layer over the doctest layer alone was not quantified.

7. **Selection bias in target repository.** The choice of `more-itertools` was informed in part by its known high density of doctest-bearing docstrings. Repositories with less documentation discipline may present integration challenges not observed here.

## Conclusion

A docstring-backed property gate was integrated into a real third-party Python repository as a local pytest hook. The gate executed 710 docstring examples and 9 docstring-anchored Hypothesis properties, killed all three synthetic behavioral mutants in a mutation probe, and did not disturb the existing upstream test suite (718 tests, 19,896 subtests). Runtime and memory overhead were modest enough for CI-style usage on this repository.

These results support the bounded hypothesis that real-repo integration is viable for docstring examples plus manually curated docstring-anchored properties. They do not support the stronger claim that fully automatic property extraction from arbitrary docstrings is feasible, nor that the approach generalizes beyond this single, favorably selected repository. The next scientific step should test semi-automatic property suggestion and ranking across a broader set of APIs or a second real repository, rather than repeating this same integration on `more-itertools`.
