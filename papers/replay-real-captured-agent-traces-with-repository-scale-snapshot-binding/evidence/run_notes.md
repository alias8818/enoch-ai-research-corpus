## Hypothesis and target metric

Hypothesis tested: captured agent shell traces become substantially more replayable when each command is bound back to its originating repository/workspace snapshot, compared with replaying the same command from a generic current workspace or a mismatched repository.

Direct target metric: command-level replay agreement for real captured Codex `command_execution` events, measured as exact exit-code match, exact stdout/stderr aggregate match, and combined exact match.

Baselines/controls:
- `origin`: run the command in the project directory inferred from the trace path.
- `current`: run the same command from this project directory, ignoring the captured repository binding.
- `shuffled`: run the same command from another fixed-seed randomly selected project directory.

Only read-only shell commands are replayed. Mutating, installation, network, process-control, and long-running commands are excluded by a conservative static filter. This is a direct replayability test for captured shell traces, but it is not a full agent-decision replay and it does not restore historical intermediate filesystem states.

## Final bounded validation result

Dataset: 775 local real Enoch/Codex JSONL trace logs under the CPU worker's project root, excluding this evaluation project from the final replay corpus. The parser saw 16,103 completed command events. The conservative read-only filter admitted 6,256 safe commands, then excluded 9 from this current project, leaving 6,247 commands. Each command was replayed under three bindings (`origin`, `current`, `shuffled`) for 18,741 total command executions.

Final aggregate metrics from `results/replay_summary.json`:

| Binding | n | exact exit+output match | exit match | output match | timeout |
| --- | ---: | ---: | ---: | ---: | ---: |
| origin repository binding | 6,247 | 58.11% | 99.50% | 58.14% | 0.29% |
| current-workspace baseline | 6,247 | 18.23% | 90.04% | 18.26% | 0.29% |
| shuffled-repository control | 6,247 | 17.62% | 90.52% | 17.66% | 0.29% |

Paired bootstrap exact-match lift from `results/replay_diagnostics.json`:
- origin minus current baseline: +39.88 percentage points, 95% bootstrap CI [38.64, 41.17].
- origin minus shuffled control: +40.48 percentage points, 95% bootstrap CI [39.23, 41.78].

Command family diagnostics:
- `sed`: 3,145 commands, origin exact 81.18%, current exact 14.69%.
- `jq`: 64 commands, origin exact 92.19%, current exact 3.13%.
- `wc`: 148 commands, origin exact 60.81%, current exact 1.35%.
- `tail`: 158 commands, origin exact 58.86%, current exact 2.53%.
- `find`: 1,448 commands, origin exact only 13.74%, current exact 9.12%, showing that repository binding alone does not reconstruct historical file inventories after later artifacts/logs are added.

## Interpretation

The mechanism is supported for the scoped claim: binding real captured shell commands back to their originating repository/workspace materially improves replayability over both a generic current workspace and mismatched repositories. This is not a small synthetic proxy; it used all safe read-only commands found in the local captured trace corpus and included real baselines/controls.

The result is not publication-ready as a general agent-trace replay system. Exact replay under origin binding is still only 58.11%. Failure examples in `results/replay_diagnostics.json` show that many mismatches are commands whose output depends on mutable workspace inventories (`find`, `rg --files`, `wc` over growing logs/results) or historical intermediate files. A repository-scale final snapshot binding is therefore helpful but insufficient: faithful replay needs per-command or periodic historical filesystem snapshots/manifests, not just a binding to the final project directory.

Directly tested:
- Real captured Codex command traces from local Enoch projects.
- Read-only shell command replay with exact output/exit-code comparison.
- Origin repository binding vs current-workspace baseline and shuffled-repository control.

Only proxied/not tested:
- Full agent-decision replay.
- Mutating command replay.
- Network/install commands.
- Restoration of historical intermediate filesystem states.
- External/non-Enoch trace corpora.

Evidence that would overturn or deepen this result:
- A replay system with pre-command Merkle manifests or filesystem snapshots that raises exact replay well beyond the 58% origin-final-snapshot ceiling on the same corpus.
- The same origin-binding lift on an independent external corpus of captured agent traces.

## Final decision

No paper-positive closure. The run produced a useful, direct, bounded signal: repository/workspace binding is necessary and gives a large replayability lift, but final-snapshot binding alone is insufficient for high-fidelity repository-scale replay.
