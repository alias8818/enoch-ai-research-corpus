# Origin Workspace Binding for Replay of Real Captured Agent Shell Traces

## Abstract

We measure how much the exact replayability of real captured agent shell commands depends on running each command in the same repository or workspace in which it was originally issued. On 6,247 read-only `command_execution` events extracted from 775 local real Enoch/Codex trace logs, we compare three bindings: (i) the inferred origin repository, (ii) a generic current workspace, and (iii) a fixed-seed shuffled project directory. Under origin binding, exact exit-code plus aggregated stdout/stderr match is 58.11%, compared with 18.23% under current-workspace and 17.62% under shuffled-project replay. Paired bootstrap 95% confidence intervals for the origin lift are [+38.64, +41.17] percentage points over the current-workspace baseline and [+39.23, +41.78] over the shuffled-project control. Exit-code-only match is 99.50% under origin binding, indicating that the residual non-exact cases are dominated by output drift rather than by command-level failure. Per-command-family diagnostics show large lift for `sed`, `jq`, `wc`, and `tail`, but only 13.74% exact match for `find` under origin binding, indicating that final-snapshot repository binding alone does not reconstruct historical file inventories after later workspace drift. The project decision for this run is `finalize_negative`: the result is a useful, bounded empirical signal that origin binding is a necessary component of faithful shell-trace replay, but final-snapshot binding leaves 41.89% of commands non-exact and is therefore not sufficient for high-fidelity repository-scale replay. A follow-up adding per-command or periodic filesystem manifests is recommended, with the success threshold defined as ≥80% exact match and ≥+20 percentage points over origin-final-snapshot binding.

## 1. Introduction

Agent coding assistants, including the local Codex and Enoch traces used in this study, issue many shell commands during a task. Faithful replay of these traces is a prerequisite for offline evaluation, regression testing, and audit. A natural first hypothesis is that the dominant source of replay non-determinism is the file and project state in which each command runs: a `find`, `rg`, `wc`, or `sed` issued inside one repository will produce a different result if executed from a different repository, or from a "current" workspace that has been mutated by later artifacts.

This paper reports a direct, bounded measurement of that hypothesis on a real captured trace corpus. The question is narrow and falsifiable:

> For a real captured read-only shell command, does binding the command to its originating project workspace materially improve exact exit-plus-output replay relative to a generic current workspace or a mismatched repository?

We do not attempt full agent-decision replay, mutating-command replay, network or install replay, or restoration of historical intermediate filesystem states. We measure one thing: whether the choice of working directory, given a fixed final-snapshot binding, moves a real captured command's output from "almost never exactly reproducible" to "frequently exactly reproducible" on the local Enoch/Codex corpus.

The run is recorded as a direct bounded validation, not as a paper-positive closure. The project decision (`finalize_negative`) explicitly records that the empirical lift is real but bounded, and that final-snapshot binding alone leaves a large residual failure rate concentrated in file-inventory commands. Both readings are honored in this draft: the lift is large and well-measured, and the paper is honest that final-snapshot repository binding is not by itself a sufficient replay system.

## 2. Method

### 2.1 Trace corpus

We use local real Enoch/Codex JSONL trace logs located under `<worker-project-root>/*/.enoch/logs/*.jsonl`. The final bounded run uses 775 such trace logs; the current evaluation project is excluded from the corpus to avoid self-referential replay. The parser saw 16,103 completed `command_execution` events.

### 2.2 Read-only filtering

A conservative static filter admits only read-only shell commands. Mutating, installation, network, process-control, and long-running commands are excluded. After filtering, 6,256 commands were admitted, of which 9 belong to the current evaluation project and are excluded, leaving 6,247 commands. Each command is replayed under three bindings, for 18,741 total command executions.

### 2.3 Bindings

For each command we recover the originating project from the trace path and define three workspace bindings:

- `origin`: run the command in the project directory inferred from the trace path.
- `current`: run the same command from this evaluation project's working directory, ignoring the captured binding.
- `shuffled`: run the same command from a fixed-seed randomly selected project directory.

The shuffled-project control is necessary because the current-workspace baseline alone could in principle be either easier or harder than a mismatched repository; the shuffled control confirms that the lift is specifically due to origin binding and not merely due to executing inside any well-formed repository. The exact shuffled seed is not transcribed in the local context excerpt and is therefore not reported here.

### 2.4 Metrics

For each execution we record:

- exact exit-code match against the captured event;
- exact aggregated stdout+stderr match against the captured event;
- combined exact exit+output match;
- timeout indicator, with a per-command timeout of 3 s in the stable final run.

We report per-binding aggregates and per-command-family diagnostics. We additionally report paired bootstrap 95% confidence intervals for the exact-match lift of `origin` over `current` and over `shuffled`. The bootstrap configuration (number of resamples, exact seed values) is not transcribed in the local context excerpt; the intervals are quoted as recorded in `results/replay_diagnostics.json`.

### 2.5 Harness and execution

The harness is `scripts/replay_trace_bindings.py`. Replay is parallelized with up to 16 worker subprocesses. Per-command timeout is 3 s. The stable final run is recorded in `logs/replay_full_exclude_current.log`. Expected wall-clock is under 15 minutes on a CPU-only worker, with peak RSS under 500 MB. Per-command results are appended to `results/replay_events.jsonl`; aggregate metrics are written to `results/replay_summary.json`; paired bootstrap intervals and failure diagnostics to `results/replay_diagnostics.json`; sample manifest and project fingerprints to `results/replay_manifest.json`.

### 2.6 What is not replayed

This is a shell-command replay system, not an agent-decision replay system. It does not restore historical intermediate filesystem states. It does not replay mutating, installation, or network commands. The harness binds to the final-snapshot state of the project directory, not to a pre-command Merkle manifest. The project decision explicitly records that the validation uses local Enoch traces only and binds to final project workspaces rather than to historical pre-command filesystem snapshots.

## 3. Results

### 3.1 Aggregate replayability

| Binding | n | exact exit+output match | exit match | output match | timeout |
| --- | ---: | ---: | ---: | ---: | ---: |
| origin repository binding | 6,247 | 58.11% | 99.50% | 58.14% | 0.29% |
| current-workspace baseline | 6,247 | 18.23% | 90.04% | 18.26% | 0.29% |
| shuffled-repository control | 6,247 | 17.62% | 90.52% | 17.66% | 0.29% |

Origin binding more than triples the exact exit+output match rate relative to either control. Exit-code match under origin binding is 99.50%, indicating that the residual non-exact cases are dominated by output drift rather than by command-level failure. Timeout rate is 0.29% across all three bindings, so the comparison is not contaminated by command-level timeouts.

### 3.2 Paired lift

From `results/replay_diagnostics.json`:

- Origin minus current baseline: +39.88 percentage points, 95% bootstrap CI [+38.64, +41.17].
- Origin minus shuffled control: +40.48 percentage points, 95% bootstrap CI [+39.23, +41.78].

The two lifts are essentially indistinguishable, which is consistent with the current workspace in this evaluation context being a poor proxy for any particular captured repository. The origin binding is doing real work, not merely selecting "some" repository. The similarity between the current and shuffled controls also indicates that the lift is not an artifact of comparing against a single weak baseline; the shuffled control partially ablates the concern that the current-workspace baseline is unusually mismatched.

### 3.3 Per-command-family diagnostics

| Family | n | origin exact | current exact |
| --- | ---: | ---: | ---: |
| `sed` | 3,145 | 81.18% | 14.69% |
| `jq` | 64 | 92.19% | 3.13% |
| `wc` | 148 | 60.81% | 1.35% |
| `tail` | 158 | 58.86% | 2.53% |
| `find` | 1,448 | 13.74% | 9.12% |

The first four families show the expected pattern: under origin binding they reach 58–92% exact match, while under current-workspace replay they collapse to single-digit percentages. `find` is the diagnostic exception: even with origin binding, exact match is only 13.74%, only modestly above the 9.12% under the current-workspace baseline. Inspection of failure cases in `results/replay_diagnostics.json` shows that the dominant source of mismatch for `find` (and, by inspection, for `rg --files` and `wc` over growing logs) is workspace drift between the time the command was originally issued and the final snapshot bound by the harness: artifacts, logs, and result files added later change the file inventory that `find` enumerates.

### 3.4 What the result supports and does not support

What the result supports, in the bounded scope above:

- Origin repository binding is a necessary ingredient of faithful shell-trace replay. Without it, exact exit+output match on a real captured corpus is below 19%.
- The lift is large (+39.88 pp over current, +40.48 pp over shuffled), tight, and consistent across the two controls.
- The mechanism is consistent with the per-family diagnostics: commands whose output is anchored to specific file paths in a specific project benefit dramatically from origin binding; commands whose output depends on the evolving workspace inventory do not.

What the result does not support:

- That final-snapshot origin binding is sufficient. 41.89% of commands remain non-exact under origin binding, concentrated in file-inventory commands affected by later workspace drift.
- That mutating, install, or network commands are replayable under this system; they are explicitly excluded.
- That the result generalizes to non-Enoch trace corpora; only local Enoch/Codex traces are evaluated.
- That full agent-decision replay is achieved; the harness replays shell commands, not higher-level agent decisions.

## 4. Limitations

The following are explicit, recorded limitations from the project decision and run notes:

1. Local traces only. The validation uses local Enoch/Codex trace logs under the CPU worker's project root. The result is not demonstrated on external or non-Enoch corpora.
2. Read-only only. Mutating, installation, network, process-control, and long-running commands are excluded by a conservative static filter. The reported lifts apply only to the read-only subset.
3. Shell commands only. The harness replays shell commands, not full agent decisions. Higher-level agent planning, tool selection, and context management are not evaluated.
4. Final-snapshot binding only. The harness binds each command to the project's final-snapshot directory state, not to a pre-command Merkle manifest or to a per-command historical filesystem snapshot. This is the direct cause of the residual 41.89% non-exact match rate, concentrated in `find` and related file-inventory commands.
5. Baseline weakness. The current-workspace baseline is the evaluation project's own working directory. It is therefore an unusually weak baseline (it is unlikely to match any of the 775 captured repositories) and may overstate the lift that would be observed against a "generic" workspace containing overlapping files. The shuffled-project control, where the lift is +40.48 pp, partly addresses this concern.
6. Corpus structure. The corpus is local and may share structure across projects (tooling, caches, common dependencies). The shuffled-project control only partially ablates project-level confounds.
7. Follow-up not executed. The follow-up hypothesis, required evidence, and success threshold are recorded in the project decision but are not executed in this run. The result is `finalize_negative` for the current run; a future bounded deepening run is recommended, not promised.
8. Configuration details not transcribed. The exact bootstrap iteration count and the exact shuffled-project seed are not present in the local context excerpt and are therefore not reported in this draft. Their absence is a limitation of the documentation excerpt, not of the underlying experiment.
9. Self-reference exclusion. The 9 commands from the current evaluation project are excluded from the 6,256 admitted commands to avoid self-referential replay. This is a small, principled exclusion, but it slightly changes the effective population of `find` and other families.
10. Negative outcome. The project decision is `finalize_negative`; this draft documents the result but does not claim paper-positive closure.

## 6. Conclusion

We presented a direct, bounded measurement of the effect of origin-repository binding on the exact replayability of real captured agent shell commands. On 6,247 read-only commands from 775 local Enoch/Codex trace logs, origin binding raises exact exit+output match from 18.23% (current-workspace) and 17.62% (shuffled-project) to 58.11%, with paired bootstrap 95% confidence intervals of [+38.64, +41.17] and [+39.23, +41.78] percentage points respectively. Per-command-family diagnostics show that the lift is concentrated in `sed`, `jq`, `wc`, and `tail`, and that `find` and other file-inventory commands remain a substantial failure mode because the harness binds to the project's final snapshot rather than to a per-command historical filesystem state.

The result supports the project decision's classification of `hypothesis_status: supported` and `research_outcome: useful_signal`: origin binding is a necessary ingredient of faithful shell-trace replay, and the measured lift is large and well-controlled. The result does not support a paper-positive closure: 41.89% of commands remain non-exact under origin binding, and the project decision is `finalize_negative` for this run. The recommended next step, recorded in the project decision as a follow-up of type `deepen` at depth 3, is to add per-command or periodic filesystem manifests or snapshots on top of origin binding and to test whether the exact-match rate rises above the final-snapshot 58.11% ceiling on the same trace corpus. The pre-registered success threshold for that follow-up is ≥80% exact match and ≥+20 percentage points over origin-final-snapshot binding, with a stop condition of <10 percentage points improvement or unavailability of the required pre-command manifests.
