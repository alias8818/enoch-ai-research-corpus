## Abstract

Reproducibility of computational workflows frequently fails when local side effects—generated files, ignored configuration, symlinks, and secret-valued settings—are omitted from handoffs. This paper reports on a lightweight reproducibility capsule mechanism that captures a directory's full state (file contents, symlink targets, command replay specifications, host metadata, and SHA-256 content hashes) into a JSON capsule, with regex-based redaction of secret-shaped values. We validate the mechanism against 80 synthetic messy workflows spanning five variant classes (cache-required, nested paths, paths with spaces, symlinks, and Unicode paths). A naive baseline that copies only source-like files failed all 80 replays; capsule replay succeeded in all 80 cases with zero raw-secret leaks. Median capture-plus-replay latency was 43.0 ms (p95: 58.6 ms) at a throughput of 9.2 cases/s. Peak RSS was 18.9 MB. These results support the capsule mechanism for the tested class of synthetic messy workflows, but generality to real-world workflows, large binaries, non-Python toolchains, and container-dependent setups remains unmeasured. The structured claim ledger for this artifact is currently in `blocked_empty_claims` status, meaning no formal claims have passed evidence audit; the findings reported here should be interpreted accordingly.

## Introduction

A common reproducibility failure mode in computational work is the "clean handoff" problem: a researcher shares source code and test scripts but omits generated artifacts, local configuration, cached state, or symlink structure that the workflow implicitly depends on. The recipient cannot replay the workflow without manually reconstructing the missing side effects.

Prior approaches to reproducibility include containerization, virtual machines, and provenance-tracking systems. These solutions are effective but impose setup cost, platform assumptions, or workflow restructuring that may not be justified for lightweight local workflows. A complementary approach is a *reproducibility capsule*: a self-contained archive that captures sufficient directory state and replay metadata to reconstruct a workflow's filesystem preconditions without requiring containers or specialized infrastructure.

The central risk for any capsule mechanism is failure under *messy* real workflows—workflows with untracked generated files, ignored configuration, symlinks, paths containing spaces or Unicode characters, and secret-valued settings that must not leak into artifacts. Toy capsules on clean directories may succeed trivially; the mechanism's value depends on whether it handles the messy cases that cause naive handoffs to fail.

This paper reports results from a targeted validation: we implement a stdlib-only capsule prototype, generate 80 synthetic messy workflows across five variant classes, and compare naive baseline replay against capsule replay. We measure pass/fail rates, secret leakage, latency, and resource consumption. We do not claim these results generalize beyond the synthetic fixture classes tested.

## Method

### Implementation

The prototype consists of three components, all written in Python using only the standard library:

1. **`src/repro_capsule.py`** — Captures a directory into a JSON capsule containing:
   - File contents (read as text or base64-encoded bytes)
   - Symlink targets
   - Command specifications for replay
   - Host and Python version metadata
   - SHA-256 hashes for each restored file and for the capsule manifest itself
   - Regex-based redaction of values matching common secret patterns (tokens, passwords, API keys) in text files

2. **`src/messy_workflow_validation.py`** — Generates disposable messy workflow fixtures, executes both a naive baseline copy and a capsule capture/replay cycle for each fixture, and records row-level and aggregate metrics. The naive baseline intentionally copies only source and test-like files while omitting ignored generated, configuration, and cache state, modeling the common failure mode where a clean-looking handoff misses local side effects.

3. **`tests/test_repro_capsule.py`** — Unit test verifying redaction and replay on a minimal capsule.

### Messy Workflow Variants

Each generated workflow exercises one of five variant classes:

| Variant | Description |
|---|---|
| `cache_required` | Workflow depends on a generated cache file that naive copy omits |
| `nested` | Deeply nested directory structure |
| `spaces` | Paths containing spaces |
| `symlink` | Symlinks that must be resolved or preserved |
| `unicode` | Paths and content containing Unicode characters |

These five variants were chosen to represent common classes of local side effects that break naive file-copy handoffs. They do not constitute an exhaustive taxonomy of messy workflow pathologies.

### Experimental Protocol

All experiments were conducted on a single Linux host with 121 GiB RAM and 116 GiB available memory (0 B swap). This workload is CPU/filesystem only; no GPU inference or CUDA calibration was applicable.

The protocol comprised six steps:

1. **Syntax check**: `python3 -m py_compile` on all source and test files — passed with no output.
2. **Unit tests**: `python3 -m unittest discover -s tests -v` — 1 test passed (log: `logs/unit_tests.log`).
3. **Memory posture**: Recorded available memory before smoke test (log: `logs/memory_posture_before_smoke.log`).
4. **Smoke test**: 5 cases, seed 20260501, output to `artifacts/smoke/` (log: `logs/smoke.log`).
5. **Calibrated pilot**: 80 cases, seed 20260501, output to `artifacts/calibrated/` (log: `logs/calibrated.log`).
6. **Invariant check**: Inline Python assertions over `artifacts/calibrated/rows.json` and `artifacts/calibrated/metrics.json` (log: `logs/invariant_check.log`).

The smoke test served as a sanity check before the larger calibrated run. The calibrated pilot is the primary evidence source.

### Metrics Collected

For each run, we recorded:

- **baseline_pass / baseline_fail**: Number of workflows where naive copy replay succeeded or failed.
- **capsule_pass / capsule_fail**: Number of workflows where capsule replay succeeded or failed.
- **secret_leak_cases**: Number of capsule JSON files containing unredacted secret-shaped values.
- **total_redactions**: Count of redacted values across all capsules.
- **latency_ms_p50 / p95_capture_plus_replay**: Wall-clock latency for the combined capture and replay cycle.
- **throughput_cases_per_sec**: Cases processed per wall-clock second.
- **mean_capsule_files**: Average number of files per capsule.
- **resource**: Peak RSS, available memory before and after, swap state.

## Limitations

1. **Synthetic workflows only.** All 80 test cases were programmatically generated disposable fixtures. Real-world messy workflows may exhibit failure modes not covered by the five variant classes tested here (e.g., large binary artifacts, multi-process coordination, network-dependent setup, non-deterministic model outputs). Generality to real external workflows remains unmeasured.

2. **Secret redaction scope.** The regex-based redaction handles common text secret shapes (tokens, passwords, API keys) but may miss non-standard patterns, binary-encoded secrets, or secrets whose exact values are semantically required for replay correctness. The invariant check verified that *generated* secret-shaped tokens were redacted; it did not test adversarial or obfuscated secret encodings.

3. **No container or system package capture.** The capsule records host and Python metadata but does not capture system packages, container images, or toolchain versions beyond what Python reports. Workflows depending on specific system libraries or non-Python runtimes may not replay correctly.

4. **Single-platform execution.** All experiments ran on a single Linux host with 121 GiB RAM. Behavior on resource-constrained or differently configured systems was not tested.

5. **No scale stress test.** Mean capsule size was 7.46 files. Workflows with thousands of files or large binary artifacts may exhibit different latency, memory, and correctness profiles.

6. **Deterministic fixture generation.** The messy workflows were generated with a fixed seed (20260501), which ensures reproducibility of the validation itself but limits the diversity of tested edge cases to those produced by the generator.

7. **No negative or mixed results observed.** The uniform 80/80 pass rate for capsules and 0/80 for the baseline reflects the deliberate design of the experiment: the baseline was constructed to fail, and the capsule was constructed to handle exactly the failure modes the baseline exhibits. The absence of edge-case failures may indicate insufficient test diversity rather than robustness.

8. **Claim ledger empty.** No structured claims have been extracted or audited for this artifact. The empirical observations have not undergone formal claim/evidence audit.

## Conclusion

A lightweight reproducibility capsule mechanism—capturing file contents, symlinks, command specs, host metadata, and SHA-256 hashes with regex-based secret redaction—was implemented and validated against 80 synthetic messy workflows spanning five variant classes. The naive baseline (copying only source-like files) failed all 80 replays; capsule replay succeeded in all 80 cases with zero raw-secret leaks and modest resource consumption (p95 latency 58.6 ms, peak RSS 18.9 MB).

These results support the capsule mechanism as viable for the tested class of synthetic messy local workflows, but they do not constitute a claim of universal scientific closure. The uniform pass/fail pattern reflects the deliberate design of the experiment and may overestimate robustness. The claim ledger remains in blocked status with no audit-approved claims. The principal remaining validity gap is measurement against real messy research and code workflows, including externally curated fixtures, to estimate generality and identify ecosystem-specific edge cases such as large binaries, container dependencies, non-Python toolchains, network-dependent setup, and secrets whose exact values are semantically required for replay.
