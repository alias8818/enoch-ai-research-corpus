## Abstract

We investigate whether retrieved source- or domain-metadata can serve as an upcycling signal for Mixture-of-Experts (MoE) expansion: selecting which experts to clone and initializing router bias so that new child experts specialize toward retrieved domains, while keeping Top-K active compute fixed. In a controlled Top-1 sparse-regression MoE proxy (3→6 expert expansion, 10 seeds, 24k train / 12k test samples), retrieval-conditioned expert expansion (RCEE) reduced test MSE by 70.4% relative to the 3-expert base router and by 43.3% relative to a 6-expert utility-clustered expansion baseline when retrieval metadata accuracy was 0.90. However, a sensitivity sweep reveals that RCEE fails to beat the utility baseline when retrieval accuracy drops below approximately 0.85, and at accuracy 0.50 it performs substantially worse (−85.8% relative to utility clustering). A no-retrieval fallback condition preserved base-router performance exactly. These results are from a synthetic sklearn-based proxy, not a language model. The mechanism appears viable for next-stage small-MoE experimentation only if retrieval-confidence gating and no-retrieval fallback are enforced.

## 1. Introduction

Mixture-of-Experts architectures allocate different parameters to different inputs via a router, enabling model capacity to grow without proportional compute increase. A recent line of work on expert upcycling explores how to expand an existing MoE by duplicating experts and extending the router, rather than training a larger sparse model from scratch.

Expert Upcycling (arXiv:2604.19835) defines progressive expert-count expansion through expert duplication plus router extension while holding Top-K fixed, and identifies utility-based expert selection as a strong baseline. Sparse Upcycling (arXiv:2212.05055) establishes dense-to-sparse checkpoint reuse as a way to avoid full sparse training from scratch. Upcycling Large Language Models into Mixture of Experts (arXiv:2410.07524) reports billion-parameter upcycling studies showing that upcycling can outperform continued dense training. Cluster-Aware Upcycling (arXiv:2604.13508) supports the general premise that structure-aware router and expert initialization can break expert symmetry and improve early specialization.

We ask: can *retrieved* domain metadata—available at inference time in retrieval-augmented systems—provide a stronger signal for choosing which experts to clone and how to bias the expanded router? The intuition is that if a retrieval system identifies the domain of the current input, newly cloned child experts can be initialized to specialize toward that domain, and the router can be biased accordingly, all without increasing active compute.

We term this mechanism Retrieval-Conditioned Expert Expansion (RCEE). This paper reports results from a controlled sparse-regression MoE proxy. We do not yet report language-model results.

## 2. Method

### 2.1 Problem Setting

We consider a Top-K MoE with K fixed before and after expansion. The base model has E experts; the expanded model has E' > E experts. Active compute (Top-K dispatch) is unchanged. The question is how to select which experts to clone and how to initialize the expanded router.

### 2.2 Baseline Conditions

We compare five conditions in a controlled proxy:

1. **Base 3-expert x-router.** A Top-1 MoE with 3 experts routed from input features only. This is the pre-expansion baseline.
2. **Utility x-cluster 6-expert.** A strong non-retrieval expansion baseline. The base 3 experts are expanded to 6 via utility-based clustering of the training data, still Top-1. This represents the best achievable expansion without retrieval metadata.
3. **RCEE retrieval-domain 6-expert.** The proposed mechanism: 6 child experts routed by noisy retrieval-domain metadata, still Top-1. Retrieval metadata accuracy is a controlled parameter.
4. **RCEE no-retrieval fallback.** An explicit fallback that uses the original base x-router when retrieval is absent. This tests whether the mechanism degrades gracefully.
5. **Oracle per-domain.** A lower-bound reference using true domain labels for routing, representing the best achievable performance with perfect domain information.

### 2.3 RCEE Mechanism

The RCEE mechanism operates as follows:

- **Expert cloning:** Experts are cloned according to per-retrieved-domain loss/utility. Domains associated with higher loss receive more expert copies.
- **Router bias:** The expanded router is biased so that child experts receive preferential routing weight when the retrieval system identifies the corresponding domain.
- **Confidence gating:** When retrieval metadata confidence falls below a threshold, the system falls back to the base router (no-retrieval fallback condition).
- **Top-K preservation:** Active compute remains fixed at Top-1 throughout.

### 2.4 Experimental Proxy

The proxy is implemented in `scripts/rcee_synthetic.py` using sklearn-based sparse regression. It is a mechanism smoke test, not a language-model pretraining run. The proxy uses synthetic input features and domain-structured regression targets to simulate the MoE routing and expert-specialization dynamics.

**Training configuration (full run):**

- Seeds: 0 through 9
- Training samples: 24,000
- Test samples: 12,000
- Retrieval accuracy: 0.90 (primary condition)
- Sensitivity sweep: retrieval accuracy ∈ {0.50, 0.70, 0.80, 0.90, 0.95}

## 3. Results

### 3.1 Primary Comparison (Retrieval Accuracy = 0.90)

Table 1 reports mean test MSE across 10 seeds at retrieval accuracy 0.90.

| Condition | Mean Test MSE | Std | n |
|---|---|---|---|
| Oracle per-domain | 0.1233 | 0.0015 | 10 |
| Base 3-expert x-router | 28.9584 | 3.9226 | 10 |
| Utility x-cluster 6-expert | 15.0903 | 3.0914 | 10 |
| RCEE retrieval-domain 6-expert | 8.5577 | 1.3206 | 10 |
| RCEE no-retrieval fallback | 28.9584 | 3.9226 | 10 |

**Effect sizes:**

- RCEE vs. base: 70.4% MSE reduction
- RCEE vs. utility x-cluster: 43.3% MSE reduction
- Utility x-cluster vs. base: 47.9% MSE reduction

RCEE substantially outperforms both baselines when retrieval metadata is accurate. The no-retrieval fallback condition exactly matches the base router (MSE = 28.9584), confirming that fallback behavior is preserved without degradation.

The oracle per-domain condition (MSE = 0.1233) shows that there is substantial room between RCEE at 0.90 retrieval accuracy and the theoretical best, indicating that improved retrieval or more sophisticated routing could yield further gains.

### 3.2 Retrieval-Quality Sensitivity

Table 2 reports RCEE performance as a function of retrieval metadata accuracy.

| Retrieval Accuracy | RCEE vs. Base MSE Reduction | RCEE vs. Utility MSE Reduction | Utility vs. Base MSE Reduction |
|---|---|---|---|
| 0.50 | 1.3% | −85.8% | 46.8% |
| 0.70 | 27.9% | −39.1% | 48.2% |
| 0.80 | 45.4% | −1.6% | 46.3% |
| 0.90 | 70.4% | 43.3% | 47.9% |
| 0.95 | 84.6% | 69.7% | 49.1% |

RCEE is only better than the non-retrieval utility baseline when retrieval metadata accuracy exceeds approximately 0.85 in this proxy. At accuracy 0.80, RCEE is roughly tied with utility clustering (−1.6% relative). At accuracy 0.70 and below, RCEE performs substantially worse than utility clustering, and at 0.50 it is nearly indistinguishable from the unexpanded base.

This sensitivity profile constitutes a concrete operational guardrail: RCEE requires calibrated retrieval-domain confidence before routing to child experts. Below the threshold, the system must fall back to utility or base-router behavior.

## 4. Limitations

1. **Synthetic proxy, not a language model.** All results come from a sklearn-based sparse-regression MoE proxy. The dynamics of expert specialization, router collapse, and domain overfitting in real transformer-based MoE language models may differ substantially. No language-model pretraining or fine-tuning results are reported.

2. **Small expert count.** The proxy uses 3 base experts expanded to 6. Real MoE language models typically have 8–64 or more experts. Scaling behavior is unknown.

3. **Top-1 only.** All conditions use Top-1 routing. The interaction between RCEE and Top-K > 1 routing is unexplored.

4. **Single proxy task.** The regression proxy has a fixed domain structure. Real retrieval-augmented language tasks exhibit far more complex, overlapping, and noisy domain structure.

5. **Retrieval modeled as noise.** Retrieval inaccuracy is modeled as uniform label noise on domain metadata. Real retrieval systems exhibit structured errors (e.g., confusions between similar domains, systematic biases) that may interact differently with RCEE routing.

6. **No training dynamics reported.** We report only final test MSE, not training curves, router entropy, expert load balance, or other diagnostic measures that would be critical in a real MoE setting.

7. **Threshold calibration is proxy-specific.** The approximate 0.85 accuracy threshold is specific to this proxy's domain structure and noise model. It should not be treated as a universal constant.

## 6. Conclusion

Retrieval-Conditioned Expert Expansion is a mechanism that uses retrieved domain metadata to select which MoE experts to clone and how to bias the expanded router, while preserving fixed Top-K active compute. In a controlled sparse-regression proxy, RCEE reduced test MSE by 70.4% relative to the base router and by 43.3% relative to a utility-clustered expansion baseline when retrieval accuracy was 0.90. The no-retrieval fallback preserved base-router performance exactly.

However, the mechanism is sharply sensitive to retrieval quality. Below approximately 0.85 retrieval accuracy in this proxy, RCEE fails to beat the utility expansion baseline, and at 0.50 accuracy it performs substantially worse. This establishes a concrete operational requirement: RCEE must be gated on calibrated retrieval confidence, with fallback to base or utility routing when confidence is insufficient.

These results are from a synthetic proxy and do not constitute a language-model result. The mechanism is viable enough to justify a next-stage experiment on a small open MoE language model, provided that (a) retrieval-confidence gating is implemented, (b) no-retrieval fallback is preserved and tested for regression, and (c) the experiment is killed if low-confidence retrieval routing behaves worse than utility expansion or if router collapse, over-specialization, or domain overfitting is observed.
