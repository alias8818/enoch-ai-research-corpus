## Abstract

We present a prototype design and implementation for rollback, audit, and escrow semantics in transactional tool calls—operations invoked by automated agents that may produce side effects on external systems. The prototype provides escrowed prepare-commit-rollback lifecycle management, a tamper-evident hash-chained audit log, and crash/restart recovery for reversible local tool calls. In a single-process benchmark of 1,000 transactions (800 commits, 200 rollbacks, 3,000 audit events), the prototype processed approximately 4,780 transactions per second with a valid audit chain. Smoke tests confirm that staged preparation does not mutate the target until commit, that rollback restores prior state, and that audit chain integrity is verifiable. However, the design does not solve rollback for arbitrary non-idempotent external APIs, and the audit log is tamper-evident but not externally anchored. These results are from a local single-writer prototype; production deployment would require concurrency control, external audit anchoring, and per-tool compensation or idempotency contracts.

## 1 Introduction

Automated agents increasingly invoke tool calls that produce side effects on external systems—file writes, API mutations, database updates. When an agent's multi-step plan fails partway through, there is often no mechanism to roll back the effects of already-executed tool calls, leaving systems in inconsistent states. Similarly, there is typically no auditable record of which tool calls were attempted, which committed, and which were rolled back.

Transaction semantics (prepare, commit, rollback) are well-established in database systems but are not generally available for arbitrary tool calls, particularly those targeting external APIs or remote services. Two-phase commit protocols require participant cooperation that most external APIs do not provide.

This work explores a pragmatic middle ground: an escrow-based approach where tool effects are staged in an intermediate buffer (the "escrow") before being committed to the target, combined with a tamper-evident audit log that records the full lifecycle of each transactional tool call. For tools whose effects can be staged locally (e.g., file writes), true rollback is achieved by discarding the escrow. For tools with irreversible external effects, the framework requires explicit compensation or idempotency metadata, acknowledging that universal rollback is not achievable without per-tool cooperation.

We implement and evaluate a dependency-free Python prototype to assess the feasibility and performance of this approach.

## 2 Method

### 2.1 Design

The prototype implements three core abstractions:

1. **Escrow buffer.** A tool call's effects are first written to an escrow staging area rather than directly to the target. The target remains unmodified until an explicit commit. On rollback, the escrow is discarded and the target retains its prior state.

2. **Transaction lifecycle.** Each tool call follows a prepare → commit | rollback lifecycle. The prepare stage writes to escrow. The commit stage atomically applies the escrowed effect to the target. The rollback stage discards the escrowed effect and restores any prior committed content.

3. **Tamper-evident audit log.** Every lifecycle event (prepare, commit, rollback) is appended to a JSONL audit log. Each entry includes a SHA-256 hash of the previous entry's hash, forming a hash chain. Chain integrity can be verified by recomputing hashes from the first entry to the last. An append-tail cache optimization avoids re-reading the full log for each append.

### 2.2 Implementation

The prototype is implemented in Python with no external dependencies, located in `src/tx_tool_escrow/`. Key components:

- **Escrow manager:** Manages staging buffers keyed by transaction ID.
- **Transaction controller:** Orchestrates the prepare-commit-rollback lifecycle.
- **Audit logger:** Appends hash-chained entries to a JSONL file with tail-cache optimization.
- **Recovery module:** On restart, replays the audit log to reconstruct the state of in-flight transactions.

### 2.3 Testing

Tests in `tests/test_transactional_escrow.py` cover:

- Escrow invisibility: staged preparation does not mutate the target.
- Commit correctness: committed content appears at the target.
- Rollback correctness: rollback restores the prior committed content.
- Crash/restart recovery: the audit log can be replayed to recover transaction state after a simulated restart.
- Audit tamper detection: modification of any audit entry is detected during chain verification.

### 2.4 Benchmark Protocol

The benchmark executes 1,000 transactions against a local file target:

- 800 transactions commit.
- 200 transactions roll back.
- 3,000 audit events are generated (prepare, commit/rollback per transaction).
- The audit chain is verified after all transactions complete.

The benchmark measures wall-clock elapsed time and computes transactions per second. It was run twice: once before and once after the audit tail-cache optimization. Only the post-optimization result is reported as the final metric.

### 3.4 Negative and Mixed Results

Several important limitations qualify the positive results:

- **No concurrency testing.** The prototype is single-process and single-writer. No concurrent access patterns were tested. The benchmark reflects a serial workload only.
- **No external API integration.** All tool calls target local file operations. The escrow-based rollback mechanism depends on the ability to stage effects before committing them. External APIs that immediately mutate state cannot use this path without tool-specific compensation or reservation semantics.
- **Audit integrity is tamper-evident, not tamper-proof.** The hash chain detects modification but does not prevent it. An adversary with write access to the audit log could rewrite the entire chain. External anchoring (e.g., signing chain heads or publishing them to an append-only store) is not implemented.
- **Recovery is replay-based.** Crash recovery depends on the audit log being intact and complete. If the log is corrupted or truncated, recovery may be partial or impossible. No checksumming or redundancy is applied to the log itself.

## 4 Limitations

1. **Single-process, single-writer.** The prototype has no locking, no concurrency control, and no support for multiple concurrent writers. Production use would require a transactional datastore or distributed locking.

2. **No universal rollback for external APIs.** The escrow approach provides true rollback only for tools whose effects can be staged before application. Non-idempotent external APIs (e.g., sending an email, executing a payment) require per-tool compensation actions or idempotency proofs that are outside the scope of this prototype.

3. **Audit log is not externally anchored.** The tamper-evident hash chain provides detection of post-hoc modification but does not provide the cryptographic guarantees of a signed or externally anchored log. In adversarial settings, an attacker who can modify the log file can also recompute the chain.

4. **Benchmark is synthetic and local.** The ~4,780 tx/s figure reflects a single-process in-memory and local-filesystem workload with no network latency, no contention, and no persistence guarantees beyond local disk. This figure should not be interpreted as a production performance estimate.

5. **No formal correctness proof.** Correctness is validated by unit tests and smoke tests only. No formal model or proof of atomicity, consistency, isolation, or durability (ACID) properties is provided.

6. **Single benchmark run.** No statistical analysis was performed. The throughput figure is from a single execution; variance across runs was not characterized.

## 6 Conclusion

This prototype demonstrates that escrow-based prepare-commit-rollback semantics and tamper-evident audit logging are feasible for local, reversible tool calls in a single-process setting. The benchmark throughput of ~4,780 transactions per second and the successful verification of a 3,000-event audit chain suggest that the overhead of the escrow and audit mechanisms is modest for local workloads.

However, the core problem—rollback for arbitrary tool calls with external side effects—remains unsolved by this approach alone. The escrow mechanism requires that effects be stageable before commitment, which is not possible for many external APIs. For such tools, the framework can only enforce that compensation or idempotency metadata is declared; it cannot guarantee correct rollback without per-tool cooperation.

The audit log provides tamper-evidence but not tamper-resistance. External anchoring of chain heads would strengthen integrity guarantees in adversarial settings.

These results are from a local research prototype and should not be generalized to production performance or correctness without further validation under concurrent, distributed, and adversarial conditions.
