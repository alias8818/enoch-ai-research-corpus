## Abstract

We present salience-backed candidate top-1 KV prefill packing, a server-side prompt compression method for structured extraction from long-context inputs. The method identifies candidate JSON spans in the input, ranks them by canonical and schema-salience cues, and packs only the top-ranked candidate plus a bounded salience window before tokenization and KV-cache prefill. On a validated 2,000-example toy JSON extraction benchmark with the Qwen2.5-3B-Instruct tokenizer, the method achieved critical-field accuracy of 1.000 while reducing average prompt tokens by 93.9% versus full-context prefill and by 32.0% versus a prior salience-only packer. In a bounded server-side generation benchmark (8 examples, Qwen2.5-3B-Instruct on CUDA), the packed policy achieved critical accuracy of 1.000 versus 0.500 for the unpacked baseline, reduced prompt tokens by 86.2%, and reduced mean request latency by 39.8%. All unpacked baseline failures were attributable to context-window truncation that removed the target JSON before the model could attend to it. These results are limited to a single model, a toy extraction task, and small sample sizes; generalization to production serving stacks and broader task distributions remains unvalidated.

## 1. Introduction

Large language models serving long-context inputs face a fundamental tension: the KV-cache prefill cost scales linearly with input length, yet many long inputs contain only small regions of task-relevant content. In structured extraction tasks—where the model must locate and reproduce specific JSON objects embedded in lengthy surrounding text—much of the input context is irrelevant to the extraction target.

Prior work on salience-based prompt compression has demonstrated that retaining only high-salience regions can preserve extraction quality while reducing token count. However, salience-only packing may still retain substantial context around the target region. When the input contains multiple candidate JSON spans, a more targeted approach is possible: identify the single most likely candidate and pack only that span plus a minimal salience window.

This paper reports on salience-backed candidate top-1 KV prefill packing, a method that combines candidate span detection with salience-guided ranking to produce a compact prefill input. We evaluate the method on a toy JSON extraction benchmark using both deterministic tokenization metrics and actual model generation through an OpenAI-compatible serving shim. The results support the hypothesis that candidate-top-1 packing preserves extraction quality while substantially reducing prompt tokens and request latency, though the evidence is bounded by sample size and task specificity.

## 2. Method

### 2.1 Salience-Backed Candidate Top-1 Packing

The packing transform (`pack_salience_candidate_top1`) operates on raw text input before chat-template rendering and tokenization. The procedure is:

1. **Candidate span detection.** Scan the input text for JSON-like candidate spans (balanced braces, syntactic plausibility).
2. **Candidate ranking.** Score each candidate by combining canonical/source-of-truth cues and stale/cache/schema salience signals. The top-ranked candidate is selected.
3. **Salience window extraction.** Extract a bounded character window around the top candidate, controlled by a configurable character budget (8,192 characters in the reported experiments).
4. **Packing.** Replace the full input with the packed candidate-plus-salience-window text. The packed text is then passed through the normal chat-template rendering and tokenization path.

The method is implemented as a reusable function in `scripts/prefill_packing.py` and integrated into an OpenAI-compatible Transformers serving shim in `scripts/openai_prefill_packing_server.py`. The server applies packing server-side, immediately before chat-template rendering, when the `--packing-policy salience_candidate_top1` flag is set.

### 2.2 Baseline Policies

Four packing policies were compared in the deterministic/tokenization benchmark:

- **full_context:** No packing; the entire input is passed to the tokenizer.
- **parent_salience:** A prior salience-only packer that retains high-salience regions without candidate selection.
- **head_tail:** A truncation baseline retaining the head and tail of the input.
- **salience_candidate_top1:** The proposed method.

In the server-side generation benchmark, two policies were compared:

- **none:** No server-side packing; the full input is sent to the model (subject to the model's 8,192-token context window limit).
- **salience_candidate_top1:** The proposed method applied server-side.

### 2.3 Benchmark Design

Two benchmark stages were conducted:

**Stage 1: Deterministic/tokenization benchmark.** This stage measures packing quality and tokenization metrics without model inference. The Qwen2.5-3B-Instruct tokenizer is applied to the packed output, and extraction accuracy is scored by comparing the packed text against ground-truth JSON fields. This benchmark ran on all 2,000 examples of the validated toy extractor dataset.

**Stage 2: Server-side generation benchmark.** This stage measures end-to-end extraction quality, latency, and throughput with actual model inference. An OpenAI-compatible Transformers shim serves Qwen2.5-3B-Instruct on CUDA. Identical full-context extraction requests are sent to the server under both packing policies. The benchmark ran on 8 examples with `max_new_tokens=128`.

Additional metrics for salience_candidate_top1: average packed characters = 2,384.8, average pack latency = 0.091 ms, throughput = 1,455.5 samples/s, prefill tokens/s = 1.11M.

The candidate-top-1 method matched parent-salience quality (critical accuracy 1.000) while reducing average prompt tokens by 32.0% (763.7 vs. 1,123.1). Versus full_context, prompt tokens were reduced by 93.9%. The full_context baseline's low critical accuracy (0.415) reflects the fact that many examples exceed the effective token budget when no packing is applied, causing the target JSON to be truncated.

Prompt tokens were reduced by 86.2% and mean request latency by 39.8% under the packed policy. The packed policy produced zero truncation events; the unpacked policy truncated 9,454.6 tokens on average (the full-context prompts exceeded the 8,192-token input limit).

**Failure analysis.** The unpacked baseline failed 4 of 8 examples. All failures were caused by long full-context prompts being truncated to the 8,192-token limit, which removed the canonical JSON target before the model could attend to it. The packed policy had zero failures and zero truncation.

### 3.4 Output Token Rate Disparity

The packed policy produced fewer output tokens per second (284.8 vs. 1,158.6). This disparity is expected: the unpacked baseline generated more tokens overall because it often produced lengthy incorrect outputs before failing, while the packed policy produced concise correct outputs within the 128-token budget. The output token rate is therefore not a direct measure of generation efficiency.

## 4. Limitations

1. **Small generation sample size.** The server-side generation benchmark evaluated only 8 examples. While the deterministic/tokenization benchmark covered 2,000 examples, the end-to-end generation results are based on a very small sample and may not reflect performance on the full distribution.

2. **Single model.** All results use Qwen2.5-3B-Instruct. Performance may differ for other model families, sizes, or architectures.

3. **Toy task.** The benchmark is a synthetic JSON extraction task with known structure. Generalization to open-ended extraction, summarization, or other task types is untested.

4. **Short generation budget.** The generation benchmark used `max_new_tokens=128`. Performance under longer generation budgets (where the model must produce extended structured output) is unknown.

5. **Single hardware environment.** All experiments ran on a single CUDA machine with ample host memory (~108 GiB available). Performance under memory pressure or on different GPU architectures is untested.

6. **No production serving validation.** The serving shim is a prototype OpenAI-compatible Transformers server. Integration with production serving stacks (vLLM, SGLang, TensorRT-LLM) and prefix-cache systems is unvalidated.

7. **Candidate ranking heuristics.** The candidate ranking relies on heuristic salience cues (canonical, stale, cache, schema signals). These heuristics may not generalize to JSON schemas or document structures not represented in the toy dataset.

8. **Deterministic benchmark does not measure model quality.** The 2,000-example deterministic benchmark scores whether the packed text contains the target fields, not whether a model would correctly extract them. The server-side generation benchmark does measure model quality but on only 8 examples.

9. **No comparison with other compression methods.** Beyond the parent-salience and head-tail baselines, no comparison with learned compression, retrieval-augmented, or other prompt-reduction methods was conducted.

## 6. Conclusion

Salience-backed candidate top-1 KV prefill packing preserves extraction quality on the tested toy JSON extraction benchmark while substantially reducing prompt tokens and request latency. In the deterministic/tokenization benchmark (2,000 examples), the method achieved critical accuracy of 1.000 with 93.9% fewer prompt tokens than full-context prefill and 32.0% fewer than a prior salience-only packer. In the server-side generation benchmark (8 examples, Qwen2.5-3B-Instruct), the method achieved critical accuracy of 1.000 versus 0.500 for the unpacked baseline, with 86.2% prompt-token reduction and 39.8% latency reduction. All baseline failures were attributable to context-window truncation that removed the target JSON.

These findings are bounded by the small generation sample size, single-model evaluation, and toy task design. The current project artifacts support this finding in the tested setting, but the method's behavior under broader conditions—different models, longer generations, production serving stacks, and diverse task distributions—remains unknown. Future work should target integration with production serving infrastructure (vLLM, SGLang prefix-cache systems) and evaluation on more demanding extraction tasks.
