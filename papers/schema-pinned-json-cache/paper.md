## Abstract

Structured-output JSON generation via large language models (LLMs) is increasingly common, yet repeated queries sharing the same JSON Schema are typically served without leveraging schema-level cache identity. This work investigates whether treating a JSON Schema as a pinned component of both local result-cache keys and LLM prompt prefixes yields correct, safe, and potentially latency-reducing cache behavior. We implement a minimal SQLite-backed prototype that canonicalizes schemas via sorted-key JSON, hashes them with SHA-256, and validates outputs against the schema before cache write and after cache read. In local microbenchmarks across 1,000 randomized key-order variants, canonical hashing produces a single stable hash where naïve `json.dumps` produces 1,000 distinct hashes. Schema revision correctly isolates cache entries, and invalid outputs are rejected before write. Median local overheads are 110.9 µs to canonicalize an 80-field schema, 82.1 µs for a cache hit with validation on a 25-field output, and 2,182.6 µs for a cache put with validation. Token-budget estimates using `tiktoken` suggest that schemas of 80 or more fields exceed the 1,024-token prefix-cache threshold documented by OpenAI, making them plausibly eligible for prompt caching when placed before dynamic content. No live provider or local model-server latency measurements were performed; provider-side cache-hit behavior remains best-effort and unvalidated in this work.

## Introduction

LLM APIs that support structured outputs—JSON constrained by a JSON Schema—enable reliable programmatic consumption of model responses. When the same schema is reused across many requests (e.g., a fixed extraction schema applied to many documents), two opportunities arise:

1. **Local result caching.** Identical `(model, schema, input)` tuples should yield cacheable responses, but only if the cache key is deterministic. Naïve JSON serialization produces different byte representations for equivalent schemas that differ only in key order, fragmenting the cache.

2. **Provider prompt-prefix caching.** APIs such as OpenAI's prompt caching and vLLM's automatic prefix caching reduce time-to-first-token (TTFT) when request prefixes match previously computed key-value caches. A stable, canonical schema placed before variable content improves prefix-match eligibility.

This work addresses the question: *Can structured-output JSON results be cached safely and usefully when the JSON Schema is treated as a pinned part of the cache identity and prompt prefix?*

We do not attempt to measure end-to-end latency improvements from provider-side caching, as no API credentials or local model server were available during the study. Instead, we establish correctness and safety properties of the local caching layer, measure its overhead, and estimate schema prefix token budgets to assess eligibility for provider-side prefix caching.

## Method

### Design

The schema-pinned cache is a key-value store (SQLite-backed) where each entry is identified by the composite key:

```
(tenant_salt, model, decoder_mode, schema_hash, input_hash)
```

Both `schema_hash` and `input_hash` are derived from canonical JSON representations to ensure that semantically equivalent schemas and inputs map to the same key regardless of key ordering or formatting differences.

### Canonicalization

Schemas and inputs are canonicalized using Python's `json.dumps(sort_keys=True, separators=(',', ':'), ensure_ascii=False)` with UTF-8 encoding. The hash is SHA-256 applied to the canonical byte string.

This approach is deterministic within a single Python process and JSON implementation but does not constitute a full RFC 8785 canonical JSON implementation; numeric edge cases (e.g., floating-point representation differences) are not addressed.

### Validation

Every value written to the cache is validated against the pinned schema using the `jsonschema` library. Every value read from the cache is re-validated before return. This double-validation discipline ensures that:

- Invalid model outputs are never stored.
- Cache corruption or schema drift does not silently serve incorrect data.

### Cache Key Isolation on Schema Revision

When a schema changes (even by a single field), the canonical hash changes, producing a different cache key. Old entries under the previous schema hash are not migrated or replayed. This isolation is by design: a schema change may alter the semantics of fields, and silent migration would be unsafe.

### Prompt-Prefix Layout

For provider-side prefix caching, the recommended request layout places static content (schema, instructions) before dynamic content (user document, query). This ordering maximizes the length of the reusable prefix. Public documentation from OpenAI states that prompt caching requires exact prefix matches and that static content should be placed at the beginning of the request. vLLM's automatic prefix caching similarly hashes token blocks with parent-hash chaining, making prefix stability a prerequisite for cache reuse.

### Implementation

The prototype consists of:

- `src/schema_pinned_cache.py` — SQLite-backed cache with canonical hashing, schema validation, and key isolation.
- `scripts/benchmark_schema_pinned_cache.py` — Correctness and microbenchmark harness (1,000 iterations).
- `tests/test_schema_pinned_cache.py` — Unit tests for canonical hashing, schema revision isolation, and invalid-output rejection.

Dependencies: `jsonschema`, `tiktoken` (for token estimation only).

### Token-Budget Estimation

Schema prefix token counts were estimated using `tiktoken` with the `o200k_base` tokenizer. Schemas of 5, 25, 80, 160, and 320 fields were generated, canonicalized, and tokenized. These estimates approximate the prefix size that would be presented to the model API; actual tokenization by the provider may differ.

## Results

### Canonical Hash Stability

Across 1,000 randomized object-key-order variants of the same schema, canonical hashing produced **1 distinct hash**. Naïve `json.dumps` (without `sort_keys`) produced **1,000 distinct hashes** over the same variants. This confirms that key-order variation is a real source of cache fragmentation and that deterministic canonicalization eliminates it for the tested cases.

### Cache Correctness

| Property | Result |
|---|---|
| Cache hit survives schema and input key reordering | true |
| Schema revision isolates cache key (old entry not replayed) | true |
| Invalid output rejected by schema validation before cache write | true |

All three unit tests passed.

### Local Overhead

Median timings over 1,000 iterations on the benchmark host:

| Operation | Median (µs) |
|---|---|
| Canonicalize + hash 80-field schema | 110.9 |
| Cache hit + validate 25-field output | 82.1 |
| Cache put + validate 25-field output | 2,182.6 |

Cache put overhead is dominated by SQLite write latency. Cache hit overhead (82.1 µs) is small relative to typical LLM API round-trip times (hundreds to thousands of milliseconds), suggesting the local caching layer adds negligible latency on the hit path. These timings are specific to the benchmark host and Python runtime; absolute microsecond values should not be generalized.

### Schema Prefix Token Budgets

| Fields | Schema bytes | Estimated prefix tokens (o200k_base) |
|---:|---:|---:|
| 5 | 741 | 171 |
| 25 | 3,181 | 671 |
| 80 | 9,891 | 2,046 |
| 160 | 19,651 | 4,046 |
| 320 | 39,171 | 8,046 |

OpenAI's documented prompt-cache activation threshold is 1,024 tokens. Schemas of 80 or more fields exceed this threshold by themselves, before any instructions or examples are added. Smaller schemas (5–25 fields) fall below the threshold and would require additional static prefix content (instructions, few-shot examples) to reach cache eligibility.

These are estimates based on `tiktoken`; actual provider tokenization may differ, and the threshold applies to the full prefix, not the schema alone.

## Limitations

1. **No live provider or local model-server latency measurements.** No API credentials or local vLLM server were present in the project environment. All provider-side claims (cache eligibility, TTFT reduction) are grounded only in public documentation and prior work, not in empirical measurement from this study. The actual cache-hit rate, TTFT improvement, and cost savings remain unvalidated.

2. **Canonicalization is not fully cross-language or cross-implementation.** The prototype uses `json.dumps(sort_keys=True)`, which is deterministic within CPython but does not implement RFC 8785 canonical JSON. Numeric representation differences (e.g., `1.0` vs. `1`, floating-point precision) could cause hash divergence across languages or JSON libraries. Production deployments requiring cross-language cache identity should adopt an explicit canonicalization standard such as RFC 8785 or canonical CBOR.

3. **Provider prompt-cache behavior is best-effort.** Local schema pinning improves prefix-match eligibility and correctness but cannot guarantee cache hits. Provider-side routing, cache eviction, and model-version changes may prevent prefix reuse. Cache-hit telemetry (e.g., `cached_tokens` in OpenAI usage) must be monitored in production.

4. **Structured Output schema support varies by provider and model.** Not all JSON Schema features are supported by all providers' structured-output modes. Unsupported features must be detected before requests are issued; this prototype does not address schema-compatibility validation against provider constraints.

5. **Microbenchmark host specificity.** Timing results are specific to the benchmark host and Python runtime. Absolute microsecond values should not be generalized; the relevant finding is that local overhead is small relative to LLM API latency, not the specific numbers.

6. **Single-prototype scope.** Only one cache implementation (SQLite-backed) was tested. Alternative backends (Redis, in-memory LRU) may exhibit different performance profiles.

7. **Claim ledger audit is blocked.** The automated claim ledger for this paper contains no structured claims and is flagged as `blocked_empty_claims`. The correctness and performance findings reported here are grounded in the local prototype artifacts and benchmark metrics, but the paper has not passed a formal claim/evidence audit.

## Conclusion

A schema-pinned JSON cache—where the canonical hash of a JSON Schema is a first-class component of the cache key and the schema is placed at the beginning of the LLM request prefix—is viable as both a safety/correctness layer and a prompt-cache-friendly request pattern for structured-output workloads with repeated schemas.

Local prototype results confirm that deterministic canonicalization eliminates key-order-induced cache fragmentation, that schema revision correctly isolates cache entries, and that schema validation prevents storage and serving of invalid outputs. Local overhead on the cache-hit path is negligible relative to LLM API latency. Token-budget estimates indicate that schemas of 80+ fields exceed the documented 1,024-token prefix-cache threshold, making them plausibly eligible for provider-side prompt caching.

However, these findings are necessary but not sufficient conditions for production benefit. The actual value of schema-pinned caching depends on provider-side cache-hit rates, TTFT reductions, and cost savings, none of which were measured in this work. We assign medium-high confidence to the viability claim: the local correctness and safety properties are empirically established, but the latency and cost benefits remain unvalidated and must be confirmed with live A/B benchmarks against real provider endpoints.
