## Abstract

We investigate whether lightweight, context-derived dynamic grammar constraints applied at decoding time can recover extraction accuracy in a retrieval-augmented evaluation setting where a small language model (Phi-4-mini) produces correct internal representations but emits incorrectly formatted or hallucinated outputs. Using llama.cpp GBNF constrained decoding, we progressively isolate failure modes across four benchmark families (abstention, code navigation, compression robustness, structured extraction) on curated-long (1,600 examples) and uncurated-control (400 examples) splits. A baseline parent-prompt configuration yields 40.50% accuracy on curated long and 39.25% on uncurated control. A context-derived dynamic grammar mechanism that constrains decoding to candidate literals extracted from the retrieved context window raises accuracy to 99.50% on both splits (+59.00 and +60.25 percentage points respectively), with per-family accuracy of 100% (abstention), 100% (code navigation), 98% (compression robustness), and 100% (structured extraction). The remaining 0.5% errors are compression cases where the topic grammar permits alternate valid-looking triples despite the correct answer being parser-visible. Throughput and latency improve modestly under the dynamic grammar. These results are specific to Phi-4-mini, the llama.cpp inference backend, and the benchmark corpus described; external replication and multi-model validation remain open.

## 1. Introduction

Retrieval-augmented generation (RAG) systems often exhibit a frustrating failure mode: the relevant evidence is present in the context window, the model's internal representations appear to encode the correct answer, but the emitted output fails to match the expected format or hallucinates plausible-but-incorrect alternatives. This gap between retrieval success and extraction accuracy is particularly acute for small language models operating under constrained decoding budgets.

This report documents a progressive series of experiments on Phi-4-mini that test whether the extraction accuracy gap can be closed not by retraining or modifying the retrieval pipeline, but by applying lightweight, context-derived grammar constraints at decoding time. The approach exploits a structural observation: when the retrieved context window contains candidate answer literals (checksums, helper identifiers, topic triples), constraining the model's output grammar to those specific literals can prevent format contamination and hallucination without altering the model, dataset, retrieval strategy, or scoring methodology.

The investigation proceeded through four stages of increasing specificity: (1) a broad prompt and grammar sweep identifying which failure families respond to output-shape constraints, (2) a targeted code-navigation pilot isolating whether prompt changes alone suffice, (3) an abstention/no-candidate pilot testing context-derived checksum grammars, and (4) a full-benchmark scale-up validating the combined mechanism. At each stage, the dataset, retrieval strategy, exact-token scoring, model, and context size were held fixed; only the decoding grammar varied.

## 2. Method

### 2.1 Benchmark and Scoring

The evaluation uses a parent benchmark corpus with four answer families:

- **Abstention**: The correct answer is `ABSTAIN` when the requested record is absent from the retrieved context, or a specific checksum when present.
- **Code navigation**: The correct answer is a helper identifier (e.g., `helper_action_###`) present in the retrieved code context.
- **Compression robustness**: The correct answer is a topic triple (e.g., `TOPIC-####:action:#`) summarizing the retrieved passage.
- **Structured extraction**: The correct answer is a checksum extracted from a structured record in the retrieved context.

Scoring is deterministic exact-token matching. The benchmark provides two splits: **curated long** (1,600 examples) and **uncurated control** (400 examples). The `parser_available_accuracy` metric indicates whether the correct answer literal is present in the retrieved context window, independent of whether the model emits it.

### 2.2 Inference Backend

All experiments use Phi-4-mini served via llama.cpp with GBNF (GGML BNF) constrained decoding. Temperature is set to 0 for all configurations reported. The llama.cpp server is launched per-run and verified absent after completion.

### 2.3 Decoding Configurations

We test the following decoding configurations, each building on the previous:

1. **`parent_temp0`**: The original parent prompt with no grammar constraint. Baseline.
2. **`copy_grammar_temp0`**: A union grammar permitting all valid answer shapes (`ABSTAIN`, checksum, `TOPIC-####:action:#`, `helper_action_###`). Tests whether broad format constraints alone improve accuracy.
3. **`copy_family_grammar_temp0`**: Family-specific grammars selecting only the valid shape for each answer family (e.g., `helper_action_###` only for code navigation). Tests whether cross-family format contamination causes errors.
4. **`code_candidates_dynamic_grammar_temp0`**: For code navigation, a dynamic grammar built from helper identifiers present in the retrieved context window. Tests whether context-derived literal constraints recover code navigation.
5. **`copy_hybrid_dynamic_grammar_temp0`**: Combines family-specific grammars for abstention, compression, and structured extraction with the dynamic code-navigation grammar. Tests the combined mechanism before full abstention repair.
6. **`copy_record_candidate_dynamic_grammar_temp0`**: The recommended configuration. Extends dynamic context-derived grammars to all families: for abstention/structured extraction, constrains decoding to checksum candidates visible in the context (or literal `ABSTAIN` if none); for code navigation, constrains to helper identifiers in the context; for compression, uses the family-level topic grammar.

### 2.4 Experimental Protocol

Each configuration was tested through a staged protocol:

1. **Smoke test** (8 examples): Verify harness correctness and server liveness.
2. **Broad sweep** (32 examples, six configurations, three splits): Identify responsive failure families.
3. **Confirmatory run** (64 examples per split): Measure accuracy, throughput, and latency for the best-performing configurations.
4. **Targeted pilots** (64 examples per split, family-isolated): Isolate remaining failure modes for code navigation and abstention.
5. **Full-benchmark scale-up** (1,600 curated + 400 uncurated examples): Validate the final recommended configuration against the parent baseline in a same-run comparison.

At each stage, wall-clock time, max RSS, CPU utilization, GPU utilization snapshots, and post-run process state were recorded.

## 3. Results

### 3.1 Broad Prompt/Grammar Sweep

The initial 64-example confirmatory run established that broad grammar constraints (`copy_grammar_temp0`) materially improve accuracy over the parent prompt, but gains are concentrated in format-sensitive families:

| Configuration | Curated Long | Uncurated Long |
|---|---|---|
| `parent_temp0` | 39.06% | 39.06% |
| `copy_grammar_temp0` | 65.62% (+26.56) | 62.50% (+23.44) |

The gain was concentrated in `compression_robustness` errors (6.25% → 100%); structured extraction remained strong, abstention improved modestly, and **code navigation remained at 0%** despite the grammar permitting the correct helper shape.

Family-specific grammars (`copy_family_grammar_temp0`) matched `copy_grammar_temp0` on both splits (65.62% curated, 62.50% uncurated) but did not recover code navigation (0%), isolating the failure to selecting the correct helper from code context rather than cross-family format contamination.

### 3.2 Code-Navigation Targeted Pilot

Prompt variants and broad code-shape grammars all remained at 0% accuracy on code navigation, despite `parser_available_accuracy=100%` (the correct helper identifier was always present in the retrieved context). The dynamic context-derived grammar (`code_candidates_dynamic_grammar_temp0`) recovered code navigation to **100%** on both curated and uncurated 64-example samples.

The hybrid configuration (`copy_hybrid_dynamic_grammar_temp0`) combining family grammars with the dynamic code-navigation grammar reached:

| Split | Accuracy | Δ from Parent |
|---|---|---|
| Curated long | 90.62% | +51.56 |
| Uncurated long | 87.50% | +48.44 |

Per-family accuracy (curated): abstention 62.5%, code navigation 100%, compression 100%, structured extraction 100%. The remaining errors were concentrated in abstention cases where the model hallucinated checksum-shaped outputs for absent records.

### 3.3 Abstention/No-Candidate Pilot

The record-candidate dynamic grammar (`copy_record_candidate_dynamic_grammar_temp0`) constrains decoding to checksum candidates visible in the context when the requested record is present, and to literal `ABSTAIN` when no matching record is found. On the abstention-only 64-example confirmatory run, this reached **100%** on both splits, while parent and broad checksum-or-ABSTAIN grammars remained near 48–52% because absent-record cases still produced hallucinated checksum-shaped outputs.

On the mixed-family 64-example confirmatory run, the record-candidate dynamic grammar reached **100%** on both curated and uncurated long splits, with per-family accuracy of 100% for all four families.

### 3.4 Full-Benchmark Scale-Up

The final validation ran `copy_record_candidate_dynamic_grammar_temp0` against the full curated long split (1,600 examples) and uncurated control split (400 examples), with a same-run `parent_temp0` comparator:

| Split | Parent Accuracy | Dynamic Grammar Accuracy | Δ |
|---|---|---|---|
| Curated long | 40.50% | 99.50% | +59.00 |
| Uncurated control | 39.25% | 99.50% | +60.25 |

Per-family accuracy under the recommended configuration (both splits):

| Family | Accuracy |
|---|---|
| Abstention | 100% |
| Code navigation | 100% |
| Compression robustness | 98% |
| Structured extraction | 100% |

The remaining 0.5% errors were all compression/topic cases where the exact answer was parser-visible in the retrieved context, but the family-level topic grammar still permitted an alternate valid-looking topic triple.

### 3.5 Throughput and Latency

The dynamic grammar configuration improved throughput and reduced latency relative to the parent baseline:

| Metric | Curated Long (Dynamic) | Curated Long (Parent) | Uncurated (Dynamic) | Uncurated (Parent) |
|---|---|---|---|---|
| Samples/s | 3.97 | 3.50 | 3.93 | 3.44 |
| p50 latency | 0.247s | 0.286s | 0.246s | 0.286s |
| p95 latency | 0.449s | 0.598s | 0.456s | 0.600s |

Full-benchmark wall time was 18:02.46 with max RSS 10,314,620 KB at 99% CPU. Mid-run GPU utilization ranged from approximately 60–96%. Post-run system state showed ~116 GiB available memory, no swap usage, and no lingering `llama-server` processes.

## 4. Limitations

1. **Single model**: All results are specific to Phi-4-mini. Whether context-derived dynamic grammars produce comparable gains on other models (larger or smaller, different architectures) is not established by the present artifacts.

2. **Single benchmark corpus**: The evaluation uses one benchmark with four answer families. Generalization to other retrieval-augmented tasks, domains, or answer formats is not demonstrated.

3. **llama.cpp prototype, not production**: These are llama.cpp hook-prototype results using GBNF constrained decoding. They are not CUDA copy calibration results or final production validation. The mechanism depends on llama.cpp's grammar-constrained sampling infrastructure, and production deployment would require equivalent constrained decoding support in the target inference stack.

4. **Dynamic grammar depends on retrieval quality**: The context-derived grammar mechanism assumes that the retrieved context window already contains the correct answer literal. When `parser_available_accuracy` is less than 100%, the grammar cannot constrain toward an absent target. The current benchmark has high retrieval quality; performance may degrade with noisier retrieval.

5. **Residual compression errors**: The 0.5% remaining errors on both full splits are compression/topic cases where the topic grammar permits multiple valid-looking triples. A context-derived topic-candidate grammar could potentially close this gap, but this has not been tested.

6. **No external replication**: All experiments were conducted in a single computational environment. Independent replication is needed to confirm the observed effect sizes.

7. **Grammar construction overhead**: The dynamic grammar must be constructed per-query by parsing candidate literals from the retrieved context. This adds a small preprocessing cost not separately measured in the current artifacts.

8. **Temperature fixed at 0**: All results use temperature 0. The interaction between dynamic grammar constraints and sampling temperature has not been explored.

## 6. Conclusion

Context-derived dynamic grammar decoding materially recovers Phi-4-mini extraction accuracy on a retrieval-augmented benchmark, raising accuracy from approximately 40% to 99.5% on both curated and uncurated splits. The mechanism operates by constraining the model's output to candidate literals extracted from the retrieved context window, preventing format contamination and hallucination without modifying the model, dataset, retrieval pipeline, or scoring methodology. The improvement holds across four answer families, with residual errors confined to compression cases where the topic grammar permits alternate valid triples.

The progressive experimental structure reveals that broad output-shape constraints alone recover format-sensitive errors but leave code navigation and abstention failures untouched; family-specific grammars isolate but do not resolve these failures; and only context-derived literal constraints close the gap. This suggests that the dominant failure mode for Phi-4-mini in this setting is not lack of knowledge but inability to select the correct output from a plausible set—a failure that prompt engineering and broad grammar constraints cannot address, but that per-query dynamic grammars can.

Whether these findings generalize to other models, benchmarks, and inference backends remains an open question requiring independent replication. The current project artifacts support the finding in the tested setting; they do not establish universal applicability.
