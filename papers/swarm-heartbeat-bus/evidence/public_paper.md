## Abstract

Long-running agent swarms require a governance and control-plane layer that can reason over compact, typed state rather than consuming full agent transcripts. This paper reports on a local prototype implementing a SQLite-backed latest-state heartbeat bus: each agent publishes a compact typed tuple (status, progress, budget, blocker identifier/severity, evidence reference, sequence, timestamp) to a shared table, and a controller process scans only the latest heartbeat rows to detect blockers, budget drift, and stale agents. We evaluate the prototype across three configurations: a 5-agent smoke test (5 s), a 5-agent 30-second calibration run, and a 20-agent stress run at approximately 100 heartbeats per second. In all configurations, the compact heartbeat mode achieved zero missed blockers and zero stale agents, with controller scan p95 latency at or below 0.105 ms. Under the 20-agent stress condition, compact heartbeats consumed 2.4% of one CPU core, compared to 6.0% for a transcript-polling baseline carrying 64 KiB synthetic payloads per heartbeat—a 2.5× CPU overhead ratio. Write p95 latency was 14.8× worse and scan p95 latency 6.7× worse for the transcript baseline. These results provide a positive local signal for the viability of compact latest-state heartbeats as a control-plane mechanism, but the evaluation is limited to a synthetic local simulation and does not validate semantic governance quality in real multi-agent workflows. The claim ledger for this artifact remains in a blocked-empty state; no structured claims have passed audit.

## 1 Introduction

As agent-based systems scale to multi-agent swarms executing long-running tasks, a control-plane challenge emerges: how does a coordinating process (e.g., a project manager or orchestrator) maintain situational awareness without reading every agent's full transcript? Transcript polling is expensive in both I/O and reasoning cost, and it does not scale with the number of agents or the length of their execution histories.

The core hypothesis of this work is that a compact, typed heartbeat schema—where each agent publishes only its latest state tuple to a shared bus—can serve as a sufficient control-plane substrate for detecting blockers, budget drift, staleness, and evidence references. The controller reads only the latest row per agent, rather than scanning full transcripts.

This paper reports on a dependency-free Python 3 prototype that implements this design using SQLite as the backing store. We evaluate the prototype's transport and control-plane mechanics across three configurations of increasing scale, comparing compact heartbeats against a transcript-polling baseline that attaches 64 KiB payloads to each heartbeat row. We measure write and scan latency, CPU overhead, blocker detection accuracy, and staleness tracking.

The contribution is an empirical characterization of the overhead and correctness properties of a compact latest-state heartbeat bus under synthetic load. We do not evaluate the design against real LLM agents or real governance decisions.

## 2 Method

### 2.1 Architecture

The prototype (`src/heartbeat_bus_experiment.py`) implements two concurrent layers:

1. **Agent threads.** Each of $N$ agent threads periodically writes a heartbeat row to a SQLite table. The heartbeat schema includes: `agent_id` (primary key, enforcing latest-state semantics via `INSERT OR REPLACE`), `status`, `progress`, `budget`, `blocker_id`, `blocker_severity`, `evidence_kind`, `evidence_ref`, `sequence`, and `timestamp`. In "transcript" mode, each heartbeat additionally carries a configurable-size byte payload simulating a transcript excerpt.

2. **Controller thread.** A single controller thread scans the heartbeat table at a fixed frequency, reading only the latest row per agent. It checks for: (a) blockers with severity above a threshold, (b) agents whose latest heartbeat is older than a configured staleness TTL, and (c) budget drift. The controller does not read any transcript payload.

### 2.2 SQLite Configuration

The SQLite database is configured with:

- **WAL mode** for concurrent read/write access.
- **`synchronous=NORMAL`** to balance durability and write throughput.
- **Latest-state semantics** enforced by using `agent_id` as the primary key with `INSERT OR REPLACE`, ensuring each agent retains exactly one current row.

### 2.3 Experimental Conditions

Three runs were executed:

| Run | Agents | Duration (s) | Heartbeat Hz | Scan Hz | Mode | Transcript payload |
|---|---:|---:|---:|---:|---|---|
| Smoke | 5 | 5 | 2 | 10 | compact | none |
| Calibration | 5 | 30 | 2 | 10 | both | none (compact) / none (transcript mode, no extra payload) |
| Stress | 20 | 20 | 5 | 20 | both | 64 KiB |

The "both" mode runs compact and transcript modes sequentially within the same process, enabling paired comparison on the same hardware.

In the calibration run, the transcript mode did not carry the 64 KiB payload; it used the same compact schema but was measured as a separate mode for overhead comparison. The stress run is the primary comparison point where transcript mode carries the full 64 KiB synthetic payload.

### 2.4 Metrics

The following metrics were captured:

- **Heartbeats per second:** aggregate write throughput.
- **CPU overhead:** percentage of one CPU core consumed by the combined agent + controller workload.
- **Write p95 latency:** 95th percentile wall-clock time for a single heartbeat write.
- **Scan p50 and p95 latency:** 50th and 95th percentile wall-clock time for a full table scan by the controller.
- **Blocker detection:** ratio of detected blockers to introduced blockers, and count of missed blockers.
- **Stale agents:** maximum number of agents simultaneously flagged as stale.
- **Memory posture:** `MemAvailable_kB` and `SwapTotal_kB` from `/proc/meminfo`, captured to verify the no-swap constraint.

### 2.5 Blocker Injection

Blockers are injected by agent threads at random intervals during the run. Each injected blocker is recorded in an event log. The controller's detection count is compared against the injection count to compute missed blockers.

## 3 Results

### 3.1 Smoke Test (5 agents, 5 seconds, compact mode)

The smoke test confirmed basic functionality:

| Metric | Value |
|---|---|
| Total writes | 50 |
| Heartbeats/s | 9.99 |
| CPU overhead (% one core) | 0.288 |
| Write p95 | 1.309 ms |
| Scan p50 | 0.014 ms |
| Scan p95 | 0.033 ms |
| Blockers detected | 1/1 |
| Missed blockers | 0 |
| Max stale agents | 0 |

The elevated write p95 (1.309 ms) relative to longer runs is attributable to SQLite cold-start and WAL initialization overhead in the short 5-second window.

### 3.2 Calibration (5 agents, 30 seconds, both modes)

| Metric | Compact | Transcript (no extra payload) |
|---|---|---|
| Total writes | 300 | 300 |
| Heartbeats/s | 10.00 | 10.00 |
| CPU overhead (% one core) | 0.143 | 0.146 |
| Write p95 | 0.082 ms | 0.100 ms |
| Scan p50 | 0.026 ms | 0.017 ms |
| Scan p95 | 0.039 ms | 0.039 ms |
| Blockers detected | 11/11 | 11/11 |
| Missed blockers | 0 | 0 |

At 5 agents and 2 Hz, both modes perform similarly. The small latency differences are within measurement noise. This confirms that at low scale, the overhead of the heartbeat bus is negligible regardless of payload size.

### 3.3 Stress Test (20 agents, 20 seconds, both modes, 64 KiB transcript payload)

| Metric | Compact | Transcript (64 KiB) | Ratio (transcript / compact) |
|---|---|---|---|
| Total writes | 2000 | 2000 | — |
| Heartbeats/s | 98.64 | 98.48 | ~1.0× |
| CPU overhead (% one core) | 2.395 | 5.996 | 2.50× |
| Write p95 | 0.085 ms | 1.257 ms | 14.80× |
| Scan p50 | 0.051 ms | 0.625 ms | 12.25× |
| Scan p95 | 0.105 ms | 0.700 ms | 6.67× |
| Blockers detected | 60/60 | 60/60 | — |
| Missed blockers | 0 | 0 | — |
| Max stale agents | 0 | 0 | — |

The stress test reveals a clear divergence. While both modes maintain perfect blocker detection and zero staleness, the transcript baseline imposes substantially higher latency and CPU cost. The write p95 ratio of 14.8× reflects the cost of writing 64 KiB payloads per heartbeat; the scan p95 ratio of 6.7× reflects the cost of the controller reading those larger rows even when it discards the payload.

### 3.4 Memory Posture

All runs reported `SwapTotal_kB = 0`, confirming no swap space was present or relied upon. `MemAvailable_kB` remained stable throughout all runs (approximately 122.7 million kB available on a machine with approximately 127.5 million kB total), indicating no memory pressure was encountered even under the 20-agent stress condition.

## 4 Limitations

1. **Synthetic workload.** The experiment uses simulated agent threads that write fixed-schema heartbeats at regular intervals. No real LLM agents were run, and no real transcripts were produced or consumed. The transcript baseline uses a synthetic 64 KiB byte payload, not actual LLM output. The results characterize transport and storage mechanics, not semantic governance quality.

2. **Single-host, single-store design.** The prototype uses SQLite on a local filesystem. This architecture is appropriate for a single-host controller but does not address distributed deployment where agents run on separate machines. Distributed scenarios would require evaluating Redis Streams, NATS JetStream, or an append-log design.

3. **No schema evolution or validation.** The current prototype uses a fixed schema with no monotonic sequence validation, no TTL enforcement at the database level, no snapshot/replay mechanism, and no backpressure signaling. A production system would need these features.

4. **No real governance decisions.** The controller detects blockers and staleness by checking heartbeat fields, but it does not make complex governance decisions (e.g., reassigning work, escalating blockers, adjusting budgets). The experiment validates detection, not decision quality.

5. **Short durations and limited scale.** The longest run was 30 seconds with at most 20 agents. Long-running production swarms may exhibit different failure modes (e.g., SQLite WAL growth, lock contention under sustained write pressure, clock drift affecting staleness detection).

6. **No failure injection.** The experiment does not simulate agent crashes, network partitions, or database corruption. Robustness under failure conditions is uncharacterized.

7. **Calibration transcript mode lacked payload.** In the 5-agent calibration run, the "transcript" mode did not carry the 64 KiB payload, making it a weak comparison point at that scale. The meaningful payload comparison is only available from the 20-agent stress run.

8. **Claim ledger audit blocked.** The structured claim ledger for this artifact is in a `blocked_empty_claims` state: no claims were extracted or passed audit. The numerical results reported here are drawn directly from the run notes and project decision JSON, not from audit-approved claims. Readers should weight the findings accordingly.

9. **Missing hardware details.** Specific CPU model, Python version, and SQLite version were not recorded in the artifacts. Absolute latency numbers should be interpreted as order-of-magnitude indicators rather than portable benchmarks.

10. **Random seed not recorded.** Blocker injection uses random timing; exact reproducibility of blocker counts would require fixing and recording the random seed, which was not done.

## 6 Conclusion

A SQLite-backed latest-state heartbeat bus was implemented and evaluated as a control-plane mechanism for agent swarm coordination. Across three configurations—5-agent smoke test, 5-agent 30-second calibration, and 20-agent stress test at ~100 heartbeats/s—the compact heartbeat mode achieved zero missed blockers, zero stale agents, sub-millisecond write and scan p95 latency, and CPU overhead below 2.4% of one core.

The 20-agent stress test provides the clearest evidence for the compact-heartbeat hypothesis: compared to a transcript-polling baseline carrying 64 KiB payloads, compact heartbeats reduced CPU overhead by 2.5×, write p95 latency by 14.8×, and scan p95 latency by 6.7×, while maintaining identical blocker detection accuracy.

These results are a positive local signal for the viability of compact typed heartbeats as a governance control-plane substrate. However, the evaluation is limited to a synthetic, single-host simulation with no real LLM agents, no failure injection, and no assessment of governance decision quality. The claim ledger remains unaudited. The recommended next step is to instrument a real multi-agent workflow (e.g., a 5-agent OMX or LangGraph pipeline) with this heartbeat schema and evaluate whether a PM controller making decisions from heartbeat-only state can match the governance quality of a transcript-aware controller under realistic task conditions.
