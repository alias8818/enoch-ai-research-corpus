## Abstract

We present a local, dependency-free controller that dynamically adjusts synthetic-user load against a bounded target service using only observable telemetry—p95 latency, error rate, a utilization proxy, and a host-memory guardrail—rather than a fixed user count. In a synthetic target model (capacity 16 concurrent service slots, p95 SLO 450 ms, request timeout 900 ms), the adaptive negotiator preserved 100% safe epochs across all experimental conditions, avoiding the unsafe overload regime that fixed high user counts (48 and 80 users) entered. However, the negotiator was conservative: over 10 epochs it achieved 96.5 steady-state requests per second (RPS), which is 6.9% below the best fixed safe baseline (static 32 users, 103.7 RPS). A longer 20-epoch run narrowed this gap to 100.5 RPS at 29 users while maintaining 100% safe epochs, still 3.0% below the best fixed safe baseline. These results support the viability of SLO-preserving adaptive load negotiation as a concept but highlight that the current safety-first policy under-explores near the SLO boundary. All results derive from a synthetic target service; production validation against a live inference stack remains necessary.

## Introduction

Synthetic load testing typically relies on a fixed number of concurrent users, chosen either by rule of thumb or by manual iteration. This approach has two failure modes. An insufficient user count underloads the target and fails to reveal performance boundaries, while an excessive count violates latency service-level objectives (SLOs) and may destabilize the system under test. On platforms where swap is disabled and memory is a shared resource—such as unified memory architectures—an uncontrolled load generator can also exhaust host memory, causing kernel-level intervention rather than graceful backoff.

We investigate whether a local controller can dynamically negotiate synthetic-user pressure using only observable telemetry—p95 latency, error rate, a utilization proxy, and a host-memory guardrail—instead of requiring a pre-selected fixed user count. The controller starts from a small smoke pressure and adjusts the user count each epoch based on whether the observed signals remain within configured guardrails.

This paper reports results from a concrete local prototype and calibrated synthetic experiments on a single aarch64 host. We do not claim production readiness. The target service is synthetic, and the utilization metric is a service-capacity proxy rather than hardware-level GPU utilization. The contribution is an empirical demonstration that adaptive negotiation can preserve SLOs while increasing load, together with a characterization of the conservatism inherent in a safety-first probing policy.

## Method

### Prototype Implementation

The controller is implemented as a deterministic asyncio harness in a single Python file (`src/synthetic_user_load_negotiator.py`) with no external dependencies. It comprises five components:

1. **Synthetic target service.** A bounded server with configurable capacity (default: 16 concurrent service slots), stochastic request durations drawn from a truncated distribution, a FIFO queue for excess requests, and per-request timeout/SLO accounting.

2. **Closed-loop synthetic users.** Each user issues a request, waits for a response or timeout, then idles for a configurable think time before issuing the next request.

3. **Static baseline scenarios.** Fixed user counts run for the full experiment duration to establish reference performance at known load levels.

4. **Adaptive negotiator.** Starting from a small smoke pressure (few users), the negotiator observes four signals at the end of each epoch:
   - **p95 latency** relative to the SLO threshold (450 ms),
   - **Error rate** relative to a tolerance,
   - **Utilization proxy** (users / capacity, clamped to [0, 1]), and
   - **MemAvailable** from `/proc/meminfo` as a host-memory safety guardrail.

   The negotiator increases user count when all signals are within safe bounds, decreases it when any signal violates its guardrail, and holds steady otherwise. The policy is intentionally safety-first: it prefers under-loading the target over risking SLO violations.

5. **Metrics emission.** Per-epoch CSV records and a JSON summary are written to disk for post-hoc analysis.

### Experimental Environment

Experiments ran on a Linux aarch64 host (kernel `gx10-efe8`) with 20 CPU cores (Cortex-X925/A725 mix), approximately 121 GiB total RAM, and swap disabled (`SwapTotal: 0 kB`). An NVIDIA GB10 GPU was visible via `nvidia-smi`, but GPU memory accounting was reported as `Not Supported`. Consequently, the memory guardrail used `/proc/meminfo` `MemAvailable` rather than GPU-specific telemetry. The environment probe is recorded in `artifacts/logs/env_probe.log`.

### Experimental Design

Four experimental conditions were run:

| Condition | Description | Epochs | Epoch Duration |
|---|---|---|---|
| Smoke test | Fixed 4 users + negotiator | 2 | 0.5 s |
| Main calibration | Fixed baselines (8, 16, 32, 48 users) + negotiator | 10 | 1.5 s |
| Longer negotiator | Negotiator only | 20 | 1.0 s |
| Deliberate overload | Fixed 80 users | 6 | 1.0 s |

The smoke test confirmed correct harness operation before calibration. The main calibration compared the negotiator against fixed baselines. The longer negotiator run tested convergence over more epochs. The deliberate overload run confirmed that the harness correctly identifies unsafe states at high fixed load.

### Metrics

The primary metrics are:

- **Mean steady-state RPS:** Average requests per second during steady-state epochs (excluding warmup).
- **Safe epoch fraction (steady):** Fraction of steady-state epochs where p95 latency remained below the 450 ms SLO.
- **Last p95 latency:** The p95 latency in the final epoch, indicating where the controller converged.
- **Last user count:** The user count at the final epoch.

## Results

### Main Calibration (10 Epochs, 1.5 s Each)

| Scenario | Mean Steady RPS | Safe Epoch Fraction (Steady) | Last Users | Last p95 (ms) |
|---|---:|---:|---:|---:|
| static_8 | 48.07 | 1.00 | 8 | 172.81 |
| static_16 | 95.81 | 1.00 | 16 | 178.63 |
| static_32 | 103.67 | 1.00 | 32 | 318.64 |
| static_48 | 106.23 | 0.00 | 48 | 459.07 |
| negotiator | 96.51 | 1.00 | 19 | 244.57 |

**Fixed baselines.** At 8 and 16 users, the target is safe but underloaded. At 32 users, the target achieves the highest safe throughput (103.67 RPS) with p95 at 318.64 ms—within the 450 ms SLO but with 29% headroom remaining. At 48 users, throughput increases marginally to 106.23 RPS but p95 exceeds the SLO at 459.07 ms, and the safe epoch fraction drops to 0.00 in steady state. This confirms that the synthetic target has a throughput ceiling near 106–108 RPS, beyond which latency degrades sharply.

**Negotiator.** The adaptive controller converged to 19 users with a last p95 of 244.57 ms and 100% safe epochs. Its steady-state RPS of 96.51 is 6.9% below the best fixed safe baseline (static_32, 103.67 RPS). The negotiator's final p95 of 244.57 ms is 46% below the 450 ms SLO, indicating substantial unused headroom that it did not exploit within 10 epochs.

### Longer Negotiator Run (20 Epochs, 1.0 s Each)

| Metric | Value |
|---|---:|
| Mean steady RPS | 100.54 |
| Safe epoch fraction (steady) | 1.00 |
| Max safe RPS | 107.93 |
| Last users | 29 |
| Last p95 (ms) | 277.31 |

With additional epochs, the negotiator increased user count to 29 and achieved 100.54 steady RPS—within 3.0% of the best fixed safe baseline. The last p95 of 277.31 ms remains 38% below the SLO. This suggests the negotiator converges toward the SLO boundary given sufficient epochs, but its ramp-up is slow and it still does not fully exploit the available safe headroom even after 20 epochs.

### Deliberate Overload (Fixed 80 Users, 6 Epochs)

| Metric | Value |
|---|---:|
| Mean steady RPS | 105.55 |
| Safe epoch fraction (steady) | 0.00 |
| Max safe RPS | 0 |
| Last p95 (ms) | 748.26 |

The 80-user fixed scenario confirms the overload regime: throughput saturates near 105.55 RPS (no meaningful improvement over 48 users), p95 latency reaches 748.26 ms (1.66× the SLO), and no steady-state epoch is safe. This validates that the harness correctly detects and reports unsafe states at high fixed load.

### Summary of Trade-offs

The negotiator's safety-first policy achieves its primary objective—zero SLO violations across all experimental conditions—but at the cost of throughput conservatism in short horizons. The gap between negotiator throughput and the best fixed safe baseline shrinks from 6.9% (10 epochs) to 3.0% (20 epochs), suggesting that convergence speed, not safety, is the limiting factor. Even at 20 epochs, the negotiator's final p95 of 277.31 ms leaves 38% SLO headroom unused, indicating that a more aggressive probing strategy near the boundary could yield further throughput gains without sacrificing safety.

## Limitations

1. **Synthetic target only.** The target service is a stochastic simulation with configurable capacity and service-time distributions. It does not model real inference-server behavior such as token generation, dynamic batching, KV-cache pressure, or GPU scheduling. Scientific closure for production claims requires running the controller against an actual serving stack with a representative request mix.

2. **Conservative convergence.** The current policy increases user count incrementally and backs off aggressively on any guardrail violation. Over 10 epochs, this leaves substantial SLO headroom unused. Faster convergence strategies—bracketed binary search, bandit-style probing, or multiplicative increase/additive decrease—could close this gap but were not evaluated.

3. **Utilization proxy, not hardware utilization.** The utilization metric is `users / capacity` clamped to [0, 1], which reflects service-slot occupancy but not GPU SM utilization, memory bandwidth, or token throughput. On a real inference server, this proxy may correlate poorly with actual resource saturation.

4. **GPU memory telemetry unavailable.** The NVIDIA GB10 on the test host reports GPU memory as `Not Supported` via `nvidia-smi`. The controller fell back to `/proc/meminfo` `MemAvailable`, which captures host-side memory pressure but not GPU-side allocation. On unified memory architectures this may be adequate as a safety guardrail; on discrete-GPU systems it would not be.

5. **Single SLO configuration.** All experiments used a 450 ms p95 SLO and 900 ms timeout. Sensitivity to SLO threshold, timeout, and capacity was not explored.

6. **Stochastic variability and single-run results.** Service times are stochastic, so individual epoch metrics vary. The reported values are single-run observations, not confidence intervals over repeated trials. The relative rankings (negotiator safe but conservative, static_48 unsafe) are expected to be robust to this variability, but the exact RPS values should not be treated as precise point estimates.

7. **Short epoch durations.** Epochs of 1.0–1.5 s may not capture slower dynamics—thermal throttling, garbage collection, memory compaction—that affect real serving stacks over longer timescales.

8. **No live inference validation.** This work constitutes a toy simulation result, not a production validation. The controller has not been tested against a real inference endpoint, and its behavior under real request mixes, real GPU scheduling, and real memory pressure is unknown.

## Conclusion

An adaptive synthetic-user load negotiator that relies solely on observable telemetry—p95 latency, error rate, a utilization proxy, and a host-memory guardrail—can preserve SLO compliance while dynamically increasing load pressure. In a synthetic target model with capacity 16 and a 450 ms p95 SLO, the negotiator maintained 100% safe epochs across all runs, successfully avoiding the unsafe overload regime that fixed high user counts (48 and 80 users) entered.

The negotiator's throughput was 3–7% below the best fixed safe baseline, depending on the number of epochs allowed for convergence. This conservatism is a direct consequence of the safety-first policy and its incremental probing strategy. The gap narrows with more epochs, suggesting that faster boundary-search algorithms—bracketed search, bandit methods, or multiplicative increase—could substantially improve throughput without sacrificing safety. However, even after 20 epochs the negotiator left 38% SLO headroom unused, indicating that the current probing strategy does not fully explore the safe operating region.

The central remaining limitation is that all results derive from a synthetic target service. Validation against a live inference endpoint—with real GPU utilization telemetry, representative request mixes, and longer timescales—is necessary before any production deployment claim. The prototype and its metrics artifacts are released to support independent reproduction and extension.
