## Abstract

We investigate whether labeling evaluation examples by their critical token type (name, number, unit, delimiter) can reveal failure clusters hidden by aggregate accuracy metrics. Using a template-synthetic dataset of 1,500 examples spanning structured extraction, code navigation, and abstention calibration, we compare four retrieval and extraction systems under both aggregate and token-type-sliced evaluation. Aggregate accuracy masks substantial per-type variation: a query-typed retriever achieving 33.60% aggregate accuracy drops to 14.02% on name-critical examples and 24.14% on number-critical examples, while a lexical window retriever at 7.53% aggregate falls to 0.46% on its worst token type. A label-aware structural extractor recovers to 100.00% across all slices, yielding +66.4 aggregate percentage points and +86.0 worst-type percentage points of uplift over the query-typed retriever. In an initial simulation with deterministic perturbation profiles, token-type macro accuracy also reversed the ranking of pruning-like and quantization-like systems compared to aggregate ranking. These results support the hypothesis that token-type importance labels expose concrete failure clusters and can guide targeted interventions, but the evidence is bounded by the template-synthetic dataset and the structural extractors' exploitation of template regularities. External replication on non-synthetic data and real neural model pipelines remains necessary.

## 1. Introduction

Standard evaluation of retrieval and extraction systems reports aggregate accuracy, treating all test examples as interchangeable. This practice can obscure systematic failures that concentrate on specific token types—names, numbers, units, or delimiters—that carry disproportionate importance in structured tasks. A system that appears adequate in aggregate may fail catastrophically on the subset of examples where a single numeric or named entity determines correctness.

We propose token-type importance labels as a lightweight annotation schema: each evaluation example is tagged with the token type whose value is critical to a correct answer. Slicing accuracy by these labels reveals whether a system's failures cluster on particular token types, and whether interventions targeted at those types produce meaningful uplift.

This paper reports results from a two-phase experiment. In the first phase, we evaluate deterministic perturbation profiles (simulating pruning-like, quantization-like, and similarity-centric retrieval degradation) against a synthetic dataset, measuring both aggregate and per-type accuracy. In the second phase, we replace simulated pass rates with four real retrieval and extraction algorithms that read each prompt and produce predictions from candidate tokens or parsed spans. We measure whether token-type labels (a) expose failure clusters invisible in aggregate metrics, (b) change system rankings, and (c) guide a targeted label-aware intervention that produces substantial uplift.

The current project artifacts support these findings in the tested setting. The evidence does not establish that token-type labels transfer to real neural model pruning, quantization, or open-domain retrieval without further validation.

## 2. Method

### 2.1 Token-Type Importance Label Schema

Each evaluation example is annotated with one of four critical token types:

- **name**: A proper noun or identifier whose exact value must be retrieved (e.g., a person name, function name, or variable identifier).
- **number**: A numeric value whose magnitude or digits must be preserved (e.g., a measurement, index, or constant).
- **unit**: A measurement unit whose identity disambiguates the answer (e.g., "kg" vs. "lb").
- **delimiter**: A structural separator whose correct parsing is necessary to segment the input (e.g., commas in CSV, brackets in code).

The label identifies which token type, if corrupted or misretrieved, would cause the example to be answered incorrectly regardless of other tokens.

### 2.2 Dataset Generation

We generate 1,500 template-synthetic examples using a dependency-free Python generator (`src/token_type_importance_experiment.py`). The dataset spans three task categories: structured extraction, code navigation, and abstention calibration. Token-type distribution: name=649, number=377, delimiter=250, unit=224.

Examples are produced from parameterized templates with randomized slot values. Abstention-calibration examples include cases where no correct answer exists in the context, testing whether systems appropriately decline to answer.

### 2.3 Phase 1: Deterministic Perturbation Profiles

In the initial phase, we simulate three degradation profiles by applying deterministic perturbation functions to correct answers:

- **Similarity-centric retrieval**: Simulates a system that preserves semantic gist but corrupts exact token values, with higher fidelity for common token types.
- **Pruning-like degradation**: Simulates aggressive pruning that disproportionately drops numeric and named-entity precision.
- **Quantization-like degradation**: Simulates quantization noise that reduces precision of numeric values and distills units.

These profiles are implemented as pass-rate tables applied per token type, not as real model outputs. They serve to test whether token-type slicing changes system rankings even under controlled, synthetic conditions.

### 2.4 Phase 2: Real Retrieval and Extraction Systems

In the second phase, we replace simulated pass rates with four actual retrieval/extraction algorithms (`src/real_retrieval_evaluation.py`):

1. **Lexical window retriever**: A naive sliding-window matcher that retrieves context windows by lexical overlap with the query. No structural parsing; no type awareness.

2. **Query-typed retriever**: Extends the lexical retriever with query-type detection, routing queries to type-appropriate matching strategies. Retains similarity-based retrieval for name and number tokens.

3. **Structural extractor**: Parses the input structure (delimiters, units, numeric fields) and extracts answers by structural position rather than similarity. Exploits template regularities in the synthetic dataset.

4. **Label-aware extractor**: Combines structural extraction with token-type importance labels, routing each query to the extraction strategy matched to its critical token type. Also exploits template regularities.

Each system processes all 1,500 examples, producing 6,000 total predictions across the four systems.

### 2.5 Evaluation Metrics

We compute:

- **Aggregate accuracy**: Fraction of correct predictions across all examples.
- **Per-type accuracy**: Accuracy restricted to examples of each critical token type.
- **Worst-token-type accuracy**: Minimum accuracy across the four token-type slices.
- **Token-type macro accuracy**: Unweighted mean of per-type accuracies.
- **Targeted uplift**: Improvement of the label-aware extractor over a baseline, measured in both aggregate and worst-type terms.

### 2.6 Weak-Label Audit

We perform a deterministic stratified spot-check of 75 weak labels (approximately 5% of the dataset), sampled to cover all token types and task categories. The audit checks for synthetic-template artifacts, inconsistent labeling, and ambiguous ground truth. This is an automated consistency check, not a human review.

## 4. Limitations

1. **Template-synthetic dataset**: All 1,500 examples are generated from parameterized templates. The structural and label-aware extractors achieve 100.00% accuracy precisely because they exploit template regularities (e.g., fixed positional structure, deterministic delimiter patterns). Performance on non-template, naturally occurring text is unknown and likely substantially lower.

2. **No neural model evaluation**: The perturbation profiles in Phase 1 are deterministic pass-rate simulations, not outputs from pruned, quantized, or retrieved neural models. The real systems in Phase 2 are rule-based retrieval and extraction algorithms, not learned models. Whether token-type labels expose failure clusters in real neural pruning, quantization, or dense retrieval pipelines remains untested.

3. **Automated label audit only**: The 75-label spot check is a deterministic consistency and artifact audit. It verifies that labels are internally consistent and free of obvious template artifacts, but it does not constitute human annotation review. Label quality on ambiguous or edge-case examples has not been assessed by human annotators.

4. **Ceiling effects**: The structural and label-aware extractors both achieve 100.00% accuracy on all slices, creating a ceiling that prevents measuring whether token-type labels provide diagnostic value beyond what structural parsing alone offers. On this dataset, structural parsing is sufficient; the label-aware routing adds no measurable benefit over the structural extractor.

5. **Limited task and domain coverage**: The three task categories (structured extraction, code navigation, abstention calibration) and four token types may not capture the full spectrum of token-type importance in other domains (e.g., legal text, medical records, multilingual input).

6. **Ranking reversal not validated on real systems**: The ranking reversal between pruning-like and quantization-like profiles observed in Phase 1 has not been replicated with real model outputs. It demonstrates the possibility of ranking changes under token-type slicing but does not confirm that such reversals occur in practice.

7. **Uplift magnitude is dataset-dependent**: The +66.4 aggregate and +86.0 worst-type uplift of the label-aware extractor over the query-typed retriever reflects the large gap between naive similarity-based retrieval and structural parsing on template data. On more diverse data, this gap may be substantially smaller.

## 6. Conclusion

Token-type importance labels expose failure clusters that aggregate accuracy conceals. In a template-synthetic evaluation of four retrieval and extraction systems, per-type slicing revealed that the query-typed retriever's name-critical accuracy (14.02%) was less than half its aggregate (33.60%), and the lexical window retriever's worst-type accuracy (0.46%) was far below its aggregate (7.53%). A label-aware structural extractor recovered performance to 100.00% across all slices, producing +66.4 aggregate and +86.0 worst-type percentage points of uplift. In a separate simulation, token-type macro accuracy reversed the ranking of two degradation profiles compared to aggregate ranking.

These findings are positive MVP evidence that token-type labels are a useful diagnostic and intervention tool, but the evidence is bounded in important ways. The dataset is template-synthetic, the strongest systems exploit template regularities to achieve ceiling performance, and no real neural model pipeline has been evaluated. The project decision recommends finalizing this MVP as positive evidence and initiating a separate project for neural pruning/quantization validation if desired.

The current project artifacts support the finding that token-type importance labels expose concrete failure clusters and guide targeted interventions in the tested setting. This does not establish that the method works universally or transfers to non-synthetic domains without further validation.
