## Abstract

We evaluate a warm-session operator design for interactive trace inspection: parse and index a trace corpus once, then serve repeated queries from the in-memory index, rather than re-parsing the raw log for each question. A no-dependency Python prototype (`trace_inspector_trial.py`) was tested on four corpora ranging from 1,000 synthetic events (138 KB) to 500,000 synthetic events (70.4 MB), plus a small real log corpus of 2,529 events (1.15 MB). Per-query latency in warm mode was 0.012–0.045 ms versus 2.0–1,426 ms in cold mode (re-parse per query), yielding per-query speedups of 166× to 31,638×. Including the one-time warm load cost (1,126 ms for the largest corpus), the full 25-query warm run completed in 1.13 s versus 35.65 s cold—a 31.6× wall-time reduction. Peak RSS was 1,015 MiB for the 500k-event corpus, approximately linear in event count, with over 121 GiB of system memory remaining on the test host. These results support warm-session indexing as a viable design direction for repeated interactive trace triage, though generalization to production-scale corpora, concurrent multi-user access, and memory-constrained environments remains unvalidated.

## Introduction

Operators inspecting trace logs—whether for debugging, performance triage, or audit—frequently pose multiple questions against the same corpus. A naive approach re-parses the raw log file for each query, incurring the full parse cost every time. An alternative is to parse and index the corpus once, then answer subsequent queries from the in-memory index. We call this the *warm-session* model, by analogy to a warm database connection that amortizes setup cost over multiple operations.

The research question is straightforward: does the warm-session model yield practically significant latency improvements for repeated trace inspection, and at what memory cost?

This question matters for interactive operator workflows where sub-second query response affects triage velocity. If warm indexing reduces per-query latency by orders of magnitude at acceptable memory cost, it argues for long-lived inspector processes rather than per-query CLI invocations.

We emphasize that this study is a prototype viability trial, not a production system evaluation. The largest corpora are synthetic, the query suite is fixed and representative rather than exhaustive, and the prototype is a single-threaded in-process Python script without concurrency, persistence, or incremental indexing. The goal is to establish whether the warm-session design direction merits further engineering investment, not to characterize a deployable system.

## Method

### Prototype

The prototype (`scripts/trace_inspector_trial.py`) is a single-file, no-dependency Python 3.12 script with two subcommands:

1. **`generate`**: Produces a deterministic synthetic NDJSON trace corpus parameterized by event count, session count, and a seed. Events are shaped to resemble operator/agent trace entries (session IDs, event types, tool invocations, agent identifiers, error flags, duration fields, token counts).

2. **`run`**: Loads an NDJSON trace file into a `TraceIndex` object, then executes a fixed five-query suite in two modes:
   - **Warm mode**: Parse and index the file once; answer all queries from the same `TraceIndex` instance.
   - **Cold mode**: For each query, re-parse and re-index the file from disk, then answer the query and discard the index.

The five queries are: (1) corpus summary, (2) session listing, (3) error sample retrieval, (4) slow operation listing, and (5) tool/agent mix breakdown. These were chosen as representative of common trace triage questions but do not constitute an exhaustive query benchmark.

### Index Structure

The `TraceIndex` builds the following in-memory structures during load:

- Session-grouped event lists
- Event-type index
- Tool-name index
- Agent-name index
- Error event sample list
- Slow operation list (sorted by duration)
- Token total accumulator

All structures are built in a single pass over the NDJSON input.

### Measurement

Each run records:

- **Warm load time**: Time to parse and index the full corpus once.
- **Warm query latency**: Per-query time answering from the in-memory index.
- **Cold query latency**: Per-query time including full re-parse, re-index, and answer.
- **Peak RSS**: Maximum resident set size of the process (via `/proc/self/status`).
- **System memory posture**: `MemAvailable` before and after the run, `SwapTotal`, `earlyoom` status.

Latencies are measured with `time.perf_counter()` (sub-microsecond resolution). Each corpus is run with multiple repetitions (2–10 depending on corpus size) and the mean is reported.

## Results

### Latency

| Corpus | Events | Warm Load (ms) | Warm Mean Query (ms) | Cold Mean Query (ms) | Cold/Warm Per-Query Speedup | Peak RSS (MiB) |
|---|---:|---:|---:|---:|---:|---:|
| smoke synthetic | 1,000 | 2.261 | 0.0122 | 2.030 | 166.6× | 17.2 |
| local OMX logs | 2,529 | 7.588 | 0.0149 | 6.588 | 440.9× | 26.0 |
| calibration 100k | 100,000 | 230.058 | 0.0135 | 270.074 | 20,024.7× | 215.7 |
| calibration 500k | 500,000 | 1,125.726 | 0.0451 | 1,426.106 | 31,637.8× | 1,015.1 |

For the largest corpus (500k events, 70.4 MB), the warm operator paid a one-time load cost of ~1,126 ms, then answered queries at ~0.045 ms mean. Cold mode averaged ~1,426 ms per query because it re-parsed the full corpus each time. The 25 cold queries (5 queries × 5 repetitions) took 35.65 s total; warm load plus all 25 queries took 1.13 s total—a 31.6× wall-time reduction for the full run.

The per-query speedup grows with corpus size because cold-mode latency is dominated by parse cost (linear in file size), while warm-mode query latency grows much more slowly (index lookup cost). At 500k events, the per-query speedup exceeds 31,000×.

### Memory

Peak RSS scales approximately linearly with event count:

- 1,000 events → 17.2 MiB
- 2,529 events → 26.0 MiB
- 100,000 events → 215.7 MiB
- 500,000 events → 1,015.1 MiB

The ratio is roughly 2 KiB per event in the index, reflecting the overhead of Python dict/list structures holding parsed event objects.

On the test host with 121.4 GiB total RAM and swap disabled, the 500k-event run consumed ~1.0 GiB peak RSS, leaving >121 GiB `MemAvailable` after completion. The `MemAvailable` delta across the 500k run was approximately 912 MiB (122,530,728 kB → 121,618,860 kB), consistent with RSS plus kernel reclaimable pages.

### Amortization Threshold

The warm-session model becomes cheaper than cold after the first query cycle. For the 500k corpus, the break-even point is:

$$N_{\text{break-even}} = \lceil T_{\text{warm-load}} / (T_{\text{cold-query}} - T_{\text{warm-query}}) \rceil = \lceil 1126 / (1426 - 0.045) \rceil = 1$$

A single query cycle (all five queries) suffices to amortize the load cost. For smaller corpora, the break-even is similarly immediate.

### Mixed and Negative Observations

1. **Warm query latency is not constant across corpus sizes.** While warm queries are consistently sub-millisecond, the mean rises from 0.012 ms (1k events) to 0.045 ms (500k events), suggesting that index traversal cost is not fully constant. For the tested sizes this is negligible, but at multi-million-event scales the trend may become material.

2. **The real corpus is too small to independently validate scaling.** The local OMX corpus (2,529 events) confirmed that the prototype works on real NDJSON, but its size is insufficient to test scaling behavior on production-shaped data.

3. **No concurrent access was tested.** The prototype is single-threaded and single-user. Whether a warm-session service can maintain low query latency under concurrent load is unaddressed.

4. **The 100k calibration shows an anomalously high per-query speedup relative to the 500k calibration.** The 100k corpus exhibits a 20,024.7× per-query speedup while the 500k corpus shows 31,637.8×. The ratio of speedups does not scale proportionally with the ratio of corpus sizes, because warm query latency itself increases with corpus size (0.0135 ms vs. 0.0451 ms). This indicates that warm query cost is not O(1) but grows, albeit slowly, with index size.

## Limitations

1. **Synthetic corpora dominate the scaling data.** The 100k and 500k calibration corpora are generated deterministically but are synthetic. Their event distributions (session counts, error rates, duration profiles) may not reflect production trace characteristics. Scaling conclusions should be treated as lower-bound estimates until validated on real data.

2. **Small real-corpus validation.** The only non-synthetic corpus was 2,529 events of operator infrastructure telemetry. No application-level or production-private trace data was available for this trial. The real corpus confirms that the prototype can parse and index actual NDJSON, but it does not independently validate the scaling trend observed in synthetic data.

3. **Prototype, not service.** The implementation is an in-process Python script. It lacks: persistence across process restarts, incremental/append-only ingestion, concurrent query handling, authentication, memory caps, eviction policies, and on-disk index formats. These are engineering concerns, not refutations of the warm-session design, but they mean the reported latencies do not account for service-level overhead.

4. **Fixed query suite.** Five queries were tested. The query suite covers common triage operations but does not stress the index with, for example, full-text search, temporal range scans, or cross-session correlation queries. Different query patterns may exhibit different warm/cold tradeoffs.

5. **Single hardware configuration.** All runs were on one aarch64 host with abundant RAM (121 GiB) and no swap. The memory cost profile may differ on memory-constrained systems, and the absence of swap means the OOM behavior under memory pressure was not exercised (though `earlyoom` was active as a safety net).

6. **Python-specific overhead.** The ~2 KiB/event index memory cost reflects CPython object overhead. A compiled-language implementation or compact serialized index would likely reduce this substantially, but that was not measured.

7. **No variance statistics reported.** The results report mean latencies across repetitions but do not include standard deviations or confidence intervals. With 2–10 repetitions per corpus, the sample sizes are small, and the stability of the reported means is not fully characterized.

## Conclusion

A warm-session trace inspection operator—parse and index once, query many times from memory—yields substantial per-query latency improvements over a cold re-parse-per-query baseline. In prototype trials on synthetic corpora up to 500,000 events (70.4 MB), per-query speedups ranged from 166× to 31,638×, and total wall time for a 25-query run was reduced by 31.6×. Memory cost is linear at approximately 2 KiB/event in CPython, reaching ~1 GiB for 500k events—acceptable on the tested host but a design consideration for memory-constrained deployments.

The result is sufficient to establish prototype viability: the warm-session design direction merits further engineering investment. However, several important qualifications remain. The scaling data rests on synthetic corpora; the real-corpus validation is small; the prototype lacks service-level features (concurrency, persistence, incremental ingestion, memory caps); the query suite, while representative, is not exhaustive; and no variance statistics are reported. Production generalization would benefit from validation on private production-shaped trace data, from implementing the warm-session design as a long-lived operator process with append-only ingestion and bounded memory, and from characterizing performance under concurrent access.
