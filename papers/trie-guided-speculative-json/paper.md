## Abstract

Constrained JSON generation by large language models frequently produces long spans of syntactically forced tokens—structural delimiters, fixed property names, constant literals, and singleton enum values—that require no probabilistic resolution by the model. On hardware where prompt prefill throughput substantially exceeds sequential decode throughput, advancing these forced spans via batched key-value cache operations rather than serial autoregressive steps could reduce generation latency. This paper presents an opportunity model that estimates the potential speedup from trie/FSM-guided speculative JSON decoding. Measurements were collected on an NVIDIA GB10 system running Qwen2.5-7B-Instruct (Q4_K_M) via llama.cpp CUDA, yielding a prefill throughput of 2609.50 tokens/s and a decode throughput of 45.23 tokens/s (a ratio of 57.70×). Across six representative JSON generation tasks, the opportunity model estimates a mean fixed-token share of 63.21%, a mean serial step reduction of 61.94%, and a mean estimated latency speedup of 3.27× (range: 1.35×–5.19×). These are analytical projections from a token-level opportunity model, not measurements from a modified decoder. The broad concept overlaps with existing compressed-FSM jump-forward decoding and constrained decoding with speculative lookaheads. A defensible contribution would require a concrete implementation and rigorous comparative benchmarks against existing constrained-decoding systems.

## Introduction

Constrained decoding is a standard technique for ensuring that large language model (LLM) outputs conform to a specified grammar or schema. In JSON generation scenarios—tool-call APIs, structured data extraction, agent pipelines—grammar-constrained decoding enforces syntactic validity by restricting the token distribution at each step to those permitted by the current finite-state machine (FSM) state. This approach is correct but potentially inefficient: when the FSM state admits exactly one valid token continuation, the model still incurs the full cost of a sequential decode step to produce that token.

JSON outputs contain substantial deterministic structure. Object delimiters (`{`, `}`), array brackets (`[`, `]`), quotes, colons, commas, fixed property names, `const` values, and singleton enum labels are all forced by the schema and require no model uncertainty resolution. On hardware with a large prefill-to-decode throughput asymmetry, these spans could theoretically be advanced in a single batched prefill operation rather than through sequential decode steps. This observation is not new: SGLang's compressed-FSM jump-forward decoding collapses deterministic FSM transitions into multi-token jumps processed via batched prefill, and recent work on constrained decoding with speculative lookaheads reports speedups of 2.2×–12.15× on constrained tasks.

This study addresses the following question: **What latency reduction does the prefill/decode throughput asymmetry afford for JSON generation, and does the opportunity justify engineering investment given the existing landscape of constrained-decoding optimizations?**

We approach this as a viability study. We measure actual prefill and decode throughput on target hardware, construct an opportunity model that attributes tokens to fixed versus variable spans across representative JSON tasks, and estimate the upper bound on speedup if fixed spans were processed at prefill throughput. We do not implement the proposed decoder; we characterize the opportunity and assess its viability relative to prior work.

### Opportunity Model

The opportunity model (`scripts/trie_spec_json_bench.py`) estimates the speedup achievable if deterministic token spans are advanced via batched prefill rather than sequential decode. The procedure is as follows:

1. **Tokenize representative JSON outputs** using the same tokenizer binary as the inference engine (`llama-tokenize`), ensuring token counts match what the model would produce.
2. **Classify each token** as fixed (deterministic, forced by schema structure) or variable (requiring model sampling). Fixed tokens include structural syntax (`{`, `}`, `[`, `]`, `:`, `,`, `"`), known property names defined by the schema, `const` values, and singleton enum labels where only one value is permitted. Variable tokens include free-form string content, numeric values, and choice-point enum selections where the model must select among multiple valid options.
3. **Estimate latency** by charging fixed tokens at the measured prefill throughput rate and variable tokens at the measured decode throughput rate. The baseline charges all tokens at decode throughput. Speedup is the ratio of baseline latency to estimated latency.
4. **Aggregate across six representative JSON generation tasks** spanning different schema complexity levels, from const/enum-heavy classifier JSON to free-form-summary-heavy JSON.

The fixed/variable attribution uses exact full-string token counts with byte-proportional splitting at boundaries. This is an analytical approximation suitable for viability assessment. It does not account for implementation overheads such as trie traversal, KV cache management at span boundaries, or checkpoint/recovery logic.

The 57.70× prefill-to-decode asymmetry is the primary driver of the estimated opportunity. Any token that can be advanced via prefill rather than sequential decode avoids the decode bottleneck entirely. The magnitude of this asymmetry is hardware-dependent; systems with different memory hierarchy characteristics or model sizes will exhibit different ratios.

### Opportunity Model Estimates

Across six representative JSON generation tasks:

| Metric | Value |
|---|---|
| Mean fixed-token share | 63.21% |
| Mean estimated latency speedup | 3.27× |
| Mean estimated serial step reduction | 61.94% |
| Speedup range (minimum) | 1.35× (free-form-summary-heavy JSON) |
| Speedup range (maximum) | 5.19× (const/enum-heavy classifier JSON) |

The estimated speedup varies substantially with schema structure. Schemas dominated by fixed property names, constant values, and singleton enum labels (e.g., classification records) show the largest opportunity, since a greater fraction of tokens can be advanced at prefill throughput. Schemas with large free-form text fields (e.g., support tickets with long summary strings) show the smallest opportunity, since most tokens in those fields are variable and must still pass through sequential decode.

The relationship between fixed-token share and estimated speedup is nonlinear due to the prefill/decode throughput ratio. Even schemas where a majority of tokens are variable still benefit from advancing the remaining fixed tokens at prefill speed, though the marginal gain diminishes.

## Limitations

This study has several limitations that materially temper its conclusions:

1. **Opportunity model, not implementation.** The speedup estimates are analytical projections based on token attribution and measured throughput rates. They assume that fixed spans can be correctly identified, batched, and processed via prefill without overhead. A real implementation would incur costs for trie traversal, KV cache management at span boundaries, and checkpoint/recovery logic that the model does not account for. Realized speedups would be lower than the estimates presented here.

2. **Approximate token attribution.** The fixed/variable classification uses byte-proportional splitting at tokenizer boundaries. In practice, tokenizer merge rules can cause a single token to span both fixed and variable content, complicating the clean separation the model assumes. Tokens that straddle fixed/variable boundaries may not be eligible for jump-forward in a real implementation, reducing the effective fixed-token share.

3. **No comparative benchmark.** The study does not compare against SGLang compressed-FSM jump-forward decoding, Outlines, XGrammar, or draft-model/ngram speculative decoding baselines. Without these comparisons, the marginal benefit of a trie-guided approach over existing constrained-decoding systems is unknown. Prior work reports speedups of 2.2×–12.15× on constrained tasks using different methods; the relationship between those results and the estimates here cannot be established without head-to-head comparison.

4. **Schema representativeness.** The six JSON tasks are representative but not exhaustive. JSONSchemaBench demonstrates substantial variation in schema structure and framework support across real-world schemas. Performance on deeply nested, recursive, or heavily conditional schemas (e.g., `oneOf`/`anyOf` with complex branching) may differ from the estimates here.

5. **Single hardware and model configuration.** All measurements are from one GPU (NVIDIA GB10) with one model (Qwen2.5-7B-Instruct Q4_K_M). The prefill/decode ratio—and thus the magnitude of the opportunity—will differ on other hardware, at other model sizes, and with different quantization levels. Systems with smaller prefill/decode asymmetry would see proportionally smaller speedups.

6. **Novelty overlap with prior work.** The broad concept of using FSM/grammar structure to skip deterministic tokens during constrained decoding is not new. SGLang's compressed-FSM jump-forward decoding and the constrained decoding with speculative lookaheads approach both target the same bottleneck. A contribution would need to differentiate on implementation specifics (e.g., llama.cpp integration, JSON Schema compilation strategy), integration with existing speculative decoding accounting, or empirical performance on a particular system and schema set.

7. **Claim ledger audit status.** The claim ledger for this artifact reports `blocked_empty_claims`, indicating that no structured claims were extracted for formal audit. The estimates and interpretations in this paper have not passed a structured claim/evidence audit.

## Conclusion

Trie/FSM-guided speculative JSON decoding is technically viable as an inference-control optimization. On the tested NVIDIA GB10 system with Qwen2.5-7B-Instruct, the 57.70× prefill-to-decode throughput asymmetry creates a substantial analytical opportunity: an estimated 3.27× mean latency speedup across six representative JSON tasks, with the largest gains (up to 5.19×) on const/enum-heavy schemas where over 63% of tokens are structurally forced.

However, the broad idea is not novel. Compressed-FSM jump-forward decoding and constrained decoding with speculative lookaheads already target the same bottleneck. The speedup estimates presented here are analytical projections from an opportunity model, not measurements from a working implementation. Realized speedups would be lower due to trie traversal overhead, KV cache management costs, tokenizer-boundary complications, and any checkpoint/recovery logic required when combining deterministic jumps with probabilistic speculation.

The defensible next step is an engineering prototype: a llama.cpp-compatible token-trie implementation for JSON Schema grammars that exposes deterministic spans as jump-forward prefill operations, integrated with existing speculative decoding checkpointing, and benchmarked against grammar-mode llama.cpp, SGLang compressed FSM, Outlines/XGrammar, and draft-model speculation on standardized schema sets (e.g., JSONSchemaBench) and agent tool-call traces. Until such a prototype demonstrates marginal gains over existing constrained-decoding systems, this study confirms viability but does not establish performance.
