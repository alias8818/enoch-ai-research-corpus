## Abstract

We present a pre-launch VRAM/UMA admission controller that estimates the memory footprint of a GGUF model configuration—comprising model weights, KV cache, compute workspace, and a safety reserve—and selects the largest context length that fits within a user-specified memory budget. The controller parses GGUF metadata to derive layer count, head dimension, KV head count, and cache data type, then computes a deterministic admission decision before any model loading occurs. We evaluate the controller on an NVIDIA GB10 system (aarch64, CUDA 13.0, unified memory, no swap) using two quantized models: Qwen2.5-7B-Instruct Q4_K_M and Phi-4-mini-instruct Q4_K_M. Under simulated budgets of 6 GiB and 8 GiB, the controller rejects context lengths whose estimated footprints exceed the budget and selects the largest admissible context. Bounded llama.cpp smoke runs confirm that the selected configurations execute without out-of-memory errors on the host hardware. However, because the GB10 platform provides approximately 118 GiB of free UMA memory, we did not induce actual hardware out-of-memory conditions; the validation is therefore formula-driven and simulation-based rather than calibrated from GPU allocator telemetry or enforced hard memory limits. The results support the viability of a pre-launch admission controller as a safety component, but closing the gap between estimation and observed allocation remains future work.

## 1. Introduction

Local inference with large language models on constrained hardware requires careful management of GPU memory. A model's memory footprint depends not only on the weight file size but also on the KV cache, which scales with context length, layer count, and head configuration, as well as compute workspace allocations and runtime overhead. Launching a model with an excessively long context on a device with insufficient VRAM typically results in an out-of-memory crash, potentially destabilizing other processes sharing the same memory.

Existing serving frameworks generally require the user to manually select context length and batch size, or they rely on runtime allocation failures to signal overcommit. A pre-launch admission controller offers an alternative: by estimating the total memory requirement before any weights are loaded, the controller can reject infeasible configurations and automatically downgrade to a safe context length.

This work investigates whether a small, deterministic pre-launch controller can predict whether a given GGUF model configuration should be admitted under a specified VRAM/UMA budget, and whether it can select a safe context length when the requested context would exceed that budget. We implement the controller, evaluate it under simulated budgets on an NVIDIA GB10 unified memory platform, and validate selected configurations with bounded llama.cpp benchmark runs. We are careful to distinguish between the simulation-level evidence (admission decisions under hypothetical budgets) and the runtime-level evidence (smoke tests confirming that admitted configurations run on hardware with ample memory), since these address different aspects of the controller's correctness.

## 2. Method

### 2.1. Memory Estimation Model

The controller computes total estimated memory as the sum of four terms:

1. **Model weights.** Read directly from the GGUF file via the `gguf-py` metadata parser. This is the file size of the quantized weight tensors.

2. **KV cache.** Computed from GGUF metadata fields: number of layers, KV head count, head dimension, context length, and cache data type (typically `f16`). The formula is:

   ```
   KV_bytes = 2 × n_layers × n_kv_heads × head_dim × ctx_len × sizeof(cache_dtype)
   ```

   The factor of 2 accounts for separate key and value caches.

3. **Compute/workspace term.** A conservative fixed estimate for batch memory, temporary buffers, and workspace allocations that are not directly visible from metadata.

4. **Safety reserve.** An additional margin to account for estimation error, runtime overhead, and allocator fragmentation.

### 2.2. Telemetry Sources

On the NVIDIA GB10 platform, standard `nvidia-smi --query-gpu=memory.*` queries return `N/A` for memory fields. The controller therefore uses two alternative telemetry sources:

- **`/proc/meminfo` `MemAvailable`**: reflects the system-level available memory on this UMA/no-swap architecture.
- **`llama-bench --list-devices`**: reports CUDA device total and free UMA memory, which on GB10 reflects the shared memory pool.

The no-swap constraint (verified via `swapon --show` returning no entries) means that `MemAvailable` is a meaningful hard bound: once physical memory is exhausted, processes are killed rather than paged.

### 2.3. Admission Decision Logic

Given a model and a budget (in GiB), the controller evaluates a list of candidate context lengths in descending order. For each candidate, it computes the total estimated footprint. The largest context whose estimate does not exceed the budget is selected. All candidates exceeding the budget are recorded as rejected.

### 2.4. Validation Strategy

Because the available GB10 UMA memory (~118 GiB free) far exceeds the simulated budgets (6–8 GiB), we cannot induce actual hardware OOM by running the admitted configurations. The validation therefore proceeds in two distinct stages:

1. **Admission under simulated budget (toy simulation).** The controller makes deterministic decisions based on its estimation model and the specified budget. These decisions are internally consistent but are not validated against actual allocation failures at the budget boundary.

2. **Smoke/calibration runs (llama.cpp hook-prototype validation).** Bounded `llama-bench` invocations confirm that the selected configurations execute successfully (exit code 0) on the host hardware and provide throughput measurements. These runs confirm that admitted configurations are runnable on the available hardware, but they do not validate the rejection decisions against actual OOM.

No CUDA copy calibration or final production validation was performed. The gap between the controller's estimates and the GPU allocator's actual behavior at the budget boundary remains uncharacterized.

## 3. Results

### 3.1. Admission Decisions (Simulated Budget)

The controller produced deterministic admission decisions under each simulated budget.

| ---: | ---: | --- |
| 512 | 5.335 | Admitted |
| 2,048 | 5.427 | Admitted |
| 8,192 | 5.795 | Admitted (selected) |
| 32,768 | 7.265 | Rejected |
| 65,536 | 9.225 | Rejected |

**Qwen2.5-7B Q4_K_M, 8 GiB budget:**

| Context | Estimated footprint (GiB) | Decision |
| ---: | ---: | --- |
| 32,768 | 7.265 | Admitted (selected) |
| 65,536 | 9.225 | Rejected |

**Phi-4-mini Q4_K_M, 6 GiB budget:**

| Context | Estimated footprint (GiB) | Decision |
| ---: | ---: | --- |
| 512 | 3.089 | Admitted |
| 2,048 | 3.299 | Admitted |
| 8,192 | 4.139 | Admitted (selected) |
| 32,768 | 7.499 | Rejected |
| 65,536 | 11.979 | Rejected |

The core signal is that context length and KV cache economics materially change the admissible launch configuration even when the same quantized model file is used. A 7B Q4_K_M model that fits at ctx=8192 under a 6 GiB budget is rejected at ctx=32768, and a mini model that fits at ctx=8192 is rejected at ctx=32768. However, these are simulation-level results: the controller's estimates were not tested against actual allocation failures at the specified budget boundaries.

### 3.2. Runtime Smoke/Calibration (llama.cpp Hook-Prototype)

Bounded `llama-bench` runs confirmed that the selected configurations execute without error on the GB10 hardware. These runs operated on hardware with ~118 GiB free UMA memory, not under the simulated 6–8 GiB budgets.

All runs completed with exit code 0. Generation throughput was approximately stable across context lengths for Qwen2.5-7B (~41.7 tok/s), while prompt processing throughput decreased at longer contexts as expected.

### 3.3. Observed vs. Estimated Memory: An Unresolved Discrepancy

A notable discrepancy exists between the controller's estimated footprints and the measured Max RSS from `llama-bench`. For Qwen2.5-7B at ctx=8192, the controller estimated 5.795 GiB while Max RSS was 4.665 GiB across all tested contexts. The Max RSS values are identical across all four context lengths for Qwen2.5-7B, which suggests that either `llama-bench` pre-allocates to a fixed maximum or that the process-level RSS metric does not fully capture per-context GPU allocation differences on this UMA platform.

This discrepancy has two implications. First, the controller's estimates appear conservative (over-predicting relative to observed RSS), which is the safer direction for an admission controller. Second, the conservatism is unquantified: we cannot determine from the available data whether the over-estimation is 10%, 25%, or varies by context length, because the RSS metric itself may not reflect the true per-context GPU allocation. The `llama-bench` JSON output does not expose per-context GPU allocation, so this gap cannot be closed with the current instrumentation.

## 4. Limitations

1. **No actual OOM validation.** The GB10 platform provides approximately 118 GiB of free UMA memory, far exceeding the simulated budgets. We did not impose a hard memory cap (e.g., via cgroups) or test on a smaller-VRAM GPU. The rejection decisions are therefore validated only by the estimation model, not by observed allocation failures. A rejected configuration might or might not actually OOM on a device matching the simulated budget. This is the most significant limitation of the present work.

2. **Formula-driven estimation, not allocator-calibrated.** The `llama-bench` JSON output does not expose per-context GPU allocation. The controller's estimates are derived from GGUF metadata and a fixed safety reserve, not fitted from observed allocator counters. The conservatism of the estimates relative to actual allocation is unknown and may vary across models, quantizations, and serving runtimes.

3. **Platform-specific telemetry.** On GB10, `nvidia-smi` memory query fields return `N/A`. The controller relies on `/proc/meminfo` and `llama-bench --list-devices` for memory telemetry. This approach is appropriate for UMA/no-swap architectures but may not generalize to discrete-GPU platforms where `nvidia-smi` works and `/proc/meminfo` does not reflect GPU memory.

4. **Static safety reserve.** The safety reserve is a fixed additive term. It does not adapt to model-specific or runtime-specific allocation patterns. An improperly sized reserve could either reject viable configurations (if too large) or admit failing ones (if too small). The present work does not characterize the sensitivity of admission decisions to the reserve size.

5. **Single-architecture evaluation.** All results are from a single GB10 system. The controller's behavior on other hardware, other CUDA versions, or other model architectures has not been tested.

6. **Max RSS as an insufficient proxy.** Process-level Max RSS on a UMA platform may not accurately reflect the GPU-side allocation. The identical Max RSS across different context lengths for Qwen2.5-7B strongly suggests the metric is inadequate for per-context memory validation. Without access to the GPU allocator's own accounting, we cannot precisely characterize the estimation error.

## 6. Conclusion

A pre-launch VRAM/UMA admission controller that parses GGUF metadata and estimates total memory footprint is viable as a safety and control-plane component for local inference. Under simulated budgets of 6 GiB and 8 GiB, the controller made deterministic, internally consistent decisions: selecting the largest admissible context and rejecting configurations whose estimated footprints exceeded the budget. Bounded llama.cpp smoke runs on an NVIDIA GB10 system confirmed that the selected configurations execute without error and provided throughput calibration data.

The strongest empirical signal is the context-dependent variation in estimated footprint: the same quantized model file may be admissible at one context length and inadmissible at another, with the transition occurring at predictable boundaries determined by KV cache scaling. This confirms that context length is a first-class admission parameter, not merely a performance knob.

However, the validation remains incomplete in a critical respect: actual hardware out-of-memory conditions were not induced, and the controller's estimates have not been calibrated against GPU allocator telemetry. The estimates appear conservative relative to observed process RSS, but the conservatism is unquantified and may not generalize. The identical Max RSS across different context lengths for Qwen2.5-7B further indicates that process-level RSS is an inadequate metric for validating per-context memory estimates on this platform.

Closing this gap requires either cgroup-enforced hard memory budgets on the current hardware or replication on a smaller-VRAM device where rejected configurations actually fail. Until such validation is performed, the controller should be treated as a conservative heuristic rather than a verified OOM-prevention mechanism.

The controller is ready for integration as a wrapper around llama.cpp or LM Studio launches, with the understanding that the safety reserve should be tuned from real allocator data as it becomes available, and that the admission decisions should be validated against hard memory limits before being relied upon in production.
